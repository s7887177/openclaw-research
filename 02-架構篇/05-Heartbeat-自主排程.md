## 引言

在探討 AI Agent 的能力時，我們通常關注的是它「回應使用者」的能力——使用者問一個問題，Agent 給出一個回答。但一個真正有用的 AI 助理不應該只是被動地等待指令。想像一下你的人類助理：她會在你要求之前就幫你整理好每天的會議摘要、會在午後提醒你即將到來的客戶拜訪、會在月底自動幫你整理當月的費用報告。

OpenClaw 的 Heartbeat（心跳）模組正是為了實現這種「主動式」Agent 行為而設計的。它賦予了 OpenClaw「自主醒來做事」的能力——即使沒有使用者的訊息，Agent 也能定期檢查待辦事項、執行維護任務、生成報告和提供提醒。

「心跳」這個名字的隱喻非常貼切：就像人的心臟即使在睡覺時也持續跳動一樣，Heartbeat 確保 OpenClaw 即使在使用者不在線時也持續「存活」和「工作」。每一次心跳都是一次「甦醒」——Agent 醒來，檢查是否有需要做的事情，做完後再回到等待狀態。

本文件將全面介紹 Heartbeat 模組的設計理念、運作機制、任務配置和最佳實踐。無論你是想要建立自動化工作流，還是想要理解 AI Agent 的自主行為原理，本文件都將提供有價值的參考。

Heartbeat 的設計哲學可以用一句話概括：**讓 Agent 主動做有用的事，但在沒事的時候保持安靜**。這個看似簡單的要求，實際上需要在多個層面進行精心的設計和權衡。

## 為什麼 Agent 需要「心跳」

### 被動 Agent 的限制

大多數 AI 助理（包括 ChatGPT、Siri、Alexa 等）都是「被動式」的——只有使用者發起互動時，它們才會運作。這種模式有幾個明顯的限制：

**限制 1：無法主動提醒**

被動 Agent 無法在使用者不主動詢問的情況下提供提醒。例如：
- 使用者昨天說「明天要開會」，但今天忘了 → 被動 Agent 不會主動提醒
- 天氣即將大變 → 被動 Agent 不會提前通知
- 重要的截止日期臨近 → 被動 Agent 沉默不語

**限制 2：無法自動執行例行任務**

很多任務是例行性的，不需要使用者每次手動觸發：
- 每日早報的生成和發送
- 待辦事項的定期回顧
- 記憶系統的維護和壓縮
- 重要資料的定期備份

**限制 3：無法持續監控**

有些場景需要 Agent 持續監控某個狀態：
- 監控特定網頁的更新
- 追蹤股票價格或加密貨幣行情
- 監控伺服器的健康狀態
- 追蹤外賣或包裹的送達進度

**限制 4：系統「遺忘」**

沒有心跳的 Agent 在沒有互動的時段可能「遺忘」重要的上下文。有了 Heartbeat，Agent 可以定期回顧和整理資訊，保持記憶的鮮活度。

### 心跳帶來的能力

有了 Heartbeat，OpenClaw 從一個「等待指令的工具」升級為一個「主動工作的助理」：

```
被動模式（無 Heartbeat）：
時間 ──────────────────────────────────────────▶
 │                                              │
 User 問                                   User 問
 Agent 答                                  Agent 答
 │                                              │
 └──── 在這段時間 Agent 完全閒置 ──────────────┘

主動模式（有 Heartbeat）：
時間 ──────────────────────────────────────────▶
 │       │       │       │       │       │     │
 User 問 💓      💓      💓      💓      💓  User 問
 Agent 答 Agent  Agent  Agent   Agent   Agent Agent 答
          檢查    壓縮   發送    備份    生成
          待辦    記憶   提醒    資料   日報
```

## 主動式 vs 被動式 Agent

### 比較表

| 維度 | 被動式 Agent | 主動式 Agent（Heartbeat） |
|------|-----------|----------------------|
| **觸發方式** | 使用者訊息 | 使用者訊息 + 定時觸發 |
| **運行模式** | 請求-回應 | 請求-回應 + 排程任務 |
| **提醒能力** | 無 | 主動提醒 |
| **例行任務** | 需手動觸發 | 自動執行 |
| **監控能力** | 無 | 定期檢查 |
| **資源消耗** | 只在互動時消耗 | 持續消耗（但可控） |
| **複雜度** | 低 | 中等 |
| **使用者干擾** | 無 | 需要降噪策略 |
| **適用場景** | 簡單問答 | 個人助理、自動化 |

### 主動式 Agent 的挑戰

引入主動行為也帶來了新的挑戰：

**挑戰 1：資源消耗**

每次心跳都可能觸發 LLM 呼叫，產生 API 費用和計算資源消耗。需要精心設計觸發頻率和任務優先級。

**挑戰 2：使用者干擾**

如果 Agent 過於「積極」，可能會在不適當的時候打擾使用者。需要實施降噪策略，確保只在有價值的時候才通知使用者。

**挑戰 3：行為可預測性**

使用者可能不清楚 Agent 什麼時候會「自己做事」。需要透明的日誌和通知機制，讓使用者了解 Agent 的自主行為。

**挑戰 4：錯誤累積**

自動執行的任務如果出錯，可能在使用者不知情的情況下累積問題。需要完善的錯誤處理和報告機制。

OpenClaw 的 Heartbeat 模組在設計中充分考慮了這些挑戰，提供了細粒度的控制和智慧的降噪策略來應對這些問題。

## Heartbeat 的運作機制

### 基本機制

Heartbeat 的運作機制非常直觀——它就是一個定時器。每隔固定的時間間隔（預設 30 分鐘），Heartbeat 模組會「觸發」一次，相當於向 Brain 模組發送一條特殊的內部訊息。

```
Heartbeat 基本流程：

┌──────────────┐
│  定時器觸發   │
│  (每 30 分鐘) │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  讀取任務配置 │
│  HEARTBEAT.md│
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  檢查條件     │
│  是否需要     │
│  執行任務？   │
└──────┬───────┘
       │
    ┌──┴──┐
   否      是
    │      │
    ▼      ▼
┌──────┐ ┌──────────────┐
│ 記錄 │ │ 組裝心跳提示詞│
│ 跳過 │ │ 發送至 Brain  │
└──────┘ └──────┬───────┘
               │
               ▼
         ┌──────────────┐
         │ Brain 處理    │
         │ 執行必要任務  │
         │ 生成報告/通知 │
         └──────┬───────┘
               │
               ▼
         ┌──────────────┐
         │ 評估結果      │
         │ 需要通知      │
         │ 使用者嗎？    │
         └──────┬───────┘
               │
           ┌───┴───┐
          否       是
           │       │
           ▼       ▼
         ┌────┐ ┌──────────────┐
         │靜默│ │ 發送通知給    │
         │完成│ │ 使用者        │
         └────┘ └──────────────┘
```

### 心跳訊息

當 Heartbeat 觸發時，它會向 Brain 發送一條特殊的「心跳訊息」：

```json
{
  "type": "heartbeat.trigger",
  "version": "1.0",
  "payload": {
    "trigger_time": "2025-01-16T10:30:00Z",
    "beat_number": 42,
    "scheduled_tasks": [
      {
        "id": "memory_compaction",
        "name": "記憶壓縮",
        "schedule": "every_6_hours",
        "last_run": "2025-01-16T04:30:00Z",
        "status": "due"
      },
      {
        "id": "daily_summary",
        "name": "每日摘要",
        "schedule": "daily_at_18:00",
        "last_run": "2025-01-15T18:00:00Z",
        "status": "not_due"
      }
    ],
    "system_status": {
      "uptime": "2d 5h 30m",
      "memory_files": 342,
      "pending_reminders": 2,
      "active_sessions": 0
    }
  }
}
```

### Brain 的心跳處理

Brain 收到心跳訊息後，會使用一個專門的「心跳提示詞」來處理：

```markdown
# 心跳處理指令

你的定時心跳已觸發。目前時間是 2025-01-16 10:30。

## 待執行任務
以下任務已到達排程時間：

1. **記憶壓縮**（每 6 小時）
   - 上次執行：2025-01-16 04:30
   - 任務：檢查並壓縮過期的會話記憶

## 系統狀態
- 運行時間：2d 5h 30m
- 記憶檔案數：342
- 待處理提醒：2 個
- 活躍會話：0（目前無人在線）

## 待處理提醒
1. 提醒 Eason 下午 3 點有產品會議
2. 提醒 Eason 明天可能下雨

## 指令
1. 執行已到期的排程任務
2. 檢查是否有需要通知使用者的事項
3. 如果有重要通知，生成通知訊息
4. 如果沒有重要事項，靜默完成

重要：遵守「降噪」原則——只有真正重要的事項才通知使用者。
```

### 心跳的時間配置

```yaml
# ~/.openclaw/config/heartbeat.yaml

heartbeat:
  # 心跳間隔
  interval: "30m"        # 每 30 分鐘觸發一次
  
  # 活躍時段
  active_hours:
    start: "07:00"       # 早上 7 點開始
    end: "23:00"         # 晚上 11 點結束
    timezone: "Asia/Taipei"
    
  # 休眠模式
  sleep_mode:
    enabled: true
    # 休眠期間（23:00-07:00）心跳間隔延長
    sleep_interval: "2h"
    # 完全靜音——不發送任何通知
    silent: true
    
  # 空閒時段優化
  idle_optimization:
    enabled: true
    # 如果使用者長時間不在線，自動降低心跳頻率
    idle_threshold: "2h"
    idle_interval: "1h"
```

### 心跳與 Brain 的互動

心跳觸發 Brain 時，Brain 有完整的能力來執行操作——它可以調用技能、查詢記憶、寫入記憶、甚至發起與使用者的對話。不過，Heartbeat 會對 Brain 的行為施加一些額外的限制：

1. **Token 預算限制**：心跳任務的 LLM 呼叫有獨立的 token 預算，避免過度消耗
2. **超時限制**：每次心跳處理有嚴格的超時限制（預設 60 秒）
3. **工具調用限制**：心跳期間可以調用的工具受限於配置的白名單
4. **通知限制**：Brain 生成的通知必須通過降噪策略的過濾

## 排程任務配置（HEARTBEAT.md）

### HEARTBEAT.md 的設計

與 OpenClaw 的其他配置一樣，心跳的排程任務也以 Markdown 檔案的形式定義。HEARTBEAT.md 既是配置檔，也是自文件——使用者可以直接閱讀它來理解排程任務的設定。

### 檔案結構

```markdown
<!-- ~/.openclaw/config/HEARTBEAT.md -->

# OpenClaw 心跳排程任務

## 系統維護任務

### 記憶壓縮
- **排程**：每 6 小時
- **描述**：檢查並壓縮過期的會話記憶和每日筆記
- **行為**：
  1. 掃描 sessions/ 目錄中超過 24 小時的活躍會話
  2. 對超過 5000 tokens 的會話執行摘要壓縮
  3. 將超過 7 天的每日筆記壓縮為摘要
  4. 更新向量索引
- **靜默**：是（不需要通知使用者）

### 記憶提升
- **排程**：每天 22:00
- **描述**：分析最近的每日筆記，將穩定資訊提升到長期記憶
- **行為**：
  1. 分析最近 7 天的每日筆記
  2. 識別反覆出現的模式和偏好
  3. 生成提升建議
  4. 自動執行高信心度的提升
- **靜默**：否（如果有重要的提升，通知使用者確認）

### 健康檢查
- **排程**：每小時
- **描述**：檢查系統各模組的健康狀態
- **行為**：
  1. 驗證 Brain 連線正常
  2. 檢查記憶檔案系統可用
  3. 確認技能列表載入正常
  4. 檢查磁碟空間
- **靜默**：是（除非發現異常）

## 使用者定義任務

### 每日晨報
- **排程**：每天 08:00
- **描述**：生成今日概覽發送給使用者
- **行為**：
  1. 查看今天的行事曆
  2. 檢查天氣預報
  3. 回顧昨天未完成的待辦事項
  4. 生成簡潔的晨報
- **頻道**：telegram
- **靜默**：否

### 每日晚報
- **排程**：每天 18:00
- **描述**：生成今日工作摘要
- **行為**：
  1. 回顧今天的所有對話
  2. 整理已完成和未完成的事項
  3. 記錄重要的決策和發現
  4. 生成精簡的晚報
- **頻道**：telegram
- **靜默**：否

### 週報生成
- **排程**：每週五 17:00
- **描述**：生成本週工作週報
- **行為**：
  1. 回顧本週五天的每日筆記
  2. 彙整重要事件和成果
  3. 識別未完成的任務
  4. 生成結構化的週報
- **輸出**：儲存到 ~/.openclaw/reports/weekly/
- **靜默**：否（發送通知告知週報已生成）

### 備份
- **排程**：每天 03:00
- **描述**：備份記憶和配置
- **行為**：
  1. 建立 ~/.openclaw/ 目錄的壓縮備份
  2. 保留最近 7 天的備份
  3. 刪除過期的備份檔案
- **靜默**：是

## 監控任務

### GitHub 通知檢查
- **排程**：每 2 小時
- **描述**：檢查 GitHub 上的新通知和 PR 更新
- **行為**：
  1. 使用 GitHub API 檢查未讀通知
  2. 檢查 watched 專案的新 PR 和 Issue
  3. 如果有重要更新，通知使用者
- **條件**：只在工作時間（09:00-18:00）執行
- **靜默**：否（只通知重要更新）
```

### HEARTBEAT.md 的解析

Memory 模組在每次心跳觸發時解析 HEARTBEAT.md，提取排程任務的定義：

```
HEARTBEAT.md 解析流程：

1. 讀取 HEARTBEAT.md 檔案
2. 解析 Markdown 結構（標題、列表、粗體標記）
3. 提取每個任務的屬性：
   - 任務名稱（### 標題）
   - 排程規則（排程欄位）
   - 描述（描述欄位）
   - 行為步驟（行為欄位的編號列表）
   - 靜默設定（靜默欄位）
   - 條件（條件欄位，可選）
   - 輸出目標（輸出欄位，可選）
   - 目標頻道（頻道欄位，可選）
4. 根據當前時間和排程規則，決定哪些任務需要執行
5. 將待執行的任務資訊傳遞給 Brain
```

### 排程規則語法

HEARTBEAT.md 支援以下排程規則：

| 語法 | 含義 | 範例 |
|------|------|------|
| `每 N 分鐘` | 每 N 分鐘執行一次 | 每 30 分鐘 |
| `每 N 小時` | 每 N 小時執行一次 | 每 6 小時 |
| `每天 HH:MM` | 每天在指定時間執行 | 每天 08:00 |
| `每週X HH:MM` | 每週指定日期和時間執行 | 每週五 17:00 |
| `每月 D HH:MM` | 每月指定日期和時間執行 | 每月 1 09:00 |
| `每小時` | 每小時執行一次 | 每小時 |
| `每 N 小時` | 每 N 小時執行一次 | 每 2 小時 |

這些規則使用自然語言而非 cron 語法，使得非技術使用者也能輕鬆理解和修改排程。

## 自動維護任務

### 內建的維護任務

Heartbeat 模組內建了一系列系統維護任務，這些任務確保 OpenClaw 長期穩定運行：

**任務 1：記憶壓縮（Memory Compaction）**

定期壓縮過期的會話記憶和每日筆記，防止記憶庫無限增長。

```
記憶壓縮流程：

1. 掃描 sessions/active/ 目錄
2. 找到超過 24 小時未活動的會話
3. 使用 LLM 生成會話摘要
4. 將摘要儲存到 sessions/recent/
5. 歸檔原始會話記錄

6. 掃描 daily/ 目錄
7. 找到超過 7 天的每日筆記
8. 使用 LLM 壓縮為精簡版本
9. 保留原始檔案到歸檔目錄

10. 更新向量索引
```

**任務 2：記憶提升（Memory Promotion）**

分析近期的每日筆記，將穩定的資訊提升到長期記憶：

```
記憶提升流程：

1. 讀取最近 7 天的每日筆記
2. 讀取當前的 MEMORY.md
3. 使用 LLM 分析：
   - 哪些資訊在多天中反覆出現？
   - 哪些資訊與現有長期記憶矛盾？
   - 哪些資訊足夠重要需要永久記錄？
4. 生成提升建議
5. 執行自動提升（高信心度）或請求使用者確認（低信心度）
6. 更新 MEMORY.md
```

**任務 3：向量索引維護（Vector Index Maintenance）**

確保向量索引與實際的記憶檔案保持同步：

```
索引維護流程：

1. 掃描所有記憶檔案的最後修改時間
2. 與向量索引中的記錄比對
3. 對新增或修改的檔案重新生成向量嵌入
4. 移除已刪除檔案的向量記錄
5. 優化向量索引（去碎片化）
```

**任務 4：日誌輪替（Log Rotation）**

管理系統日誌檔案的大小和數量：

```
日誌輪替流程：

1. 檢查各日誌檔案的大小
```

## Cron Job vs Heartbeat 的差異

### 比較表

很多人可能會問：Heartbeat 不就是 Cron Job 嗎？其實兩者有本質的差異：

| 維度 | Cron Job | Heartbeat |
|------|---------|-----------|
| **本質** | 系統級定時器 | AI 驅動的智慧排程 |
| **任務定義** | Shell 命令 | 自然語言描述 + Markdown |
| **執行方式** | 直接執行命令 | 透過 LLM 理解並執行 |
| **上下文感知** | 無 | 有（知道記憶、當前狀態） |
| **條件判斷** | 僅時間條件 | 時間 + 上下文 + 智慧判斷 |
| **輸出處理** | 寫入日誌或郵件 | 智慧通知（降噪） |
| **錯誤處理** | 簡單（重試或通知） | 智慧（LLM 分析錯誤並調整） |
| **互動能力** | 無 | 可以與使用者對話確認 |
| **適應性** | 固定 | 可以根據上下文調整行為 |
| **配置門檻** | 需要 cron 語法 | 自然語言 |

### Heartbeat 的「智慧」特性

**智慧特性 1：上下文感知**

Heartbeat 在觸發時可以存取完整的記憶上下文。例如，晨報生成可以根據昨天的對話內容調整報告的重點。

```
Cron: 每天 08:00 執行 generate_report.sh
→ 每次生成相同格式的報告

Heartbeat: 每天 08:00 生成晨報
→ 如果昨天是密集開會日，報告重點放在會議摘要
→ 如果昨天主要是寫程式，報告重點放在開發進度
→ 如果今天是假日，不生成報告
```

**智慧特性 2：條件判斷**

Heartbeat 可以根據當前狀態決定是否執行任務：

```
Cron: 不管三七二十一，到時間就執行
Heartbeat:
  - 如果使用者正在對話中 → 延遲非緊急任務
  - 如果今天是假日 → 跳過工作相關任務
  - 如果昨天已經沒有新內容 → 跳過記憶壓縮
  - 如果使用者已經手動完成了任務 → 不重複執行
```

**智慧特性 3：錯誤適應**

```
Cron: 任務失敗 → 記錄錯誤 → 下次繼續嘗試
Heartbeat:
  - 天氣 API 失敗 → 使用備用 API
  - 備用 API 也失敗 → 在晨報中標記「天氣資訊暫時不可用」
  - 連續三天失敗 → 建議使用者更換天氣服務提供者
```

**智慧特性 4：自然語言配置**

```
Cron 配置：
30 8 * * 1-5 /usr/local/bin/openclaw-report --type=morning --format=markdown

Heartbeat 配置（HEARTBEAT.md）：
### 每日晨報
- 排程：每天 08:00（僅工作日）
- 描述：生成今日概覽，包含天氣、行程和待辦事項
```

### 何時用 Cron，何時用 Heartbeat

**用 Cron**：
- 不需要 AI 判斷的純機械式任務
- 系統級的任務（如備份腳本、日誌輪替）
- 需要在 OpenClaw 未運行時也能執行的任務

**用 Heartbeat**：
- 需要上下文感知的任務（如生成個性化報告）
- 需要智慧判斷的任務（如「有事才通知」）
- 需要與使用者互動的任務（如確認記憶提升）
- 與 OpenClaw 記憶和技能系統整合的任務

## 降噪策略：「沒事不打擾」

### 降噪的重要性

如果 Heartbeat 每次觸發都發送通知給使用者，很快使用者就會開始忽略這些通知——這就是「通知疲勞」（Notification Fatigue）。更糟的是，使用者可能會因為過多的打擾而關閉 Heartbeat 功能。

降噪策略的目標是：**只在真正重要的時候才打擾使用者**。

### 降噪規則

OpenClaw 實施了多層次的降噪機制：

**規則 1：靜默任務不通知**

標記為「靜默」的任務永遠不會產生使用者可見的通知：

```yaml
# 靜默任務範例
- name: "記憶壓縮"
  silent: true  # 使用者不需要知道記憶壓縮的細節
  
- name: "健康檢查"
  silent: true  # 除非發現異常
  silent_exceptions:
    - "error"   # 發現錯誤時仍然通知
    - "warning" # 發現警告時仍然通知
```

**規則 2：合併通知**

如果多個任務同時產生通知，將它們合併為一條：

```
不合併（打擾 3 次）：
[10:30] 記憶壓縮完成
[10:30] 發現 2 個新提醒
[10:30] 天氣變化通知

合併（打擾 1 次）：
[10:30] 心跳報告：
  - ✅ 記憶壓縮完成
  - 🔔 2 個提醒等待處理
  - ⛅ 天氣可能轉變
```

**規則 3：時段感知**

根據時間段調整通知策略：

```yaml
notification_policy:
  work_hours:         # 09:00-18:00
    allow: true
    priority: "normal"
    
  evening:            # 18:00-23:00
    allow: true
    priority: "high_only"  # 只推送高優先級通知
    
  night:              # 23:00-07:00
    allow: false            # 不推送任何通知
    queue: true             # 排入佇列，早上一起推送
```

**規則 4：頻率限制**

限制通知的頻率，避免在短時間內連續打擾：

```yaml
notification_rate_limit:
  max_per_hour: 3      # 每小時最多 3 條通知
  min_interval: "10m"  # 兩條通知之間至少間隔 10 分鐘
  
  # 如果超過限制，排入佇列
  overflow: "queue"    # queue / drop / merge
```

**規則 5：智慧評估**

Heartbeat 使用 LLM 評估通知的重要性：

```
LLM 評估提示詞：

以下是 Heartbeat 產生的通知候選項。請評估每一項的重要性，
只保留真正需要打擾使用者的項目。

候選通知：
1. 記憶壓縮完成了 3 個會話
2. 明天下午 3 點有重要客戶會議
3. GitHub 上有 2 個新的 PR Review 請求
4. 磁碟空間使用率達到 72%

評估結果：
1. ❌ 不重要——記憶壓縮是常規操作
2. ✅ 重要——明天的會議需要提醒
3. ✅ 重要——PR Review 影響工作進度
4. ❌ 不重要——72% 還很充足
```

### 降噪的可配置性

使用者可以根據個人偏好調整降噪策略：

```yaml
noise_reduction:
  # 降噪等級：low（多通知）、medium（平衡）、high（少通知）
  level: "medium"
  
  # 永遠通知的事件類型
  always_notify:
    - "calendar_reminder"
    - "urgent_alert"
    - "system_error"
    
  # 永遠不通知的事件類型
  never_notify:
    - "routine_maintenance"
    - "backup_complete"
    
  # 使用者可以用指令即時調整
  # /heartbeat quiet  → 靜音 2 小時
  # /heartbeat loud   → 暫時降低過濾
```

### 最佳實踐總結

**實踐 1：從簡單開始**

不要一開始就配置大量排程任務。先從最基本的開始（晨報 + 維護），根據需要逐步添加。

**實踐 2：善用靜默模式**

大多數維護任務都應該設為靜默。使用者需要知道的是結果和異常，而非每個日常操作。

**實踐 3：調整降噪等級**

使用 OpenClaw 一週後，根據實際體驗調整降噪等級。如果通知太多，提高降噪等級；如果錯過重要資訊，降低降噪等級。

**實踐 4：監控資源消耗**

定期查看 Heartbeat 消耗的 token 數和 API 費用。如果成本過高，考慮：
- 延長心跳間隔
- 減少需要 LLM 的任務
- 使用本地模型處理簡單任務

**實踐 5：利用條件執行**

善用「條件」欄位，避免在不需要的時候執行任務。例如：
- 假日不生成工作報告
- 沒有新內容時跳過壓縮
- 使用者離線超過一天時暫停非維護任務

**實踐 6：定期審查 HEARTBEAT.md**

每月審查一次排程配置，刪除不再需要的任務，添加新的需求。保持配置精簡是減少資源消耗的關鍵。

## 總結

Heartbeat 模組是 OpenClaw 從「被動工具」升級為「主動助理」的關鍵。它讓 Agent 能夠自主地執行維護任務、生成報告、提供提醒，同時透過智慧的降噪策略確保不過度打擾使用者。

### 核心設計要點

1. **定時觸發**：預設每 30 分鐘一次心跳，可配置
2. **HEARTBEAT.md**：自然語言定義排程任務，易讀易改
3. **LLM 驅動**：心跳任務由 Brain 執行，具有完整的 AI 能力
4. **降噪策略**：多層次的過濾機制，只推送真正重要的通知
5. **自動維護**：記憶壓縮、提升、索引維護等自動完成
6. **靈活配置**：從最小配置到完整自動化，適應不同需求
