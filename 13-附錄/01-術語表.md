# 術語表

> **說明**：本術語表收錄 OpenClaw 相關的所有技術術語，按字母順序排列。每個術語包含中文名稱、英文原文和詳細解釋。

---

## A

### Agent（代理）
在 OpenClaw 中，Agent 是一個獨立的 AI 助手實體，擁有自己的模型配置、人格定義（SOUL.md）、工具權限和記憶空間。一個 Gateway 可以配置多個 Agent，各自負責不同的任務。例如，一個 Agent 負責社交對話，另一個負責程式碼輔助。

### API Key（API 金鑰）
用於身份認證的密鑰字串，通常由 AI 模型提供者（如 OpenAI、Anthropic）發放。在 OpenClaw 中，API Key 儲存在環境變數中，透過 `${ENV_VAR}` 語法引用，避免直接寫入配置檔。

### Approval（人工批准）
九層權限模型中的第九層安全機制。當工具調用被標記為需要人工批准時，Gateway 會暫停執行，等待操作者明確同意後才繼續。適用於高風險操作如檔案刪除、系統命令執行等。

### Auto-Memorize（自動記憶）
OpenClaw 根據對話內容自動判斷並存儲重要資訊到長期記憶的功能。不需要使用者明確要求「記住這個」，系統會根據 MEMORY.md 中定義的規則自動執行記憶寫入。

---

## B

### Backend（後端）
在沙箱語境中，指執行工具的隔離環境類型。OpenClaw 支援三種沙箱後端：Docker（容器化）、SSH（遠端主機）和 Local（本地 namespace）。

### Bearer Token（Bearer 令牌）
一種 HTTP 認證方式，透過在請求標頭中包含 `Authorization: Bearer <token>` 來進行身份驗證。OpenClaw 的 API 通道和 MCP SSE 模式使用此方式進行認證。

---

## C

### Channel（通道）
OpenClaw 與外部通訊平台連接的介面。每個通道對應一個平台，例如 Discord、Telegram、Matrix、CLI 或 API。通道負責接收使用者訊息和發送 Agent 回應。

### Channel Adapter（通道適配器）
將特定平台的訊息格式轉換為 OpenClaw 內部格式的元件。每個通道都有對應的 Adapter，處理該平台特有的訊息類型、附件、回覆等功能。

### Chain of Thought / CoT（思維鏈）
一種 LLM 推理策略，讓模型逐步展示推理過程。CoT 是純粹在模型內部進行的思考，不與外部世界互動。ReAct 模式在 CoT 的基礎上增加了與外部工具的互動能力。

### Context Assembly（上下文組裝）
推理迴圈的第二階段，負責將系統提示、使用者資料、相關記憶、會話歷史和工具定義組裝成完整的模型輸入。此階段需要在 Token 預算內做出取捨。

### Context Window（上下文窗口）
AI 模型一次能處理的最大 Token 數量。例如 GPT-4-turbo 的上下文窗口為 128K tokens。OpenClaw 需要在此限制內管理系統提示、記憶、歷史和工具定義。

---

## D

### Diarization（說話者分離）
語音處理技術，用於識別音頻中不同說話者的身份。在 Discord 語音場景中，Discord 原生提供使用者 ID，因此不需要複雜的聲紋分析。

### DM / Direct Message（直接訊息）
平台上的一對一私密訊息。在 OpenClaw 中，DM 是與 Agent 進行私密對話的主要方式，透過 Pairing（配對）機制將特定使用者的 DM 綁定到特定 Agent。

### Docker（Docker 容器）
OpenClaw 推薦的沙箱後端。透過 Docker 容器隔離工具執行環境，提供檔案系統、網路和進程的完整隔離。每個 Session 可以有獨立的容器。

---

## E

### Elevated Tool（提升權限工具）
在沙箱環境中，某些工具需要突破沙箱限制才能執行（例如存取主機檔案系統）。這些工具被標記為 Elevated，需要額外的權限配置。

### Embedding（嵌入向量）
將文字轉換為高維度數值向量的技術。OpenClaw 的記憶系統使用嵌入向量進行語義搜尋，找到與查詢語義相似的記憶，而非僅靠關鍵字匹配。

---

## F

### Fade Out（漸弱）
禮貌打斷機制的一部分。當使用者在 Bot 說話時開始說話，Bot 會逐漸降低音量並停止播放，而非突然中斷，提供更自然的對話體驗。

### Function Calling（函數調用）
AI 模型的一項能力，模型可以在回應中指定要調用的工具/函數及其參數。OpenClaw 的工具系統建立在此能力之上。

---

## G

### Gateway（閘道）
OpenClaw 的核心進程，負責管理所有 Agent、通道、工具和推理迴圈。一個 Gateway 實例代表一個完整的 AI 助手系統。Gateway 是信任邊界的最小單位。

### Grounding（接地/錨定）
ReAct 模式中，透過工具調用獲得真實世界資料來驗證和補充模型推理的過程。Grounding 確保 AI 的回答基於事實而非幻覺。

---

## H

### Hallucination（幻覺）
AI 模型生成看似合理但實際上不正確的資訊的現象。ReAct 模式透過工具調用（Grounding）來減少幻覺，讓模型在不確定時查詢外部資訊源。

### Heartbeat（心跳/自主任務）
OpenClaw 的排程任務系統。透過 HEARTBEAT.md 或 cron 排程定義，Agent 可以在沒有使用者訊息觸發的情況下自主執行任務，例如每日摘要、記憶整理等。

### Hook（鉤子）
推理迴圈中的擴展點，允許開發者在特定階段注入自訂邏輯。包括 preInference、postInference、preTool、postTool、preStream 等。

### Human-in-the-Loop（人在迴圈中）
一種安全策略，要求人類操作者在 AI 執行高風險操作前進行確認。OpenClaw 九層權限模型的第九層（Human Approval）實現了這一模式。

---

## I

### Inference（推理）
AI 模型處理輸入並生成輸出的過程。在 OpenClaw 中，一次推理包含上下文組裝、模型調用、回應解析等步驟。

### Inference Loop（推理迴圈）
OpenClaw 的核心處理流程，是 ReAct 模式的實作。包含多次迭代：模型推理 → 工具調用 → 觀察結果 → 再次推理，直到模型生成最終回應。

### Intake（接收）
推理迴圈的第一階段，負責接收通道訊息、解析格式、識別使用者、路由到目標 Agent 和管理 Session。

---

## J

### JSON5
JSON 的超集格式，支援註解、尾逗號、單引號字串等更寬鬆的語法。OpenClaw 使用 JSON5 作為配置檔格式（openclaw.json5），提高可讀性和維護性。

---

## L

### Layer / Permission Layer（權限層）
OpenClaw 九層權限模型中的每一層安全檢查。從第一層（全域配置）到第九層（人工批准），每一層都可以獨立允許或拒絕工具調用。

### LLM / Large Language Model（大型語言模型）
驅動 AI 助手的核心技術，如 GPT-4、Claude、Llama 等。OpenClaw 透過 Provider 抽象層支援多種 LLM。

---

## M

### MCP / Model Context Protocol（模型上下文協議）
由 Anthropic 提出的標準協議，定義 AI 助手如何與外部工具和資料源通訊。OpenClaw 可以作為 MCP Server 運行，讓 Copilot CLI 等 MCP Client 調用其能力。

### MEMORY.md（記憶指引）
定義 Agent 記憶行為規則的 Markdown 文件，包括何時自動記憶、記憶的優先級分類、記憶衛生規則等。

### Memory Recall（記憶檢索）
在上下文組裝階段，根據當前對話內容搜尋相關長期記憶的過程。通常使用嵌入向量的語義搜尋來找到最相關的記憶片段。

### Memory Store（記憶儲存）
OpenClaw 的長期記憶資料庫，支援 SQLite 和 PostgreSQL 等後端。儲存的記憶包含內容、類別、重要性、嵌入向量和時間戳等元資料。

---

## N

### Namespace（命名空間）
Linux 核心的隔離機制，OpenClaw 的 Local 沙箱後端使用 mount namespace、pid namespace 和 network namespace 來隔離工具執行環境。

---

## O

### Observation（觀察）
ReAct 模式中，工具執行後返回的結果。觀察被加入對話上下文，供模型在下一輪推理中參考。觀察是模型與真實世界之間的橋樑。

### Ollama
一個本地運行 LLM 的工具，支援 Llama、Mistral 等開源模型。OpenClaw 支援 Ollama 作為 Provider，適合需要本地推理或離線使用的場景。

### OpenClaw
一個開源的 AI Agent 框架，核心設計理念是「個人 AI 助手」而非多租戶平台。提供 Gateway 架構、九層權限模型、ReAct 推理迴圈、多通道支援等完整功能。

### Operator（操作者）
部署和管理 OpenClaw Gateway 的人。在個人使用場景中，Operator 通常就是使用者本人。Operator 擁有最高信任等級。

### Opus
一種音頻編解碼器，Discord 語音通道使用 Opus 格式傳輸音頻。OpenClaw 的語音處理管線需要在 Opus 和 PCM 之間轉換。

---

## P

### Pairing（配對）
將特定使用者在特定通道上的訊息綁定到特定 Agent 的過程。配對後，該使用者的 DM 會自動路由到指定的 Agent。

### PCM / Pulse Code Modulation（脈衝編碼調變）
一種未壓縮的音頻格式，STT 和 TTS 引擎通常以 PCM 格式處理音頻。OpenClaw 的語音管線在接收到 Opus 音頻後先解碼為 PCM。

### Persistence（持久化）
推理迴圈的最後階段，負責儲存對話歷史、寫入記憶、記錄使用統計和日誌。確保所有重要資料在推理結束後被安全保存。

### Prompt Injection（提示注入）
一種安全攻擊，惡意使用者透過精心構造的輸入來改變 AI 模型的行為，繞過安全限制。OpenClaw 透過輸入清理、角色隔離和權限模型來防禦此類攻擊。

### Provider（提供者）
OpenClaw 中對 AI 模型服務的抽象。每個 Provider 對應一個模型提供商（OpenAI、Anthropic、Ollama 等），管理 API 連接、認證和模型列表。

---

## R

### Rate Limiting（速率限制）
限制 API 請求頻率的安全機制。OpenClaw 支援每分鐘最大請求數和每小時最大 Token 數的限制，防止過度使用或攻擊。

### ReAct（Reasoning + Acting）
一種結合推理和行動的 AI Agent 模式。模型先思考（Thought），然後執行行動（Action），觀察結果（Observation），再繼續思考。這種循環讓 AI 能基於真實資訊做出更準確的回應。

### Resource（資源）
MCP 協議中的概念，代表 MCP Server 可以提供給 Client 讀取的資料，如配置檔、記憶摘要等。

---

## S

### Sandbox（沙箱）
工具執行的隔離環境，防止不受信任的操作影響主系統。OpenClaw 支援 Docker、SSH 和 Local 三種沙箱後端，可按 Agent 或 Session 配置。

### Scope（作用域）
沙箱的生命週期範圍。per-session 表示每次會話一個新的沙箱（最安全），per-agent 表示同一 Agent 的所有會話共享沙箱，shared 表示全局共享。

### seccomp（Secure Computing Mode）
Linux 核心的安全機制，限制進程可以使用的系統呼叫。OpenClaw 的 Local 沙箱後端使用 seccomp 來限制工具進程的能力。

### Session（會話）
使用者與 Agent 之間的一次連續對話。Session 管理對話歷史、上下文和狀態。同一 Session 內的推理是串列執行的。

### SOUL.md（靈魂文件）
定義 Agent 人格、說話風格、行為規則的 Markdown 文件。SOUL.md 是 OpenClaw 人格系統的核心，對應每個 Agent 一個。

### Speaker Diarization
見 [Diarization](#diarization說話者分離)。

### SSE / Server-Sent Events
一種基於 HTTP 的單向串流技術。伺服器可以透過 SSE 持續向客戶端推送事件。MCP 協議和 OpenClaw 的即時回應都可以使用 SSE。

### STT / Speech-to-Text（語音轉文字）
將語音音頻轉換為文字的技術。OpenClaw 支援 OpenAI Whisper、Azure Speech 和 Google STT 等引擎。

### Streaming（串流）
將模型回應逐步（token by token）發送給使用者的技術，而非等待完整回應生成後才發送。串流模式能顯著降低使用者感知的回應延遲。

---

## T

### TAO Cycle（思考-行動-觀察循環）
ReAct 模式的核心循環：Thought（思考）→ Action（行動）→ Observation（觀察）。模型重複此循環直到獲得足夠資訊來生成最終回應。

### Temperature（溫度）
控制 LLM 輸出隨機性的參數。值越低（如 0.1）輸出越確定和一致，值越高（如 1.0）輸出越多樣和創意。程式碼生成通常用低溫度，社交對話用較高溫度。

### Token（令牌）
LLM 處理文字的基本單位。一個中文字通常為 1-2 個 token，一個英文單字通常為 1-3 個 token。Token 是計算 API 費用和上下文限制的基礎單位。

### Tool（工具）
Agent 可以調用的外部功能。工具可以是內建的（search_web、memory_read）或自訂的（weather_query、calendar_read）。工具透過九層權限模型進行訪問控制。

### Trust Boundary（信任邊界）
安全模型中的關鍵概念。一個 OpenClaw Gateway 實例構成一個信任邊界，邊界內的元件（Agent、工具、記憶）受到統一的安全策略保護。

### TTS / Text-to-Speech（文字轉語音）
將文字轉換為語音音頻的技術。OpenClaw 支援 Azure TTS、ElevenLabs 和 Edge TTS 等引擎，用於語音通道的回應輸出。

### Turn Taking（輪流機制）
語音對話中的對話管理策略，確保 Bot 和使用者不會同時說話。OpenClaw 透過 VAD 和靜默檢測來實現自然的輪流對話。

---

## U

### USER.md（使用者檔案）
記錄使用者基本資訊、偏好和上下文的 Markdown 文件。USER.md 在每次推理時被載入系統提示，幫助 Agent 個性化回應。

---

## V

### VAD / Voice Activity Detection（語音活動檢測）
偵測音頻中是否有人在說話的技術。VAD 用於確定語音的起始和結束，將連續音頻切分為獨立的語句段落。

---

## W

### WebSocket
一種全雙工通訊協議，允許客戶端和伺服器之間進行即時雙向通訊。OpenClaw 使用 WebSocket 進行串流回應和即時事件通知。

### Whisper
OpenAI 開發的語音識別模型，支援多語言轉錄。OpenClaw 支援將 Whisper 作為 STT 引擎使用。

---

## Z

### Zero-Trust（零信任）
一種安全架構理念，假設沒有任何元件是完全可信的，所有存取都需要驗證和授權。OpenClaw 的九層權限模型部分採用了零信任的設計思想。

### Zod
一個 TypeScript-first 的 schema 驗證庫。OpenClaw 的 MCP Server 使用 Zod 來定義和驗證工具參數。
