## 1. 語音能力概述

### 1.1 為什麼語音能力很重要

文字是人類與電腦互動的傳統方式，但語音是人類最自然的溝通方式。一個真正好用的 AI 助手必須能聽、能說。語音能力讓 OpenClaw 從一個「聊天機器人」進化為一個「對話夥伴」。

語音能力的核心價值：

- **自然互動**：人類每分鐘能說 150 個字，但只能打 40 個字
- **解放雙手**：開車、做飯、運動時都能與 AI 互動
- **情感傳達**：語音攜帶情緒、語氣、韻律等豐富資訊
- **無障礙**：讓視障使用者或不擅打字的人也能使用
- **多場景適用**：Discord 語音頻道、WhatsApp 語音訊息、電話整合

### 1.2 語音的三大模組

OpenClaw 的語音能力由三個核心模組組成，每個模組獨立運作但可以協同工作：

```
┌─────────────────────────────────────────────────┐
│                 OpenClaw 語音系統                  │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌────────────────┐ │
│  │   STT    │  │   TTS    │  │   Talk Mode    │ │
│  │ 語音轉文字 │  │ 文字轉語音 │  │   即時對話      │ │
│  │          │  │          │  │                │ │
│  │ 聽 → 文字 │  │ 文字 → 說 │  │ 聽 + 想 + 說  │ │
│  └──────────┘  └──────────┘  └────────────────┘ │
│                                                   │
│       獨立使用        獨立使用       整合 STT+TTS    │
│   WhatsApp 語音    Discord 回覆    Discord 語音頻道  │
│   Telegram 語音    [[tts]] 標記    電話/Twilio       │
└─────────────────────────────────────────────────┘
```

- **STT（Speech-to-Text）**：將語音訊息轉換為文字，送入 ReAct 引擎處理
- **TTS（Text-to-Speech）**：將 AI 的文字回應合成為語音，供使用者聆聽
- **Talk Mode**：整合 STT + LLM + TTS 的即時語音對話系統

### 1.3 語音管線全景圖

完整的語音處理管線：

```
使用者說話                                        使用者聽到回應
    │                                                ▲
    ▼                                                │
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 音訊擷取  │───▶│   STT    │───▶│  ReAct   │───▶│   TTS    │
│          │    │ 語音轉文字 │    │  引擎    │    │ 文字轉語音 │
│ 麥克風    │    │          │    │          │    │          │
│ 語音訊息  │    │ Whisper  │    │ 理解意圖  │    │ 合成語音  │
│ 串流音訊  │    │ Deepgram │    │ 執行技能  │    │ 串流輸出  │
│          │    │          │    │ 生成回應  │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     ↕                ↕               ↕               ↕
  音訊格式          轉錄文字        指令/回應         音訊格式
  PCM/Opus        + 時間戳         文字            PCM/MP3/Opus
  16kHz/48kHz     + 信心分數                       + SSML 標記
```

**兩種使用模式**：

1. **非即時模式**：語音訊息 → 完整轉錄 → 處理 → 回覆（WhatsApp、Telegram）
2. **即時模式（Talk Mode）**：串流音訊 → 即時轉錄 → 即時處理 → 即時語音回覆（Discord 語音頻道）

## 2. 語音架構設計

### 2.1 模組化設計原則

OpenClaw 的語音系統遵循模組化設計，每個組件都可以獨立替換：

```
┌────────────────────────────────────────┐
│              語音管理器                  │
│         (VoiceManager)                  │
├────────────────────────────────────────┤
│  STT Provider     │  TTS Provider       │
│  ├─ Whisper       │  ├─ ElevenLabs      │
│  ├─ Deepgram      │  ├─ OpenAI TTS      │
│  ├─ Azure Speech  │  ├─ Azure Speech    │
│  ├─ Google STT    │  ├─ Edge TTS        │
│  └─ 自定義        │  ├─ MiniMax         │
│                   │  └─ 自定義          │
├────────────────────────────────────────┤
│           音訊處理引擎                   │
│  ├─ 格式轉換 (FFmpeg)                   │
│  ├─ 降噪 (RNNoise)                      │
│  ├─ 音量正規化                           │
│  └─ 靜音偵測 (VAD)                       │
└────────────────────────────────────────┘
```

### 2.2 Provider 抽象層

所有語音供應商都實作統一的介面：

```typescript
// STT Provider 介面
interface STTProvider {
  name: string;
  
  // 非串流轉錄（語音訊息）
  transcribe(audio: AudioBuffer, options: STTOptions): Promise<Transcript>;
  
  // 串流轉錄（Talk Mode）
  transcribeStream(stream: AudioStream, options: STTOptions): AsyncIterator<PartialTranscript>;
}

// TTS Provider 介面
interface TTSProvider {
  name: string;
  
  // 非串流合成
  synthesize(text: string, options: TTSOptions): Promise<AudioBuffer>;
  
  // 串流合成（Talk Mode）
  synthesizeStream(text: string, options: TTSOptions): AsyncIterator<AudioChunk>;
  
  // 可用的語音列表
  listVoices(): Promise<Voice[]>;
}
```

### 2.3 音訊格式與編碼

OpenClaw 內部統一使用 PCM 格式處理音訊，在輸入/輸出時自動轉換：

| 格式 | 用途 | 特性 |
|------|------|------|
| PCM 16-bit | 內部處理 | 無壓縮，高品質 |
| Opus | Discord/WhatsApp/Telegram | 高壓縮率，低延遲 |
| MP3 | TTS 輸出 | 廣泛相容 |
| WAV | 本地存儲/除錯 | 無壓縮 |
| OGG | Telegram 語音訊息 | 標準容器格式 |
| WebM | 瀏覽器 | 網頁相容 |

```bash
# OpenClaw 依賴 FFmpeg 進行格式轉換
# 確認 FFmpeg 已安裝
ffmpeg -version
```

### 2.4 延遲與品質的權衡

語音系統的關鍵指標是**延遲**和**品質**的平衡：

```
延遲組成（Talk Mode 完整循環）：

  音訊擷取    ≈ 100ms（緩衝區）
  + STT 轉錄  ≈ 200-800ms（取決於供應商）
  + LLM 推理  ≈ 500-2000ms（取決於模型）
  + TTS 合成  ≈ 200-500ms（取決於供應商）
  + 網路傳輸  ≈ 50-200ms
  ──────────────────────
  總延遲      ≈ 1-3.5 秒
```

**人類對話的自然停頓約為 200-600ms**。要做到自然對話，需要將延遲控制在 1.5 秒以內。

優化策略：
- **串流處理**：STT 和 TTS 都使用串流模式，不等待完整結果
- **First Token 優先**：TTS 在收到第一個字就開始合成
- **預測性準備**：在使用者說話時就開始準備回應
- **快速模型**：使用更快的 STT/TTS 供應商

## 3. STT 語音轉文字概覽

### 3.1 STT 的角色

STT 是語音系統的「耳朵」，負責將音訊轉換為文字。轉錄品質直接影響 AI 理解使用者意圖的能力。

| 供應商 | 模型 | 中文支援 | 串流 | 延遲 | 費用 |
|--------|------|----------|------|------|------|
| OpenAI | Whisper | ✅ 優秀 | ❌ | 中 | $0.006/min |
| Deepgram | Nova-2 | ✅ 良好 | ✅ | 低 | $0.0043/min |
| Azure | Speech Service | ✅ 優秀 | ✅ | 低 | $1/hr |

### 3.3 STT 選擇指南

```
需要最佳中文品質？
  → Whisper（OpenAI API 或本地 whisper.cpp）

需要即時串流轉錄？
  → Deepgram Nova-2 或 Azure Speech

預算有限？
  → 本地 whisper.cpp（免費，需要 GPU）

Talk Mode 使用？
  → Deepgram（延遲最低 + 串流支援）
```

## 5. Talk Mode 即時對話概覽

### 5.1 什麼是 Talk Mode

Talk Mode 是 OpenClaw 最進階的語音功能 — 即時語音對話。不是「錄音 → 轉文字 → 回覆」的異步模式，而是像打電話一樣的即時對話體驗。

```
傳統語音訊息模式（異步）：
  使用者錄音 → 發送 → STT → LLM → 文字/語音回覆
  延遲：5-10 秒

Talk Mode（即時）：
  使用者邊說 → 邊轉錄 → 邊思考 → 邊說話回覆
  延遲：1-2 秒
```

### 5.2 Talk Mode 架構

```
┌─────────────────────────────────────────────────┐
│                  Talk Mode Engine                 │
│                                                   │
│  ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐     │
│  │ VAD  │──▶│ STT  │──▶│ LLM  │──▶│ TTS  │     │
│  │      │   │Stream│   │Stream│   │Stream│     │
│  └──────┘   └──────┘   └──────┘   └──────┘     │
│      │                                  │        │
│      │         ┌──────────┐             │        │
│      └────────▶│ Barge-In │◀────────────┘        │
│                │ Detector │                       │
│                └──────────┘                       │
│                                                   │
│  ┌────────────────────────────────────────────┐  │
│  │           Turn Management                   │  │
│  │  ├─ 誰在說話？（Speaker Detection）          │  │
│  │  ├─ 何時該回應？（Endpointing）              │  │
│  │  └─ 如何處理打斷？（Barge-In）               │  │
│  └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 5.3 Talk Mode 的技術挑戰

1. **VAD（Voice Activity Detection）**：判斷使用者是否在說話
2. **Endpointing**：判斷使用者是否說完了
3. **Barge-In**：使用者打斷 AI 說話的處理
4. **回聲消除**：防止 AI 聽到自己的聲音
5. **延遲控制**：保持對話的自然節奏

## 8. 語音安全與隱私

### 8.1 語音資料的敏感性

語音資料比文字更敏感，因為它包含：

- **生物特徵**：聲紋可以用於身份識別
- **情緒狀態**：語音攜帶情緒資訊
- **環境資訊**：背景噪音可能洩露位置
- **健康狀態**：某些語音特徵與健康狀況相關

### 8.2 隱私保護措施

```json
{
  "voice": {
    "privacy": {
      "deleteAudioAfterProcessing": true,
      "noAudioLogging": true,
      "localProcessingPreferred": true,
      "consentRequired": true,
      "anonymizeVoicePrint": true
    }
  }
}
```

### 8.3 本地 vs 雲端處理

| 面向 | 本地處理 | 雲端處理 |
|------|----------|----------|
| 隱私 | 最高 | 依賴供應商 |
| 延遲 | 取決於硬體 | 穩定低延遲 |
| 品質 | 取決於模型大小 | 最高 |
| 成本 | 硬體成本 | 按量付費 |
| 維護 | 需要自行維護 | 供應商維護 |

## 9. 效能優化概述

語音系統效能優化的關鍵點：

1. **預載入模型**：啟動時載入 STT/TTS 模型，避免首次使用的冷啟動延遲
2. **連接池**：維護到雲端供應商的持久連接
3. **快取**：常用短語的 TTS 結果可以快取
4. **串流處理**：所有階段都使用串流，不等待完整結果
5. **並行處理**：STT 和前一段的處理可以並行
6. **品質自適應**：根據網路狀況自動調整音訊品質
