## 1. 通道（Channel）的概念

### 1.1 什麼是通道

在 OpenClaw 的架構中，**通道（Channel）** 是 AI 助手與外部世界溝通的介面。每個通道代表一個通訊平台或協議——Discord 是一個通道、Telegram 是一個通道、WhatsApp 是一個通道，甚至命令行終端（CLI）也是一個通道。

通道的本質是一個「翻譯層」：它把各種不同平台的訊息格式翻譯成 OpenClaw 內部的統一格式，同時把 OpenClaw 的回應翻譯回各個平台的原生格式。

想像一下：你的 AI 助手 EasonClaw 同時活躍在 Discord、Telegram 和 WhatsApp 上。在 Discord 上，使用者通過 Slash Command 互動；在 Telegram 上，使用者直接發送文字訊息；在 WhatsApp 上，使用者可能還會發送語音訊息。儘管介面和交互方式各不相同，但 EasonClaw 在每個平台上都表現出一致的人格和能力——這就是通道系統的魔力。

### 1.2 通道在 OpenClaw 架構中的位置

```
┌──────────────────────────────────────────────────────────┐
│                     外部世界                              │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │ Discord │ │Telegram │ │WhatsApp │ │  CLI    │  ...  │
│  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘       │
│       │           │           │           │              │
├───────┼───────────┼───────────┼───────────┼──────────────┤
│       ▼           ▼           ▼           ▼              │
```

通道位於系統的最外層，是使用者觸及 OpenClaw 的第一個接觸點。

### 1.3 為什麼需要多通道支援

多通道支援帶來以下關鍵優勢：

**1. 使用者在哪裡，AI 就在哪裡**

不同使用者有不同的通訊偏好。有人住在 Discord，有人離不開 Telegram，有人只用 WhatsApp。多通道支援讓 AI 助手不需要使用者改變習慣——而是融入使用者已有的通訊生態。

**2. 場景適配**

不同的場景適合不同的通道。在工作中使用 Slack 與 AI 協作，在社交時通過 Discord 語音頻道互動，出門在外用 WhatsApp 快速查詢。每個通道都有其最適合的使用情境。

**3. 可用性保證**

如果只依賴單一通道，該通道故障就意味著 AI 助手完全不可用。多通道設計提供了天然的冗餘——即使 Discord 宕機，你還可以通過 Telegram 聯繫你的 AI。

**4. 功能互補**

不同通道有不同的能力。Discord 有語音頻道，Telegram 有 Inline Bot，WhatsApp 有端到端加密。多通道讓你能利用各平台的獨特優勢。

## 3. 通道適配器架構

### 3.1 適配器模式

OpenClaw 使用**適配器模式（Adapter Pattern）** 來統一不同通道的介面。每個通道都有一個對應的適配器，負責：

1. **接收**：監聽平台事件，接收使用者訊息
2. **轉換**：將平台特定的訊息格式轉換為 OpenClaw 內部格式
3. **路由**：將轉換後的訊息送到 OpenClaw 核心處理
4. **回傳**：將 OpenClaw 的回應轉換回平台格式並發送

```
Discord 訊息（Discord.js 格式）
    │
    ▼
Discord Adapter
    │
    ├── 提取內容：message.content
    ├── 提取作者：message.author.id
    ├── 提取頻道：message.channel.id
    ├── 提取附件：message.attachments
    │
    ▼
OpenClaw 標準訊息格式（StandardMessage）
{
  "id": "msg_unique_id",
  "channel": "discord",
  "channelId": "1234567890",
  "userId": "discord:user123",
  "userName": "Eason",
  "content": "今天天氣如何？",
  "attachments": [],
  "timestamp": "2024-03-15T10:00:00Z",
  "metadata": {
    "guildId": "guild123",
    "isDirectMessage": false
  }
}
```

### 3.2 訊息標準化流程

```
外部訊息 → 適配器接收 → 格式轉換 → 安全過濾 → 權限檢查 → 路由分發
    │                                                        │
    │                                                        ▼
    │                                              OpenClaw Core
    │                                                        │
    │                                                        ▼
    │                                              產生回應
    │                                                        │
外部回應 ← 適配器發送 ← 格式轉換 ← 長度裁剪 ← 安全過濾 ←──┘
```

**StandardMessage 結構**：

```typescript
interface StandardMessage {
  id: string;                    // 訊息唯一 ID
  channel: ChannelType;          // 通道類型
  channelId: string;             // 通道內的頻道/對話 ID
  userId: string;                // 統一使用者 ID（channel:platformId）
  userName: string;              // 顯示名稱
  content: string;               // 文字內容
  contentType: 'text' | 'voice' | 'image' | 'file';
  attachments: Attachment[];     // 附件列表
  replyTo?: string;              // 回覆的訊息 ID
  timestamp: string;             // ISO 8601 時間戳
  metadata: Record<string, any>; // 通道特定的元數據
}
```

### 3.3 適配器生命週期

```
初始化 → 連線 → 監聽 → 處理 → 斷線重連 → 關閉
  │       │       │       │         │         │
  │    建立連線  等待事件  訊息處理   自動重連   清理資源
  │    認證驗證  心跳檢查  回應發送   指數退避   關閉連線
  │    註冊事件  狀態更新  錯誤處理   最大重試   記錄狀態
  │
  └── 載入配置
      驗證參數
      建立客戶端
```

每個適配器的狀態機：

```
INITIALIZING → CONNECTING → CONNECTED → LISTENING
                   ↑                        │
                   │                        ▼
              RECONNECTING ← ─ ─ ─ ─ DISCONNECTED
                   │
                   ▼ (max retries)
                 FAILED
```

### 3.4 自定義適配器開發

如果你需要支援 OpenClaw 尚未內建的通道，可以開發自定義適配器：

```typescript
// custom-adapters/my-platform-adapter.ts
import { ChannelAdapter, StandardMessage, AdapterConfig } from '@openclaw/core';

export class MyPlatformAdapter extends ChannelAdapter {
  name = 'my-platform';
  
  async initialize(config: AdapterConfig): Promise<void> {
    // 初始化客戶端
    this.client = new MyPlatformClient(config.token);
  }
  
  async connect(): Promise<void> {
    await this.client.connect();
    this.client.on('message', this.handleMessage.bind(this));
  }
  
  private async handleMessage(rawMessage: any): Promise<void> {
    // 轉換為標準格式
    const standardMessage: StandardMessage = {
      id: rawMessage.id,
      channel: 'my-platform',
      channelId: rawMessage.chatId,
      userId: `my-platform:${rawMessage.senderId}`,
      userName: rawMessage.senderName,
      content: rawMessage.text,
      contentType: 'text',
      attachments: [],
      timestamp: new Date().toISOString(),
      metadata: {},
    };
    
    // 送到核心處理
    await this.routeMessage(standardMessage);
  }
  
  async sendResponse(channelId: string, response: string): Promise<void> {
    await this.client.sendMessage(channelId, response);
  }
  
  async disconnect(): Promise<void> {
    await this.client.disconnect();
  }
}
```

## 4. 通道配置的通用模式

所有通道都遵循相同的配置模式，定義在 `openclaw.json` 的 `channels` 區段中。

### 4.1 enabled——啟用與停用

```json
{
  "channels": {
    "discord": {
      "enabled": true
    },
    "telegram": {
      "enabled": true
    },
    "whatsapp": {
      "enabled": false
    }
  }
}
```

`enabled: false` 的通道不會建立連線，也不會消耗系統資源。你可以隨時切換而不需要重新部署。

### 4.2 allowFrom——存取控制

`allowFrom` 控制哪些使用者或群組可以與 AI 互動：

```json
{
  "channels": {
    "discord": {
      "enabled": true,
      "allowFrom": {
        "users": [
          "discord:123456789",
          "discord:987654321"
        ],
        "guilds": [
          "guild:111222333"
        ],
        "channels": [
          "channel:444555666"
        ]
      }
    }
  }
}
```

**存取控制邏輯**：

```
收到訊息
    │
    ├── allowFrom 未設定？ → 允許所有人（危險！）
    │
    ├── 檢查 users 列表 → 符合？ → 允許
    │
    ├── 檢查 guilds 列表 → 符合？ → 允許
    │
    ├── 檢查 channels 列表 → 符合？ → 允許
    │
    └── 全部不符合 → 拒絕（靜默忽略）
```

### 4.3 dmPolicy——私訊策略

`dmPolicy` 控制 AI 如何處理私訊：

```json
{
  "channels": {
    "discord": {
      "dmPolicy": {
        "enabled": true,
        "allowUnknownUsers": false,
        "requirePairing": true,
        "maxConcurrentDMs": 10,
        "rateLimitPerUser": {
          "messages": 30,
          "period": "1m"
        }
      }
    }
  }
}
```

| 選項 | 類型 | 預設 | 說明 |
|------|------|------|------|
| `enabled` | bool | `true` | 是否接受私訊 |
| `allowUnknownUsers` | bool | `false` | 是否接受未在 allowFrom 中的使用者的私訊 |
| `requirePairing` | bool | `false` | 是否需要配對流程 |
| `maxConcurrentDMs` | number | `20` | 最大同時私訊數 |
| `rateLimitPerUser` | object | - | 每使用者的速率限制 |

### 4.4 groupPolicy——群組策略

`groupPolicy` 控制 AI 在群組中的行為：

```json
{
  "channels": {
    "discord": {
      "groupPolicy": {
        "enabled": true,
        "triggerMode": "mention",
        "triggerPrefix": "!claw",
        "respondInThread": true,
        "maxGroupSize": 100,
        "cooldown": 5
      }
    }
  }
}
```

| 選項 | 類型 | 說明 |
|------|------|------|
| `triggerMode` | string | `"mention"`: @提及才回應; `"prefix"`: 指令前綴; `"always"`: 所有訊息 |
| `triggerPrefix` | string | 當 triggerMode 為 "prefix" 時的前綴字串 |
| `respondInThread` | bool | 在討論串中回應，避免刷頻 |
| `maxGroupSize` | number | 群組人數上限，超過不回應 |
| `cooldown` | number | 同一群組的回應冷卻時間（秒） |

### 4.5 完整配置範例

```json
{
  "channels": {
    "discord": {
      "enabled": true,
      "botToken": "${DISCORD_BOT_TOKEN}",
      "allowFrom": {
        "users": ["discord:123456789"],
        "guilds": ["guild:111222333"]
      },
      "dmPolicy": {
        "enabled": true,
        "requirePairing": false
      },
      "groupPolicy": {
        "enabled": true,
        "triggerMode": "mention",
        "respondInThread": true
      },
      "voice": {
        "enabled": true,
        "autoJoin": false
      }
    },
    "telegram": {
      "enabled": true,
      "botToken": "${TELEGRAM_BOT_TOKEN}",
      "allowFrom": {
        "users": ["telegram:123456"]
      },
      "dmPolicy": {
        "enabled": true
      },
      "groupPolicy": {
        "enabled": true,
        "triggerMode": "mention"
      }
    },
    "cli": {
      "enabled": true,
      "promptStyle": "interactive"
    }
  }
}
```

## 5. 通道隔離與容錯

### 5.1 通道隔離原則

OpenClaw 的通道系統遵循嚴格的隔離原則：**一個通道的故障不應該影響其他通道**。

```
Discord 斷線               Telegram 正常
    │                          │
    ▼                          ▼
Discord Adapter → FAILED   Telegram Adapter → LISTENING
    │                          │
    │                          ▼
    │                    正常處理訊息
    │
    ▼
  自動重連中...
  不影響其他通道
```

每個適配器運行在獨立的進程/執行緒中，記憶體、錯誤處理、重連機制都是獨立的。

### 5.2 容錯機制

```json
{
  "channels": {
    "discord": {
      "resilience": {
        "reconnect": {
          "enabled": true,
          "maxRetries": 10,
          "baseDelay": 1000,
          "maxDelay": 30000,
          "backoffMultiplier": 2
        },
        "circuitBreaker": {
          "enabled": true,
          "failureThreshold": 5,
          "recoveryTimeout": 60000
        }
      }
    }
  }
}
```

**斷路器模式（Circuit Breaker）**：

```
CLOSED（正常運作）
    │
    ├── 連續 5 次失敗
    │
    ▼
OPEN（停止嘗試）
    │
    ├── 等待 60 秒
    │
    ▼
HALF-OPEN（嘗試恢復）
    │
    ├── 成功 → 回到 CLOSED
    └── 失敗 → 回到 OPEN
```

### 5.3 通道健康監控

```bash
# 查看所有通道狀態
openclaw channels status

# 輸出：
# ┌───────────┬──────────┬──────────┬────────────┐
# │ Channel   │ Status   │ Uptime   │ Messages   │
# ├───────────┼──────────┼──────────┼────────────┤
# │ discord   │ 🟢 ACTIVE │ 3d 12h  │ 1,234      │
# │ telegram  │ 🟢 ACTIVE │ 3d 12h  │ 567        │
# │ whatsapp  │ ⬜ OFF    │ -       │ -          │
# │ cli       │ 🟢 ACTIVE │ 2h 30m  │ 45         │
# └───────────┴──────────┴──────────┴────────────┘
```

## 6. 通道間的一致性保證

### 6.1 身份統一

OpenClaw 使用統一的身份系統，將不同平台的使用者映射到同一個身份：

```json
{
  "identity": {
    "user": "eason",
    "linkedAccounts": {
      "discord": "discord:123456789",
      "telegram": "telegram:987654",
      "whatsapp": "whatsapp:+886912345678"
    }
  }
}
```

當使用者在 Discord 上說「我剛剛在 Telegram 提到的那件事」，OpenClaw 能夠透過身份映射，找到同一使用者在 Telegram 上的對話歷史。

### 6.2 記憶共享

所有通道共享同一個記憶系統。這意味著：

- 你在 Discord 上告訴 AI 你的名字，它在 Telegram 上也知道
- 你在 WhatsApp 上設定的偏好，在 Slack 上同樣生效
- 你在任何通道上的對話歷史，AI 都能存取

```
Discord 對話: "我叫 Eason，我喜歡喝咖啡"
    │
    ▼
OpenClaw Memory System
    │
    ▼
Telegram 對話: "你好 Eason！想喝杯咖啡嗎？"
```

### 6.3 人格一致性

無論通過哪個通道互動，AI 的人格設定保持一致。`personality.md` 中定義的角色、語氣、行為準則，在所有通道上都是統一的。

但有一些通道特定的適配——例如，在 Discord 群組中回應可能更簡短活潑，在私訊中更詳細親切。這些差異通過通道策略（而非人格設定）來控制。

### 6.4 跨通道通知

OpenClaw 支援跨通道通知：

```json
{
  "notifications": {
    "crossChannel": true,
    "preferredChannel": "discord",
    "fallbackOrder": ["discord", "telegram", "whatsapp"]
  }
}
```

例如，一個定時提醒可以設定為：先嘗試在 Discord 上通知，如果使用者不在線，就嘗試 Telegram，最後嘗試 WhatsApp。

## 7. 通道配置最佳實踐

**1. 最小權限原則**

只啟用你需要的通道，只授權必要的使用者。

```json
{
  "channels": {
    "discord": {
      "enabled": true,
      "allowFrom": {
        "users": ["discord:your-id"]
      }
    }
  }
}
```

**2. 先 CLI，後通道**

先在 CLI 通道上確認 AI 的行為符合預期，再啟用其他通道。

**3. 漸進式開放**

先只允許自己使用（測試），再開放給信任的朋友，最後才（如果需要）開放給公眾。

**4. 設定速率限制**

避免被濫用或意外產生高額 API 費用。

**5. 監控和日誌**

定期檢查通道的健康狀態和使用情況。

## 8. 總結

通道系統是 OpenClaw 的「窗口」，讓 AI 助手能夠在使用者習慣的平台上與他們互動。通過適配器模式、統一的訊息格式、和完善的容錯機制，OpenClaw 實現了「一個 AI，多個通道」的願景。

關鍵要點：

- **通道是翻譯層**，不是 AI 的核心——核心邏輯對所有通道統一
- **適配器模式**讓新增通道變得簡單
- **通道隔離**確保一個通道的故障不會拖垮整個系統
- **身份統一和記憶共享**讓使用者在任何通道都有一致的體驗
- **最小權限**是安全的第一步
