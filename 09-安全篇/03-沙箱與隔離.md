## 1. 沙箱的目的與原理

### 1.1 為什麼需要沙箱？

在 AI Agent 系統中，Agent 可以被指示（或被注入）執行各種操作。即使有九層權限模型，仍然存在一個根本性的問題：**一旦工具被允許執行，它就在 Gateway 的完整 OS 環境中運行**。

```
沒有沙箱的世界：

Agent 調用 file_write("/workspace/code.py", "print('hello')")
→ 在 Gateway 的 OS 環境中執行
→ file_write 程式碼有完整的 OS 權限
→ 如果 file_write 有漏洞，攻擊者可能：
  • 讀取 /etc/passwd
  • 讀取環境變數中的 API 金鑰
  • 修改 Gateway 配置
  • 安裝後門
  • 存取網路中的其他服務
```

沙箱的作用就是**將工具的執行環境隔離**，即使工具被濫用或有漏洞，影響也被限制在沙箱內。

### 1.2 隔離原理

```
有沙箱的世界：

Agent 調用 file_write("/workspace/code.py", "print('hello')")
→ 工具在 Docker 容器中執行
→ 容器只有 /workspace 目錄（掛載的 volume）
→ 容器沒有 Gateway 的環境變數
→ 容器的網路被限制
→ 容器的 CPU/記憶體被限制
→ 即使工具被濫用：
  • ❌ 無法讀取 /etc/passwd（容器的 /etc 是獨立的）
  • ❌ 無法讀取 API 金鑰（環境變數未傳入）
  • ❌ 無法修改 Gateway 配置（不可見）
  • ❌ 無法安裝後門（容器會被銷毀）
  • ❌ 無法存取其他服務（網路被限制）
```

### 1.3 沙箱 vs 權限模型

```
權限模型（L1-L6, L8-L9）：
  → 決定「是否允許」執行工具
  → 就像保全在門口檢查你的身份
  → 如果你通過檢查，你進入大樓後可以自由行動

沙箱（L7）：
  → 決定「在哪裡」執行工具
  → 就像把你帶到一個會客室
  → 你可以做被允許的事，但只在這個房間裡
  → 即使你行為不當，也不會影響大樓的其他部分

兩者結合 = 最佳安全：
  → 先檢查是否允許（權限模型）
  → 再限制執行環境（沙箱）
```

## 2. 爆炸半徑的概念

### 2.1 什麼是爆炸半徑？

爆炸半徑（Blast Radius）是一個安全工程術語，指的是**一個安全事件影響的最大範圍**。

```
無沙箱的爆炸半徑：

Agent 被 compromise
├── 可以存取整個檔案系統
├── 可以讀取所有環境變數
├── 可以存取所有網路服務
├── 可以修改 Gateway 配置
├── 可以影響所有其他 Agent
└── 爆炸半徑 = 整個 Gateway 伺服器
    ┌──────────────────────────┐
    │ █████████████████████████ │ ← 全部受影響
    │ █████████████████████████ │
    │ █████████████████████████ │
    └──────────────────────────┘

有沙箱的爆炸半徑：

Agent 被 compromise
├── 只能存取沙箱內的檔案系統
├── 只能讀取傳入沙箱的環境變數
├── 網路被限制或禁止
├── 無法修改 Gateway 配置
├── 無法影響其他 Agent 或 Session
└── 爆炸半徑 = 一個沙箱容器
    ┌──────────────────────────┐
    │ ░░░░░░░░░░░░░░░░░░░░░░░ │ ← 不受影響
    │ ░░░░░░░█████░░░░░░░░░░░ │ ← 只有沙箱受影響
    │ ░░░░░░░░░░░░░░░░░░░░░░░ │ ← 不受影響
    └──────────────────────────┘
```

### 2.2 爆炸半徑等級

| 等級 | 沙箱配置 | 爆炸半徑 |
|------|----------|----------|
| 無沙箱 | sandbox.enabled = false | 整個 OS |
| 共享沙箱 | scope = "shared" | 所有 Agent 的共享環境 |
| per-agent 沙箱 | scope = "per-agent" | 一個 Agent 的所有 session |
| per-session 沙箱 | scope = "per-session" | 一個 session |
| per-call 沙箱 | scope = "per-call" | 一次工具調用 |

```
爆炸半徑大小：

  無沙箱 > shared > per-agent > per-session > per-call
  最危險                                      最安全
  最高效能                                    最低效能
```

## 3. 沙箱後端類型

### 3.1 Docker 後端

Docker 是最常用的沙箱後端，利用 Linux 容器技術提供隔離。

**工作原理**：

```
Gateway 程序
│
├── 工具調用請求
│
├── Docker API
│   └── 建立容器
│       ├── 映像：python:3.11-slim
│       ├── 掛載：/workspace
│       ├── 網路：受限
│       ├── 資源：受限
│       └── 生命週期：與 session 綁定
│
├── 在容器中執行工具
│
├── 取得結果
│
└── 銷毀容器（或保留給下一次調用）
```

**配置範例**：

```json5
{
  sandbox: {
    enabled: true,
    backend: "docker",

    docker: {
      // Docker daemon 連線
      socket: "/var/run/docker.sock",
      // 或者使用 TCP 連線
      // host: "tcp://localhost:2376",
      // tls: true,

      // 基礎映像
      image: "openclaw/sandbox:latest",
      // 或者自訂映像
      // image: "my-registry/my-sandbox:v1",

      // 容器建立選項
      create_options: {
        // 安全選項
        security_opt: ["no-new-privileges:true"],
        // 唯讀根檔案系統
        read_only: true,
        // 臨時檔案系統
        tmpfs: {
          "/tmp": "size=100M,noexec,nosuid"
        },
        // 停用特權模式
        privileged: false,
        // 停用所有 Linux capabilities
        cap_drop: ["ALL"],
        // 只加回需要的
        cap_add: [],
        // PID 命名空間隔離
        pid_mode: "private",
        // 停用網路（除非明確允許）
        network_mode: "none"
      },

      // 資源限制
      resources: {
        memory: "512M",
        memory_swap: "512M",  // 等於 memory = 禁用 swap
        cpu_period: 100000,
        cpu_quota: 50000,     // 0.5 CPU
        pids_limit: 100       // 最多 100 個程序
      },

      // 掛載設定
      volumes: [
        {
          host: "./sandbox-workspace",
          container: "/workspace",
          mode: "rw"
        },
        {
          host: "./sandbox-data",
          container: "/data",
          mode: "ro"  // 唯讀
        }
      ],

      // 環境變數（只傳入必要的）
      environment: {
        "LANG": "en_US.UTF-8",
        "TZ": "Asia/Taipei"
        // ⚠️ 絕不傳入 API 金鑰或密碼！
      },

      // 生命週期管理
      lifecycle: {
        // 容器閒置多久後銷毀
        idle_timeout: 300,  // 5 分鐘
        // 容器最大存活時間
        max_lifetime: 3600,  // 1 小時
        // 是否在 session 結束時銷毀
        destroy_on_session_end: true
      }
    }
  }
}
```

**優點**：
- 強隔離（Linux namespaces + cgroups）
- 成熟的技術，穩定可靠
- 支援自訂映像（可安裝特定工具）
- 資源限制精確

**缺點**：
- 需要安裝 Docker
- 容器啟動有一定延遲（約 1-3 秒）
- 佔用更多記憶體
- Docker socket 存取本身是一個安全風險

### 3.2 SSH 後端

SSH 後端透過 SSH 連線到遠端機器執行工具，適合將執行環境完全分離到另一台機器。

**配置範例**：

```json5
{
  sandbox: {
    enabled: true,
    backend: "ssh",

    ssh: {
      // SSH 連線設定
      host: "sandbox.internal.lan",
      port: 22,
      user: "sandbox",
      // 使用金鑰認證
      private_key: "${SSH_PRIVATE_KEY_PATH}",
      // 或者使用密碼
      // password: "${SSH_PASSWORD}",

      // 遠端工作目錄
      remote_workspace: "/home/sandbox/workspace",

      // 連線池
      connection_pool: {
        max_connections: 5,
        idle_timeout: 60
      },

      // 安全設定
      security: {
        // 限制可執行的命令
        allowed_commands: ["python3", "node", "ls", "cat"],
        // 限制環境變數
        clean_environment: true,
        // 設定 ulimit
        ulimits: {
          nofile: 1024,
          nproc: 50,
          fsize: "100M"
        }
      }
    }
  }
}
```

**優點**：
- 物理隔離（不同機器）
- 不需要 Docker
- 適合已有基礎設施的環境

**缺點**：
- 需要額外的機器
- SSH 連線延遲
- 配置較複雜
- 隔離等級取決於遠端機器的配置

### 3.3 Local（OpenShell）後端

Local 後端在同一台機器上使用 OS 層級的隔離（chroot、seccomp、namespaces）但不使用 Docker。

**配置範例**：

```json5
{
  sandbox: {
    enabled: true,
    backend: "local",

    local: {
      // 使用 Linux namespaces（需要 root 或 CAP_SYS_ADMIN）
      namespaces: {
        // 檔案系統隔離
        mount: true,
        // PID 隔離
        pid: true,
        // 網路隔離
        network: true,
        // 使用者隔離（允許 rootless）
        user: true
      },

      // chroot 根目錄
      root: "/opt/openclaw/sandbox-root",

      // seccomp 設定
      seccomp: {
        // 使用預設的限制設定檔
        profile: "default",
        // 或自訂
        // profile: "/etc/openclaw/seccomp-profile.json"
      },

      // 資源限制（使用 cgroups v2）
      resources: {
        memory: "512M",
        cpu: "0.5",
        pids: 50
      },

      // 掛載設定
      mounts: [
        { src: "./workspace", dst: "/workspace", type: "bind", options: "rw" },
        { src: "tmpfs", dst: "/tmp", type: "tmpfs", options: "size=100M" }
      ]
    }
  }
}
```

**優點**：
- 不需要 Docker
- 啟動速度快（接近原生）
- 資源消耗低

**缺點**：
- 隔離等級可能不如 Docker
- 需要 Linux（不支援 macOS/Windows）
- 配置較複雜
- 可能需要 root 權限

### 3.4 後端選擇指南

```
你需要什麼等級的隔離？
│
├─ 最高隔離 → SSH 到獨立機器
│              適合：生產環境、處理敏感資料
│
├─ 高隔離 → Docker 容器
│            適合：大多數場景
│
├─ 中等隔離 → Local（namespaces + seccomp）
│              適合：資源受限的環境
│
└─ 無隔離 → sandbox.enabled = false
              適合：完全信任的環境（個人開發機）
```

## 4. 沙箱範圍

### 4.1 per-session 範圍

每個對話 session 有自己獨立的沙箱實例。

```
Session A ──▶ 沙箱容器 A（Session A 結束後銷毀）
Session B ──▶ 沙箱容器 B（Session B 結束後銷毀）
Session C ──▶ 沙箱容器 C（Session C 結束後銷毀）

特點：
  ✅ 不同 session 完全隔離
  ✅ session 結束後自動清理
  ⚠️ 每個 session 都要啟動一個容器
  ⚠️ session 間的檔案不共享
```

```json5
{
  sandbox: {
    scope: "per-session",
    // 每個 session 一個獨立容器
    docker: {
      lifecycle: {
        destroy_on_session_end: true
      }
    }
  }
}
```

### 4.2 per-agent 範圍

每個 Agent 有自己的沙箱，同一 Agent 的所有 session 共享。

```
Agent A
├── Session 1 ──┐
├── Session 2 ──┼──▶ 沙箱容器 A（Agent A 共享）
└── Session 3 ──┘

Agent B
├── Session 4 ──┐
└── Session 5 ──┼──▶ 沙箱容器 B（Agent B 共享）
                 └

特點：
  ✅ 不同 Agent 隔離
  ✅ 同一 Agent 的 session 可以共享檔案
  ⚠️ 同一 Agent 的 session 間沒有隔離
  ⚠️ 容器生命週期較長
```

```json5
{
  sandbox: {
    scope: "per-agent",
    docker: {
      lifecycle: {
        destroy_on_session_end: false,
        idle_timeout: 1800  // 30 分鐘無活動後銷毀
      }
    }
  }
}
```

### 4.3 shared 範圍

所有 Agent 和 session 共享一個沙箱實例。

```
Agent A Session 1 ──┐
Agent A Session 2 ──┼──▶ 一個共享的沙箱容器
Agent B Session 3 ──┤
Agent B Session 4 ──┘

特點：
  ⚠️ 所有 Agent 和 session 之間沒有隔離
  ✅ 資源效率最高
  ✅ 所有工具可以共享檔案
  ⚠️ 一個 Agent 被 compromise 影響所有其他
```

```json5
{
  sandbox: {
    scope: "shared",
    docker: {
      lifecycle: {
        max_lifetime: 86400  // 24 小時
      }
    }
  }
}
```

### 4.4 範圍選擇指南

```
安全需求               推薦範圍
───────────────────────────────
最高安全              per-session
多 Agent 需要隔離     per-agent
資源受限              shared
開發測試              shared
```

## 5. 配置詳解

### 5.1 完整配置範例

```json5
{
  sandbox: {
    // 啟用沙箱
    enabled: true,

    // 後端類型
    backend: "docker",

    // 範圍
    scope: "per-session",

    // Docker 後端配置
    docker: {
      socket: "/var/run/docker.sock",
      image: "openclaw/sandbox:latest",

      create_options: {
        security_opt: ["no-new-privileges:true"],
        read_only: true,
        privileged: false,
        cap_drop: ["ALL"],
        network_mode: "none",
        tmpfs: {
          "/tmp": "size=100M,noexec,nosuid"
        }
      },

      resources: {
        memory: "512M",
        memory_swap: "512M",
        cpu_period: 100000,
        cpu_quota: 100000,  // 1 CPU
        pids_limit: 100
      },

      volumes: [
        {
          host: "./workspace/${session_id}",
          container: "/workspace",
          mode: "rw"
        }
      ],

      environment: {
        LANG: "en_US.UTF-8",
        TZ: "Asia/Taipei"
      },

      lifecycle: {
        idle_timeout: 300,
        max_lifetime: 3600,
        destroy_on_session_end: true,
        cleanup_on_error: true
      }
    },

    // 哪些工具在沙箱中執行
    sandboxed_tools: [
      "exec",
      "shell",
      "code_execute",
      "file_write",
      "file_delete",
      "browser"
    ],

    // 哪些工具不在沙箱中（在主環境中執行）
    unsandboxed_tools: [
      "memory_*",
      "search_web",
      "discord_*",
      "telegram_*",
      "time_now"
    ],

    // 預設行為：未列出的工具
    default_behavior: "sandbox"  // "sandbox" | "unsandbox" | "deny"
  }
}
```

### 5.2 自訂沙箱映像

```dockerfile
# Dockerfile for custom sandbox
FROM ubuntu:22.04

# 安裝需要的工具
RUN apt-get update && apt-get install -y \
    python3 python3-pip \
    nodejs npm \
    git \
    curl \
    && rm -rf /var/lib/apt/lists/*

# 安裝 Python 套件
RUN pip3 install \
    requests \
    numpy \
    pandas

# 建立工作目錄
RUN mkdir -p /workspace && \
    useradd -r -s /bin/bash -d /workspace sandbox && \
    chown sandbox:sandbox /workspace

# 以非 root 使用者運行
USER sandbox
WORKDIR /workspace

# 健康檢查
HEALTHCHECK CMD test -d /workspace
```

```bash
# 建立自訂映像
docker build -t my-openclaw-sandbox:v1 .
```

```json5
{
  sandbox: {
    docker: {
      image: "my-openclaw-sandbox:v1"
    }
  }
}
```

### 5.3 網路設定

```json5
{
  sandbox: {
    docker: {
      // 方案一：完全禁止網路（最安全）
      create_options: {
        network_mode: "none"
      },

      // 方案二：受限網路（允許特定目標）
      // create_options: {
      //   network_mode: "openclaw-sandbox-net"  // 自訂 Docker 網路
      // },

      // 方案三：完整網路（最不安全）
      // create_options: {
      //   network_mode: "bridge"
      // }
    }
  }
}
```

**建立受限網路**：

```bash
# 建立隔離的 Docker 網路
docker network create \
  --driver bridge \
  --internal \
  openclaw-sandbox-net

# 如果需要允許特定外部存取，
# 使用 iptables 規則
iptables -A FORWARD -s 172.18.0.0/16 -d api.openai.com -j ACCEPT
iptables -A FORWARD -s 172.18.0.0/16 -j DROP
```

## 6. 提權工具（tools.elevated）

### 6.1 概念

某些工具需要超出沙箱預設限制的權限才能正常運作。`tools.elevated` 機制允許特定工具在沙箱中獲得額外權限，同時仍然保持在沙箱環境內。

```
普通沙箱工具：
  在沙箱中執行
  受所有限制（CPU、記憶體、網路、檔案系統）

提權工具：
  在沙箱中執行
  但可以存取額外的資源
  例如：掛載額外的目錄、存取網路
```

### 6.2 配置範例

```json5
{
  tools: {
    elevated: [
      {
        tool: "code_execute",
        // 額外的掛載
        additional_volumes: [
          {
            host: "./project-data",
            container: "/data",
            mode: "ro"  // 唯讀
          }
        ],
        // 額外的資源
        additional_resources: {
          memory: "+256M",  // 額外 256MB
          cpu: "+0.5"       // 額外 0.5 CPU
        },
        // 允許網路存取
        network: {
          enabled: true,
          allowlist: ["pypi.org", "npmjs.com"]  // 只允許套件管理
        }
      },
      {
        tool: "browser",
        // 瀏覽器需要更多資源
        additional_resources: {
          memory: "+1G",
          cpu: "+1.0"
        },
        // 需要網路
        network: {
          enabled: true,
          allowlist: ["*"]  // 瀏覽器需要存取任何網站
        },
        // 需要顯示（headless chrome）
        environment: {
          DISPLAY: ":99",
          CHROME_FLAGS: "--headless --no-sandbox --disable-gpu"
        }
      },
      {
        tool: "git_push",
        // Git push 需要網路和 SSH 金鑰
        network: {
          enabled: true,
          allowlist: ["github.com", "gitlab.com"]
        },
        additional_volumes: [
          {
            host: "${HOME}/.ssh/id_rsa",
            container: "/home/sandbox/.ssh/id_rsa",
            mode: "ro"
          }
        ]
      }
    ]
  }
}
```

### 6.3 提權的安全考量

```
提權工具的風險等級：

低風險提權：
  額外掛載唯讀目錄 → 只能讀取，不能寫入
  額外 CPU/記憶體 → 不增加攻擊面
  
中風險提權：
  額外掛載讀寫目錄 → 可能修改重要檔案
  有限網路存取 → 可能外傳資料

高風險提權：
  完整網路存取 → 可能聯繫 C2 伺服器
  掛載 SSH 金鑰 → 可能存取其他系統
  掛載 Docker socket → 可能逃逸沙箱
```

**⚠️ 絕對不要提權的項目**：

```json5
{
  tools: {
    elevated: [
      {
        tool: "some-tool",
        // ❌ 絕對不要！
        additional_volumes: [
          // 不要掛載 Docker socket
          // { host: "/var/run/docker.sock", container: "/var/run/docker.sock" },
          // 不要掛載整個根目錄
          // { host: "/", container: "/host" },
          // 不要掛載 /etc
          // { host: "/etc", container: "/etc" }
        ],
        // ❌ 絕對不要！
        // privileged: true
        // cap_add: ["SYS_ADMIN"]
      }
    ]
  }
}
```

### 7.1 Gateway 本身

**最重要的事實：OpenClaw Gateway 程序本身不在沙箱中運行。** 沙箱只保護工具執行環境。

```
┌────────────────────────────────────────────┐
│              Gateway 程序（主環境）          │
│                                            │
│  ┌──────────────────────────────────────┐  │
│  │ • 推理迴圈                           │  │
│  │ • 權限檢查引擎                       │  │
│  │ • 記憶系統                           │  │
│  │ • 通道管理                           │  │
│  │ • 配置管理                           │  │
│  │ • 日誌系統                           │  │
│  │                                      │  │
│  │ ⚠️ 這些都不在沙箱中！               │  │
│  └──────────────────────────────────────┘  │
│                                            │
│  ┌───────────────┐  ┌───────────────┐     │
│  │  沙箱容器 A   │  │  沙箱容器 B   │     │
│  │  (Session 1)  │  │  (Session 2)  │     │
│  │  工具在這裡   │  │  工具在這裡   │     │
│  │  執行         │  │  執行         │     │
│  └───────────────┘  └───────────────┘     │
└────────────────────────────────────────────┘
```

### 7.2 不在沙箱中的元件清單

| 元件 | 在沙箱中？ | 保護方式 |
|------|------------|----------|
| Gateway 程序 | ❌ | OS 權限、systemd 安全設定 |
| 配置檔案 | ❌ | 檔案權限（600） |
| 記憶資料庫 | ❌ | 檔案權限 + 可選加密 |
| API 金鑰（環境變數） | ❌ | OS 環境變數保護 |
| 通道連線（WebSocket） | ❌ | TLS 加密 |
| 日誌檔案 | ❌ | 檔案權限 |
| AI 模型 API 呼叫 | ❌ | HTTPS 加密 |
| 內建安全工具 | ❌ | 程式碼審計 |

### 7.3 保護 Gateway 本身

既然 Gateway 不在沙箱中，如何保護它？

```bash
# 1. 以專用使用者運行
sudo useradd -r -s /bin/bash -d /opt/openclaw openclaw
sudo chown -R openclaw:openclaw /opt/openclaw
sudo chmod 700 /opt/openclaw

# 2. 使用 systemd 安全功能
cat > /etc/systemd/system/openclaw.service <<EOF
[Unit]
Description=OpenClaw Gateway

[Service]
Type=simple
User=openclaw
Group=openclaw

# 防止權限提升
NoNewPrivileges=true

# 保護系統目錄
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/opt/openclaw

# 限制可用的核心能力
CapabilityBoundingSet=
AmbientCapabilities=

# 限制系統呼叫
SystemCallFilter=@system-service
SystemCallFilter=~@mount @reboot @swap @clock

# 私有 /tmp
PrivateTmp=true

# 私有 /dev
PrivateDevices=true

# 限制網路存取
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX

ExecStart=/usr/local/bin/openclaw start
Restart=always

[Install]
WantedBy=multi-user.target
EOF

# 3. 設定嚴格的檔案權限
chmod 600 /opt/openclaw/openclaw.json5
chmod 600 /opt/openclaw/.env
chmod 700 /opt/openclaw/memory/
chmod 700 /opt/openclaw/logs/
```

## 8. 沙箱逃逸風險

### 8.1 什麼是沙箱逃逸？

沙箱逃逸（Sandbox Escape）指的是攻擊者找到方法從沙箱環境中突破出來，存取到沙箱外部的資源。

```
正常情況：
  攻擊者 → 沙箱 → 被限制在沙箱內

沙箱逃逸：
  攻擊者 → 沙箱 → 找到漏洞 → 突破沙箱 → 存取主環境
```

### 8.2 Docker 沙箱的已知風險

#### 8.2.1 Docker Socket 掛載

```
⚠️ 最危險的錯誤配置：

如果將 Docker socket 掛載到容器中：
  volumes: [{ host: "/var/run/docker.sock", container: "/var/run/docker.sock" }]

攻擊者可以：
  1. 在容器中使用 docker 命令
  2. 建立一個掛載了主機根目錄的新容器
  3. 在新容器中擁有主機的完整存取權限
  → 完全逃逸！

防禦：
  ❌ 絕對不要將 Docker socket 掛載到沙箱容器中
```

#### 8.2.2 核心漏洞

```
Docker 容器共享主機核心。
如果核心有漏洞，容器內的攻擊者可能利用核心漏洞逃逸。

歷史案例：
  - CVE-2019-5736 (runc 漏洞)
  - CVE-2020-15257 (containerd 漏洞)
  - CVE-2022-0185 (Linux 核心漏洞)

防禦：
  ✅ 保持核心和 Docker 更新
  ✅ 使用 seccomp 限制系統呼叫
  ✅ 使用 AppArmor/SELinux
  ✅ 禁用不需要的核心功能 (cap_drop: ALL)
```

#### 8.2.3 資源耗盡攻擊

```
即使在沙箱中，攻擊者可能嘗試耗盡主機資源：
  - Fork 炸彈（建立大量程序）
  - 記憶體炸彈（分配大量記憶體）
  - 磁碟填滿（寫入大量資料）

防禦：
  ✅ 設定 pids_limit（限制程序數量）
  ✅ 設定 memory 和 memory_swap 限制
  ✅ 使用唯讀根檔案系統 + 限制 tmpfs 大小
  ✅ 使用 disk quota
```

### 8.3 緩解措施清單

```json5
{
  sandbox: {
    docker: {
      create_options: {
        // 1. 禁止權限提升
        security_opt: ["no-new-privileges:true"],

        // 2. 唯讀根檔案系統
        read_only: true,

        // 3. 禁用特權模式
        privileged: false,

        // 4. 移除所有 Linux capabilities
        cap_drop: ["ALL"],

        // 5. 限制程序數量
        // (在 resources 中設定 pids_limit)

        // 6. 禁用網路
        network_mode: "none",

        // 7. 私有 PID 命名空間
        pid_mode: "private",

        // 8. 限制臨時檔案系統
        tmpfs: {
          "/tmp": "size=50M,noexec,nosuid,nodev"
        }
      },

      resources: {
        // 9. 記憶體限制
        memory: "512M",
        memory_swap: "512M",

        // 10. CPU 限制
        cpu_quota: 50000,

        // 11. 程序數量限制
        pids_limit: 50
      }
    }
  }
}
```

### 8.4 防禦深度

```
即使沙箱被逃逸，還有其他防線：

沙箱逃逸後...
├── Gateway 以專用低權限使用者運行
│   → 即使逃逸，仍受 OS 使用者權限限制
│
├── 系統使用 systemd 安全功能
│   → ProtectSystem=strict 限制了檔案系統存取
│
├── 網路防火牆
│   → 限制了可存取的網路服務
│
├── SELinux/AppArmor
│   → 強制存取控制限制了程序能力
│
└── 監控和告警
    → 異常行為會被偵測和報警
```

## 9. 最佳實踐

### 9.1 沙箱配置最佳實踐

```
1. 預設在沙箱中執行所有有副作用的工具
   → sandboxed_tools 包含所有寫入/執行類工具
   → default_behavior: "sandbox"

2. 使用 per-session 範圍
   → 每個對話獨立隔離
   → session 結束自動清理

3. 最小化沙箱的資源
   → 根據實際需求設定限制
   → 不要給過多的記憶體/CPU

4. 禁用不需要的功能
   → network_mode: "none"（除非需要）
   → read_only: true
   → cap_drop: ["ALL"]

5. 不傳入敏感資料
   → 不傳入 API 金鑰
   → 不傳入密碼
   → 不掛載包含敏感資料的目錄

6. 定期清理
   → 設定 idle_timeout 和 max_lifetime
   → 定期刪除舊的沙箱 workspace

7. 監控沙箱健康狀態
   → 檢查容器資源使用
   → 監控異常行為（大量程序、高 CPU 等）

8. 保持映像更新
   → 定期更新沙箱映像
   → 修補已知漏洞
```

### 9.2 沙箱配置範本

**最安全配置（公開伺服器）**：

```json5
{
  sandbox: {
    enabled: true,
    backend: "docker",
    scope: "per-session",
    default_behavior: "deny",  // 未列出的工具直接拒絕
    sandboxed_tools: ["code_execute"],
    unsandboxed_tools: ["memory_read", "search_web"],
    docker: {
      image: "openclaw/sandbox:minimal",
      create_options: {
        security_opt: ["no-new-privileges:true"],
        read_only: true,
        privileged: false,
        cap_drop: ["ALL"],
        network_mode: "none",
        tmpfs: { "/tmp": "size=10M,noexec,nosuid,nodev" }
      },
      resources: {
        memory: "128M",
        memory_swap: "128M",
        cpu_quota: 25000,
        pids_limit: 20
      },
      lifecycle: {
        idle_timeout: 60,
        max_lifetime: 300,
        destroy_on_session_end: true
      }
    }
  }
}
```

**平衡配置（辦公室使用）**：

```json5
{

    enabled: true,
    backend: "docker",
    scope: "per-agent",
```

### 10.1 效能影響分析

| 操作 | 沒有沙箱 | Docker 沙箱 | SSH 沙箱 |
|------|----------|-------------|----------|
| 首次啟動 | 0ms | 1-3s | 0.5-1s |

| 檔案操作 | ~1ms | ~10ms | ~50ms |
| 記憶體額外開銷 | 0 | ~50MB/容器 | ~5MB/連線 |

### 10.2 效能調校建議

```json5
{
  sandbox: {

      pre_warm: {
        enabled: true,
        pool_size: 3  // 預先建立 3 個容器
      },

      // 2. 使用輕量映像
      image: "openclaw/sandbox:alpine",  // Alpine 比 Ubuntu 小很多

      // 減少容器建立/銷毀頻率
    }
  }
```

## 11. 疑難排解

### 11.1 常見問題

**問題：Docker 容器啟動失敗**

```bash
# 檢查 Docker daemon 是否運行
sudo systemctl status docker

# 檢查 Gateway 使用者是否在 docker 群組中
groups openclaw
# 如果不在，添加到 docker 群組
sudo usermod -aG docker openclaw

# 檢查映像是否存在
docker images | grep openclaw/sandbox
# 如果不存在，拉取映像
docker pull openclaw/sandbox:latest
```

**問題：容器內無法存取檔案**

```bash
# 檢查掛載目錄的權限
ls -la ./sandbox-workspace/
# 確保 Gateway 使用者有讀寫權限
chmod 777 ./sandbox-workspace/
```

**問題：容器記憶體不足**

```json5
{
  sandbox: {
    docker: {
      resources: {
        memory: "1G",        // 增加記憶體限制
        memory_swap: "2G"    // 允許使用 swap
      }
    }
  }
}
```

**問題：沙箱中無法安裝套件**

```json5
{
  sandbox: {
    docker: {
      // 允許網路存取（安裝套件需要）
      create_options: {
        network_mode: "bridge"
      },
      // 或者在自訂映像中預裝套件
      // image: "my-sandbox-with-packages:v1"
    }
  }
}
```

```bash
docker ps --filter "label=openclaw-sandbox"
```

## 總結

沙箱是 OpenClaw 安全架構中不可或缺的一環。它與九層權限模型互補：

- **權限模型**：決定「是否允許」
- **沙箱**：決定「在哪裡執行」

兩者結合，確保即使安全防線被突破，影響範圍也被限制在最小。

**關鍵要點**：
1. 所有有副作用的工具都應在沙箱中執行
2. Docker 是最推薦的沙箱後端
3. per-session 範圍提供最好的隔離
4. Gateway 本身不在沙箱中——需要 OS 層級保護
5. 沙箱不是萬能的——需要配合其他安全措施
