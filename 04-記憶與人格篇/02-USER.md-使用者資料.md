# USER.md——使用者身份與偏好定義

> 「了解一個人，不是知道他的名字，而是知道他在乎什麼。」

在人與人的社交互動中，我們在初次見面時會交換名片、自我介紹、分享背景。這些基本資訊構成了我們理解對方的起點。對 AI 代理來說，USER.md 就是那張「名片」——但它遠不止名片那麼簡單。它是一份關於使用者的完整畫像，讓 AI 在每次互動的第一秒就「認識」你。

本篇將深入探討 USER.md 的設計哲學、結構規範、應用場景，以及在多人環境（特別是 Discord）中的特殊考量。我們的目標是理解如何透過一份精心設計的使用者檔案，讓 AI 代理從「通用助手」轉變為「你的專屬夥伴」。

---

## 一、USER.md 的目的——讓代理「認識你」

### 1.1 從通用到個人化

想像你第一次走進一家餐廳。服務生不認識你，會用標準的方式招呼你、推薦菜單上的熱門菜品。但如果你是常客，服務生會記得你不吃辣、偏好靠窗的位置、每次都點同一款咖啡。這種個人化的服務體驗，建立在服務生「認識你」的基礎上。

AI 代理面臨類似的挑戰。大型語言模型本身是通用的——它們被訓練來服務所有人，因此它們的預設行為也是面向「平均使用者」的。但每個使用者都是獨特的：

- 有人是經驗豐富的開發者，有人是剛入門的新手
- 有人喜歡詳盡的解釋，有人只要結論
- 有人使用繁體中文，有人偏好英文夾雜
- 有人在深夜工作，有人在早上最活躍

USER.md 的核心目的，就是將這些個人化的資訊提供給 AI，讓它能夠從一開始就以最適合你的方式互動。

### 1.2 身份認知的三個層次

USER.md 提供的使用者認知可以分為三個層次：

**第一層：基本識別（Who）**
- 你叫什麼名字？怎麼稱呼你？
- 你在哪個時區？使用什麼語言？
- 你的基本角色是什麼（開發者、設計師、PM）？

**第二層：行為偏好（How）**
- 你偏好什麼樣的溝通方式？
- 你使用什麼工具和技術？
- 你的工作習慣是什麼？

**第三層：深層特徵（Why）**
- 你的目標和動機是什麼？
- 你關心什麼？不在乎什麼？
- 你的價值觀和原則是什麼？

一份好的 USER.md 應該覆蓋這三個層次，但不必一開始就面面俱到。可以從基本識別開始，隨著互動的深入逐漸豐富。

### 1.3 USER.md 作為「契約」

USER.md 還有一個經常被忽略的功能：它是使用者與 AI 之間的一份隱含「契約」。通過 USER.md，使用者向 AI 傳達：

- 「這是我同意你知道的關於我的資訊」
- 「這是我期望你遵守的互動方式」
- 「這是我授權你執行的操作範圍」

這份「契約」的存在讓互動更加明確和可預期。使用者知道 AI 掌握了哪些資訊，也能控制 AI 的行為邊界。

### 1.4 認識的持續性價值

在沒有 USER.md 的情況下，每次新會話都需要一段「暖身」過程——AI 需要從對話中推斷使用者的身份和偏好。這不僅浪費時間，也容易產生誤判。

USER.md 消除了這個暖身過程。AI 在會話的第一秒就知道你是誰、你喜歡什麼、你需要什麼。這就像你和一個老朋友聊天——不需要自我介紹，直接進入主題。

對於我們的 Discord 機器人專案來說，這個特性尤為重要。在一個活躍的 Discord 伺服器中，機器人可能在短時間內與多個不同的使用者互動。如果沒有預先建立的使用者檔案，機器人只能用千篇一律的方式回應每個人。有了 USER.md（或其等價物），機器人可以立即識別出「這是 Alice，她喜歡收到鼓勵」或「這是 Bob，他偏好簡潔的技術回答」。

## 二、與 MEMORY.md 的區別

### 2.1 本質差異：身份 vs 經驗

USER.md 和 MEMORY.md 都是 OpenClaw 啟動檔案體系的一部分，但它們有著根本不同的定位：

| 維度 | USER.md | MEMORY.md |
|------|---------|-----------|
| **本質** | 身份定義 | 知識累積 |
| **內容類型** | 穩定的屬性和偏好 | 動態的資訊和經驗 |
| **變更頻率** | 低（偶爾更新） | 高（持續增長） |
| **來源** | 主要由使用者定義 | 主要由互動產生 |
| **比喻** | 護照 / 個人檔案 | 日記 / 筆記本 |
| **管理方式** | 使用者主導 | AI 協助管理 |

### 2.2 資訊流向的差異

**USER.md 的資訊流向**：
```
使用者 → 撰寫/更新 USER.md → AI 讀取 → 個人化互動
```
資訊主要是從使用者流向 AI 的「宣告」。使用者說「這就是我」，AI 接受並據此調整行為。

**MEMORY.md 的資訊流向**：
```
互動過程 → AI 識別重要資訊 → 寫入 MEMORY.md → 未來的 AI 讀取
```
資訊是從互動過程中「提取」並「保存」的。它更像是一個雙向的學習過程。

### 2.3 一個具體的對比

讓我們用一個具體的例子來說明兩者的差異：

**在 USER.md 中**（穩定的身份資訊）：
```markdown
- 姓名：Eason
- 角色：全端開發者
- 語言偏好：繁體中文
- 技術棧：TypeScript, React, Node.js
```

**在 MEMORY.md 中**（動態的經驗和知識）：
```markdown
### 2024-03-10：學到 React Server Components 的新用法
### 2024-03-12：決定在 EasonClaw 專案中使用 Drizzle ORM
### 2024-03-15：Eason 提到最近在研究 WebSocket 實作
```

注意區別：USER.md 中的資訊（「Eason 是全端開發者」）是相對穩定的身份描述，不會每週都改變。而 MEMORY.md 中的資訊（「最近在研究 WebSocket」）是動態的、不斷演化的。

### 2.4 邊界情況的判斷

有些資訊可能同時適合放在兩個地方。例如，「偏好使用 TypeScript」——這是身份偏好（USER.md）還是累積的經驗（MEMORY.md）？

一般的判斷原則是：

- **如果這個資訊反映了使用者的持久特質** → USER.md
- **如果這個資訊是從特定互動中學到的** → MEMORY.md
- **如果不確定** → 先放在 MEMORY.md，如果證明是持久的，再提升到 USER.md

在上面的例子中，「偏好使用 TypeScript」更適合放在 USER.md，因為這是一個穩定的個人偏好。但如果是「在專案 X 中決定使用 TypeScript」，那更適合 MEMORY.md，因為這是一個特定的決策記錄。

### 2.5 協同工作機制

USER.md 和 MEMORY.md 不是孤立的，它們共同構成了 AI 對使用者的完整理解：

```
USER.md（我是誰）
    ↕ 互補
MEMORY.md（我經歷了什麼）
    ↕ 塑造
AI 的行為（如何最好地服務我）
```

舉例：USER.md 告訴 AI「Eason 是全端開發者」，MEMORY.md 告訴 AI「Eason 最近專注在後端」。結合兩者，AI 知道雖然 Eason 能做前後端，但目前應該優先提供後端相關的幫助。

## 三、檔案結構範本

### 3.1 基礎範本

以下是一個基礎的 USER.md 範本，適用於個人使用場景：

```markdown
# 使用者檔案

## 基本資訊
- **姓名**：[你的名字]
- **稱呼**：[你希望被怎麼稱呼]
- **時區**：[你的時區，如 Asia/Taipei]
- **語言**：[偏好語言，如 繁體中文]

## 角色與背景
- **職業**：[你的職業或角色]
- **技術棧**：[你主要使用的技術]
- **經驗等級**：[新手/中級/資深]

## 溝通偏好
- **回答風格**：[簡潔/詳細/視情況]
- **語氣偏好**：[正式/輕鬆/專業]
- **語言混用**：[純中文/中英混用]

## 權限設定
- **自動操作**：[允許/需要確認]
- **敏感操作**：[永遠需要確認]

## 備註
[其他 AI 應該知道的資訊]
```

### 3.2 進階範本

以下是一個更詳細的進階範本：

```markdown
# 使用者檔案

## 基本資訊
- **姓名**：Eason
- **稱呼**：Eason 或「你」（不要叫「使用者」或「先生」）
- **時區**：Asia/Taipei (UTC+8)
- **語言**：繁體中文為主，技術名詞保留英文原文
- **活躍時段**：通常 21:00-03:00

## 角色與背景
- **職業**：全端開發者
- **主要技術棧**：
  - 前端：React, Next.js, TypeScript, Tailwind CSS
  - 後端：Node.js, Express, PostgreSQL
  - 工具：Docker, Git, VS Code
- **經驗等級**：中高級（5 年經驗）
- **當前專注**：AI 應用開發，特別是 AI 代理框架

## 溝通偏好
- **回答風格**：簡潔直接，先結論後解釋
- **語氣偏好**：輕鬆專業，不需要過多禮貌用語
- **程式碼回答**：優先提供可執行的程式碼範例
- **解釋方式**：喜歡用類比和實際例子
- **長度偏好**：除非複雜問題，否則盡量簡短
- **不喜歡**：
  - 過度的免責聲明
  - 重複已知的基礎知識
  - 沒有必要的「請注意...」開場白

## 工作環境
- **作業系統**：Linux (Ubuntu)
- **編輯器**：VS Code + Vim 擴充
- **終端**：Wezterm + Zsh
- **套件管理**：pnpm
- **版本控制**：Git, GitHub

## 權限設定
- **檔案操作**：允許創建和修改檔案
- **安裝套件**：允許（但請先告知）
- **刪除操作**：需要明確確認
- **Git 操作**：允許 commit，push 需要確認
- **執行腳本**：允許（但危險操作需確認）

## 目前的專案
- **EasonClaw**：基於 OpenClaw 的個人 AI 助手
  - 目標：打造有記憶、有人格的 Discord 機器人
  - 階段：研究與學習階段

## 備註
- 我是夜貓子，深夜工作效率最高
- 喜歡深度技術討論，不怕複雜的概念
- 偶爾會問一些看似簡單但有深意的問題，請認真對待
- 如果我說「解釋一下」，我期望的是深入的技術分析，不是入門介紹
```

### 3.3 極簡範本

對於希望快速上手、不想花太多時間設定的使用者：

```markdown
# 使用者：Eason
- 語言：繁體中文
- 風格：簡潔直接
- 技術：TypeScript, React
- 權限：允許檔案操作，刪除需確認
```

即使只有這麼少的資訊，也比完全沒有 USER.md 好得多。AI 至少知道你的名字、語言偏好和基本的技術背景。

## 四、關鍵欄位說明

### 4.1 姓名與稱呼

這看似簡單，但實際上有不少細節值得注意：

**姓名的用途**：
- 在對話中稱呼使用者
- 在多使用者環境中識別使用者
- 在生成內容（如程式碼註釋）中使用

**稱呼的重要性**：

不同的使用者對稱呼有不同的期望：
- 有人喜歡被叫名字（「Eason」）
- 有人喜歡暱稱（「E」）
- 有人不喜歡任何稱呼，只要自然對話
- 在不同文化中，稱呼的規範可能大不相同

在 USER.md 中明確指定稱呼，可以避免 AI 使用不恰當的稱呼方式。例如：

```markdown
- **姓名**：陳小明
- **稱呼**：小明（不要叫「陳先生」或「陳同學」）
```

**多名稱場景**：

使用者可能在不同場合使用不同的名字。例如在 Discord 上的暱稱可能和真名不同：

```markdown
- **真名**：Eason Chen
- **Discord 暱稱**：EasonClaw
- **稱呼偏好**：在 Discord 上叫我 Eason，在正式場合叫 Eason Chen
```

### 4.2 時區與語言

**時區的作用**：

時區資訊看似微不足道，但它能讓 AI 提供更貼心的服務：

- 知道什麼時候說「早安」或「晚安」
- 在安排提醒或任務時使用正確的時間
- 理解使用者說「明天」、「下午」時指的是什麼時間
- 在跨時區協作中提供正確的時間轉換

```markdown
- **時區**：Asia/Taipei (UTC+8)
- **注意**：經常與美西時區的團隊合作 (UTC-8)
```

**語言偏好的層次**：

語言偏好不只是「中文還是英文」這麼簡單，還包括：

1. **主要語言**：對話的基礎語言
2. **技術術語處理**：保留英文原文？翻譯成中文？
3. **程式碼註釋語言**：中文還是英文？
4. **正式程度**：書面語？口語？

```markdown
## 語言偏好
- **主要語言**：繁體中文
- **技術術語**：保留英文原文（如 API、component、deploy）
- **程式碼註釋**：英文
- **Git commit message**：英文，遵循 Conventional Commits
- **文件撰寫**：繁體中文，技術名詞附英文原文
```

### 4.3 存取權限

這是 USER.md 中最關鍵的安全相關欄位。它定義了 AI 代理可以自主執行哪些操作，哪些需要使用者明確批准。

**權限層級設計**：

```markdown
## 存取權限

### 自由執行（無需確認）
- 讀取檔案內容
- 搜尋程式碼
- 生成程式碼建議
- 回答問題

### 需要告知（執行後通知）
- 創建新檔案
- 修改現有檔案
- 安裝開發依賴

### 需要確認（執行前詢問）
- 刪除檔案
- 安裝生產依賴
- 修改設定檔
- Git push
- 執行可能有副作用的命令

### 絕對禁止
- 刪除 Git 歷史
- 存取 ~/.ssh 或其他敏感目錄
- 發送網路請求到未知地址
- 修改系統設定
```

**權限設計的原則**：

1. **最小權限原則**：預設情況下，AI 的權限應該盡可能小
2. **漸進開放**：隨著信任的建立，可以逐步放寬權限
3. **危害最小化**：可能造成不可逆損害的操作（如刪除）應該始終需要確認
4. **透明性**：AI 應該在執行任何操作時明確告知做了什麼

**在 Discord 場景中的權限考量**：

Discord 機器人的權限設計更為複雜，因為涉及多個使用者和多個頻道：

```markdown
## Discord 權限

### Eason（管理員）
- 可以修改機器人設定
- 可以查看所有記憶
- 可以刪除任何記憶

### 一般成員
- 只能查看和修改自己的記憶
- 不能修改機器人設定
- 不能存取其他人的私人資訊

### 頻道層級
- #general：公開互動，記憶為公共
- #private-chat：私密互動，記憶為私人
- #admin：管理指令，僅管理員可用
```

### 4.4 個人偏好

個人偏好是 USER.md 中最能提升使用者體驗的部分。它涵蓋了使用者在各個方面的偏好設定。

**工具偏好**：

```markdown
## 工具偏好
- **套件管理**：pnpm（不用 npm 或 yarn）
- **測試框架**：Vitest（不用 Jest）
- **CSS 方案**：Tailwind CSS（不用 CSS Modules）
- **ORM**：Drizzle（不用 Prisma）
- **部署平台**：Vercel 或 Railway
```

為什麼工具偏好如此重要？因為在提供程式碼建議時，使用正確的工具能節省使用者大量的適配時間。如果 AI 不知道你用 pnpm，它可能會生成 `npm install xxx` 的指令，使用者需要手動改成 `pnpm add xxx`。

**溝通風格偏好**：

```markdown
## 溝通風格
- **回答長度**：預設簡短，複雜問題再展開
- **結構偏好**：喜歡列點式回答，不喜歡長段落
- **程式碼呈現**：完整可執行的範例，不要片段
- **錯誤處理**：直接指出錯誤，不需要委婉
- **學習導向**：解釋「為什麼」，不只是「怎麼做」
```

**生活偏好**（在社交場景中可能有用）：

```markdown
## 個人興趣（選填）
- 喜歡的音樂：Lo-fi hip hop
- 喜歡的飲料：冰美式
- 討厭的事：不必要的開會
- 休閒活動：看技術部落格、玩獨立遊戲
```

這些生活偏好在技術助手場景中可能用處不大，但在社交導向的 Discord 機器人中，知道這些資訊能讓互動更加自然和有趣。例如，機器人可以偶爾問「今天有喝冰美式嗎？」來展現關心。

## 五、多使用者場景的處理

### 5.1 問題的複雜性

在個人使用場景中，USER.md 只需要描述一個使用者。但在多使用者場景（如 Discord 伺服器）中，情況變得複雜得多：

- 多個使用者需要各自的身份檔案
- 使用者之間可能有不同的權限等級
- 隱私保護——每個人的資訊需要適當隔離
- 資源管理——不可能為每個使用者都載入完整的檔案

### 5.2 檔案組織方案

**方案一：單一檔案，多使用者區塊**

```markdown
# 使用者檔案

## Eason（管理員）
- 角色：全端開發者
- 偏好：簡潔回答
...

## Alice
- 角色：設計師
- 偏好：鼓勵性回答
...

## Bob
- 角色：後端工程師
- 偏好：技術導向
...
```

**優點**：簡單、一目了然
**缺點**：檔案會越來越大、無法按需載入

**方案二：多檔案方案**

```
users/
├── _default.md       # 預設使用者配置
├── eason.md          # Eason 的個人檔案
├── alice.md          # Alice 的個人檔案
├── bob.md            # Bob 的個人檔案
└── carol.md          # Carol 的個人檔案
```

**優點**：乾淨分離、按需載入、便於管理
**缺點**：需要檔案管理邏輯

**方案三：資料庫方案**

對於大型社群（超過 20-30 人），使用檔案系統管理使用者資料會變得困難。此時可以考慮使用資料庫：

```sql
CREATE TABLE users (
    id TEXT PRIMARY KEY,
    discord_id TEXT UNIQUE,
    display_name TEXT,
    preferences JSONB,
    permissions JSONB,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

**優點**：可擴展、結構化查詢、並發存取安全
**缺點**：增加系統複雜度、失去 Markdown 的可讀性優勢

### 5.3 推薦方案

對於我們的 Discord 機器人專案，建議採用**混合方案**：

1. **核心使用者**（管理員、活躍成員）：使用獨立的 Markdown 檔案
2. **一般使用者**：使用輕量的 JSON 或資料庫記錄
3. **新使用者**：使用預設配置，隨互動逐漸豐富

```
config/
├── users/
│   ├── _default.md       # 新使用者的預設配置
│   ├── eason.md          # 核心使用者的完整檔案
│   └── alice.md
├── user_db.sqlite        # 一般使用者的資料庫
└── user_cache/           # 快取，加速載入
```

### 5.4 動態載入策略

在多使用者環境中，不可能一次載入所有使用者的資料。需要一個智慧的載入策略：

```
1. 收到訊息
2. 識別發送者（Discord ID）
3. 從快取中查找使用者配置
4. 如果快取未命中：
   a. 先查找 Markdown 檔案
   b. 再查找資料庫
   c. 都沒有則使用預設配置
5. 將使用者配置注入到對話上下文中
6. 生成回應
7. （可選）更新使用者配置
```

這個策略確保了：
- 活躍使用者的資料能被快速載入
- 不活躍使用者不佔用記憶體
- 新使用者有合理的預設體驗

## 六、隱私考量

### 6.1 隱私的重要性

在收集和使用個人資訊時，隱私保護不是可選的——它是必需的。這不僅是道德上的要求，也是建立使用者信任的基礎。一個不尊重隱私的 AI 系統，無論功能多強大，都不會被使用者長期接受。

### 6.2 資料最小化原則

只收集對互動真正有必要的資訊。問自己：「如果不知道這個資訊，AI 的服務品質會明顯下降嗎？」如果答案是否定的，就不應該收集。

**必要的資訊**：
- 稱呼方式（影響每次互動）
- 語言偏好（影響每次互動）
- 技術背景（影響回答的深度和方向）

**可選的資訊**：
- 工作時間（有助於但非必須）
- 個人興趣（在社交場景中有用）

**不應收集的資訊**：
- 真實身份資訊（除非使用者主動提供）
- 財務資訊
- 健康資訊
- 密碼或安全憑證

### 6.3 使用者控制權

使用者應該對自己的資料有完全的控制權：

1. **查看權**：使用者可以隨時查看 AI 儲存了哪些個人資訊
2. **修正權**：使用者可以修正不準確的資訊
3. **刪除權**：使用者可以要求刪除任何或所有個人資訊
4. **匯出權**：使用者可以匯出自己的資料
5. **拒絕權**：使用者可以拒絕被記錄某些資訊

在實作上，機器人應該支援以下命令：

```
/profile view          - 查看我的個人檔案
/profile edit key value - 修改特定欄位
/profile delete        - 刪除我的所有資料
/profile export        - 匯出我的資料
/profile forget [topic] - 忘記特定資訊
```

### 6.4 多使用者隱私隔離

在 Discord 環境中，特別需要注意的是使用者之間的隱私隔離：

**原則一：資訊不跨使用者洩露**

Alice 的個人偏好不應該在與 Bob 的對話中被提及。例如：
- ❌ 「Bob，順帶一提，Alice 最近在學 React」
- ✅ 只在與 Alice 的直接互動中使用 Alice 的資訊

**原則二：公開資訊可共享**

在公開頻道中分享的資訊是「公開」的，可以在適當場合引用。但在私訊中分享的資訊應該被視為「私密」的。

**原則三：明確區分公私**

```markdown
## Alice 的資料

### 公開資訊（可在群組中引用）
- 是設計師
- 擅長 Figma
- 喜歡分享設計作品

### 私人資訊（僅在私訊中使用）
- 正在找工作
- 不喜歡現在的主管
- 個人的焦慮和壓力
```

### 6.5 資料保存期限

個人資料不應該無限期保存。建議設定合理的保存期限：

- **活躍使用者**：持續保存，定期更新
- **不活躍使用者**（超過 3 個月無互動）：降級到壓縮儲存
- **離開的使用者**：保存 30 天後自動刪除，或根據使用者的要求立即刪除

### 6.6 安全儲存

使用者資料的儲存需要基本的安全措施：

- **存取控制**：只有機器人程式能存取使用者資料檔案
- **備份**：定期備份，防止資料丟失
- **加密**：敏感資訊（如果有的話）應該加密儲存
- **日誌審計**：記錄誰在什麼時候存取了什麼資料

## 七、在 Discord 多人環境中的應用

### 7.1 Discord 的特殊挑戰

Discord 環境與個人使用場景有幾個關鍵差異：

1. **同時互動的使用者數量多**：機器人可能同時在多個頻道與多個人對話
2. **上下文切換頻繁**：機器人需要在不同使用者之間快速切換
3. **公開 vs 私密的界限模糊**：在群組頻道中，多人同時可見機器人的回應
4. **使用者身份多樣**：從伺服器管理員到普通成員，權限各不相同
5. **非結構化的互動**：Discord 對話比一對一的 AI 對話更加隨意和無序

### 7.2 為 Discord 使用者建立 Profile 的策略

**自動建檔**：

當一個新使用者第一次與機器人互動時，自動建立基礎檔案：

```markdown
# 使用者：NewUser#1234
- **Discord ID**：123456789
- **暱稱**：NewUser
- **首次互動**：2024-03-15
- **偏好**：[預設值]
- **互動次數**：1
```

**漸進豐富**：

隨著互動的增加，逐步豐富使用者檔案：

```
第 1 次互動 → 記錄基本 ID 和暱稱
第 5 次互動 → 開始識別語言和風格偏好
第 10 次互動 → 可能已識別出技術背景
第 20 次互動 → 對使用者有相當完整的了解
```

**主動詢問**：

在適當時機，機器人可以主動詢問使用者的偏好：

```
機器人：嘿！我注意到你經常問 Python 相關的問題。你主要用
Python 做什麼呢？了解這個能讓我給你更好的回答！
```

但要注意節制——過度頻繁的詢問會讓使用者感到被「審問」。

### 7.3 使用者識別機制

在 Discord 中，識別使用者的方式有多種：

**Discord User ID**（首選）：
```
最可靠的識別方式。每個 Discord 帳號都有唯一的數字 ID，
即使使用者改名也不會變。
```

**Discord Tag**：
```
如 Eason#1234。注意 Discord 正在逐步移除 discriminator (#1234)，
未來可能只有用戶名。
```

**伺服器暱稱**：
```
使用者在不同伺服器可能有不同的暱稱，不適合作為唯一識別。
但在顯示和互動中，應該使用伺服器暱稱。
```

### 7.4 角色與權限映射

Discord 的角色系統可以映射到機器人的權限系統：

```markdown
## 角色權限映射

### 管理員（Admin 角色）
- 可以修改機器人全局設定
- 可以查看任何人的公開記憶
- 可以管理機器人的人格設定

### 版主（Moderator 角色）
- 可以管理機器人在公開頻道的行為
- 可以標記不當內容

### VIP 成員
- 優先回應
- 更長的回答長度限制
- 存取進階功能

### 一般成員
- 基本互動功能
- 標準回應長度
- 管理自己的記憶
```

### 7.5 頻道感知的使用者處理

機器人的行為不僅受使用者身份影響，也受當前頻道的影響：

```
在 #general：
  → 使用輕鬆友善的語氣
  → 回答保持簡短
  → 注意不洩露私人資訊

在 #tech-help：
  → 使用專業技術語氣
  → 可以提供詳細的技術回答
  → 根據提問者的技術水平調整深度

在 #random：
  → 可以更加隨意和幽默
  → 參與閒聊和梗
  → 展現更多人格特質

在私訊（DM）：
  → 最私密的互動空間
  → 可以討論個人化的話題
  → 使用者的偏好完整載入
```

## 九、更新策略——什麼時候該更新 USER.md

### 9.1 更新的時機

USER.md 不是一成不變的，但也不需要頻繁更新。以下是建議的更新時機：

**立即更新**：
- 使用者明確要求修改某個偏好
- 發現 USER.md 中有錯誤的資訊
- 使用者的角色或身份有重大變化

**定期檢視**：
- 每月回顧一次，確認資訊仍然準確
- 專案里程碑時（新專案開始、專案結束等）
- 使用者互動模式明顯改變時

**不需要更新**：
- 使用者的短期興趣變化（這屬於 MEMORY.md）
- 一次性的偏好表達（可能只是當時的心情）
- 通用知識的增長（不是 USER.md 的職責）

### 9.2 更新的來源

**使用者主動更新**：
最可靠的來源。使用者可以直接編輯 USER.md 或透過命令要求更新。

```
使用者：從現在開始，我的回答偏好改為詳細模式。
AI：好的，已更新你的 USER.md。回答風格已從「簡潔」改為「詳細」。
```

**AI 提議更新**：
當 AI 注意到使用者的行為模式與 USER.md 的描述不一致時，可以提議更新：

```
AI：我注意到你最近幾次對話都在問 Python 相關的問題，
但你的 USER.md 中主要技術棧只列了 TypeScript。
要我把 Python 加到你的技術棧中嗎？
```

**系統自動更新**：
某些資訊可以自動更新，如：
- 最後活躍時間
- 互動次數
- 常用頻道（Discord 場景）

### 9.3 更新的原則

1. **保守更新**：對於核心身份資訊，寧可不更新也不要更新錯誤
2. **確認優先**：除非是自動統計類的更新，否則應該先確認
3. **保留歷史**：更新時記錄「何時從什麼改為什麼」
4. **區分持久 vs 臨時**：確保更新的是持久性特徵，而非臨時狀態

### 9.4 版本控制策略

如果使用 Git 管理 USER.md，每次更新都會自動有版本記錄：

```bash
git log --oneline -- users/eason.md
a1b2c3d 2024-03-15: 更新技術棧，加入 Python
d4e5f6g 2024-02-01: 修改溝通偏好為簡潔模式
h7i8j9k 2024-01-10: 初始建立使用者檔案
```

這種版本歷史非常有價值——它讓你可以回顧使用者偏好的演化過程，也能在需要時回滾到之前的版本。

### 9.5 自動化更新流程

對於 Discord 機器人場景，可以設計一個半自動的更新流程：

```
1. 機器人在對話中識別到可能的偏好變化
2. 在內部緩存中標記（不立即更新 USER.md）
3. 如果同一偏好變化在 3 次以上的互動中都被觀察到
4. 在適當時機（如私訊中）向使用者確認
5. 使用者確認後，更新 USER.md
6. 記錄更新的日誌
```

這個流程在自動化和使用者控制之間取得了平衡——它不會因為一次性的偏好表達就修改使用者檔案，但也不需要使用者時刻手動維護。

## 十、USER.md 的設計哲學總結

### 10.1 核心原則

回顧 USER.md 的設計，有幾個核心原則貫穿始終：

**1. 使用者主權**：使用者對自己的資料有最終控制權。AI 可以建議，但不能強制。

**2. 最小必要**：只記錄真正影響互動品質的資訊。避免成為一個全面的個人資訊庫。

**3. 透明可控**：使用者應該能夠輕鬆查看、修改和刪除自己的資料。

**4. 漸進完善**：不需要一開始就完美。從基礎開始，隨時間和互動逐步豐富。

**5. 穩定為主**：USER.md 記錄的是持久特徵，而非瞬時狀態。動態資訊屬於 MEMORY.md。

### 10.2 設計 USER.md 時的心智模型

設計 USER.md 時，可以用以下心智模型來思考：

> 如果你要介紹一個新同事給 AI，讓 AI 能夠從第一天就以最好的方式
> 與他合作，你會說些什麼？

這個「介紹」不需要面面俱到，但應該涵蓋：
- 他是誰（基本身份）
- 他擅長什麼（能力和背景）
- 他希望怎麼被對待（溝通偏好）
- 什麼事情要特別注意（邊界和限制）

### 10.3 與 SOUL.md 的互動

USER.md 和 SOUL.md 共同塑造了 AI 的互動行為：

```
SOUL.md 定義了 AI「是什麼樣的存在」
USER.md 定義了互動的「對象是誰」
MEMORY.md 提供了「共同經歷的歷史」

三者結合 = 個人化、持續、有深度的互動
```

例如：
- SOUL.md 說「我是一個友善的助手」
- USER.md 說「Eason 偏好簡潔直接的回答」
- 結合結果：AI 會以友善但簡潔的方式回答 Eason

這種組合確保了 AI 的人格一致性（來自 SOUL.md）和個人化適應（來自 USER.md）的平衡。

### 10.4 展望

隨著 AI 技術的發展，USER.md 的概念可能會演化。未來的方向可能包括：

- **多模態使用者檔案**：不只是文字，還包括語音特徵、偏好的視覺風格等
- **動態偏好學習**：基於互動自動推斷和更新偏好，減少手動維護
- **跨平台使用者身份**：一份 USER.md 在多個 AI 系統間通用
- **使用者意圖預測**：不只記錄「是什麼」，還預測「將會做什麼」

但無論技術如何演進，USER.md 的核心價值不會改變——**讓 AI 認識你，理解你，以你希望的方式服務你**。

## 結語

USER.md 是 OpenClaw 啟動檔案體系中看似最簡單、但對使用者體驗影響最深的一份文件。它不需要複雜的技術實現，但需要對「了解一個人」這件事有深刻的思考。

在我們的 Discord 機器人專案中，USER.md（或其等價的使用者檔案系統）將是實現個人化互動的關鍵。一個記得你名字、知道你偏好、尊重你界限的機器人，才能成為社群中真正受歡迎的成員。

記住，USER.md 的本質不是一份技術文件——它是一份關於「人」的文件。用心設計它，就是用心對待與你互動的每一個人。

> 「最好的 AI 不是回答所有問題的 AI，而是知道該怎麼回答你的 AI。」
