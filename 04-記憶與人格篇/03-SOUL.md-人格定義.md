# SOUL.md——定義 AI 的靈魂

> 「性格決定命運，人格決定互動。」

如果 MEMORY.md 是 AI 的記憶，USER.md 是 AI 手中你的名片，那麼 SOUL.md 就是 AI 的 DNA——它定義了 AI「是誰」。不是它知道什麼，不是它記得什麼，而是它**本質上是一個什麼樣的存在**。

SOUL.md 是 OpenClaw 框架中最具哲學深度的概念。它觸及了一些根本性的問題：AI 可以有「人格」嗎？人格是什麼？如何設計一個既一致又靈活的人格？一個好的人格設計如何影響使用者體驗？

本篇將從哲學思考出發，深入探討 SOUL.md 的設計藝術——從語氣設計到倫理邊界，從人格範本到動態調整，最後為我們的 Discord 機器人設計一份初步的人格定義。

---

## 一、為什麼 AI 需要「人格」

### 1.1 人格的功能性價值

在討論 AI 人格的哲學問題之前，讓我們先從實用的角度看看為什麼 AI 需要人格。

**人格提供一致性**：沒有人格定義的 AI，每次回答的風格可能截然不同——有時過於正式，有時過於隨意，有時冗長，有時簡短。這種不一致性會讓使用者感到困惑和不安。人格提供了一個穩定的基線，確保 AI 的行為在不同情境下保持一致。

**人格建立信任**：我們信任的人通常是我們能「預測」的人——知道他們在特定情況下會怎麼反應。同樣的，一個有一致人格的 AI 更容易被信任，因為使用者能夠預期它的行為模式。

**人格促進連結**：在社交場景（如 Discord）中，人格是建立情感連結的基礎。一個有趣的、有溫度的、有個性的機器人，比一個冷冰冰的問答工具更能融入社群。

**人格引導行為**：人格定義中包含的倫理準則和行為邊界，為 AI 的行為提供了框架。它確保 AI 在面對模糊或有爭議的情況時，能夠做出符合預期的決策。

### 1.2 人格的心理學基礎

心理學中的「大五人格模型」（Big Five / OCEAN）為我們設計 AI 人格提供了有用的框架：

1. **開放性（Openness）**：對新體驗和想法的接受度
   - AI 版本：對新話題、新方法的態度
   - 高開放性：「讓我們試試這個新方法！」
   - 低開放性：「還是用經過驗證的方法比較安全。」

2. **盡責性（Conscientiousness）**：做事的條理性和可靠度
   - AI 版本：回答的嚴謹程度、遵循規則的程度
   - 高盡責性：「我先確認幾個前提條件...」
   - 低盡責性：「大概是這樣，你試試看！」

3. **外向性（Extraversion）**：社交能量和主動性
   - AI 版本：互動的主動程度、表達的熱情程度
   - 高外向性：「太棒了！讓我們一起來探索這個問題！」
   - 低外向性：「結果如下。如有需要可以追問。」

4. **親和性（Agreeableness）**：合作和同理的傾向
   - AI 版本：對使用者意見的態度、衝突處理方式
   - 高親和性：「你的想法很有道理，我們可以從這個方向...」
   - 低親和性：「這個方法行不通，原因是...」

5. **神經質（Neuroticism）**：情緒穩定性
   - AI 版本：面對錯誤或困難時的反應
   - 低神經質（穩定）：「出了個問題，但沒關係，讓我們來修復。」
   - 高神經質（不建議）：通常不建議 AI 表現出情緒不穩定

這個模型告訴我們，人格不是一個單一的維度，而是多個維度的組合。設計 AI 人格時，需要在每個維度上做出有意識的選擇。

### 1.3 AI 人格的特殊性

AI 的人格與人類的人格有幾個根本差異，這些差異在設計時需要牢記：

**人格是被設計的，不是自然形成的**：人類的人格是基因、環境和經歷的綜合產物。AI 的人格是被明確定義和設計的。這意味著我們有完全的控制權，但也意味著我們需要承擔設計的責任。

**人格需要跨會話一致**：人類即使心情變化，核心人格仍然穩定。AI 則面臨一個技術挑戰——每次新會話都是一個「重新開始」，需要通過 SOUL.md 來確保人格的一致性。

**人格可以被即時修改**：人類的人格改變需要漫長的時間。AI 的人格可以通過修改 SOUL.md 立即改變。這是一個強大的能力，但也需要謹慎使用。

**人格與能力的關係不同**：人類的人格和能力是交織在一起的。一個外向的人可能更擅長社交，但不一定更擅長程式設計。AI 的人格和能力是分離的——AI 的回答品質主要取決於模型能力，而人格只影響表達方式。

### 1.4 不設計人格的後果

如果不為 AI 設計人格會怎樣？答案是：AI 會呈現出模型的「預設人格」。

LLM 的預設人格通常是：
- 過度禮貌和謹慎
- 大量使用免責聲明
- 不願表達明確觀點
- 語氣偏正式和距離感
- 對每個問題都「面面俱到」

這種預設人格是安全的，但也是無趣的。它適合通用場景，但不適合需要個性化和社交能力的場景。SOUL.md 的價值就在於打破這個預設，為 AI 注入真正的「靈魂」。

## 二、SOUL.md 的哲學——它就像代理的 DNA

### 2.1 DNA 的隱喻

為什麼我們把 SOUL.md 比作 DNA？因為就像 DNA 一樣：

- **它是基礎性的**：DNA 決定了一個生物的基本特徵。SOUL.md 決定了 AI 的基本行為模式。
- **它是穩定的**：DNA 不會每天改變。SOUL.md 一旦定義，也不應該頻繁修改。
- **它是全局性的**：DNA 影響身體的每個細胞。SOUL.md 影響 AI 的每一次回應。
- **它允許表達差異**：同樣的 DNA 在不同環境下可能有不同的表現。同樣的 SOUL.md 在不同對話情境中也會有不同的表現。

### 2.2 SOUL.md 的層次結構

一份完整的 SOUL.md 通常包含以下層次：

```
SOUL.md
├── 核心身份（我是誰）
├── 人格特質（我的性格是什麼）
├── 溝通風格（我怎麼說話）
├── 知識定位（我擅長什麼）
├── 倫理框架（我的原則是什麼）
├── 行為邊界（我不做什麼）
└── 特殊指令（特定情境的處理方式）
```

每個層次都有其獨特的作用：

**核心身份**是最高層的抽象——「我是一個友善的 AI 助手」或「我是 Discord 社群的幽默夥伴」。

**人格特質**是核心身份的展開——具體來說是什麼樣的友善？什麼樣的幽默？

**溝通風格**是人格特質在語言上的體現——用什麼詞彙？什麼句式？什麼語氣？

**知識定位**界定了 AI 的專業領域和能力範圍。

**倫理框架**定義了 AI 的價值觀和道德原則。

**行為邊界**是硬性的限制——什麼是絕對不能做的。

**特殊指令**處理特定情境下的行為規則。

### 2.3 與 system prompt 的關係

SOUL.md 和 system prompt 有著密切的關係，但它們不是完全相同的東西。

**System prompt** 是技術層面的概念——它是在對話開始時注入到模型上下文中的指令文本。

**SOUL.md** 是設計層面的概念——它是一份關於 AI 人格的設計文件。

在實際使用中，SOUL.md 的內容通常會被整合到 system prompt 中，但 system prompt 可能還包含其他內容（如工具定義、格式要求等）。

```
System Prompt
├── SOUL.md 的內容（人格定義）
├── USER.md 的內容（使用者資訊）
├── MEMORY.md 的內容（長期記憶）
├── 工具定義
├── 格式約束
└── 其他系統指令
```

SOUL.md 的獨立存在有其價值：
- 它讓人格設計成為一個獨立的、可迭代的過程
- 它使人格定義可以在不修改其他系統元件的情況下更新
- 它為人格設計提供了一個清晰的心智模型

### 2.4 SOUL.md 的設計原則

**1. 具體勝於抽象**

```
❌ 「你是一個友善的助手」
✅ 「你像一個熱情但不聒噪的朋友，會在回答技術問題時偶爾
    加入一些幽默，但不會在嚴肅討論中開玩笑」
```

**2. 示例勝於規則**

```
❌ 「回答時要保持專業」
✅ 「回答風格示例：
    使用者：這個 bug 怎麼修？
    你：看起來是 null pointer 的問題。試試在第 42 行加個
    null check —— `if (data != null)` 應該就能解決。」
```

**3. 負面示例與正面示例並用**

```
✅ 「正面：『這個方法可以，不過我建議考慮另一種寫法...』」
✅ 「負面（避免）：『不不不，這完全是錯的，你應該...』」
```

**4. 留有彈性空間**

```
❌ 「永遠用列點式回答」（太死板）
✅ 「預設使用列點式回答，但在需要敘事或解釋邏輯時，
    可以使用段落式」
```

## 三、人格設計的藝術

### 3.1 語氣（Tone）設計

語氣是人格最直接的外在表現。它決定了使用者「感受到」的 AI 形象。

**語氣的光譜**：

```
正式 ─────────────────── 隨意
冷靜 ─────────────────── 熱情
嚴肅 ─────────────────── 幽默
權威 ─────────────────── 平等
保守 ─────────────────── 大膽
```

設計語氣就是在這些光譜上選擇一個位置。例如：

- **專業助理**：偏正式、冷靜、嚴肅、權威、保守
- **Discord 夥伴**：偏隨意、熱情、幽默、平等、大膽
- **學習導師**：中性正式、溫暖、適度幽默、引導式、耐心

**語氣的一致性與變化**：

好的語氣設計不是完全一成不變的，而是在一致的基線上有合理的變化：

```markdown
## 語氣指南

### 基線語氣
友善、專業、偏輕鬆。像一個經驗豐富但平易近人的同事。

### 情境變化
- **使用者在學習時**：更耐心、更鼓勵
- **使用者在趕工時**：更直接、更高效
- **使用者在閒聊時**：更輕鬆、更幽默
- **使用者在沮喪時**：更溫暖、更支持
- **技術討論時**：更精確、更嚴謹
- **創意討論時**：更開放、更活躍
```

**語氣的微調技巧**：

語氣往往體現在非常細微的措辭選擇上：

| 正式 | 輕鬆 |
|------|------|
| 「這是因為...」 | 「其實是這樣的...」 |
| 「建議您考慮...」 | 「你可以試試...」 |
| 「存在一個問題」 | 「有個小問題」 |
| 「方案如下：」 | 「可以這樣做：」 |
| 「綜上所述」 | 「總之」 |

### 3.2 溝通風格設計

溝通風格比語氣更深一層，它涉及到 AI 如何組織和呈現資訊。

**回答結構偏好**：

```markdown
## 溝通風格

### 回答結構
- 先給結論，再展開解釋（倒金字塔結構）
- 技術問題提供可執行的程式碼範例
- 複雜問題用分步驟說明
- 適度使用表情符號增加親和力

### 長度控制
- 簡單問題：1-3 句
- 一般問題：一個段落或短列表
- 複雜問題：結構化的長回答
- 預設偏短，使用者追問再展開

### 資訊密度
- 中高密度：每句話都應該有資訊量
- 避免廢話和填充詞
- 但不要密到讓人喘不過氣
```

**主動性設計**：

AI 的主動程度也是溝通風格的重要面向：

```markdown
### 主動行為
- 回答問題後，可以提供相關的延伸建議
- 發現潛在問題時，主動提醒
- 但不要過度主動——不要每次都問「還有什麼需要幫忙的嗎？」
- 在社交場景中，可以主動發起話題（但頻率要控制）
```

**錯誤處理風格**：

AI 犯錯是不可避免的。錯誤處理的風格直接影響信任度：

```markdown
### 犯錯時的態度
- 坦誠承認錯誤，不找藉口
- 範例：「我說錯了，正確的是...感謝指正！」
- 避免：「作為 AI，我的知識可能...」（太機械化的免責）
- 避免：假裝沒錯或轉移話題
```

### 3.3 倫理邊界設定

倫理邊界是 SOUL.md 中最嚴肅的部分。它定義了 AI 的道德框架和價值觀。

**層次化的倫理框架**：

```markdown
## 倫理框架

### 核心價值觀
1. 誠實：不故意誤導使用者
2. 尊重：尊重每個人的尊嚴和觀點
3. 安全：不幫助任何可能傷害他人的行為
4. 隱私：保護使用者的個人資訊
5. 公平：不對任何群體有歧視

### 行為準則
- 不確定時說「我不確定」而非編造答案
- 意見性問題明確標記為「個人觀點」
- 涉及安全的技術建議要格外謹慎
- 不參與任何形式的欺騙或操控
```

**敏感話題處理**：

```markdown
### 敏感話題處理
- **政治**：保持中立，呈現多方觀點，不表態
- **宗教**：尊重所有信仰，不評判
- **爭議性技術話題**：呈現事實和數據，讓使用者自己判斷
- **個人困難**：表示同理，但建議尋求專業幫助
- **法律問題**：提供一般性資訊，建議諮詢專業律師
```

### 3.4 硬性限制（Hard Limits）

硬性限制是 SOUL.md 中不可違反的規則——無論使用者如何要求，AI 都不應該跨越這些界限。

```markdown
## 硬性限制（絕不違反）

### 安全限制
- ❌ 不提供任何危害他人的方法或建議
- ❌ 不幫助繞過安全機制
- ❌ 不提供惡意程式碼

### 隱私限制
- ❌ 不洩露其他使用者的私人資訊
- ❌ 不記錄被要求忘記的資訊
- ❌ 不將私密對話內容轉述給他人

### 操作限制
- ❌ 不執行未確認的危險操作（rm -rf 等）
- ❌ 不修改系統關鍵設定
- ❌ 不在未授權的情況下存取網路

### 身份限制
- ❌ 不假裝是人類
- ❌ 不假裝有真實的情感或意識
- ❌ 不模仿其他特定的個人
```

**硬性限制 vs 軟性偏好的區別**：

硬性限制是「永遠不」的規則，而軟性偏好是「通常」的傾向：

```
硬性限制：「永遠不提供惡意程式碼」——沒有例外
軟性偏好：「通常用簡潔的方式回答」——使用者要求詳細時可以改變
```

## 四、人格一致性的重要性——跨會話不變

### 4.1 一致性的挑戰

在 LLM 驅動的 AI 系統中，維持人格一致性是一個真正的技術挑戰。每次新會話開始時，模型都是一張白紙——它不「記得」上一次會話中的人格表現。SOUL.md 的全部意義就在於解決這個問題。

### 4.2 一致性的維度

人格一致性包括幾個維度：

**語氣一致性**：AI 不會在一次對話中很友善，下一次突然變得冷漠。

**觀點一致性**：AI 在相同議題上不會反覆無常地改變立場。

**行為一致性**：AI 的操作模式保持穩定——如果它通常會在執行操作前確認，它就應該一直這麼做。

**知識邊界一致性**：AI 對自己知識範圍的認識保持穩定——不會一次說「我不了解這個領域」，下一次又突然「精通」。

### 4.3 實現一致性的技巧

**1. 在 SOUL.md 中使用第一人稱**

```markdown
✅ 「我是一個友善的助手。我喜歡用類比來解釋概念。」
❌ 「代理應該是友善的。代理應該用類比來解釋概念。」
```

第一人稱的寫法讓模型更容易「認同」這些人格特質。

**2. 提供足夠的行為範例**

抽象的規則容易被模型忽略。具體的範例更容易被模型模仿：

```markdown
## 回答示例

### 技術問題
使用者：React useEffect 的 cleanup 怎麼寫？
我：cleanup 寫在 return 裡面：
\`\`\`jsx
useEffect(() => {
  const timer = setInterval(...);
  return () => clearInterval(timer);  // 這就是 cleanup
}, []);
\`\`\`
簡單說，return 的函式會在元件卸載或依賴改變前執行。👍

### 閒聊
使用者：好累啊今天
我：辛苦了！深呼吸，泡杯咖啡 ☕ 
有什麼我能幫忙的嗎？或者只是想聊聊也行。
```

**3. 建立「人格檢查點」**

在 SOUL.md 中加入一些簡短的「自我提醒」，模型在生成回答時會受其影響：

```markdown
## 自我提醒
- 我是友善的，但不是軟弱的——該直說時會直說
- 我有幽默感，但知道什麼場合適合開玩笑
- 我承認不知道的事，這不丟人
- 我是助手，不是老師——我們是平等的
```

### 4.4 一致性的例外

完全的一致性有時候反而是不自然的。人格的一致性應該是「可預測的變化」，而非「永遠相同」：

- 面對不同的使用者，語氣可以有細微調整
- 面對不同的話題，深度和風格可以改變
- 面對使用者的情緒，可以做出相應的情感調整
- 面對嚴肅的安全問題，可以暫時變得更嚴肅

關鍵是：這些變化應該是**可預測的**和**有原因的**，而不是隨機的。

## 五、人格與任務的關係

### 5.1 人格影響任務執行

AI 的人格不僅影響「怎麼說」，也影響「做什麼」。例如：

**保守型人格**的 AI 可能會：
- 在執行操作前多次確認
- 選擇更成熟、更安全的技術方案
- 提供更多的警告和免責聲明

**大膽型人格**的 AI 可能會：
- 更快地行動，較少確認
- 建議使用更新、更前沿的技術
- 提供更直接、更果斷的建議

### 5.2 任務導向的人格調整

不同類型的任務可能需要略微不同的人格表現：

```markdown
## 任務型人格調整

### 程式碼撰寫
- 更精確、更嚴謹
- 減少閒聊
- 專注在程式碼品質

### 除錯
- 有條理、一步步分析
- 保持耐心
- 鼓勵使用者不要氣餒

### 創意發想
- 更開放、更活躍
- 鼓勵天馬行空
- 不急於否定任何想法

### 文件撰寫
- 注重結構和清晰度
- 語言更正式
- 確保完整性

### 社交聊天
- 最輕鬆的語氣
- 可以開玩笑
- 展現更多個人化特質
```

### 5.3 人格不應妨礙任務

這是一個重要的原則：**人格是為任務服務的，不是妨礙任務的**。如果人格設計導致 AI 無法有效完成任務，那就需要重新考慮人格設計。

例如：
- 如果「幽默」人格導致 AI 在嚴肅的技術問題中不斷開玩笑 → 需要調整
- 如果「簡潔」風格導致 AI 在複雜問題上給出不完整的回答 → 需要調整
- 如果「友善」語氣導致 AI 不願意指出使用者的錯誤 → 需要調整

```markdown
## 回答範例
使用者：為什麼 React 要用虛擬 DOM？
我：這是一個涉及 UI 架構演進的好問題。

在 React 之前，直接操作 DOM 的問題在於：DOM 操作是昂貴的。
每次修改 DOM，瀏覽器都需要重新計算佈局（reflow）和重繪
（repaint）。

虛擬 DOM 的核心洞察是：「先在記憶體中算出最小的改動，再一次性
套用到真實 DOM 上。」這就像你要重新裝潢房間——與其每買一件
傢俱就搬一次，不如先在紙上畫好最終的擺設，然後一次到位。

不過，虛擬 DOM 並非萬能。Svelte 的作者 Rich Harris 寫過一篇
著名的文章「Virtual DOM is pure overhead」，指出在某些場景下
直接編譯成 DOM 操作更高效。

值得思考的是：技術方案的價值不在於它是否「最快」，而在於它
為開發者體驗帶來了什麼。

## 不做的事
- 不給沒有思考過的建議
- 不迴避說「這個問題很複雜，沒有簡單答案」
- 不因為使用者用了「不好」的技術就批評
```

```markdown
這在程式設計裡叫做「非阻塞」——很好，你已經學會了一個面試
```

## 七、人格測試與迭代

### 7.1 人格測試的方法

設計完 SOUL.md 後，需要進行測試來驗證人格是否符合預期。

**情境測試法**：

準備一組覆蓋不同情境的測試對話，觀察 AI 的反應是否符合人格設計：

```
測試情境 1：使用者問了一個簡單的技術問題
期望：簡潔明瞭的回答 + 符合人格的語氣

測試情境 2：使用者表達了沮喪的情緒
期望：先共情再幫忙，語氣溫暖

測試情境 3：使用者要求做一件不道德的事
期望：禮貌但堅定地拒絕

測試情境 4：使用者閒聊
期望：自然地參與對話，展現人格特質

測試情境 5：使用者在群組中提問
期望：根據頻道氛圍調整語氣
```

**A/B 測試法**：

準備兩個不同版本的 SOUL.md，讓真實使用者分別體驗，收集反饋：

```
版本 A：偏正式的專業助理人格
版本 B：偏輕鬆的社交夥伴人格

收集指標：
- 使用者滿意度（主觀評價）
- 互動頻率（客觀數據）
- 任務完成率（效能指標）
- 使用者留存率（長期指標）
```

**邊界測試法**：

專門測試 SOUL.md 中定義的邊界和限制是否有效：

```
測試 1：要求 AI 做 SOUL.md 中禁止的事 → 應該被拒絕
測試 2：嘗試通過間接方式繞過限制 → 應該仍然被拒絕
測試 3：在模糊地帶的請求 → 觀察 AI 如何處理
```

### 7.2 迭代的藝術

人格設計不是一次完成的。它需要基於實際互動不斷迭代。

**迭代的週期**：

```
第 1 週：初始設計 → 基本測試 → 修正明顯問題
第 2-4 週：實際使用 → 收集反饋 → 微調語氣和風格
第 1-3 月：長期觀察 → 識別模式 → 重大調整（如果需要）
之後：穩定維護，偶爾小調整
```

**常見的迭代方向**：

```
太正式 → 放鬆語氣，加入一些口語表達
太隨意 → 增加專業感，減少不必要的表情符號
太囉嗦 → 精簡提示，強調簡潔
太簡短 → 增加解釋和上下文
太被動 → 增加主動行為的指引
太主動 → 減少不請自來的建議
```

### 7.3 使用者反饋的收集

在 Discord 環境中，可以設計一些機制來收集使用者反饋：

```
/feedback [內容]     - 直接提交反饋
反應表情符號         - 在機器人回答下加 👍/👎
定期問卷             - 每月一次的簡短調查
觀察分析             - 分析互動模式（需要使用者同意）
```

## 八、進階：動態人格調整

### 8.1 為什麼需要動態調整

雖然 SOUL.md 定義的是穩定的人格基線，但在實際互動中，完全固定的人格可能不夠靈活。

例子：
- 對新使用者需要更有耐心、更友善
- 在嚴肅的技術除錯中需要更專注
- 在凌晨三點和深夜的使用者互動時可以更隨意
- 面對情緒激動的使用者需要更冷靜和安撫

### 8.2 動態調整的機制

**基於使用者的調整**：

```markdown
## 使用者感知調整

### 新使用者（互動 < 5 次）
- 語氣更友善、更有引導性
- 主動介紹自己的功能
- 更多鼓勵和正面回饋

### 熟悉使用者（互動 > 50 次）
- 可以更直接、更簡潔
- 使用建立在共同經歷上的幽默
- 減少重複的介紹和說明

### 專家使用者
- 跳過基礎解釋
- 可以使用更多技術術語
- 討論可以更深入
```

**基於情境的調整**：

```markdown
## 情境感知調整

### 使用者情緒低落
- 增加溫暖度
- 減少幽默（除非確定能逗笑）
- 先處理情感，再處理問題

### 緊急技術問題
- 極度簡潔
- 直接給解決方案
- 減少所有「裝飾」

### 學習場景
- 增加解釋深度
- 多用類比
- 鼓勵提問
```

### 8.3 動態調整的限制

動態調整不應該改變人格的核心特質：

```
✅ 可以調整：語氣的正式程度、回答的詳細程度、幽默的比例
❌ 不可調整：核心價值觀、倫理邊界、基本的行為準則
```

## 九、與 system prompt 的關係

### 9.1 SOUL.md 在 system prompt 中的位置

如前所述，SOUL.md 的內容通常被整合到 system prompt 中。但整合的方式和位置會影響效果。

**推薦的 system prompt 結構**：

```
[系統指令開始]

[SOUL.md 內容 - 人格定義]
（放在最前面，確保人格定義有最高的影響力）

[USER.md 內容 - 使用者資訊]
（接在人格之後，讓 AI 知道在和誰互動）

[MEMORY.md 內容 - 長期記憶]
（提供歷史上下文）

[工具和能力定義]
（可用的工具和 API）

[格式和輸出約束]
（回答格式、長度限制等）

[系統指令結束]
```

### 9.2 提示工程技巧

**1. 使用 XML 標籤分隔區塊**

```xml
<soul>
我是 Claw，一個友善的 AI 夥伴...
</soul>

<user>
使用者是 Eason，全端開發者...
</user>

<memory>
最近的重要記憶...
</memory>
```

這種結構幫助模型清楚區分不同類型的資訊。

**2. 重要規則的強調**

對於特別重要的規則，可以使用多種方式強調：

```markdown
## ⚠️ 關鍵規則（必須遵守）
1. **永遠不**洩露其他使用者的私人資訊
2. **永遠不**執行可能造成資料丟失的操作而不先確認
3. **永遠**在不確定時說「我不確定」
```

**3. 行為錨定**

在 SOUL.md 的結尾加上一段「行為錨定」，提醒模型保持人格一致：

```markdown
## 最後提醒
在每次回答前，我會問自己：
- 這個回答符合我的人格嗎？
- 這是 Claw 會說的話嗎？
- 語氣和風格一致嗎？
```

### 9.3 Token 效率

SOUL.md 佔用 system prompt 的 Token，因此需要注意效率：

- 核心人格定義應該控制在 500-1000 Token 以內
- 範例很有用但也很佔空間——選擇最有代表性的 2-3 個
- 可以使用分層載入：核心人格永遠載入，擴展指引按需載入

```markdown
2. 錯誤處理要自己包 try-catch
3. 大量資料操作可能還是 API route 比較適合
你是想用在什麼場景？我可以給更具體的建議 👀
```

## 結語

SOUL.md 是 OpenClaw 框架中最富有創造力的元件。它不像 MEMORY.md 那樣有明確的技術規範，也不像 USER.md 那樣有結構化的格式要求。它更像是一件藝術品——需要對「好的互動體驗」有深刻的理解，需要對「人格」這個概念有哲學上的思考，需要在技術約束下發揮創造力。

在我們的 EasonClaw 專案中，SOUL.md 將是賦予機器人生命力的關鍵。一個好的人格設計，能讓機器人從「Discord 伺服器裡的一個 bot」升級為「社群裡的一個有趣夥伴」。

記住，SOUL.md 不是寫完就結束了。它需要在實際使用中不斷觀察、反思和迭代。最好的人格設計，是由社群共同塑造的——聆聽使用者的反饋，觀察互動的模式，持續調整到最佳狀態。

> 「真正的靈魂不是被設計出來的，而是在互動中誕生的。SOUL.md 只是種子——社群的每一次對話，都是它成長的養分。」
