# 記憶管理最佳實踐——讓你的 AI 真正「記住」

> 「記憶的藝術不在於記住一切，而在於記住對的事情，在對的時間想起來。」

前三篇我們分別探討了 MEMORY.md（長期記憶）、USER.md（使用者身份）和 SOUL.md（人格定義）。這些是 OpenClaw 記憶與人格體系的三大支柱。但知道每個元件「是什麼」還不夠——我們需要知道如何將它們整合成一個高效、可維護、可擴展的記憶管理系統。

本篇是一份實戰指南。我們將從記憶預算規劃開始，到記憶架構設計，再到每日筆記系統、記憶清理策略、效能優化，最後針對我們的 Discord 機器人專案設計一套完整的記憶方案。

這不是理論探討——這是讓你的 AI 真正「記住」的操作手冊。

## 一、記憶預算規劃

### 1.1 啟動檔案的 Token 成本計算

在設計記憶系統時，第一個必須面對的現實問題是：**Token 是有限的資源**。每一個寫入 SOUL.md、USER.md、MEMORY.md 的字，都佔用了寶貴的上下文窗口空間。

讓我們先建立對 Token 成本的直覺：

**中文 Token 計算經驗法則**：
- 一個常用中文字 ≈ 1-2 個 Token
- 一個中文標點符號 ≈ 1 個 Token
- 一行 Markdown 格式的中文（約 30 字）≈ 40-60 個 Token
- 一個完整的記憶條目（5-8 行）≈ 100-200 個 Token

**英文 Token 計算經驗法則**：
- 一個常見英文單詞 ≈ 1-1.5 個 Token
- 程式碼通常比自然語言更「Token 高效」
- 一行程式碼 ≈ 10-30 個 Token

**各啟動檔案的建議預算**：

| 檔案 | 建議 Token 預算 | 說明 |
|------|----------------|------|
| SOUL.md | 500-1,000 | 核心人格定義，精簡但完整 |
| USER.md | 300-800 | 使用者身份，依複雜度而定 |
| MEMORY.md | 1,000-2,500 | 長期記憶，是最大的變數 |
| 每日筆記 | 200-500 | 最近幾天的摘要 |
| 工具定義 | 500-2,000 | 可用工具的描述 |
| **總計** | **2,500-7,000** | 全部啟動內容 |

### 1.2 Token 預算的計算實踐

讓我們用實際的例子來計算：

**SOUL.md 範例（精簡版）**：
```markdown
# 我是 Claw
- 友善的 Discord 機器人
- 語氣輕鬆、會用表情符號
- 技術問題認真回答
- 不洩露隱私、不嘲笑他人
```
預估 Token：約 150-200

**SOUL.md 範例（完整版）**：
```markdown
# 我是 Claw
[包含核心身份、人格特質、語氣指南、行為準則、
 回答範例、硬性限制、特殊情境處理...]
```
預估 Token：約 800-1,200

差距是顯著的。精簡版節省了大量 Token，但可能犧牲了人格的一致性和豐富度。完整版提供了更好的人格表現，但佔用了更多的上下文空間。

**如何選擇？**

這取決於你的上下文窗口大小和使用場景：

- **4k-8k 上下文**（較舊的模型）：必須使用精簡版，總啟動成本控制在 1,000 Token 以內
- **32k 上下文**：可以使用適中版本，總啟動成本 3,000-5,000 Token
- **128k+ 上下文**（如 GPT-4 Turbo、Claude 3）：可以使用完整版，但仍需注意長上下文的效能衰減

### 1.3 長期記憶 vs 上下文窗口的權衡

這是記憶系統設計中最核心的權衡：**載入更多的長期記憶 vs 保留更多的對話空間**。

**載入過多記憶的後果**：
- 對話歷史被壓縮，AI 可能「忘記」剛才說的話
- 回應生成空間被壓縮，回答品質可能下降
- 模型在「注意力」分散後，可能忽略某些記憶
- 長文本處理速度變慢，回應延遲增加

**載入過少記憶的後果**：
- AI 不了解使用者，互動感受差
- 缺乏歷史上下文，無法延續之前的話題
- 人格表現可能不一致

**最佳平衡點**：

根據經驗，啟動檔案（包括所有記憶）佔用總上下文的 3%-8% 是一個合理的範圍。以 128k Token 的上下文為例：

```
128,000 × 3% = 3,840 Token  （最小）
128,000 × 8% = 10,240 Token （最大）
推薦：5,000-7,000 Token
```

### 1.4 Token 預算分配策略

不同的使用場景需要不同的預算分配策略：

**策略一：人格優先（社交機器人）**

```
SOUL.md：40%（800 Token）
USER.md：15%（300 Token）
MEMORY.md：30%（600 Token）
其他：15%（300 Token）
總計：2,000 Token
```
適合我們的 Discord 機器人——人格表現是第一優先。

**策略二：記憶優先（個人助理）**

```
SOUL.md：15%（450 Token）
USER.md：20%（600 Token）
MEMORY.md：50%（1,500 Token）
其他：15%（450 Token）
總計：3,000 Token
```
適合長期使用的個人助理——累積的記憶是最大的價值。

**策略三：任務優先（技術助手）**

```
SOUL.md：10%（200 Token）
USER.md：10%（200 Token）
MEMORY.md：20%（400 Token）
工具定義：60%（1,200 Token）
總計：2,000 Token
```
適合專注於技術任務的助手——工具能力比人格更重要。

### 1.5 實際的預算監控

建議建立一個簡單的預算監控機制：

```markdown
## Token 預算報告

| 檔案 | 目前 Token | 預算 | 使用率 |
|------|-----------|------|--------|
| SOUL.md | 650 | 800 | 81% |
| USER.md | 280 | 400 | 70% |
| MEMORY.md | 1,120 | 1,500 | 75% |
| 每日筆記 | 180 | 300 | 60% |
| **總計** | **2,230** | **3,000** | **74%** |

⚠️ MEMORY.md 接近預算上限，建議進行記憶清理。
```

定期（例如每週）檢查這個報告，可以在記憶膨脹成為問題之前及時處理。

## 二、記憶架構設計

### 2.1 單一代理記憶策略

最簡單的記憶架構是一個代理對一個使用者的一對一模式。這是個人使用場景中最常見的模式。

**架構圖**：

```
使用者 ↔ 代理
              ├── SOUL.md（固定）
              ├── USER.md（偶爾更新）
              ├── MEMORY.md（持續增長）
              └── daily_notes/（每日更新）
```

**優勢**：
- 最簡單的實現
- 沒有隔離和權限問題
- 所有記憶都為一個使用者服務

**劣勢**：
- 不支援多使用者
- 不支援代理間的知識共享
- 記憶完全綁定在一個代理上

**適用場景**：
- 個人 AI 助手
- 個人學習夥伴
- 個人生產力工具

**實作建議**：

在單一代理模式中，記憶管理相對簡單。主要需要關注的是：
1. 記憶的增長速度控制
2. 定期的清理和整合
3. Token 預算的監控

### 2.2 多代理共享記憶

在更複雜的場景中，可能有多個代理需要共享同一套記憶。例如，一個技術助手和一個社交助手可能需要共享使用者的基本資訊。

**架構圖**：

```
                 ┌── 代理 A（技術助手）
                 │     ├── SOUL_A.md
                 │     └── 私有記憶
                 │
共享記憶層 ─────┤
│   USER.md    │
│   共享 MEMORY │
                 │
                 └── 代理 B（社交助手）
                       ├── SOUL_B.md
                       └── 私有記憶
```

**共享記憶的設計原則**：

1. **只共享穩定的資訊**：使用者的基本身份、核心偏好等穩定資訊適合共享。動態的、情境相關的記憶應該保持私有。

2. **讀多寫少**：共享記憶應該主要被讀取，寫入操作應該受到嚴格控制，以避免衝突。

3. **版本控制**：共享記憶的每次修改都應該有記錄，便於追蹤和回滾。

4. **存取控制**：不是每個代理都應該有相同的存取權限。

**實作方案**：

```
config/
├── shared/
│   ├── USER.md           # 共享的使用者資訊
│   └── SHARED_MEMORY.md  # 共享的長期記憶
├── agent_tech/
│   ├── SOUL.md           # 技術助手人格
│   └── MEMORY.md         # 技術相關的私有記憶
└── agent_social/
    ├── SOUL.md           # 社交助手人格
    └── MEMORY.md         # 社交相關的私有記憶
```

### 2.3 分層記憶體系

最完整的記憶架構是分層設計，模仿人類記憶的多層次結構。

**三層記憶模型**：

```
第一層：即時記憶（Immediate Memory）
├── 當前對話歷史
├── 最新的工具輸出
└── 生存期：會話內

第二層：工作記憶（Working Memory）
├── 啟動檔案（SOUL.md, USER.md, MEMORY.md）
├── 最近幾天的每日筆記
├── 當前任務的上下文
└── 生存期：數天到數週

第三層：長期儲存（Long-term Storage）
├── 歷史每日筆記歸檔
├── 完整的對話歷史資料庫
├── 語意搜尋索引
└── 生存期：永久
```

**層間互動**：

```
寫入路徑（Write Path）：
對話 → 即時記憶
           ↓ 每日總結
      工作記憶 → 每日筆記
           ↓ 提升機制
      長期儲存 → MEMORY.md / 資料庫

讀取路徑（Read Path）：
啟動時 → 載入第二層到即時記憶
對話中 → 需要時從第三層語意搜尋
         結果注入到即時記憶
```

**為什麼分層很重要？**

分層記憶解決了一個核心矛盾：**我們需要大量的記憶，但只能載入有限的上下文**。

通過分層，我們可以：
- 在有限的上下文窗口中保持最重要的資訊（第一、二層）
- 在需要時從海量的歷史資料中檢索（第三層）
- 自動管理記憶的生命週期（從即時到長期的提升）

## 三、每日筆記系統

### 3.1 自動日誌的價值

每日筆記（Daily Notes）是連接短期對話和長期記憶的橋樑。它記錄了每天互動的重要摘要，作為記憶系統的「原始素材」。

**為什麼需要每日筆記？**

1. **緩衝層**：不是所有對話中的資訊都值得寫入長期記憶。每日筆記作為一個緩衝層，先記錄「可能重要」的資訊，經過一段時間的沉澱後，再決定是否提升到長期記憶。

2. **短期回顧**：在連續幾天的工作中，AI 可以回顧昨天的筆記來恢復上下文，而不需要重新載入完整的對話歷史。

3. **趨勢識別**：通過分析一段時間的每日筆記，可以識別出重複出現的主題和模式，這些往往是值得長期記憶的資訊。

4. **記憶考古**：當需要回憶某個具體的事件時，每日筆記提供了比長期記憶更詳細的記錄。

### 3.2 每日筆記的格式

```markdown
# 每日筆記 - 2024-03-15

## 互動摘要
- 與 Eason 討論了 EasonClaw 的記憶架構設計（14:00-15:30）
- 幫 Alice 解決了 CSS Grid 的對齊問題（16:00）
- 在 #general 參與了關於最佳 IDE 的討論（18:00）

## 重要資訊
- Eason 決定使用 SQLite 作為長期記憶的後端資料庫
- Alice 提到她下週有一個設計評審，可能會比較忙

## 技術知識
- 學到了 CSS Grid 的 `place-items: center` 是最簡潔的置中方法
- 討論了 SQLite 在並發寫入時的限制（WAL 模式可以緩解）

## 情感觀察
- Eason 今天精神很好，對專案進展很滿意
- Alice 有點焦慮，可能與下週的評審有關

## 待跟進
- [ ] 下次和 Eason 討論語意搜尋的實作方案
- [ ] Alice 下週評審後問候一下

## 可能提升到長期記憶
- Eason 決定使用 SQLite（重要的技術決策）
- Alice 下週有設計評審（時效性資訊，暫不提升）
```

### 3.3 壓縮策略

每日筆記不能無限制地保存——隨著時間推移，需要進行壓縮。壓縮的目標是保留核心資訊，去除不再相關的細節。

**壓縮的時間軸**：

```
Day 0（今天）：完整的每日筆記
Day 1-3：保持完整
Day 4-7：壓縮為摘要版本（約 50% 大小）
Day 8-30：壓縮為要點版本（約 20% 大小）
Day 31+：重要資訊已提升到 MEMORY.md，筆記歸檔到長期儲存
```

**壓縮的實際範例**：

原始筆記（Day 0）：
```markdown
## 互動摘要
- 14:00 與 Eason 開始討論記憶架構。他先提出了用純 Markdown
  的方案，但我指出在多使用者場景下可能有並發問題。
  我們討論了 30 分鐘，最後決定用 SQLite 作為後端。
  SQLite 的 WAL 模式可以支援同時讀寫。
  Eason 很滿意這個方案。
```

壓縮版本（Day 7）：
```markdown
## 3/15 要點
- 與 Eason 確定使用 SQLite 作為記憶後端（WAL 模式）
```

最終版本（Day 30+ → 提升到 MEMORY.md）：
```markdown
### 技術決策：SQLite 作為記憶後端
- **日期**：2024-03-15
- **內容**：使用 SQLite（WAL 模式）作為長期記憶的後端儲存
- **原因**：支援並發讀寫，適合多使用者場景
```

### 3.4 從日誌提升到長期記憶

提升（Promotion）是將每日筆記中的重要資訊寫入 MEMORY.md 的過程。這是記憶系統中最需要「判斷力」的環節。

**提升的觸發條件**：

1. **重複出現**：某個資訊在多天的筆記中反覆出現
2. **明確的決策**：使用者做出了明確的選擇或決定
3. **持久的偏好**：觀察到的穩定偏好模式
4. **重要的里程碑**：專案的重大進展或變化
5. **人際動態**：團隊成員的重要變化或關係動態

**提升的流程**：

```
每週一次的提升審查流程：

1. 回顧過去一週的每日筆記
2. 標記所有「可能值得提升」的資訊
3. 對每條標記的資訊評估：
   a. 這個資訊在未來的互動中會被用到嗎？
   b. 它會影響 AI 的行為或回答嗎？
   c. 它是持久的還是臨時的？
4. 對通過評估的資訊進行壓縮
5. 檢查 MEMORY.md 中是否有相關的舊記憶需要更新
6. 寫入 MEMORY.md
7. 更新 Token 預算報告
```

**提升的品質控制**：

每次提升時，問自己三個問題：

1. **相關性**：這條記憶在下個月還會有用嗎？
2. **獨特性**：這條記憶提供了新的資訊嗎？還是已有記憶的重複？
3. **可操作性**：這條記憶能改變 AI 的行為嗎？

如果三個問題的答案都是「是」，就值得提升。如果任何一個是「否」，就要重新考慮。

### 3.5 自動化每日筆記的生成

在 Discord 場景中，每日筆記的生成可以高度自動化：

```python
# 偽代碼：每日筆記自動生成流程

async def generate_daily_note(date):
    # 1. 收集今天的所有互動
    interactions = await get_today_interactions(date)
    
    # 2. 分類和摘要

    summaries['interactions'] = summarize_interactions(interactions)
    summaries['decisions'] = extract_decisions(interactions)
    summaries['emotions'] = analyze_emotions(interactions)
    summaries['learnings'] = extract_learnings(interactions)
    
    # 3. 識別待跟進事項
    follow_ups = identify_follow_ups(interactions)
    
    # 4. 識別可能提升的記憶
    promotion_candidates = identify_promotion_candidates(
        interactions, 
        existing_memory
    )
    
    # 5. 格式化為 Markdown
    note = format_daily_note(date, summaries, follow_ups, 
                             promotion_candidates)
    
    # 6. 保存
    save_daily_note(date, note)
```

## 四、記憶清理與維護

### 4.1 定期審計

記憶審計是一個系統化的過程，目的是確保 MEMORY.md 中的每條記憶仍然準確、相關和有價值。

**審計清單**：

```markdown
## 記憶審計清單

### 準確性檢查
- [ ] 所有記憶條目的事實是否仍然正確？
- [ ] 技術決策是否已經改變？
- [ ] 使用者偏好是否已經更新？

### 相關性檢查
- [ ] 每條記憶是否仍然對未來的互動有用？
- [ ] 是否有已完成的專案記憶可以歸檔？
- [ ] 是否有過於具體的臨時性資訊？

### 組織性檢查
- [ ] 分類結構是否仍然合理？
- [ ] 是否有記憶放錯了類別？
- [ ] 是否有可以合併的重複記憶？

### 預算檢查
- [ ] 當前 Token 使用量是否在預算內？
- [ ] 是否有低優先級的記憶可以移除？
- [ ] 是否有更簡潔的表述方式？
```

**審計頻率建議**：

| 使用強度 | 審計頻率 | 每次時間 |
|---------|---------|---------|
| 每天使用 | 每週一次 | 15-30 分鐘 |
| 每週使用 | 每月一次 | 20-40 分鐘 |
| 偶爾使用 | 每季一次 | 30-60 分鐘 |

### 4.2 過時資訊的處理

過時的資訊是記憶系統最大的敵人。一條過時的記憶不僅浪費 Token，還可能誤導 AI 做出錯誤的假設。

**過時的類型**：

1. **事實過時**：「使用者正在學 React 17」→ 使用者已經在用 React 18
2. **決策過時**：「專案使用 REST API」→ 已經改為 GraphQL
3. **狀態過時**：「Alice 正在找工作」→ Alice 已經找到了
4. **偏好過時**：「偏好 npm」→ 已經改用 pnpm

**處理策略**：

```
過時的事實 → 更新為最新的事實
過時的決策 → 更新決策記錄，保留舊決策的原因（作為歷史參考）
過時的狀態 → 如果新狀態重要則更新，否則直接刪除
過時的偏好 → 更新為新的偏好
```

**範例**：

```markdown
[更新前]
### 專案技術棧
- 前端：React 17 + Redux
- 後端：Express.js
- 資料庫：MongoDB

[更新後]
### 專案技術棧
- 前端：React 18 + Zustand（2024-02 從 React 17 + Redux 遷移）
- 後端：Express.js
- 資料庫：PostgreSQL（2024-03 從 MongoDB 遷移，因為需要複雜查詢）
```

注意更新時保留了變更的歷史——這有助於 AI 理解技術選擇的演化脈絡。

### 4.3 衝突解決

當同一件事在不同的記憶來源中有不同的記錄時，就產生了衝突。

**衝突的來源**：

1. **時間差異**：MEMORY.md 和每日筆記中對同一件事的記錄不一致（因為其中一方更新了）
2. **來源差異**：使用者在不同時間說了矛盾的話
3. **推斷錯誤**：AI 自動記憶的內容與使用者的實際意圖不符

**衝突解決的優先級**：

```
最高優先：使用者最近的明確表態
    ↓
次高優先：使用者之前的明確表態
    ↓
中等優先：AI 從多次互動中推斷的結論
    ↓
最低優先：AI 從單次互動中推斷的結論
```

**衝突解決的實作**：

當發現衝突時：
1. 標記衝突的記憶條目
2. 在下次互動中，禮貌地向使用者確認
3. 根據確認結果更新記憶
4. 刪除錯誤的記憶版本

```
AI：我記得你之前說偏好 MongoDB，但最近我們又決定用 PostgreSQL。
    要我更新記憶嗎？現在主要用的是 PostgreSQL？
使用者：對，已經全部改用 PostgreSQL 了。
AI：好的，已更新。
```

### 4.4 記憶歸檔策略

不是所有舊記憶都應該被刪除。有些記憶雖然不再是「活躍」的，但作為歷史記錄仍有價值。

**歸檔標準**：

```
活躍記憶（在 MEMORY.md 中）：
  ← 最近 3 個月內被引用過
  ← 影響當前的互動行為
  ← 與進行中的專案相關

歸檔記憶（移到 archives/）：
  ← 超過 3 個月未被引用
  ← 已完成的專案相關記憶
  ← 歷史性的決策記錄（保留作為參考）

刪除：
  ← 完全過時且無歷史價值的資訊
  ← 重複的記憶
  ← 使用者要求刪除的資訊
```

## 五、效能優化

### 5.1 語意搜尋的調校

語意搜尋是記憶系統中最能體現「智慧」的部分。它決定了 AI 能否在正確的時間想起正確的記憶。

**語意搜尋的核心流程**：

```
使用者提問
    ↓
提取查詢關鍵字 / 生成查詢向量
    ↓
在記憶向量資料庫中搜尋
    ↓
排序結果（相關度 × 新鮮度 × 重要性）
    ↓
選取 Top-K 結果
    ↓
注入到對話上下文中
```

**調校參數**：

1. **Top-K 值**：返回多少條相關記憶
   - K 太小（1-2）：可能遺漏重要的相關記憶
   - K 太大（10+）：引入太多噪音，佔用 Token
   - 建議：3-5 條

2. **相關度閾值**：低於多少相關度的結果不返回
   - 太高（0.9+）：只返回幾乎完全匹配的結果，可能錯過相關但措辭不同的記憶
   - 太低（0.5-）：返回很多弱相關的結果
   - 建議：0.65-0.75

3. **新鮮度權重**：較新的記憶是否應該被優先返回
   - 建議：給予適度的新鮮度加分，但不要壓倒相關度

4. **重要性權重**：高優先級的記憶是否應該被優先返回
   - 建議：P0/P1 的記憶可以有 10-20% 的加分

**調校的實際方法**：

```python
# 綜合排序分數計算
def calculate_memory_score(memory, query):
    relevance = cosine_similarity(query_embedding, memory_embedding)
    freshness = 1.0 / (1 + days_since_update(memory) * 0.01)
    importance = priority_weight(memory.priority)  # P0=1.2, P1=1.1, P2=1.0, P3=0.9
    
    score = relevance * 0.6 + freshness * 0.2 + importance * 0.2
    return score
```

### 5.2 記憶索引

當記憶條目數量增多時，全文掃描會變得低效。建立索引可以加速檢索。

**標籤索引**：

```markdown
## 標籤索引

#技術選型 → [記憶 #3, #7, #12, #15]
#偏好 → [記憶 #1, #2, #5, #8]
#人際 → [記憶 #4, #9, #13]
#決策 → [記憶 #3, #7, #10, #15]
#TypeScript → [記憶 #1, #3, #7]
```

**類別索引**：

```markdown
## 類別索引

使用者偏好（5 條）：#1, #2, #5, #8, #11
專案知識（4 條）：#3, #7, #12, #15
技術經驗（3 條）：#6, #10, #14
人際關係（4 條）：#4, #9, #13, #16
```

**時間索引**：

```markdown
## 時間索引（最近優先）

2024-03：#15, #16
2024-02：#12, #13, #14
2024-01：#8, #9, #10, #11
2023-12：#5, #6, #7
```

### 5.3 載入順序優化

啟動檔案的載入順序會影響模型的「注意力分配」。研究表明，放在上下文開頭和結尾的資訊更容易被模型「注意到」（primacy and recency effect）。

**推薦的載入順序**：

```
1. SOUL.md（最重要——定義基本行為模式）
2. USER.md（其次——定義互動對象）
3. 高優先級記憶（P0, P1）
4. 工具定義
5. 中低優先級記憶（P2, P3）
6. 最近的每日筆記
7. [對話開始]
```

**為什麼這個順序？**

- SOUL.md 放最前面，確保人格定義有最強的影響力
- 高優先級記憶緊跟在使用者資訊後面，被引用的機率更高
- 低優先級記憶和每日筆記放在對話開始之前，即使被忽略影響也最小
- 每日筆記放在最後（最接近對話開始），因為它是最新的資訊，recency effect 有利

### 5.4 延遲載入策略

不是所有記憶都需要在啟動時載入。延遲載入（Lazy Loading）可以在需要時才檢索特定的記憶。

**啟動時載入（Eager Loading）**：
- SOUL.md（完整）
- USER.md（完整）
- MEMORY.md 的高優先級部分
- 今天和昨天的每日筆記

**按需載入（Lazy Loading）**：
- MEMORY.md 的低優先級部分
- 舊的每日筆記
- 歸檔的記憶
- 語意搜尋結果

**按需載入的觸發機制**：

```
1. 使用者提到了一個特定的人名 → 載入該人的記憶
2. 使用者提到了一個舊專案 → 載入該專案的記憶
3. 使用者使用了「上次」「之前」等回憶性詞語 → 觸發語意搜尋
4. AI 識別到當前話題可能有相關記憶 → 主動搜尋
```

## 六、針對我們目標的記憶設計

### 6.1 Discord 社交記憶——記住每個人的特徵和歷史

在 Discord 社交場景中，最重要的記憶是「人」的記憶。機器人需要記住每個活躍成員的特徵，才能提供個人化的互動。

**社交記憶的結構**：

```markdown
## 成員記憶：Alice

### 基本印象
一個充滿熱情的設計師，正在從設計轉向前端開發。
性格開朗，喜歡用表情符號。

### 互動模式
- 活躍時段：工作日 10:00-18:00
- 訊息風格：短句、多表情符號、常用驚嘆號
- 偏好的互動方式：鼓勵性、具體讚美

### 技能圖譜
- 精通：Figma, 設計原理, 色彩理論
- 學習中：HTML, CSS, JavaScript
- 感興趣：前端動畫, SVG

### 最近動態
- 完成了第一個前端專案（3/10）
- 下週有設計評審（3/20）

### 互動歷史亮點
- 幫她解決 CSS Grid 問題 → 她非常開心
- 讚美她的配色方案 → 她特別感動
- 她不喜歡被指出設計上的「問題」→ 要用「建議」的方式

### 與其他成員的關係
- 和 Eason 經常討論設計趨勢
- 偶爾和 Bob 合作前後端對接
```

**社交記憶的管理**：

```
新成員加入 → 建立基本檔案（暱稱、角色、首次互動日期）
    ↓
前 10 次互動 → 觀察並記錄互動模式
    ↓
持續互動 → 不斷豐富技能圖譜和互動歷史
    ↓
定期更新 → 更新最近動態、清理過時資訊
    ↓
不活躍（>3 個月）→ 歸檔記憶，保留基本檔案
```

### 6.2 對話情境記憶——記住話題脈絡

除了「人」的記憶，對話情境的記憶也非常重要。這讓機器人能夠延續之前的話題，而不是每次都從頭開始。

**對話情境的層次**：

```
短期情境（當前會話）：
  「Alice 正在問 CSS Flexbox 的問題」
  → 由對話歷史自動處理

中期情境（最近幾天）：
  「這週大家主要在討論 Next.js 14 的新特性」
  → 由每日筆記記錄

長期情境（持續的話題線索）：
  「Alice 有一個長期的學習路線：設計 → CSS → JavaScript → React」
  → 由 MEMORY.md 記錄
```

**話題追蹤的實作**：

```markdown
## 活躍話題追蹤

### 頻道：#tech-help
- 最近熱門：Next.js 14 App Router 問題
- 持續話題：Alice 的前端學習之旅
- 新話題：Bob 在調研 Bun 運行時

### 頻道：#general
- 最近熱門：下週的線下聚會規劃
- 持續話題：大家的 2024 學習目標
- 文化梗：「又是 CSS 的問題」→ 每次有人問 CSS 問題時的固定反應
```

### 6.3 情感記憶——記住互動時的情感狀態

情感記憶是最「人性化」的記憶類型。它讓機器人能夠感知和記住互動中的情感動態，從而提供更有溫度的回應。

**情感記憶的結構**：

```markdown
## 情感記憶

### Eason 的情感模式
- 整體情緒狀態：穩定偏積極
- 壓力信號：深夜連續發問、語氣變得簡短
- 開心信號：使用更多表情符號、分享成果
- 觸發點：技術問題被卡住太久會焦躁
- 最佳回應方式：直接給解決方案 + 一句鼓勵

### Alice 的情感模式
- 整體情緒狀態：活潑但有時焦慮
- 壓力信號：發言變少、不再用表情符號
- 開心信號：連發多條訊息、分享截圖
- 觸發點：設計被否定會沮喪
- 最佳回應方式：先肯定再建議

### 群組情感動態
- 3/10：Alice 完成專案，群組氛圍很好，大家互相鼓勵
- 3/12：有一場關於技術選型的激烈討論，但最後友好解決
- 群組文化：鼓勵嘗試、包容錯誤、互相幫助
```

**情感感知的實作**：

情感感知可以通過以下信號來實現：

```
文字信號：
  - 表情符號的使用量和類型
  - 標點符號的使用（多個驚嘆號 = 興奮，省略號 = 猶豫）
  - 詞彙選擇（消極詞彙 vs 積極詞彙）
  - 訊息長度和頻率的變化

行為信號：
  - 活躍時間的變化
  - 回應速度的變化
  - 話題選擇的變化
  - 互動對象的變化
```

**情感記憶的倫理考量**：

情感記憶是最敏感的記憶類型，需要特別注意：

1. **不做心理分析**：記錄觀察到的情感信號，但不做深度的心理學推斷
2. **尊重隱私**：情感狀態是非常私人的資訊，不應在公開場合提及
3. **不操控情感**：情感記憶的目的是更好地服務，不是操控使用者
4. **允許刪除**：使用者可以要求刪除任何情感相關的記憶

### 7.2 常見問題與解決方案

| 問題 | 症狀 | 解決方案 |
|------|------|----------|
| 記憶膨脹 | Token 持續超預算 | 立即審計，刪除低優先級記憶 |

| 冷啟動慢 | 新使用者互動體驗差 | 優化預設配置，加快學習曲線 |

## 結語

記憶管理不是一個「設定好就不用管」的系統——它是一個需要持續維護、調優和演化的有機體。就像人類的記憶一樣，AI 的記憶也需要不斷地整理、更新和反思。

在我們的 EasonClaw 專案中，記憶系統將是最核心的差異化能力。一個有記憶的 Discord 機器人和一個沒有記憶的機器人，帶給社群的體驗是天壤之別。前者是一個真正的社群成員，後者只是一個便利的工具。

本篇覆蓋的最佳實踐——從預算規劃到架構設計，從每日筆記到效能優化——提供了一份完整的操作指南。但最重要的建議其實很簡單：

**從小開始，持續迭代。**

不要試圖一開始就建立完美的記憶系統。先用一個簡單的 MEMORY.md 開始，觀察什麼有效、什麼無效，然後逐步添加更多的功能。記憶系統的價值不在於它的複雜度，而在於它為使用者帶來的體驗改善。

> 「最好的記憶系統不是記住最多的系統，而是在最需要的時候，想起最重要的事情的系統。」
