# MEMORY.md——OpenClaw 的長期記憶系統

> 「記憶不是過去的倉庫，而是未來的建築材料。」

在人工智慧代理（AI Agent）的世界中，記憶是一切智慧行為的基石。沒有記憶的 AI，就像每次醒來都失去所有經歷的人——它可以回答問題、執行任務，但永遠無法真正「理解」你、適應你、成長為你的專屬夥伴。OpenClaw 框架中的 MEMORY.md 系統，正是為了解決這個根本性問題而設計的。

本篇將深入探討 OpenClaw 長期記憶系統的方方面面：從哲學思考到技術實現，從檔案格式到管理策略，從個人使用到多人 Discord 環境的應用。我們的目標不僅是理解 MEMORY.md 的運作方式，更是掌握如何設計一套真正有效的 AI 記憶體系。

---

## 一、長期記憶的重要性——為什麼 AI 需要「記得」

### 1.1 無記憶 AI 的困境

讓我們先想像一個場景：你每天早上都和同一個人打招呼，告訴他你的名字、你的工作、你喜歡什麼咖啡。但每天他都忘記了一切，你必須從頭來過。這不僅令人沮喪，更重要的是——你們之間永遠無法建立真正的關係。

現今絕大多數 AI 系統正面臨這樣的困境。大型語言模型（LLM）本質上是無狀態的（stateless）——每次對話都是一個全新的開始。即使在同一個對話會話中，模型能夠記住先前的內容，但一旦會話結束，所有的上下文都會消失。這就像一個極度短暫的工作記憶，完全沒有長期記憶的能力。

這種無記憶的狀態帶來了幾個根本性的問題：

**重複性溝通成本**：使用者必須在每次新對話中重新解釋自己的背景、偏好和需求。想像你每次找同事討論專案時，都要先花十分鐘介紹專案背景——效率會有多低？

**無法建立個人化體驗**：沒有記憶，AI 無法累積對使用者的理解。它不知道你偏好簡潔的回答還是詳細的解釋，不知道你是前端開發者還是資料科學家，不知道你上週遇到了什麼問題。每次互動都是「面對陌生人」的體驗。

**缺乏連續性思維**：真正有價值的工作往往跨越多個會話。一個複雜的專案可能需要數天甚至數週的持續推進。沒有長期記憶，AI 無法追蹤任務進度、記住已經做出的決策、或者理解問題的演化歷史。

**無法從經驗中學習**：人類的學習很大程度上依賴於記憶——我們從過去的成功和失敗中汲取教訓。缺乏記憶的 AI 無法記住哪種方法對特定使用者有效，哪些建議被採納了，哪些被拒絕了。

### 1.2 人類記憶的啟示

在設計 AI 記憶系統之前，我們值得回顧一下人類記憶的運作方式，因為這為我們提供了寶貴的設計靈感。

認知心理學告訴我們，人類的記憶系統大致可以分為以下幾個層次：

**感覺記憶（Sensory Memory）**：持續時間極短（毫秒到秒級），容量巨大但極度短暫。對應到 AI 系統中，這就像模型接收到輸入 token 的最初處理階段。

**短期記憶 / 工作記憶（Short-term / Working Memory）**：容量有限（著名的「7±2」法則），持續數秒到數分鐘。在 AI 系統中，這對應於當前對話的上下文窗口（context window）。

**長期記憶（Long-term Memory）**：容量幾乎無限，持續時間從數小時到一生。又可以細分為：
- **語意記憶（Semantic Memory）**：關於世界的一般性知識和事實
- **情節記憶（Episodic Memory）**：關於個人經歷的記憶
- **程序記憶（Procedural Memory）**：關於如何做事情的記憶

OpenClaw 的 MEMORY.md 系統主要對應的是長期記憶中的語意記憶和情節記憶。它試圖為 AI 建立一個持久的知識庫，記錄與使用者相關的重要資訊和互動歷史。

### 1.3 記憶作為關係的基礎

在我們規劃的 Discord 機器人專案中，記憶的重要性尤為突出。一個能「記住」社群成員的機器人，能夠：

- 記住每個成員的專長和興趣，在相關討論中主動 @他們
- 記住之前的對話脈絡，延續未完成的話題
- 記住成員的情緒模式，在適當時機給予關懷
- 記住社群的文化規範和內部梗，融入社群而不是像個外人

這不僅僅是技術上的便利，更是建立「社交存在感」的關鍵。一個有記憶的機器人，才能成為社群的真正成員，而不只是一個工具。

### 1.4 記憶的第一性原理思考

從第一性原理的角度思考，AI 記憶系統需要解決的核心問題是：**如何在有限的上下文窗口中，呈現出最相關、最有價值的歷史資訊**。

這個問題可以分解為四個子問題：

1. **寫入問題**：什麼資訊值得記住？如何從海量的對話中篩選出值得長期保存的內容？
2. **儲存問題**：記憶應該以什麼格式和結構保存？如何在可讀性和機器處理效率之間取得平衡？
3. **讀取問題**：在每次會話開始時，應該載入哪些記憶？如何確保最相關的資訊被優先呈現？
4. **維護問題**：記憶如何隨時間演化？過時的資訊如何處理？如何避免記憶膨脹到無法管理的程度？

MEMORY.md 系統對這四個問題都提供了明確的答案，我們將在後續章節中逐一深入探討。

## 二、MEMORY.md 的定位與角色

### 2.1 啟動檔案體系中的位置

在 OpenClaw 框架中，代理的「身份」和「知識」由一組啟動檔案（boot files）共同定義。這些檔案在每次會話開始時被載入，為代理提供持久的上下文。主要的啟動檔案包括：

| 檔案 | 角色 | 比喻 |
|------|------|------|
| SOUL.md | 人格與行為準則 | DNA / 性格 |
| USER.md | 使用者身份與偏好 | 對方的名片 |
| MEMORY.md | 累積的知識與經驗 | 日記與筆記本 |
| daily_notes/ | 每日互動日誌 | 當天的便條紙 |

MEMORY.md 在這個體系中扮演的角色，就像一本不斷更新的「筆記本」——它記錄了代理在與使用者互動過程中學到的所有重要資訊。與 SOUL.md（定義代理是誰）和 USER.md（定義使用者是誰）不同，MEMORY.md 是動態的、成長的、不斷演化的。

### 2.2 MEMORY.md 與其他記憶機制的關係

值得注意的是，MEMORY.md 並不是 OpenClaw 中唯一的記憶機制。完整的記憶體系包括：

**對話歷史（Conversation History）**：當前會話中的完整對話記錄。這是最詳細但也是最短暫的記憶形式，會話結束後即消失。

**每日筆記（Daily Notes）**：每天的互動摘要，記錄當天發生的重要事件和決策。這是中期記憶，保存數天到數週。

**MEMORY.md**：經過篩選和提煉的長期記憶，只保留最重要的資訊。這是長期記憶，理論上可以永久保存。

**外部知識庫（External Knowledge Base）**：透過 MCP 協議連接的外部資料來源，如資料庫、API、文件系統等。這不算嚴格意義上的「記憶」，但提供了額外的知識來源。

這些記憶層次之間存在一個「提升」（promotion）機制：對話中的重要資訊 → 每日筆記 → MEMORY.md。每一次提升都伴隨著資訊的壓縮和篩選，確保長期記憶中只保留最有價值的內容。

### 2.3 為什麼是 Markdown 檔案

選擇 Markdown 作為記憶的儲存格式，是一個經過深思熟慮的設計決策。相比於資料庫或其他結構化儲存方式，Markdown 有以下優勢：

**人類可讀性**：MEMORY.md 是一個普通的文字檔案，使用者可以直接打開閱讀和編輯。這種透明性對於建立信任至關重要——使用者可以清楚地看到 AI 記住了什麼，並且可以隨時修正或刪除不準確的記憶。

**LLM 友好性**：Markdown 是 LLM 訓練資料中最常見的格式之一。模型對 Markdown 的理解和生成能力極強，能夠自然地讀取和寫入 Markdown 格式的記憶。

**版本控制友好**：作為純文字檔案，MEMORY.md 可以完美地融入 Git 版本控制系統。這意味著記憶的每次變更都有完整的歷史記錄，可以追溯、比較和回滾。

**零依賴**：不需要任何資料庫伺服器或額外的軟體。一個文字編輯器就能管理所有的記憶，極大地降低了系統的複雜度。

**便於遷移**：Markdown 檔案可以輕鬆地在不同系統和平台之間遷移，沒有任何鎖定效應。

當然，Markdown 也有其局限性（我們將在第九節詳細討論），但對於大多數使用場景來說，其優勢遠大於劣勢。

## 三、檔案結構與內容格式

### 3.1 推薦的整體結構

一個設計良好的 MEMORY.md 應該有清晰的組織結構，便於人類閱讀和 AI 解析。以下是推薦的整體結構：

```markdown
# 長期記憶

## 使用者偏好
<!-- 關於使用者的個人偏好和習慣 -->

## 專案知識
<!-- 關於當前專案的重要資訊 -->

## 技術知識
<!-- 學到的技術知識和經驗 -->

## 人際關係
<!-- 關於團隊成員和合作者的資訊 -->

## 重要決策
<!-- 記錄重要的決策及其理由 -->

## 待辦事項
<!-- 長期追蹤的任務和目標 -->

## 經驗教訓
<!-- 從過去的互動中學到的教訓 -->
```

### 3.2 記憶條目的格式

每個記憶條目應該遵循一致的格式，包含足夠的上下文資訊：

```markdown
### [簡短標題]
- **日期**：YYYY-MM-DD
- **來源**：[使用者告知 / 自動識別 / 日誌提升]
- **重要性**：[高 / 中 / 低]
- **內容**：[記憶的具體內容]
- **上下文**：[這條記憶的背景和產生原因]
```

例如：

```markdown
### 使用者偏好 TypeScript
- **日期**：2024-01-15
- **來源**：使用者告知
- **重要性**：高
- **內容**：Eason 偏好使用 TypeScript 而非 JavaScript，所有新專案都應預設使用 TypeScript。
- **上下文**：在討論新專案技術選型時，Eason 明確表示了對 TypeScript 的偏好。
```

### 3.3 結構化標籤系統

為了便於語意搜尋和分類管理，建議為記憶條目添加標籤：

```markdown
### 使用者偏好 TypeScript
**標籤**：#偏好 #技術選型 #TypeScript #程式語言
...
```

標籤系統使得 AI 可以更精確地檢索相關記憶。當使用者詢問技術相關問題時，AI 可以優先搜尋帶有 #技術選型 標籤的記憶。

### 3.4 記憶的粒度控制

記憶條目的粒度是一個需要仔細拿捏的問題。太粗糙的記憶缺乏實用性，太細緻的記憶則會消耗過多 Token。以下是一些指導原則：

**適合記錄的粒度**：
- ✅ 「使用者偏好 Vim 鍵位綁定」——具體、可操作
- ✅ 「專案使用 Next.js 14 + App Router」——明確的技術決策
- ✅ 「Alice 是團隊的設計師，擅長 Figma」——有用的人際資訊

**過於粗糙的記憶**：
- ❌ 「使用者喜歡程式設計」——太模糊，沒有實用價值
- ❌ 「專案用了很多技術」——缺乏具體資訊

**過於細緻的記憶**：
- ❌ 「使用者在 2024-01-15 14:23:45 提到他今天午餐吃了三明治」——除非與飲食偏好相關，否則不值得記錄
- ❌ 「使用者問了一個關於 CSS flexbox 的問題」——除非這代表一個持續的學習主題

### 3.5 實際的 MEMORY.md 範例

以下是一個更完整的、實際使用中的 MEMORY.md 範例：

```markdown
# 長期記憶

> 最後更新：2024-03-15
> 條目數量：23
> 預估 Token：約 1,200

## 使用者偏好

### 程式語言偏好
- **日期**：2024-01-15
- **重要性**：高
- **內容**：偏好 TypeScript，所有新專案預設使用。對 Python 也很熟悉，
  主要用於資料分析和腳本。不喜歡 Java 的冗長語法。
  
### 編輯器與工具
- **日期**：2024-01-20
- **重要性**：中
- **內容**：主要使用 VS Code，安裝了 Vim 擴充套件。終端使用 Wezterm + Zsh。
  偏好深色主題。

### 溝通風格
- **日期**：2024-02-01
- **重要性**：高
- **內容**：偏好簡潔直接的回答，不喜歡過多的客套話。
  技術討論時希望看到程式碼範例。對於複雜概念，偏好用類比來解釋。

## 專案知識

### EasonClaw 專案
- **日期**：2024-03-01
- **重要性**：高
- **內容**：個人 AI 助手專案，基於 OpenClaw 框架。目標是建立一個
  有記憶、有人格的 Discord 機器人。技術棧：Node.js + TypeScript，
  計劃整合 MCP 協議。
- **上下文**：這是使用者當前的主要學習專案。

## 經驗教訓

### Token 超出限制的處理
- **日期**：2024-02-15
- **重要性**：中
- **內容**：當 MEMORY.md 超過 2000 Token 時，會明顯影響回應速度。
  需要定期清理不再相關的記憶條目。
- **教訓**：記憶不是越多越好，品質比數量重要。
```

## 四、記憶的寫入機制

記憶的寫入是整個記憶系統中最關鍵的環節之一。寫入什麼、何時寫入、以什麼格式寫入，直接決定了記憶系統的品質和實用性。OpenClaw 支援三種主要的寫入機制。

### 4.1 使用者明確要求記住

這是最直接、最可靠的寫入方式。當使用者明確告訴 AI「記住這個」時，AI 應該將相關資訊寫入 MEMORY.md。

**觸發方式**：

使用者可能使用各種自然語言表達來觸發記憶寫入：
- 「記住，我偏好使用 TypeScript」
- 「請記下，Alice 的生日是 3 月 15 日」
- 「以後記得，我不喜歡在程式碼中使用分號」
- 「把這個加到你的記憶中」

AI 需要能夠識別這些意圖，並將資訊正確地格式化後寫入 MEMORY.md。

**寫入流程**：

1. **意圖識別**：AI 識別到使用者希望記住某條資訊
2. **資訊提取**：從使用者的話語中提取核心資訊
3. **分類歸檔**：將記憶歸入適當的類別（偏好、知識、人際等）
4. **格式化**：按照標準格式組織記憶條目
5. **去重檢查**：檢查是否已存在類似的記憶，避免重複
6. **寫入確認**：將記憶寫入檔案，並向使用者確認

**範例交互**：

```
使用者：記住，我們的 Discord 伺服器叫做「深夜程式碼」，主要成員有
我、Alice（設計師）、Bob（後端工程師）和 Carol（PM）。

AI：好的，我已經將以下資訊記入長期記憶：
- Discord 伺服器名稱：深夜程式碼
- 成員：Eason（你）、Alice（設計師）、Bob（後端工程師）、Carol（PM）
這些資訊已保存在「人際關係」分類下。
```

### 4.2 代理自動識別重要資訊

除了使用者明確要求外，一個智慧的代理應該能夠自動識別對話中值得記住的資訊。這需要更複雜的判斷邏輯。

**自動識別的信號**：

以下類型的資訊通常值得自動記憶：

1. **修正性資訊**：當使用者糾正 AI 的錯誤假設時，這通常代表一個重要的偏好或事實。例如：「不，我用的不是 npm，我用的是 pnpm。」

2. **重複出現的主題**：如果使用者在多次對話中反覆提到同一件事，這可能是一個重要的持續關注點。

3. **明確的偏好表達**：即使使用者沒有說「記住」，像「我討厭...」、「我總是用...」這類強烈的偏好表達也值得記錄。

4. **重要的決策和承諾**：「我決定用 PostgreSQL 而不是 MongoDB」、「我們下週要交付 v1.0」——這類決策性陳述值得記憶。

5. **關鍵的上下文資訊**：專案名稱、團隊成員、技術棧等結構性資訊，一旦出現就應該記錄。

**自動記憶的風險與控制**：

自動記憶雖然便利，但也帶來了一些風險：

- **隱私侵犯**：自動記住敏感資訊可能讓使用者不舒服
- **噪音問題**：過度積極的自動記憶會導致記憶膨脹
- **準確性問題**：AI 對資訊重要性的判斷可能不準確

因此，自動記憶應該遵循以下原則：

1. **保守原則**：寧可遺漏也不要記錄不確定的資訊
2. **透明原則**：告知使用者自動記住了什麼
3. **可撤銷原則**：使用者可以隨時要求刪除任何記憶
4. **分級原則**：對於不確定的記憶，可以先標記為「待確認」

**實現策略**：

在技術實現上，自動識別可以透過在對話結束時的「反思」步驟來實現：

```
[對話結束時的內部流程]
1. 回顧整場對話
2. 識別新出現的重要資訊
3. 與現有記憶進行比對
4. 如果發現新的、重要的、不重複的資訊
5. 生成記憶條目草稿
6. （可選）向使用者確認
7. 寫入 MEMORY.md
```

### 4.3 每日筆記的壓縮提升

這是 OpenClaw 記憶系統中最精妙的機制之一——將每日筆記（daily notes）中的重要資訊「提升」（promote）到長期記憶中。

**每日筆記的角色**：

每日筆記是對每天互動的詳細記錄，包含：
- 當天討論的主題
- 完成的任務
- 遇到的問題
- 做出的決策
- 有趣的觀察

每日筆記的特點是詳細但短暫——它們記錄了每天的「原始素材」，但不是所有素材都值得長期保存。

**提升（Promotion）流程**：

提升是一個定期（通常是每天或每週）執行的流程：

1. **回顧期間的每日筆記**：AI 閱讀最近一段時間的所有每日筆記
2. **識別長期價值資訊**：篩選出具有長期參考價值的資訊
3. **壓縮與提煉**：將詳細的描述壓縮為簡潔的記憶條目
4. **整合到 MEMORY.md**：將新的記憶條目合併到長期記憶中
5. **標記已處理的筆記**：避免重複處理

**壓縮的藝術**：

壓縮是提升流程中最具挑戰性的步驟。好的壓縮應該保留核心資訊，去除不必要的細節：

```
[每日筆記原文 - 2024-03-10]
今天花了 3 個小時嘗試用 Puppeteer 做網頁爬蟲，但遇到了很多
反爬蟲機制的問題。後來改用 Playwright，發現它對 headless 模式
的支援更好，而且 API 更直覺。最後成功完成了爬蟲任務。
Eason 說他以後都想用 Playwright 而不是 Puppeteer。

[提升到 MEMORY.md 的記憶]
### 偏好 Playwright 而非 Puppeteer
- **日期**：2024-03-10
- **重要性**：中
- **內容**：偏好使用 Playwright 做網頁自動化/爬蟲，因為 headless 模式
  支援更好，API 更直覺。不再使用 Puppeteer。
```

注意壓縮過程中發生了什麼：
- 保留了核心偏好（Playwright > Puppeteer）
- 保留了原因（headless 支援好、API 直覺）
- 去除了過程細節（花了 3 小時、遇到反爬蟲問題等）
- 去除了具體任務的資訊（除非任務本身值得長期記憶）

**提升的判斷標準**：

並非所有每日筆記中的資訊都需要提升。以下是判斷標準的參考框架：

| 標準 | 提升 | 不提升 |
|------|------|--------|
| 持續性 | 長期有效的偏好或知識 | 一次性的具體問題 |
| 頻率 | 多次出現的主題 | 偶爾提到的瑣事 |
| 影響力 | 影響未來決策的資訊 | 不影響未來互動的資訊 |
| 獨特性 | 使用者特有的知識 | 通用的常識 |

## 五、記憶的讀取機制

寫入記憶只是第一步，更重要的是如何在恰當的時機讀取和利用這些記憶。OpenClaw 提供了兩種主要的讀取機制。

### 5.1 每次會話啟動時載入

這是最基礎的讀取機制——在每次新會話開始時，自動將 MEMORY.md 的全部（或部分）內容載入到對話的上下文中。

**載入流程**：

```
1. 使用者發起新會話
2. 系統讀取 MEMORY.md 檔案
3. 將記憶內容插入到 system prompt 中
4. （可選）根據 Token 預算截斷或摘要
5. AI 在回應第一個訊息時，已經「知道」所有長期記憶
```

**載入策略**：

根據 MEMORY.md 的大小和 Token 預算，可以採用不同的載入策略：

**完整載入**：當 MEMORY.md 較小（< 1000 Token）時，直接全部載入。這是最簡單也最可靠的策略。

**優先級載入**：當 MEMORY.md 較大時，根據記憶條目的重要性和相關性，優先載入高優先級的內容。

**分類載入**：根據對話的預期主題，只載入相關分類的記憶。例如，如果使用者一般在工作時間問技術問題，則優先載入技術相關的記憶。

**摘要載入**：對 MEMORY.md 進行摘要，用較少的 Token 傳達最核心的資訊。這是 Token 預算極度有限時的最後手段。

### 5.2 語意搜尋輔助回憶

僅僅在啟動時載入記憶是不夠的。在對話過程中，AI 可能需要根據當前的對話內容，動態地搜尋和檢索特定的記憶。這就是語意搜尋（semantic search）的用武之地。

**語意搜尋的工作原理**：

1. **向量化**：將 MEMORY.md 中的每個記憶條目轉換為向量表示（embedding）
2. **查詢匹配**：當需要回憶某條資訊時，將查詢文本轉換為向量，並與記憶向量進行相似度比較
3. **排序返回**：按相似度排序，返回最相關的記憶條目

**實際應用場景**：

```
使用者：上次我們討論過要用什麼資料庫來做，你還記得嗎？

[AI 的內部流程]
1. 識別到使用者在嘗試回憶之前的討論
2. 以「資料庫選擇」作為查詢進行語意搜尋
3. 在 MEMORY.md 中找到相關記憶條目
4. 返回結果

AI：是的，我記得。在 2024-02-20 的討論中，你決定使用 PostgreSQL
而不是 MongoDB，主要原因是你的專案需要複雜的關聯查詢，而且你更
熟悉 SQL。
```

**與 MCP 的整合**：

在 OpenClaw 框架中，語意搜尋可以透過 MCP（Model Context Protocol）工具來實現。代理可以使用專門的記憶搜尋工具來查詢長期記憶：

```json
{
  "tool": "memory_search",
  "parameters": {
    "query": "資料庫選擇",
    "top_k": 3,
    "min_relevance": 0.7
  }
}
```

**搜尋與載入的互補關係**：

啟動時的完整載入確保了 AI 對使用者有基本的了解，而語意搜尋則在需要時提供更精確的記憶檢索。兩者互補，共同構成了完整的記憶讀取體系。

可以用一個類比來理解：啟動載入就像你每天早上回顧今天要做的事情清單，而語意搜尋則像你在需要時翻閱筆記本尋找特定的資訊。

## 六、記憶管理策略

隨著使用時間的增長，MEMORY.md 會不斷膨脹。如果不加以管理，記憶系統最終會變得臃腫、混亂、充滿過時資訊。因此，有效的記憶管理策略至關重要。

### 6.1 定期清理與整合

就像我們會定期整理書桌一樣，MEMORY.md 也需要定期清理和整合。

**清理的時機**：

- **定期審計**：建議每 1-2 週進行一次記憶審計
- **Token 超標時**：當 MEMORY.md 的 Token 數量接近或超過預算時
- **使用者要求**：當使用者明確要求清理記憶時
- **專案轉換時**：當使用者開始一個新專案或進入新階段時

**清理的原則**：

1. **過時性清除**：刪除已經不再準確或相關的記憶（例如：「當前使用 React 17」如果已經升級到 React 18）
2. **合併相似條目**：將多個相關但重複的記憶合併為一個更完整的條目
3. **降級不活躍記憶**：將長時間未被引用的記憶降低優先級
4. **刪除臨時性記憶**：刪除原本就是短期有效的資訊

**整合的範例**：

```markdown
[整合前]
### TypeScript 偏好 (2024-01-15)
- 偏好使用 TypeScript

### 嚴格模式偏好 (2024-01-20)
- TypeScript 要用 strict 模式

### ESLint 設定 (2024-02-01)
- 使用 @typescript-eslint/recommended

[整合後]
### TypeScript 開發環境偏好 (2024-01-15, 更新於 2024-02-01)
- 偏好 TypeScript（strict 模式）
- ESLint 使用 @typescript-eslint/recommended
- 所有新專案預設啟用
```

### 6.2 Token 預算控制

Token 預算是 MEMORY.md 設計中最重要的技術約束。每個 Token 都是寶貴的上下文空間資源。

**預算規劃的基本原則**：

以 128k Token 的上下文窗口為例，一個典型的 Token 分配可能是：

| 用途 | Token 預算 | 佔比 |
|------|-----------|------|
| System Prompt（SOUL.md 等） | 2,000 | 1.6% |
| USER.md | 500 | 0.4% |
| MEMORY.md | 1,500 | 1.2% |
| 對話歷史 | 100,000 | 78% |
| 工具輸出 | 20,000 | 15.6% |
| 回應生成空間 | 4,000 | 3.1% |

MEMORY.md 的預算通常在 1,000-3,000 Token 之間。超過這個範圍，載入記憶的邊際效益會急劇下降，因為它佔用了本可以用於對話歷史或工具輸出的空間。

**Token 計算技巧**：

中文的 Token 效率通常低於英文——一個中文字大約佔 1-2 個 Token，而一個英文單詞約佔 1-1.5 個 Token。因此，中文環境下的 MEMORY.md 需要更加注意簡潔性。

一些節省 Token 的技巧：
- 使用簡潔的表述方式（列點式而非段落式）
- 避免冗餘的格式標記
- 使用縮寫和慣用簡寫（在不影響理解的前提下）
- 合併相關的資訊點

### 6.3 優先級排序

不是所有記憶都同等重要。一個有效的優先級排序系統能確保最重要的記憶永遠被優先載入。

**優先級層級**：

- **P0（關鍵）**：直接影響 AI 行為的資訊。例如：硬性限制（「永遠不要自動執行 rm -rf」）、核心偏好（「回答時使用繁體中文」）
- **P1（重要）**：經常引用的資訊。例如：專案技術棧、團隊成員資訊
- **P2（有用）**：偶爾有用的資訊。例如：特定技術的經驗教訓、次要偏好
- **P3（背景）**：提供額外上下文但很少直接影響互動的資訊

當 Token 預算緊張時，從 P3 開始裁切，逐級向上。P0 級別的記憶應該永遠保留在載入內容中。

**動態優先級調整**：

記憶的優先級不是固定的，應該根據使用情況動態調整：
- 頻繁被引用的記憶 → 優先級上升
- 長時間未被引用的記憶 → 優先級下降
- 使用者明確標記為重要的 → 優先級上升
- 已被新資訊覆蓋的 → 優先級下降或刪除

## 八、最佳實踐與常見陷阱

### 8.1 最佳實踐

**1. 保持記憶的高信噪比**

記憶系統的價值在於信號（有用資訊）與噪音（無用資訊）的比例。一條高品質的記憶勝過十條模糊的記憶。

**2. 定期審計記憶**

建議設定一個固定的記憶審計流程：
- 每週快速掃描，移除明顯過時的條目
- 每月深度審計，整合和重組記憶結構
- 每季度重新評估記憶分類體系是否仍然合適

**3. 讓使用者參與記憶管理**

記憶不應該完全由 AI 自主管理。使用者應該能夠：
- 查看所有記憶內容
- 修正不準確的記憶
- 刪除不想被記住的資訊
- 標記特別重要的記憶

**4. 版本控制記憶**

將 MEMORY.md 納入 Git 版本控制，這樣可以：
- 追蹤記憶的變更歷史
- 在記憶被錯誤修改時回滾
- 比較不同時期的記憶狀態

**5. 為記憶添加來源標記**

每條記憶都應該標記其來源，這有助於：
- 評估記憶的可靠性（使用者直接告知 vs AI 自動推斷）
- 在記憶衝突時做出判斷
- 追蹤資訊的演化過程

### 8.2 常見陷阱

**1. 記憶膨脹（Memory Bloat）**

最常見的陷阱是記憶檔案不斷增長，直到嚴重影響效能。預防措施：
- 設定明確的 Token 預算上限
- 實施定期清理機制
- 優先記錄「改變行為的」資訊，而非「有趣但不改變行為的」資訊

**2. 記憶幻覺（Memory Hallucination）**

AI 可能基於不準確的記憶做出錯誤的假設。例如，如果 MEMORY.md 中有一條過時的記憶「使用者正在學習 React」，AI 可能會在使用者已經精通 React 之後仍然用入門的方式回答 React 問題。

預防措施：
- 為記憶添加日期標記
- 定期驗證記憶的準確性
- 允許使用者輕鬆修正記憶

**3. 隱私洩露（Privacy Leak）**

在多使用者環境中（如 Discord），需要特別注意記憶的隔離。一個使用者的記憶不應該在與另一個使用者的對話中被引用。

預防措施：
- 為每個使用者維護獨立的記憶空間
- 明確區分「公共記憶」和「私人記憶」
- 定期審計記憶存取日誌

**4. 記憶偏見（Memory Bias）**

AI 可能過度依賴記憶中的資訊，忽略了使用者當前的實際需求。例如，記憶中記錄了「使用者偏好簡短回答」，但使用者當前的問題可能需要詳細的解釋。

預防措施：
- 記憶應該作為參考而非硬性規則
- 關注使用者當前的信號（如追問、要求展開等）
- 在不確定時，以當前對話中的信號為準

**5. 冷啟動問題（Cold Start Problem）**

新使用者沒有任何記憶，AI 需要從零開始了解他們。這可能導致初始互動品質較低。

緩解措施：
- 提供合理的預設值（default settings）
- 在早期互動中主動收集關鍵資訊
- 提供「快速設定」引導，讓使用者一次性提供基本資訊

## 九、與傳統資料庫的比較

### 9.1 為什麼 OpenClaw 選擇 Markdown 而不是 SQLite

這是一個經常被問到的問題：既然有成熟的資料庫技術，為什麼要用一個簡單的文字檔案來管理記憶？

**Markdown 的優勢**：

| 方面 | Markdown | SQLite/資料庫 |
|------|----------|--------------|
| 可讀性 | 人類直接可讀 | 需要查詢工具 |
| 編輯成本 | 文字編輯器即可 | 需要 SQL 知識 |
| 部署複雜度 | 零依賴 | 需要資料庫引擎 |
| Git 整合 | 完美（純文字） | 困難（二進位檔案） |
| LLM 處理 | 原生支援 | 需要 ORM 層 |
| 遷移性 | 極高 | 中等 |
| 透明度 | 完全透明 | 需要額外工具查看 |

**SQLite/資料庫的優勢**：

| 方面 | Markdown | SQLite/資料庫 |
|------|----------|--------------|
| 查詢效率 | 需完整掃描 | 索引支援 |
| 結構化查詢 | 不支援 | SQL 查詢 |
| 大量資料處理 | 效能差 | 效能優良 |
| 並發存取 | 可能衝突 | 原生支援 |
| 資料完整性 | 無保證 | 約束和事務 |
| 關聯查詢 | 不支援 | 原生支援 |

### 9.2 什麼時候應該考慮資料庫

在以下情況下，可能需要考慮用資料庫替代或輔助 MEMORY.md：

1. **記憶條目超過數百條**：Markdown 檔案的管理和搜尋會變得困難
2. **多代理共享記憶**：多個 AI 代理需要同時讀寫同一套記憶
3. **複雜的查詢需求**：需要按多個維度篩選和排序記憶
4. **高並發場景**：多個使用者同時產生記憶寫入

### 9.3 混合方案

最務實的做法可能是混合方案：

- **MEMORY.md**：用於核心記憶（高頻存取、最重要的資訊）
- **資料庫（如 SQLite）**：用於擴展記憶（大量的歷史記錄、語意搜尋索引）

這種方案結合了兩者的優勢：MEMORY.md 提供高可讀性和透明度的核心記憶，而資料庫在背後支撐大規模的記憶檢索需求。

在我們的 Discord 機器人專案中，可能的架構是：

```
[啟動時載入]
├── MEMORY.md（核心記憶，< 2000 Token）
│   ├── 使用者核心偏好
│   ├── 重要人際關係摘要
│   └── 關鍵決策記錄
│
[需要時查詢]
├── SQLite 資料庫
│   ├── 完整的對話歷史
│   ├── 所有使用者的詳細 Profile
│   ├── 語意搜尋向量索引
│   └── 每日筆記歸檔
```

### 9.4 向量資料庫的角色

隨著 RAG（Retrieval Augmented Generation）技術的普及，向量資料庫（如 ChromaDB、Pinecone、Qdrant）在 AI 記憶系統中扮演著越來越重要的角色。

向量資料庫的核心價值在於**語意搜尋**——它能根據含義（而非精確的關鍵字匹配）來檢索記憶。這對於記憶系統特別有價值，因為使用者回憶某件事時，通常不會使用和原始記憶完全相同的措辭。

例如：
- 記憶：「使用者偏好 PostgreSQL 因為需要複雜的關聯查詢」
- 使用者問：「我之前選了什麼資料庫？」
- 傳統搜尋：可能匹配不到（因為「選了什麼資料庫」和「偏好 PostgreSQL」沒有共同關鍵字）
- 語意搜尋：能正確匹配（因為兩者在語意上相關）

在 OpenClaw 框架中，可以透過 MCP 工具整合向量資料庫，為 MEMORY.md 提供語意搜尋能力。這是一個值得深入探索的進階主題。

## 十、進階思考：記憶系統的未來

### 10.1 自我反思記憶

未來的記憶系統可能不僅僅記錄「事實」，還能記錄「反思」——AI 對自身行為和決策的回顧和評估。例如：

```markdown
### 反思：回答風格調整
- **日期**：2024-03-12
- **觀察**：在最近三次對話中，Eason 都要求我「說重點」。
- **推斷**：我的回答可能過於冗長。
- **調整**：將預設回答長度縮短 30%，先給結論再展開解釋。
- **效果追蹤**：待觀察
```

### 10.2 情感記憶

記憶不僅僅是事實的記錄，情感維度同樣重要。在 Discord 社交環境中，記住互動時的情感狀態能讓機器人更有溫度：

```markdown
### Alice 的情感時刻
- 2024-03-05：分享設計作品時非常興奮，大家的正面回饋讓她很開心
- 2024-03-08：提到工作壓力很大，可能需要更多鼓勵和支持
- 2024-03-12：完成了一個大專案，心情很好
```

### 10.3 記憶的遺忘曲線

靈感來自艾賓浩斯（Ebbinghaus）的遺忘曲線，未來的記憶系統可能會模擬人類的自然遺忘過程：

- 新鮮的記憶更容易被回憶
- 長時間未被訪問的記憶逐漸「褪色」
- 被多次回憶的記憶變得更加穩固
- 與強烈情感相關的記憶保持更久

這種機制可以自動解決記憶膨脹的問題——不重要的記憶會自然地被「遺忘」，而重要的記憶則因為反覆使用而被強化。

### 10.4 共享記憶與群體智慧

在多代理或多使用者的場景中，記憶的共享和整合是一個有趣的挑戰。例如，在一個 Discord 伺服器中：

- **個人記憶**：每個使用者的私人偏好和歷史
- **群組記憶**：整個社群的共同知識和文化
- **代理記憶**：AI 自身的經驗和學習成果

這三個層次的記憶如何有效地整合和管理，是一個值得深入研究的方向。

## 結語

MEMORY.md 作為 OpenClaw 框架的長期記憶系統，雖然在技術實現上相對簡單（就是一個 Markdown 檔案），但其設計背後蘊含著深刻的思考——關於記憶的本質、關於人機互動的哲學、關於如何在有限的技術約束下實現最大的使用者價值。

記憶是連結過去與未來的橋樑。一個有記憶的 AI，才能成為一個真正的「夥伴」而不僅僅是一個「工具」。在我們的 Discord 機器人專案中，MEMORY.md 將是賦予機器人「靈魂」的關鍵組件之一——讓它記住每個成員、理解社群文化、在每次互動中展現出持續成長的智慧。

在下一篇中，我們將探討 USER.md——另一個關鍵的啟動檔案，專注於定義使用者的身份和偏好。如果說 MEMORY.md 是 AI 的「日記」，那 USER.md 就是它手中關於你的「名片」——一份穩定的、結構化的使用者畫像。

> 「真正的智慧不在於知道所有事情，而在於記住重要的事情。」
