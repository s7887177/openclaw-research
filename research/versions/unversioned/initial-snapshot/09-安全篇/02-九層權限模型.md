## 1. 九層模型概覽

### 1.1 為什麼需要九層？

在傳統的 Web 應用中，權限通常只有兩三層（認證 → 授權 → 資源存取）。但 AI Agent 系統面臨的挑戰更加複雜：

- **多模型**：同一個系統可能使用 OpenAI、Anthropic、Ollama 等多個 AI 提供者
- **多代理**：同一個 Gateway 可能運行多個具有不同職責的 Agent
- **多工具**：每個 Agent 可以使用數十甚至上百個工具
- **動態性**：AI 模型的輸出不可預測，工具調用是動態決定的
- **鏈式攻擊**：一個被 compromise 的元件可能影響其他元件

九層模型的設計理念是：**在工具調用到達執行環境之前，必須通過九道獨立的安全閘門，每一道都有權拒絕**。

### 1.3 檢查流程

當一個工具調用請求進入九層檢查時，流程如下：

```
工具調用請求 (tool: "exec", args: {command: "ls -la"})
│
├─▶ 第 1 層：全域設定檔檢查
│   問：全域是否允許工具使用？
│   └── 通過 → 繼續
│
├─▶ 第 2 層：提供者基礎設定
│   問：這個 AI 提供者是否被允許使用工具？
│   └── 通過 → 繼續
│
├─▶ 第 3 層：全域工具清單
│   問：這個工具是否在全域允許清單中？
│   └── 通過 → 繼續
│
├─▶ 第 4 層：提供者工具清單
│   問：這個提供者是否被允許使用這個工具？
│   └── 通過 → 繼續
│
├─▶ 第 5 層：代理權限
│   問：這個 Agent 是否被允許使用這個工具？
│   └── 通過 → 繼續
│
├─▶ 第 6 層：代理+提供者組合
│   問：這個 Agent 使用這個提供者時，是否被允許使用這個工具？
│   └── 通過 → 繼續
│
├─▶ 第 7 層：沙箱隔離
│   問：這個工具是否需要在沙箱中執行？
│   └── 通過 → 繼續（可能重導向到沙箱）
│
├─▶ 第 8 層：參數約束
│   問：工具的參數是否在允許範圍內？
│   └── 通過 → 繼續
│
├─▶ 第 9 層：人工批准
│   問：這個工具是否需要人工確認？
│   └── 確認 → 執行工具
│
└── ✅ 執行工具
```

**關鍵原則**：**任何一層拒絕 = 整個請求被拒絕**。不存在「繞過」某一層的方式。

### 1.4 拒絕優先原則

```
規則解析優先順序：

1. denied 清單永遠優先於 allowed 清單
2. 如果一個工具同時出現在 allowed 和 denied 中，它會被拒絕
3. 明確拒絕（explicit deny）優先於隱式允許（implicit allow）
4. 萬用字元（*）的優先級最低

示例：
  allowed: ["*"]      → 允許所有
  denied: ["exec"]    → 但拒絕 exec
  結果：exec 被拒絕，其他都允許

  allowed: ["exec"]   → 允許 exec
  denied: ["exec"]    → 但也拒絕 exec
  結果：exec 被拒絕（denied 優先）
```

## 2. 第一層：全域設定檔/權限基礎層

### 2.1 概念

第一層是最基礎的安全閘門，定義了整個 Gateway 實例的基本安全態勢。它控制的是**全域開關**——是否允許工具使用、是否啟用安全功能等。

```
第 1 層的角色：
┌────────────────────────────────────┐
│ 「這個 Gateway 的基本安全設定是什麼？」 │
│                                      │
│ • 是否允許工具使用？                  │
│ • 安全模式是什麼（寬鬆/標準/嚴格）？  │
│ • 基本的速率限制？                    │
│ • 日誌等級？                          │
└────────────────────────────────────┘
```

### 2.2 配置範例

```json5
// openclaw.json5 — 第 1 層配置
{
  // 全域安全設定
  security: {
    // 安全模式
    // "permissive" = 寬鬆（適合開發）
    // "standard"   = 標準（推薦生產使用）
    // "strict"     = 嚴格（高安全需求）
    mode: "standard",

    // 全域工具開關
    tools_enabled: true,  // false = 完全禁用所有工具

    // 全域速率限制
    rate_limiting: {
      enabled: true,
      global_max_tool_calls_per_minute: 60,
      global_max_messages_per_minute: 30
    },

    // 內容安全
    content_safety: {
      enabled: true,
      // 過濾輸入中的可疑提示注入模式
      prompt_injection_detection: true,
      // 過濾輸出中的敏感資訊
      output_filtering: true
    },

    // 日誌設定
    logging: {
      level: "info",          // debug/info/warn/error
      tool_calls: true,       // 記錄所有工具調用
      permission_checks: true  // 記錄所有權限檢查
    }
  }
}
```

### 2.3 安全模式詳解

#### 寬鬆模式（Permissive）

```json5
{
  security: {
    mode: "permissive",
    // 效果：
    // • 未配置的工具預設允許
    // • 速率限制較寬鬆
    // • 警告但不阻止可疑操作
    // ⚠️ 僅用於開發和測試！
  }
}
```

**寬鬆模式的行為**：
- 未在 allowed/denied 清單中的工具 → **預設允許**
- 提示注入偵測 → 記錄警告但不阻止
- 速率限制 → 較高的限額

#### 標準模式（Standard）

```json5
{
  security: {
    mode: "standard",
    // 效果：
    // • 未配置的工具需要明確允許
    // • 危險工具（exec、browser）預設拒絕
    // • 啟用速率限制
    // • 啟用內容安全
  }
}
```

**標準模式的行為**：
- 未在 allowed 清單中的工具 → **預設拒絕**
- 危險工具類別（exec、shell、browser、file_write）→ 需要明確允許
- 提示注入偵測 → 阻止可疑請求
- 速率限制 → 標準限額

#### 嚴格模式（Strict）

```json5
{
  security: {
    mode: "strict",
    // 效果：
    // • 所有工具都需要明確允許
    // • 敏感操作需要人工確認
    // • 嚴格的速率限制
    // • 完整的審計日誌
  }
}
```

**嚴格模式的行為**：
- 所有工具 → 必須在 allowed 清單中才能使用
- 敏感工具 → 需要第 9 層人工確認
- 速率限制 → 較低的限額
- 所有操作 → 完整審計日誌

### 2.4 實際應用情境

**情境：開發階段快速測試**

```json5
{
  security: {
    mode: "permissive",   // 開發時用寬鬆模式
    tools_enabled: true,
    logging: {
      level: "debug"     // 詳細日誌幫助除錯
    }
  }
}
```

**情境：生產環境部署**

```json5
{
  security: {
    mode: "strict",
    tools_enabled: true,
    rate_limiting: {
      enabled: true,
      global_max_tool_calls_per_minute: 30,
      global_max_messages_per_minute: 10
    },
    content_safety: {
      enabled: true,
      prompt_injection_detection: true,
      output_filtering: true
    },
    logging: {
      level: "warn",
      tool_calls: true,
      permission_checks: true,
      audit_trail: true
    }
  }
}
```

### 2.5 最小權限原則在第一層的應用

```
✅ 正確做法：
  從 "strict" 模式開始
  → 根據需求逐一啟用功能
  → 每次啟用都記錄原因

❌ 錯誤做法：
  從 "permissive" 模式開始
  → 發現問題後逐一限制
  → 可能遺漏危險配置
```

## 3. 第二層：每個 AI 提供者基礎設定

### 3.1 概念

第二層控制每個 AI 模型提供者（OpenAI、Anthropic、Ollama 等）的基礎能力。不同的提供者可能有不同的安全特性和風險等級。

```
第 2 層的角色：
┌──────────────────────────────────────────┐
│ 「這個 AI 提供者被允許做什麼？」            │
│                                            │
│ • 這個提供者是否可以使用工具？              │
│ • 這個提供者的 token 限制是多少？           │
│ • 這個提供者是否需要額外的安全措施？        │
│ • 這個提供者的費用限制是多少？              │
└──────────────────────────────────────────┘
```

### 3.2 為什麼需要這一層？

不同的 AI 提供者有不同的安全風險：

| 提供者 | 風險等級 | 原因 |
|--------|----------|------|
| 本地 Ollama | 低 | 資料不離開本機 |
| OpenAI | 中 | 資料傳到 OpenAI 伺服器 |
| Anthropic | 中 | 資料傳到 Anthropic 伺服器 |
| Groq | 中 | 第三方服務 |
| 未知第三方 | 高 | 安全性未知 |

### 3.3 配置範例

```json5
{
  providers: {
    // OpenAI 提供者
    openai: {
      apiKey: "${OPENAI_API_KEY}",
      // 基礎安全設定
      security: {
        // 允許使用工具
        tools_enabled: true,
        // token 限制（每次請求）
        max_tokens_per_request: 4096,
        // 費用限制（每日美元）
        max_daily_cost: 10.00,
        // 允許的模型
        allowed_models: ["gpt-4", "gpt-4-turbo", "gpt-3.5-turbo"],
        // 禁止的模型
        denied_models: ["gpt-4-vision-preview"]  // 不允許視覺模型
      }
    },

    // Anthropic 提供者
    anthropic: {
      apiKey: "${ANTHROPIC_API_KEY}",
      security: {
        tools_enabled: true,
        max_tokens_per_request: 4096,
        max_daily_cost: 15.00,
        allowed_models: ["claude-3-opus", "claude-3-sonnet", "claude-3-haiku"]
      }
    },

    // 本地 Ollama（更寬鬆，因為資料不離開本機）
    ollama: {
      baseUrl: "http://localhost:11434",
      security: {
        tools_enabled: true,
        max_tokens_per_request: 8192,  // 本地可以更寬鬆
        // 無費用限制（本地運行）
      }
    },

    // 不受信任的第三方提供者
    "sketchy-provider": {
      apiKey: "${SKETCHY_API_KEY}",
      baseUrl: "https://ai.example.com",
      security: {
        tools_enabled: false,     // ⚠️ 禁止使用工具
        max_tokens_per_request: 1024,
        max_daily_cost: 1.00,
        // 不允許發送敏感上下文
        strip_system_prompt: true,
        strip_memory_context: true
      }
    }
  }
}
```

### 3.4 提供者級別的資料保護

```json5
{
  providers: {
    openai: {
      security: {
        // 資料保護設定
        data_protection: {
          // 不發送記憶內容（可能包含個人資訊）
          strip_memory: false,

          // 不發送系統提示（保護 SOUL.md 內容）
          strip_system_prompt: false,

          // 過濾敏感資料（電話、信用卡等）
          filter_pii: true,

          // 日誌中遮蔽 API 金鑰
          mask_api_key_in_logs: true
        }
      }
    },

    // 對於不信任的提供者，啟用所有保護
    "untrusted-provider": {
      security: {
        data_protection: {
          strip_memory: true,         // 不發送記憶
          strip_system_prompt: true,  // 不發送系統提示
          filter_pii: true,           // 過濾個人資訊
          max_context_tokens: 1024    // 限制上下文大小
        }
      }
    }
  }
}
```

### 3.5 實際應用情境

**情境：混合使用雲端和本地模型**

```json5
{
  providers: {
    // 雲端模型用於複雜任務，但限制敏感資料
    openai: {
      security: {
        tools_enabled: true,
        data_protection: {
          filter_pii: true
        },
        max_daily_cost: 20.00
      }
    },
    // 本地模型用於處理敏感資料
    ollama: {
      security: {
        tools_enabled: true
        // 無限制——資料不離開本機
      }
    }
  },

  agents: {
    // 一般助手使用雲端
    "general": {
      model: "openai/gpt-4"
    },
    // 處理敏感資料時使用本地模型
    "private": {
      model: "ollama/llama3",
      system_prompt_suffix: "你處理的資料是敏感的，不要嘗試使用外部 API。"
    }
  }
}
```

### 3.6 最小權限原則在第二層的應用

```
✅ 為每個提供者設定適當的限制
✅ 對不信任的提供者禁用工具
✅ 設定費用上限防止意外支出
✅ 對雲端提供者啟用 PII 過濾

❌ 所有提供者使用相同的寬鬆設定
❌ 允許所有提供者使用所有工具
❌ 不設定費用限制
❌ 向不信任的提供者發送完整上下文
```

## 4. 第三層：全域工具允許/拒絕清單

### 4.1 概念

第三層定義了**整個 Gateway 範圍內**哪些工具可以使用、哪些被禁止。這是工具權限的最外層防線。

```
第 3 層的角色：
┌────────────────────────────────────────────┐
│ 「在這個 Gateway 中，哪些工具是可用的？」     │
│                                              │
│ • 全域允許清單：明確允許的工具                │
│ • 全域拒絕清單：絕對禁止的工具                │
│ • 適用於所有 Agent、所有提供者                │
└────────────────────────────────────────────┘
```

### 4.2 配置範例

```json5
{
  // 全域工具設定
  tools: {
    // 全域允許清單
    // 只有在這個清單中的工具才可能被使用
    allowed: [
      "memory_*",       // 所有記憶相關工具
      "search_web",     // 網路搜尋
      "discord_*",      // 所有 Discord 工具
      "telegram_*",     // 所有 Telegram 工具
      "calendar_*",     // 日曆工具
      "reminder_*",     // 提醒工具
      "file_read"       // 只允許讀取，不允許寫入
    ],

    // 全域拒絕清單
    // 在這個清單中的工具絕對不能使用
    // denied 永遠優先於 allowed
    denied: [
      "exec",           // 禁止執行系統命令
      "shell",          // 禁止 shell 存取
      "file_write",     // 禁止寫入檔案
      "file_delete",    // 禁止刪除檔案
      "browser",        // 禁止瀏覽器（防止 SSRF）
      "email_send",     // 禁止發送郵件
      "admin_*"         // 禁止所有管理工具
    ]
  }
}
```

### 4.3 萬用字元匹配規則

```
萬用字元語法：
  *          → 匹配所有工具
  memory_*   → 匹配 memory_read, memory_write, memory_search 等
  *_read     → 匹配 file_read, memory_read, config_read 等
  discord_*  → 匹配 discord_send, discord_react, discord_voice 等

解析順序：
  1. 先檢查 denied 清單（包括萬用字元展開）
  2. 再檢查 allowed 清單（包括萬用字元展開）
  3. 如果都不匹配，根據安全模式決定：
     - permissive → 允許
     - standard/strict → 拒絕

範例：
  allowed: ["*"]           → 允許所有
  denied: ["exec"]         → 但拒絕 exec
  工具 "exec" → ❌ 拒絕（denied 優先）
  工具 "search" → ✅ 允許

  allowed: ["memory_*"]    → 只允許記憶工具
  denied: []               → 無特別拒絕
  工具 "memory_read" → ✅ 允許
  工具 "exec" → ❌ 拒絕（不在 allowed 中）
```

### 4.4 工具分類參考

```
安全等級：低風險
  memory_read      → 讀取記憶
  search_web       → 網路搜尋
  time_now         → 取得當前時間
  weather_*        → 天氣查詢

安全等級：中風險
  memory_write     → 寫入記憶（可能被注入虛假記憶）
  discord_send     → 發送 Discord 訊息
  telegram_send    → 發送 Telegram 訊息
  file_read        → 讀取檔案（可能讀到敏感資料）
  calendar_create  → 建立日曆事件

安全等級：高風險
  file_write       → 寫入檔案
  file_delete      → 刪除檔案
  browser          → 控制瀏覽器（SSRF 風險）
  email_send       → 發送郵件（社交工程風險）

安全等級：極高風險
  exec             → 執行系統命令（等同於 shell 存取）
  shell            → Shell 存取
  admin_*          → 管理操作
  config_write     → 修改配置（可能自我提權）
```

### 4.5 實際應用情境

**情境：只讀助手**

```json5
{
  tools: {
    allowed: [
      "memory_read",
      "memory_search",
      "search_web",
      "time_now",
      "weather_query"
    ],
    denied: ["*_write", "*_delete", "*_create", "exec", "shell", "browser"]
  }
}
```

**情境：開發助手**

```json5
{
  tools: {
    allowed: [
      "memory_*",
      "search_web",
      "file_read",
      "file_write",
      "code_*",
      "git_*"
    ],
    denied: ["exec", "shell", "browser", "email_*", "admin_*"]
  }
}
```

### 4.6 最小權限原則在第三層的應用

```
步驟 1：列出 Agent 需要的所有功能
  → 查詢天氣、管理記憶、回應 Discord

步驟 2：找到對應的最小工具集
  → weather_query, memory_read, memory_write, discord_send, discord_react

步驟 3：只允許這些工具
  → allowed: ["weather_query", "memory_*", "discord_send", "discord_react"]

步驟 4：明確拒絕危險工具
  → denied: ["exec", "shell", "browser", "file_*", "admin_*"]
```

## 5. 第四層：每個提供者允許/拒絕清單

### 5.1 概念

第四層在第三層的基礎上，為每個 AI 提供者設定額外的工具限制。即使一個工具在全域允許清單中，特定提供者也可能被禁止使用它。

```
第 4 層的角色：
┌──────────────────────────────────────────────┐
│ 「這個提供者被允許使用哪些工具？」              │
│                                                │
│ • 不信任的提供者 → 禁止使用任何工具             │
│ • 雲端提供者 → 禁止存取敏感檔案                │
│ • 本地提供者 → 可以存取本地資源                 │
└──────────────────────────────────────────────┘
```

### 5.2 配置範例

```json5
{
  providers: {
    openai: {
      apiKey: "${OPENAI_API_KEY}",
      tools: {
        // OpenAI 被允許使用的工具（在全域允許的前提下）
        allowed: [
          "memory_*",
          "search_web",
          "discord_*"
        ],
        // OpenAI 被禁止使用的工具
        denied: [
          "file_read",    // 不允許 OpenAI 模型讀取本地檔案
          "file_write"    // 不允許 OpenAI 模型寫入本地檔案
          // 原因：不希望本地檔案內容被送到 OpenAI 伺服器
        ]
      }
    },

    ollama: {
      baseUrl: "http://localhost:11434",
      tools: {
        // 本地模型可以使用更多工具（資料不外傳）
        allowed: ["*"],
        denied: ["admin_*"]  // 仍然禁止管理操作
      }
    },

    groq: {
      apiKey: "${GROQ_API_KEY}",
      tools: {
        // Groq 只允許使用搜尋和記憶工具
        allowed: ["search_web", "memory_read"],
        denied: ["*_write", "*_delete", "file_*"]
      }
    }
  }
}
```

### 5.3 層級疊加效果

```
全域（第 3 層）允許：["memory_*", "file_read", "search_web", "exec"]
全域（第 3 層）拒絕：["exec"]

OpenAI（第 4 層）允許：["memory_*", "search_web"]
OpenAI（第 4 層）拒絕：["file_read"]

有效權限計算：
  memory_read:
    全域允許 ✅ + 全域未拒絕 ✅ + OpenAI 允許 ✅ + OpenAI 未拒絕 ✅
    → ✅ 允許

  file_read:
    全域允許 ✅ + 全域未拒絕 ✅ + OpenAI 拒絕 ❌
    → ❌ 拒絕（第 4 層拒絕）

  exec:
    全域拒絕 ❌
    → ❌ 拒絕（第 3 層就被拒絕了，不會到第 4 層）

  search_web:
    全域允許 ✅ + 全域未拒絕 ✅ + OpenAI 允許 ✅ + OpenAI 未拒絕 ✅
    → ✅ 允許
```

### 5.4 實際應用情境

**情境：資料主權合規**

某些地區的法規要求敏感資料不得傳輸到境外：

```json5
{
  providers: {
    // 美國提供者——不允許存取個人資料
    openai: {
      tools: {
        denied: [
          "file_read",        // 不讀取本地檔案
          "memory_read",      // 不讀取記憶（可能包含個人資料）
          "contacts_*"        // 不存取聯絡人
        ]
      }
    },
    // 本地模型——可以存取個人資料
    ollama: {
      tools: {
        allowed: ["*"],
        denied: ["admin_*"]
      }
    }
  }
}
```

### 5.5 最小權限原則在第四層的應用

```
原則：每個提供者只應獲得其功能所需的最小工具集

OpenAI（用於對話和推理）：
  → 需要：memory_read, search_web, discord_send
  → 不需要：file_*, exec, admin_*

Ollama（用於本地資料處理）：
  → 需要：file_read, file_write, memory_*
  → 不需要：discord_*, telegram_*（本地不需要通道工具）
```

## 6. 第五層：每個代理權限策略

### 6.1 概念

第五層為每個 Agent 定義獨立的權限策略。不同的 Agent 有不同的職責，因此應該有不同的工具權限。

```
第 5 層的角色：
┌────────────────────────────────────────────────┐
│ 「這個 Agent 被允許做什麼？」                    │
│                                                  │
│ • Agent "社交助手" → 可以使用 Discord 工具       │
│ • Agent "程式助手" → 可以使用程式碼工具          │
│ • Agent "管理員" → 可以使用管理工具              │
│ • 每個 Agent 只有完成其任務所需的最小權限        │
└────────────────────────────────────────────────┘
```

### 6.2 配置範例

```json5
{
  agents: {
    // 社交助手——友善的聊天機器人
    "social-bot": {
      model: "openai/gpt-4",
      persona: "SOUL.md",
      tools: {
        allowed: [
          "memory_read",
          "memory_write",
          "search_web",
          "discord_send",
          "discord_react",
          "discord_voice_join",
          "discord_voice_speak"
        ],
        denied: [
          "file_*",
          "exec",
          "shell",
          "browser",
          "admin_*",
          "config_*"
        ]
      }
    },

    // 程式助手——幫助寫程式碼
    "code-helper": {
      model: "anthropic/claude-3-opus",
      tools: {
        allowed: [
          "file_read",
          "file_write",
          "code_analyze",
          "code_format",
          "git_status",
          "git_diff",
          "search_web",
          "memory_read"
        ],
        denied: [
          "exec",           // 不允許直接執行（用沙箱代替）
          "discord_*",      // 不需要通道工具
          "admin_*"
        ]
      },
      // 額外安全設定
      sandbox: {
        enabled: true,
        backend: "docker"
      }
    },

    // 管理員助手——需要更高權限
    "admin-agent": {
      model: "ollama/llama3",  // 用本地模型處理管理操作
      tools: {
        allowed: [
          "admin_status",
          "admin_logs",
          "config_read",
          "memory_*",
          "file_read"
        ],
        denied: [
          "exec",         // 即使管理員也不能直接執行命令
          "config_write"  // 不能修改配置（需要直接編輯檔案）
        ]
      },
      // 必須通過 DM 配對才能使用
      require_pairing: true
    }
  }
}
```

### 6.3 代理繼承與覆蓋

```json5
{
  // 預設代理設定（所有代理繼承）
  agents: {
    // "default" 是特殊鍵，定義所有代理的預設值
    default: {
      tools: {
        allowed: ["memory_read", "search_web"],
        denied: ["exec", "shell", "admin_*"]
      },
      sandbox: {
        enabled: false
      }
    },

    // 具體代理可以覆蓋預設值
    "code-helper": {
      // 這個 Agent 在 default 基礎上添加工具
      tools: {
        allowed: [
          "memory_read",     // 從 default 繼承
          "search_web",      // 從 default 繼承
          "file_read",       // 新增
          "file_write",      // 新增
          "code_*"           // 新增
        ],
        denied: [
          "exec",            // 從 default 繼承
          "shell",           // 從 default 繼承
          "admin_*"          // 從 default 繼承
        ]
      },
      sandbox: {
        enabled: true        // 覆蓋 default 的 false
      }
    }
  }
}
```

### 6.4 實際應用情境

**情境：一個 Gateway 服務多個 Discord 頻道**

```json5
{
  agents: {
    // #general 頻道的 Agent——寬鬆、友善
    "general-chat": {
      model: "openai/gpt-4",
      tools: {
        allowed: ["memory_*", "search_web", "discord_send", "discord_react"],
        denied: ["file_*", "exec", "admin_*"]
      }
    },
    // #code-review 頻道的 Agent——專業、有程式碼權限
    "code-review": {
      model: "anthropic/claude-3-opus",
      tools: {
        allowed: ["file_read", "code_*", "git_*", "memory_read", "discord_send"],
        denied: ["file_write", "exec", "admin_*"]  // 只讀！
      }
    },
    // #admin 頻道的 Agent——限制存取
    "admin-channel": {
      model: "ollama/llama3",
      tools: {
        allowed: ["admin_status", "admin_logs", "memory_read"],
        denied: ["exec", "file_*"]
      },
      require_pairing: true
    }
  },

  channels: {
    discord: {
      channel_routing: {
        "${GENERAL_ID}": { agent: "general-chat" },
        "${CODE_REVIEW_ID}": { agent: "code-review" },
        "${ADMIN_ID}": { agent: "admin-channel" }
      }
    }
  }
}
```

### 6.5 最小權限原則在第五層的應用

```
設計每個 Agent 時問自己：
1. 這個 Agent 的職責是什麼？
2. 完成這個職責需要哪些工具？
3. 有沒有更安全的替代方案？
4. 如果這個 Agent 被 compromise，影響範圍是什麼？

範例：
  Agent: "weather-bot"
  職責：回答天氣相關問題
  需要的工具：weather_query, discord_send
  不需要的工具：其他所有
  → allowed: ["weather_query", "discord_send"]
  → denied: ["*"]  // 其他全部拒絕
```

## 7. 第六層：代理+提供者組合限制

### 7.1 概念

第六層處理一個更細粒度的問題：**特定 Agent 使用特定提供者時**的工具限制。這允許你為同一個 Agent 在不同模型提供者下設定不同的權限。

```
第 6 層的角色：
┌──────────────────────────────────────────────────────┐
│ 「這個 Agent 使用這個提供者時，被允許做什麼？」        │
│                                                        │
│ • Agent A + OpenAI → 限制檔案存取                     │
│ • Agent A + Ollama → 允許檔案存取                     │
│ • Agent B + OpenAI → 允許搜尋                         │
│ • Agent B + Ollama → 允許搜尋 + 檔案                  │
└──────────────────────────────────────────────────────┘
```

### 7.2 為什麼需要這一層？

考慮以下場景：

```
Agent "code-helper" 可能使用 OpenAI 或 Ollama：

使用 OpenAI 時：
  → 不希望本地程式碼被傳到 OpenAI
  → 限制 file_read 工具
  → 只允許分析已公開的程式碼

使用 Ollama 時：
  → 資料不離開本機
  → 可以自由讀取本地檔案
  → 可以使用完整的程式碼分析工具
```

### 7.3 配置範例

```json5
{
  agents: {
    "code-helper": {
      // 預設模型
      model: "openai/gpt-4",
      // 可以回退到本地模型
      fallback_model: "ollama/codellama",

      tools: {
        // Agent 層級的基本權限
        allowed: ["code_*", "memory_read", "search_web"],
        denied: ["exec", "admin_*"]
      },

      // 第 6 層：per-provider 覆蓋
      provider_overrides: {
        openai: {
          tools: {
            // 使用 OpenAI 時的額外限制
            denied: [
              "file_read",     // 不讀取本地檔案
              "file_write",    // 不寫入本地檔案
              "git_diff"       // 不傳送 diff 資訊
            ]
          }
        },
        ollama: {
          tools: {
            // 使用 Ollama 時放寬限制
            allowed: [
              "code_*",
              "memory_read",
              "search_web",
              "file_read",     // 允許讀取
              "file_write",    // 允許寫入
              "git_*"          // 允許完整 git 操作
            ]
          }
        }
      }
    }
  }
}
```

### 7.4 權限計算示例

```
Agent "code-helper" 調用工具 "file_read"：

使用 OpenAI 時：
  L1: 全域啟用工具 ✅
  L2: OpenAI 啟用工具 ✅
  L3: 全域允許 file_read ✅
  L4: OpenAI 未拒絕 file_read ✅
  L5: Agent 允許 code_* ✅（假設 file_read 不在 code_* 中）
      → ❌ 拒絕（Agent 層級未允許 file_read）

  即使 L5 允許了，L6 也會拒絕：
  L6: OpenAI override 拒絕 file_read ❌

使用 Ollama 時：
  L1: 全域啟用工具 ✅
  L2: Ollama 啟用工具 ✅
  L3: 全域允許 file_read ✅
  L4: Ollama 未拒絕 file_read ✅
  L5: Agent 基本允許 ✅（假設添加了 file_read）
  L6: Ollama override 允許 file_read ✅
  → ✅ 允許
```

### 7.5 最小權限原則在第六層的應用

```
對於雲端提供者（OpenAI、Anthropic、Groq）：
  → 限制可能外傳敏感資料的工具
  → 限制 file_read（防止原始碼外傳）
  → 限制 memory_read（防止個人資料外傳）

對於本地提供者（Ollama）：
  → 可以放寬限制
  → 但仍要限制 exec、admin 等危險工具
```

## 8. 第七層：沙箱隔離權限

### 8.1 概念

第七層不是傳統的允許/拒絕檢查，而是一種**執行環境隔離**。即使工具通過了前六層的檢查，第七層決定工具是在主環境還是在沙箱環境中執行。

```
第 7 層的角色：
┌──────────────────────────────────────────────┐
│ 「這個工具應該在哪裡執行？」                    │
│                                                │
│ • 安全工具（memory_read）→ 在主環境中執行      │
│ • 有風險工具（file_write）→ 在沙箱中執行       │
│ • 高風險工具（exec）→ 在嚴格限制的沙箱中執行   │
└──────────────────────────────────────────────┘
```

### 8.2 配置範例

```json5
{
  sandbox: {
    // 全域沙箱設定
    enabled: true,
    backend: "docker",      // docker | ssh | local
    scope: "per-session",   // per-agent | per-session | shared

    // 沙箱資源限制
    limits: {
      memory: "512M",
      cpu: "1.0",
      disk: "1G",
      timeout: 120,         // 秒
      network: true,        // 允許網路存取
      network_allowlist: [  // 只允許這些網路目標
        "api.openai.com",
        "api.anthropic.com",
        "*.googleapis.com"
      ]
    },

    // 哪些工具在沙箱中執行
    sandboxed_tools: [
      "exec",
      "shell",
      "file_write",
      "file_delete",
      "code_execute",
      "browser"
    ],

    // 哪些工具不在沙箱中（直接在主環境執行）
    unsandboxed_tools: [
      "memory_*",
      "search_web",
      "discord_*",
      "telegram_*"
    ]
  },

  // 提權工具——在沙箱中需要額外權限的工具
  tools: {
    elevated: [
      {
        tool: "file_write",
        // 允許寫入的路徑
        allowed_paths: ["/workspace/*"],
        // 禁止寫入的路徑
        denied_paths: ["/etc/*", "/usr/*", "/var/*"]
      }
    ]
  }
}
```

### 8.3 沙箱與權限的交互

```
工具 "exec" 被調用：

沒有沙箱：
  exec("rm -rf /important")
  → 直接在 Gateway 的 OS 環境中執行
  → ⚠️ 危險！可能刪除重要檔案

有沙箱：
  exec("rm -rf /important")
  → 在 Docker 容器中執行
  → 容器只有 /workspace 目錄
  → /important 不存在或不可寫
  → 影響範圍限制在容器內
```

### 8.4 實際應用情境

**情境：程式碼執行沙箱**

```json5
{
  agents: {
    "code-runner": {
      model: "openai/gpt-4",
      tools: {
        allowed: ["code_execute", "file_read", "file_write"],
        denied: ["exec", "shell"]  // 只允許 code_execute，不允許任意命令
      },
      sandbox: {
        enabled: true,
        backend: "docker",
        scope: "per-session",
        image: "python:3.11-slim",  // 使用 Python 映像
        limits: {
          memory: "256M",
          cpu: "0.5",
          timeout: 30,
          network: false  // 禁止網路（程式碼執行不需要）
        },
        volumes: {
          // 只掛載工作目錄
          "/workspace": { host_path: "./sandbox-workspace", mode: "rw" }
        }
      }
    }
  }
}
```

### 8.5 最小權限原則在第七層的應用

```
沙箱設計原則：
1. 預設在沙箱中執行所有有副作用的工具
2. 沙箱資源限制應盡可能嚴格
3. 網路存取預設禁止，需要時明確允許
4. 檔案系統存取限制在工作目錄
5. 沙箱生命週期與 session 綁定
```

> 詳細的沙箱配置請參閱 [第 3 章：沙箱與隔離](./03-沙箱與隔離.md)

## 9. 第八層：每個工具參數與約束

### 9.1 概念

第八層是最細粒度的權限控制——它不僅檢查「是否允許使用這個工具」，還檢查「使用這個工具時的參數是否在允許範圍內」。

```
第 8 層的角色：
┌──────────────────────────────────────────────────┐
│ 「這個工具的參數是否合法？」                       │
│                                                    │
│ • file_read(path="/etc/passwd") → ❌ 路徑不允許   │
│ • file_read(path="/workspace/code.py") → ✅ 允許  │
│ • exec(command="rm -rf /") → ❌ 命令不允許        │
│ • exec(command="ls -la") → ⚠️ 需要進一步檢查     │
└──────────────────────────────────────────────────┘
```

### 9.2 配置範例

```json5
{
  tools: {
    constraints: {
      // file_read 工具的約束
      file_read: {
        params: {
          path: {
            // 允許的路徑模式
            allowed_patterns: [
              "/workspace/**",
              "/home/eason/projects/**",
              "./data/**"
            ],
            // 禁止的路徑模式
            denied_patterns: [
              "/etc/**",
              "/root/**",
              "/var/**",
              "**/.env",
              "**/*secret*",
              "**/*password*",
              "**/*key*"
            ]
          }
        }
      },

      // file_write 工具的約束
      file_write: {
        params: {
          path: {
            allowed_patterns: ["/workspace/**"],
            denied_patterns: ["**/*.exe", "**/*.sh", "**/*.bat"]
          },
          // 檔案大小限制
          max_size: "10MB"
        }
      },

      // exec 工具的約束
      exec: {
        params: {
          command: {
            // 允許的命令模式
            allowed_patterns: [
              "ls *",
              "cat *",
              "grep *",
              "python3 *.py",
              "node *.js"
            ],
            // 禁止的命令模式
            denied_patterns: [
              "rm *",
              "sudo *",
              "chmod *",
              "chown *",
              "curl *",
              "wget *",
              "nc *",
              "dd *",
              "> *",           // 重定向
              "| *",           // 管道（可能被濫用）
              "&& *",          // 命令鏈
              "$(*)  ",        // 命令替換
              "`*`"            // 反引號命令替換
            ]
          },
          // 執行時間限制
          timeout: 30
        }
      },

      // search_web 工具的約束
      search_web: {
        params: {
          query: {
            // 限制搜尋查詢長度
            max_length: 200,
            // 禁止的搜尋模式（防止資訊洩漏）
            denied_patterns: [
              "*api_key*",
              "*password*",
              "*secret*",
              "*credentials*"
            ]
          }
        },
        // 速率限制
        rate_limit: {
          max_calls_per_minute: 10,
          max_calls_per_hour: 100
        }
      },

      // discord_send 工具的約束
      discord_send: {
        params: {
          content: {
            max_length: 2000,  // Discord 訊息長度限制
            denied_patterns: [
              "@everyone",     // 禁止 @everyone
              "@here"          // 禁止 @here
            ]
          },
          channel_id: {
            // 只允許發送到特定頻道
            allowed_values: [
              "${GENERAL_CHANNEL_ID}",
              "${BOT_CHANNEL_ID}"
            ]
          }
        }
      },

      // memory_write 工具的約束
      memory_write: {
        params: {
          content: {
            max_length: 10000,
            // 防止寫入可能影響 Agent 行為的內容
            denied_patterns: [
              "*ignore previous*",
              "*system prompt*",
              "*你的指令*",
              "*忽略之前*"
            ]
          },
          // 每日寫入次數限制
          rate_limit: {
            max_calls_per_day: 100
          }
        }
      }
    }
  }
}
```

### 9.3 參數驗證引擎

```
參數驗證流程：

1. 類型檢查
   → 參數是否是預期的類型（string, number, boolean）？

2. 長度/範圍檢查
   → 字串長度、數字範圍是否在限制內？

3. 模式匹配檢查
   → 參數值是否匹配 allowed_patterns？
   → 參數值是否匹配 denied_patterns？（被拒絕）

4. 特殊驗證
   → 路徑穿越檢測（../../../etc/passwd）
   → 命令注入檢測（; rm -rf /）
   → 編碼繞過檢測（%2e%2e%2f = ../）
```

### 9.4 路徑穿越防護

```json5
{
  tools: {
    constraints: {
      file_read: {
        params: {
          path: {
            // 基本路徑限制
            allowed_patterns: ["/workspace/**"],

            // 自動防護（內建）
            auto_protection: {
              // 拒絕包含 .. 的路徑
              block_path_traversal: true,
              // 解析符號連結後再檢查
              resolve_symlinks: true,
              // 標準化路徑（移除 ./ 和 //）
              normalize_path: true
            }
          }
        }
      }
    }
  }
}
```

```
嘗試路徑穿越：
  file_read(path="/workspace/../../etc/passwd")
  → 標準化後：/etc/passwd
  → 不在 allowed_patterns 中
  → ❌ 拒絕

嘗試符號連結繞過：
  /workspace/link → /etc/passwd
  file_read(path="/workspace/link")
  → 解析符號連結：/etc/passwd
  → 不在 allowed_patterns 中
  → ❌ 拒絕
```

### 9.5 實際應用情境

**情境：限制 Agent 只能操作特定專案**

```json5
{

    constraints: {
      file_read: {
        params: {
          path: {
            allowed_patterns: ["/home/eason/projects/my-app/**"],
            denied_patterns: [
              "**/.env",
              "**/.git/config",
              "**/node_modules/**",
              "**/*.key",
              "**/*.pem"
            ]
          }
        }
      },
      file_write: {
        params: {
          path: {
            allowed_patterns: ["/home/eason/projects/my-app/src/**"],
            denied_patterns: ["**/*.js", "**/*.json"]  // 只允許修改 TypeScript
          }
        }
      }
    }
  }
```

### 9.6 最小權限原則在第八層的應用

```
原則：每個工具的參數都應該被約束到最小必要範圍

file_read:
  ✅ 只允許讀取工作目錄
  ❌ 允許讀取整個檔案系統

exec:
  ✅ 只允許特定安全命令（ls, cat, grep）
  ❌ 允許任意命令

discord_send:
  ✅ 只允許發送到特定頻道
  ❌ 允許發送到任何頻道

search_web:
  ✅ 限制查詢長度和頻率
  ❌ 無限制的搜尋
```

## 10. 第九層：批准工作流（人工確認）

### 10.1 概念

第九層是最後一道安全閘門，也是最強大的一道——**人工批准**。即使工具通過了前八層的所有檢查，仍然可以要求人工確認後才執行。

```
第 9 層的角色：
┌──────────────────────────────────────────────────┐
│ 「操作者是否批准這個操作？」                       │
│                                                    │
│ • 敏感工具 → 每次使用都需要確認                   │
│ • 高風險參數 → 根據參數值決定是否需要確認         │
│ • 金額超過閾值 → 需要確認                         │
│ • 不可逆操作 → 需要確認                           │
└──────────────────────────────────────────────────┘
```

### 10.2 配置範例

```json5
{
  security: {
    approval: {
      enabled: true,

      // 需要人工批准的工具
      required_for: [
        "file_write",
        "file_delete",
        "exec",
        "email_send",
        "admin_*",
        "config_write"
      ],

      // 批准通知方式
      notify: {
        // 透過 DM 通知操作者
        method: "dm",
        // 或者透過特定頻道
        // method: "channel",
        // channel_id: "${APPROVAL_CHANNEL_ID}",
      },

      // 批准超時
      timeout: 120,  // 秒，超時後自動拒絕

      // 批准訊息模板
      message_template: `
⚠️ 需要批准

工具：{tool_name}
參數：{tool_args}
Agent：{agent_name}
原因：{reason}

回覆 ✅ 批准 或 ❌ 拒絕
      `,

      // 條件式批准（只在特定條件下需要批准）
      conditional: [
        {
          tool: "file_write",
          // 只有在非工作目錄寫入時需要批准
          condition: "args.path NOT MATCHES '/workspace/**'",
          required: true
        },
        {
          tool: "exec",
          // 所有 exec 調用都需要批准
          required: true
        },
        {
          tool: "discord_send",
          // 當訊息包含 @mention 時需要批准
          condition: "args.content MATCHES '.*@.*'",
          required: true
        }
      ]
    }
  }
}
```

### 10.3 批准流程

```
Agent 想要執行 file_write("/important/config.yaml", "new content")

L1-L8: 全部通過 ✅

L9: 檢查是否需要批准
  → file_write 在 required_for 清單中
  → 需要人工批准

流程：
  1. Gateway 暫停工具執行
  2. 發送 DM 給操作者：
     "⚠️ Agent 想要寫入 /important/config.yaml，是否批准？"
  3. 等待操作者回應（最多 120 秒）
  4a. 操作者回覆 ✅ → 執行工具
  4b. 操作者回覆 ❌ → 拒絕執行
  4c. 超時 → 自動拒絕

           ┌──────────┐
           │ Agent 請求│
           │ 工具調用  │
           └────┬─────┘
                │
                ▼
           ┌──────────┐
           │ L1-L8    │
           │ 權限檢查  │──── ❌ 拒絕
           └────┬─────┘
                │ ✅
                ▼
           ┌──────────┐
           │ L9 需要   │──── 不需要 ──▶ 執行工具
           │ 批准？    │
           └────┬─────┘
                │ 需要
                ▼
           ┌──────────┐
           │ 發送通知  │
           │ 給操作者  │
           └────┬─────┘
                │
                ▼
           ┌──────────┐
           │ 等待回應  │──── 超時 ──▶ 拒絕
           └────┬─────┘
                │
           ┌────┴────┐
           │         │
           ▼         ▼
       ✅ 批准    ❌ 拒絕
           │         │
           ▼         ▼
       執行工具   回報錯誤
```

### 10.4 批准記憶——智慧批准

```json5
{
  security: {
    approval: {
      // 智慧批准功能
      smart_approval: {
        enabled: true,

        // 記住之前的批准決定
        remember_decisions: true,

        // 記住多久
        decision_ttl: "1h",  // 1 小時內相同的操作不再詢問

        // 自動批准已知安全模式
        auto_approve_patterns: [
          {
            tool: "file_write",
            pattern: "args.path MATCHES '/workspace/temp/**'",
            // 寫入暫存目錄自動批准
          }
        ],

        // 永遠不自動批准的操作
        never_auto_approve: [
          "file_delete",
          "exec",
          "admin_*"
        ]
      }
    }
  }
}
```

### 10.5 批量批准

```
場景：Agent 需要寫入 50 個檔案來完成一個任務

逐一批准 → 操作者要確認 50 次 → 體驗極差

批量批准方案：
  Gateway 發送：
  "⚠️ Agent 請求批量寫入 50 個檔案到 /workspace/output/
   檔案列表：file1.py, file2.py, ...（顯示前 10 個）
   全部批准 / 全部拒絕 / 逐一審查"

  操作者選擇 "全部批准" → 50 個檔案都執行
```

```json5
{
  security: {
    approval: {
      // 批量批准設定
      batch_approval: {
        enabled: true,
        // 同一工具、相同模式的連續調用可以批量處理
        max_batch_size: 100,
        // 批量批准超時
        timeout: 300  // 5 分鐘
      }
    }
  }
}
```

### 10.6 最小權限原則在第九層的應用

```
原則：不可逆和高風險操作必須有人工批准

必須批准的操作：
  ✅ file_delete    → 刪除不可逆
  ✅ exec           → 命令可能有破壞性
  ✅ email_send     → 發送後無法撤回
  ✅ admin_*        → 管理操作影響大
  ✅ config_write   → 可能影響安全設定

可以不批准的操作：
  ✅ memory_read    → 只讀操作
  ✅ search_web     → 只讀操作
  ✅ discord_send   → 可以刪除訊息（可逆）

條件式批准：
  ✅ file_write(path="/workspace/*") → 不需要批准
  ✅ file_write(path="/etc/*") → 需要批准
```

## 11. 九層協同運作機制

### 11.1 完整檢查流程範例

讓我們追蹤一個完整的工具調用，看九層如何協同運作：

```
場景：Agent "code-helper" 使用 OpenAI 模型，
      想要執行 file_write("/workspace/output.py", "print('hello')")

L1: 全域設定
  → security.mode = "standard" ✅
  → tools_enabled = true ✅
  → 通過

L2: OpenAI 提供者設定
  → openai.security.tools_enabled = true ✅
  → 通過

L3: 全域工具清單
  → tools.allowed = ["*"]（假設允許所有）✅
  → tools.denied = ["exec", "shell"]（file_write 不在拒絕清單）✅
  → 通過

L4: OpenAI 提供者工具清單
  → openai.tools.denied = ["file_read"]（file_write 不在拒絕清單）✅
  → 通過

L5: Agent 權限
  → code-helper.tools.allowed = ["file_read", "file_write", "code_*"] ✅
  → code-helper.tools.denied = ["exec"] ✅
  → 通過

L6: Agent + OpenAI 組合
  → code-helper.provider_overrides.openai.tools.denied 未包含 file_write ✅
  → 通過

L7: 沙箱
  → file_write 在 sandboxed_tools 中
  → 重導向到 Docker 沙箱執行
  → 通過（在沙箱中）

L8: 參數約束
  → path = "/workspace/output.py"
  → allowed_patterns = ["/workspace/**"] ✅
  → denied_patterns 不匹配 ✅
  → 內容長度檢查 ✅
  → 通過

L9: 批准工作流
  → file_write 在 conditional 中
  → condition: "args.path MATCHES '/workspace/**'" → 自動批准
  → 通過

最終結果：✅ 在 Docker 沙箱中執行 file_write
```

### 11.2 拒絕案例追蹤

```
場景：Agent "social-bot" 被提示注入，
      嘗試執行 exec("cat /etc/passwd")

L1: 全域設定
  → security.mode = "strict" ✅
  → tools_enabled = true ✅
  → 通過

L2: OpenAI 提供者設定
  → 通過

L3: 全域工具清單
  → tools.denied = ["exec"] ❌
  → ❌ 在第 3 層被拒絕！

結果：❌ 拒絕
日誌：[SECURITY] Tool call denied at Layer 3 (Global Tool List):
      agent=social-bot, tool=exec, reason=tool in global denied list
```

### 11.3 九層效能影響

```
每次工具調用的權限檢查時間：

L1: < 0.1ms（記憶體中的配置查詢）
L2: < 0.1ms（記憶體中的配置查詢）
L3: < 0.5ms（萬用字元匹配）
L4: < 0.5ms（萬用字元匹配）
L5: < 0.5ms（萬用字元匹配）
L6: < 0.5ms（萬用字元匹配）
L7: < 1ms（沙箱路由決策）
L8: < 2ms（正則表達式匹配和路徑驗證）
L9: 0ms（如果不需要批准）或 0-120s（等待人工）

總計（不含 L9）：< 5ms
→ 對使用者體驗幾乎無影響
```

## 12. 最小權限原則實踐指南

### 12.1 逐步限縮法

```
步驟 1：從全部拒絕開始
  allowed: []
  denied: ["*"]

步驟 2：列出 Agent 需要的功能
  功能清單：查天氣、記住事情、在 Discord 回覆

步驟 3：找到對應的最小工具集
  weather_query, memory_read, memory_write, discord_send

步驟 4：只允許這些工具
  allowed: ["weather_query", "memory_read", "memory_write", "discord_send"]
  denied: ["*"]

步驟 5：設定參數約束
  discord_send:
    channel_id: 只允許特定頻道
    content: 最大長度 2000
  memory_write:

步驟 6：決定是否需要批准
  memory_write: 不需要（低風險）
  discord_send: 不需要（可逆）

步驟 7：測試和驗證
  openclaw security audit
```

## 13. 常見配置模式

### 13.1 模式一：只讀助手

```json5
// 最安全的配置——Agent 只能讀取和回覆
{
  security: { mode: "strict" },
  tools: {
    allowed: ["memory_read", "search_web", "time_now"],
    denied: ["*_write", "*_delete", "*_create", "exec", "shell", "browser"]
  },
  agents: {
    default: {
      tools: { denied: ["*_write", "*_delete", "exec"] }
    }
  }
}
```

### 13.2 模式二：開發助手

```json5
// 適合程式碼輔助的配置

  },

    enabled: true,
    sandboxed_tools: ["file_write", "code_execute"]
  }
}
```

### 13.3 模式三：社交機器人

```json5
// 適合 Discord/Telegram 社交場景

```

## 14. 除錯與疑難排解

### 14.1 工具被拒絕但不知道是哪一層

```bash
# 啟用詳細的權限檢查日誌
openclaw config set logging.permission_checks true
openclaw config set logging.level debug

# 查看日誌
tail -f ./logs/openclaw.log | grep "PERMISSION"

# 日誌格式：
# [DEBUG] PERMISSION CHECK: tool=file_write agent=code-helper provider=openai
# [DEBUG]   L1 (Global Settings): PASS
# [DEBUG]   L2 (Provider Settings): PASS
# [DEBUG]   L3 (Global Tool List): PASS
# [DEBUG]   L4 (Provider Tool List): FAIL - tool in provider denied list
# [DEBUG] RESULT: DENIED at Layer 4
```

### 14.2 權限配置衝突

```
問題：設定了 allowed: ["*"] 但工具仍被拒絕

可能原因：
1. 更高層級的 denied 清單包含該工具
2. 安全模式為 "strict" 且沒有明確允許
3. 工具不存在或名稱拼寫錯誤
4. 提供者級別的限制

除錯步驟：
1. openclaw security audit --deep
2. 檢查每一層的配置
3. 使用 openclaw call --dry-run tool_name 測試權限
```

### 14.3 常見錯誤

```
錯誤：Cannot call tool 'exec': denied by Layer 3 (Global Tool List)
原因：exec 在全域拒絕清單中
修復：如果確實需要，從 denied 中移除（⚠️ 考慮風險）

錯誤：Cannot call tool 'file_write': denied by Layer 8 (Parameter Constraints)
原因：file_write 的 path 參數不符合允許模式
修復：檢查 path 是否在 allowed_patterns 中

錯誤：Tool call timed out waiting for approval
原因：第 9 層的人工批准超時
修復：增加 timeout 或調整需要批准的工具列表
```

## 總結

九層權限模型是 OpenClaw 安全架構的核心，它提供了從粗粒度到細粒度的完整權限控制：

| 層 | 粒度 | 控制範圍 |
|----|------|----------|
| L1 | 最粗 | 整個 Gateway |
| L2 | 粗 | 每個提供者 |
| L3 | 中粗 | 全域工具 |
| L4 | 中 | 每個提供者的工具 |
| L5 | 中細 | 每個 Agent |
| L6 | 細 | Agent + 提供者組合 |
| L7 | 細 | 執行環境 |
| L8 | 很細 | 工具參數 |
| L9 | 最細 | 逐次操作批准 |

**核心原則**：
- 拒絕優先（denied > allowed）
- 最小權限（只允許需要的）
- 縱深防禦（九層獨立檢查）
- 可審計（每層都記錄日誌）
