## 引言

在人工智慧代理（AI Agent）領域快速演進的今天，如何構建一個既強大又可控的個人 AI 助理系統，已成為技術社群的核心議題。市面上多數 AI Agent 框架選擇雲端優先（Cloud-first）的路線，將使用者資料託管於遠端伺服器，透過 API 呼叫完成所有運算。這種方式雖然降低了部署門檻，卻犧牲了隱私性、可控性與透明度。

OpenClaw 的誕生正是為了回應這些挑戰。它是一個以「本地優先」為核心理念的 AI Agent 架構，旨在讓使用者能在自己的硬體上運行完整的 AI 助理系統，同時保持模組化的彈性與完全的可審計性。本文件將全面介紹 OpenClaw 的系統架構，從設計哲學到具體實作，從資料流到進程模型，提供一份完整的技術參考。

OpenClaw 的命名靈感來自於「爪」（Claw）的意象——一個能夠靈巧抓取、操作、執行各種任務的機械臂。而「Open」則強調其開源、開放的本質。整個系統的設計目標是：讓任何有基本技術能力的使用者，都能在自己的電腦上擁有一個功能完整、可自訂、可擴展的 AI 助理。

本文件是架構篇的第一份文件，後續章節將深入探討每個核心模組的設計細節。建議讀者先完整閱讀本文件，建立對系統全貌的理解，再依興趣深入個別模組。

## 設計哲學：本地優先、模組化、透明可審計

### 本地優先（Local-first）

OpenClaw 的第一設計原則是「本地優先」。這不僅僅是一個技術選擇，更是一個價值觀的體現。在這個原則下，系統做出了以下承諾：

**資料主權（Data Sovereignty）**

所有使用者資料——包括對話記錄、記憶檔案、設定檔、技能定義——全部儲存在使用者的本地檔案系統中。沒有任何資料會在未經使用者明確授權的情況下離開本機。這意味著：

- 對話歷史以純文字檔案（Markdown）格式儲存於 `~/.openclaw/memory/` 目錄
- 系統設定以 YAML 格式存放於 `~/.openclaw/config/` 目錄
- 技能定義以 Markdown 與腳本檔案形式存放於 `~/.openclaw/skills/` 目錄
- 所有檔案都可以用任何文字編輯器直接查看和修改

這種「檔案即資料庫」的設計，使得使用者可以隨時透過 `cat`、`grep`、`find` 等基本 Unix 工具來檢視系統狀態。不需要特殊的管理工具或 GUI 介面就能理解系統正在做什麼。

**離線能力（Offline Capability）**

當使用本地模型（如 Ollama）時，OpenClaw 可以完全離線運作。即使使用雲端 LLM 服務（如 Claude、GPT），系統的核心功能——包括記憶管理、排程、技能執行——都不依賴外部服務。網路中斷只影響 LLM 推理，不影響系統運行。

**運算本地化（Compute Locality）**

除了 LLM 推理可能需要遠端 API 外，所有其他運算都在本地完成：
- 訊息路由在本地 Gateway 進程中處理
- 記憶壓縮與提升在本地文件系統上執行
- 技能腳本在本地 Shell 環境中運行
- 向量搜尋可使用本地向量資料庫

```
┌─────────────────────────────────────────────┐
│              使用者的電腦                      │
│                                               │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │ Gateway  │  │  Brain  │  │ Memory  │      │
│  │ Process  │  │ Engine  │  │  Files  │      │
│  └─────────┘  └─────────┘  └─────────┘      │
│  ┌─────────┐  ┌─────────┐                    │
│  │Heartbeat│  │ Skills  │                    │
│  │ Daemon  │  │ Scripts │                    │
│  └─────────┘  └─────────┘                    │
│                                               │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─   │
│  唯一的外部連線：LLM API（可選）               │
└─────────────────────────────────────────────┘
          │ HTTPS（僅 LLM 推理）
          ▼
   ┌──────────────┐
   │  LLM Provider │
   │  (Claude/GPT) │
   └──────────────┘
```

### 模組化（Modularity）

OpenClaw 的第二設計原則是嚴格的模組化。系統被分解為五個核心模組，每個模組都有清晰定義的職責邊界和介面契約：

**單一職責原則（Single Responsibility Principle）**

每個模組只負責一個核心功能：
- **Gateway**：訊息路由與連線管理
- **Brain**：LLM 推理與回應生成
- **Memory**：資訊儲存與檢索
- **Heartbeat**：定時排程與自動觸發
- **Skills**：外部能力擴展與工具調用

**介面契約（Interface Contracts）**

模組之間透過明確定義的 JSON Schema 訊息格式進行通訊。這意味著：
- 每個模組都可以獨立開發、測試和部署
- 模組的內部實作可以自由更換，只要符合介面契約
- 新模組可以輕鬆接入系統，只需實作對應的介面
- 第三方開發者可以替換任何模組而不影響其他部分

```yaml
# 訊息介面範例
message:
  type: "brain.request"
  version: "1.0"
  payload:
    session_id: "sess_abc123"
    user_input: "幫我查看今天的行程"
    context:
      memory_snippets: [...]
      active_skills: [...]
```

**插件式架構（Plugin Architecture）**

Skills 模組本身就是插件式架構的體現。每個技能都是一個獨立的目錄，包含定義檔（SKILL.md）、執行腳本（scripts/）和參考資料（references/）。新增技能就像新增一個目錄一樣簡單，移除技能只需刪除對應目錄。

**鬆耦合（Loose Coupling）**

模組之間透過 Gateway 進行間接通訊，避免直接依賴。這種設計的好處是：
- Brain 不需要知道訊息來自哪個頻道（Telegram、LINE、CLI）
- Skills 不需要知道是哪個 LLM 模型決定調用它
- Memory 不需要知道記憶是被哪個流程觸發寫入
- Heartbeat 不需要知道 Brain 的內部狀態

### 透明可審計（Transparent & Auditable）

OpenClaw 的第三設計原則是透明可審計。這個原則確保使用者能夠完全理解和控制系統的行為：

**人類可讀的格式（Human-readable Formats）**

系統中的每一份資料都以人類可讀的格式儲存：
- 記憶檔案使用 Markdown 格式
- 設定檔使用 YAML 格式
- 技能定義使用 Markdown + YAML frontmatter 格式
- 日誌檔案使用結構化純文字格式

不使用任何二進位格式或私有格式。使用者可以用任何文字編輯器打開和修改系統中的任何檔案。

**完整的日誌追蹤（Full Audit Trail）**

系統的每一個操作都會被記錄：
- 每次 LLM 呼叫的完整提示詞（Prompt）和回應
- 每次技能調用的參數和結果
- 每次記憶寫入的內容和觸發原因
- 每次 Heartbeat 觸發的時間和執行動作

```
~/.openclaw/
├── logs/
│   ├── gateway.log        # Gateway 操作日誌
│   ├── brain.log          # LLM 呼叫日誌
│   ├── memory.log         # 記憶操作日誌
│   ├── heartbeat.log      # 排程執行日誌
│   └── skills.log         # 技能調用日誌
├── audit/
│   ├── prompts/           # 完整提示詞歷史
│   └── decisions/         # 決策追蹤記錄
```

**決策可解釋性（Decision Explainability）**

OpenClaw 的 Brain 模組在做出每個決策時，都會記錄推理過程。使用者可以查看：
- 為什麼系統選擇調用某個技能
- 提示詞是如何被組裝的
- 記憶檢索返回了哪些上下文
- 模型的原始回應（未經後處理）

這種透明度使得系統行為完全可審計，也方便開發者進行除錯和優化。

**版本控制友好（VCS-friendly）**

由於所有資料都是純文字格式，整個 `~/.openclaw/` 目錄可以直接用 Git 進行版本控制。這意味著：
- 設定變更可以追蹤和回滾
- 記憶演化可以被記錄
- 技能修改有完整的變更歷史
- 多設備間可以透過 Git 同步

## 五大核心模組簡介

OpenClaw 系統由五個核心模組構成，這五個模組各司其職，共同形成一個完整的 AI Agent 架構。以下是每個模組的概覽。

### Gateway（閘道器）——中央編排器

Gateway 是 OpenClaw 的「神經中樞」，負責所有訊息的接收、路由和分派。它是系統中唯一暴露對外介面的元件，所有外部通訊都必須經過 Gateway。

**核心職責：**
- 啟動並維護 WebSocket 伺服器（預設 `ws://localhost:18789`）
- 管理所有客戶端連線的生命週期
- 驗證入站訊息的格式與安全性（JSON Schema 驗證）
- 將訊息路由到正確的內部處理器
- 管理頻道適配器（Channel Adapters）的註冊與健康監測
- 處理錯誤回覆和超時管理

Gateway 採用 Hub/Spoke 拓撲設計，作為中央 Hub 連接所有 Spoke（頻道適配器和內部模組）。這種設計確保了單點管控和一致的安全策略。

```
         ┌──────────┐
         │ Telegram │
         │ Adapter  │───┐
         └──────────┘   │
         ┌──────────┐   │    ┌──────────┐
         │   LINE   │───┼───▶│          │
         │ Adapter  │   │    │ Gateway  │
         └──────────┘   │    │  (Hub)   │
         ┌──────────┐   │    │          │
         │   CLI    │───┘    └──────────┘
         │ Adapter  │
         └──────────┘
```

Gateway 模組的詳細設計請參閱 [02-Gateway-中央編排器.md](./02-Gateway-中央編排器.md)。

### Brain（大腦）——推理引擎

Brain 是 OpenClaw 的「思考中心」，封裝了與大型語言模型（LLM）的所有互動。它負責將使用者輸入轉化為結構化的提示詞，發送給 LLM，並解析模型回應為可執行的動作。

**核心職責：**
- 組裝完整的提示詞（系統提示 + 記憶上下文 + 對話歷史 + 使用者輸入）
- 管理上下文窗口，確保不超過模型的 token 限制
- 執行 ReAct（Reasoning + Acting）代理迴圈
- 抽象化不同的 LLM 提供者（Anthropic、OpenAI、Google、Ollama）
- 處理串流回應（Streaming Response）
- 管理模型備援（Fallback）策略
- 控制推理參數（溫度、最大 token、top-p 等）

Brain 模組的一個重要設計特點是「提供者抽象層」（Provider Abstraction Layer）。無論底層使用哪個 LLM 服務，Brain 對外呈現的介面都是一致的。這使得切換模型——從 Claude 換到 GPT，或從雲端模型換到本地 Ollama——只需修改配置檔，不需要改動任何程式碼。

```
┌─────────────────────────────────────────┐
│                 Brain                    │
│                                          │
│  ┌──────────┐  ┌────────────────────┐   │
│  │ Prompt   │  │   Provider Layer   │   │
│  │ Assembly │  │                    │   │
│  │          │  │  ┌──────┐ ┌─────┐ │   │
│  │ System + │  │  │Claude│ │ GPT │ │   │
│  │ Memory + │  │  └──────┘ └─────┘ │   │
│  │ History +│  │  ┌──────┐ ┌─────┐ │   │
│  │ Input    │  │  │Gemini│ │Ollam│ │   │
│  └──────────┘  │  └──────┘ └─────┘ │   │
│                │                    │   │
│  ┌──────────┐  └────────────────────┘   │
│  │  ReAct   │                           │
│  │  Agent   │                           │
│  │  Loop    │                           │
│  └──────────┘                           │
└─────────────────────────────────────────┘
```

Brain 模組的詳細設計請參閱 [03-Brain-推理引擎.md](./03-Brain-推理引擎.md)。

### Memory（記憶）——記憶系統

Memory 是 OpenClaw 的「長期記憶庫」，負責資訊的持久化儲存與智慧檢索。它實作了一個多層次的記憶架構，從短暫的會話上下文到長久的核心記憶，確保 Agent 能夠累積和利用歷史資訊。

**核心職責：**
- 管理多層次記憶：會話記憶、每日筆記、長期記憶
- 執行記憶壓縮（Compaction），將冗長的對話壓縮為摘要
- 執行記憶提升（Promotion），將重要資訊從每日筆記提升到長期記憶
- 管理 Token 預算，確保檢索的記憶不超過上下文窗口限制
- 提供語意搜尋（Semantic Search）和關鍵字搜尋
- 維護「檔案即記憶」的設計原則

Memory 模組最獨特的設計是「檔案優先」（File-first Approach）。所有記憶都以 Markdown 檔案形式儲存，而非存放在傳統資料庫中。這樣的設計帶來了極高的透明度和可控性——使用者可以直接閱讀、編輯、搜尋記憶檔案，就像操作任何普通的文字檔案一樣。

```
~/.openclaw/memory/
├── MEMORY.md              # 長期核心記憶
├── daily/
│   ├── 2025-01-15.md     # 每日筆記
│   ├── 2025-01-16.md
│   └── ...
├── sessions/
│   ├── sess_abc123.md    # 會話記憶
│   └── ...
└── vectors/               # 向量索引（可選）
    └── embeddings.db
```

Memory 模組的詳細設計請參閱 [04-Memory-記憶系統.md](./04-Memory-記憶系統.md)。

### Heartbeat（心跳）——自主排程系統

Heartbeat 是 OpenClaw 的「自主驅動器」，讓 Agent 能夠不依賴使用者輸入而主動執行任務。它定期「喚醒」系統，檢查是否有需要執行的排程任務或維護工作。

**核心職責：**
- 定期觸發系統自檢（預設每 30 分鐘一次）
- 執行配置的排程任務（定義於 HEARTBEAT.md）
- 自動進行記憶維護（壓縮、清理過期會話）
- 生成每日報告和健康摘要
- 執行備份和資料整理
- 實施「降噪」策略——只在有重要事項時通知使用者

Heartbeat 模組解決了大多數 AI Agent 的一個根本限制：它們只能在使用者主動提問時才能運作。有了 Heartbeat，OpenClaw 變成了一個「主動式 Agent」，能夠自發地執行任務、監控狀態、提供提醒。

```
時間線：
──────────────────────────────────────────────▶
 │       │       │       │       │       │
 ▼       ▼       ▼       ▼       ▼       ▼
 💓      💓      💓      💓      💓      💓
 心跳    心跳    心跳    心跳    心跳    心跳
 #1      #2      #3      #4      #5      #6
 │               │                       │
 ▼               ▼                       ▼
 執行            執行                    執行
 記憶壓縮        每日報告                備份
```

Heartbeat 模組的詳細設計請參閱 [05-Heartbeat-自主排程.md](./05-Heartbeat-自主排程.md)。

### Skills（技能）——可擴展技能系統

Skills 是 OpenClaw 的「工具箱」，提供了一個標準化的框架來擴展 Agent 的能力。每個技能都是一個獨立的單元，定義了 Agent 可以執行的特定操作。

**核心職責：**
- 管理技能的發現與註冊
- 提供技能調用的標準化介面
- 執行技能的安全門控（Gating）
- 管理技能的啟用/停用狀態
- 支援技能市集（ClawHub）的整合
- 處理技能的依賴管理

Skills 模組最突出的設計特色是「用 Markdown 定義工具」。每個技能的核心定義檔是一個 `SKILL.md` 文件，其中包含 YAML frontmatter（定義元資料和參數）和 Markdown 正文（定義指令和範例）。這種設計使得技能的開發和分享變得極其簡單。

```
~/.openclaw/skills/
├── weather/
│   ├── SKILL.md           # 技能定義
│   ├── scripts/
│   │   └── get_weather.sh # 執行腳本
│   └── references/
│       └── api_docs.md    # 參考文件
├── calendar/
│   ├── SKILL.md
│   └── scripts/
│       └── manage_cal.py
└── git-helper/
    ├── SKILL.md
    └── scripts/
        └── git_ops.sh
```

Skills 模組的詳細設計請參閱 [06-Skills-技能系統.md](./06-Skills-技能系統.md)。

## 系統資料流

理解 OpenClaw 的運作方式，最直觀的方法是追蹤一條訊息從使用者輸入到系統回應的完整路徑。以下是詳細的資料流圖和每個步驟的說明。

### 完整資料流圖

```
使用者輸入（例如：Telegram 訊息 "明天天氣如何？"）
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│ ① 頻道適配器（Channel Adapter）                          │
│    - 接收原始訊息                                        │
│    - 轉換為統一的 OpenClaw 訊息格式                       │
│    - 透過 WebSocket 發送至 Gateway                        │
└──────────────────────────────────────────────────────────┘
    │ WebSocket（ws://localhost:18789）
    ▼
┌──────────────────────────────────────────────────────────┐
│ ② Gateway（中央編排器）                                   │
│    - JSON Schema 驗證訊息格式                              │
│    - 認證發送者身份                                       │
│    - 記錄審計日誌                                        │
│    - 路由訊息至 Router                                    │
```

### 資料流步驟詳解

**步驟 ①：頻道適配器接收**

頻道適配器是 OpenClaw 與外部世界的橋樑。每個支援的通訊頻道（Telegram、LINE、Discord、CLI 等）都有一個對應的適配器。適配器的職責是：

1. 監聽來自外部頻道的原始訊息
2. 將頻道特有的訊息格式轉換為 OpenClaw 統一格式
3. 處理頻道特有的功能（如 Telegram 的 inline keyboard、LINE 的 Flex Message）

```json
// 統一訊息格式
{
  "type": "message.incoming",
  "channel": "telegram",
  "sender": {
    "id": "user_12345",
    "name": "Eason"
  },
  "content": {
    "type": "text",
    "text": "明天天氣如何？"
  },
  "metadata": {
    "timestamp": "2025-01-16T10:30:00Z",
    "message_id": "msg_67890",
    "channel_specific": {
      "chat_id": -100123456,
      "telegram_message_id": 42
    }
  }
}
```

**步驟 ②：Gateway 驗證與路由**

Gateway 收到訊息後，首先進行一系列驗證：
- JSON Schema 驗證：確認訊息格式符合預定義的 Schema
- 身份驗證：確認發送者是否為已授權的使用者或裝置
- 速率限制：檢查是否超過配置的訊息頻率限制
- 大小限制：確認訊息不超過允許的最大大小

通過驗證後，Gateway 將訊息寫入審計日誌，並將其轉發給 Router。

**步驟 ③：Router 分析與分派**

Router 分析訊息內容，決定處理路徑：
- 以 `/` 開頭的訊息被視為系統指令（如 `/status`、`/memory`）
- 包含 `@` 提及的訊息可能需要特殊處理
- 一般文字訊息被路由至 Brain 進行推理

**步驟 ④：Brain 推理**

Brain 接收到使用者輸入後，執行以下流程：

1. **記憶檢索**：向 Memory 模組查詢與當前輸入相關的記憶片段
2. **提示詞組裝**：將系統提示、記憶上下文、對話歷史和使用者輸入組合成完整的提示詞
3. **Token 管理**：檢查組裝後的提示詞是否超過模型的上下文窗口，必要時進行截斷
4. **LLM 呼叫**：將提示詞發送給配置的 LLM 服務
5. **回應解析**：分析 LLM 回應，判斷是純文字回應還是包含工具調用

**步驟 ⑤⑥：工具調用與 ReAct 迴圈**

如果 LLM 回應中包含工具調用（例如查詢天氣的請求），Brain 會啟動 ReAct 迴圈：

1. 解析工具調用請求（工具名稱和參數）
2. 將請求轉發給 Skills 模組
3. Skills 模組執行對應的技能腳本
4. 將執行結果返回給 Brain
5. Brain 將結果加入上下文，再次呼叫 LLM
6. 重複直到 LLM 生成最終的自然語言回應

```
ReAct 迴圈示意：

Brain: "使用者問明天天氣，我需要調用天氣查詢工具"
  → 調用 weather.get_forecast(location="台北", date="明天")
  → 結果：{temp: "25-30°C", condition: "晴天", humidity: "65%"}
Brain: "我已獲得天氣資料，可以生成回應了"
  → 生成："明天台北天氣晴，氣溫 25-30°C，濕度約 65%"
```

**步驟 ⑦：記憶寫入**

對話完成後，Memory 模組執行以下操作：
- 將完整的對話記錄附加到當前會話記憶檔
- 分析對話內容，提取可能重要的資訊
- 將重要資訊寫入今天的每日筆記
- 如果會話記憶超過設定的大小閾值，觸發壓縮

**步驟 ⑧⑨：回應返回**

最終回應沿著相反的路徑返回：
- Brain → Gateway → 頻道適配器 → 使用者
- 頻道適配器負責將統一格式的回應轉換為頻道特定的格式
- 例如，在 Telegram 中可能會使用 Markdown 格式化，在 LINE 中可能會使用 Flex Message

### 併行處理

值得注意的是，OpenClaw 支援多個請求的併行處理。Gateway 可以同時處理來自不同頻道或不同使用者的多個請求。每個請求都有獨立的會話上下文，不會互相干擾。

```
User A (Telegram)  ─┐
                    ├──▶ Gateway ──▶ Brain Instance 1 ──▶ Response A
User B (LINE)      ─┤
                    ├──▶ Gateway ──▶ Brain Instance 2 ──▶ Response B
User A (CLI)       ─┘
                    └──▶ Gateway ──▶ Brain Instance 3 ──▶ Response C
```

不過，由於 LLM 呼叫通常是系統的瓶頸，預設配置下 Brain 會對併行的 LLM 呼叫進行限流，避免超過 API 的速率限制或耗盡本地計算資源。

## 進程模型：單一 Gateway 守護程式與 Hub/Spoke 拓撲

### 進程架構概覽

OpenClaw 採用了精簡的進程模型。在最小部署情況下，整個系統只需要一個進程——Gateway 守護程式。所有核心模組（Brain、Memory、Heartbeat、Skills）都作為 Gateway 進程內的模組運行，而非獨立進程。

```
┌─────────────────────────────────────────┐
│           Gateway Process (PID 1234)     │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │         WebSocket Server           │  │
│  │      ws://localhost:18789          │  │
│  └────────────────────────────────────┘  │
│                                          │
│  ┌──────┐  ┌──────┐  ┌──────────┐      │
│  │Router│  │Brain │  │ Memory   │      │
│  └──────┘  └──────┘  └──────────┘      │
│                                          │
│  ┌──────────┐  ┌──────────┐             │
│  │Heartbeat │  │ Skills   │             │
│  └──────────┘  └──────────┘             │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │        Event Loop (async)          │  │
│  └────────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

這種「單一進程」的設計帶來了顯著的優勢：

**部署簡單**：只需啟動一個進程，不需要編排多個服務。沒有容器、沒有 Kubernetes、沒有服務發現。一條指令即可啟動整個系統：

```bash
$ openclaw start
# 或
$ openclaw daemon --port 18789
```

**資源高效**：避免了多進程之間的通訊開銷（IPC overhead）。所有模組共享同一個進程的記憶體空間，模組間的函數調用比跨進程通訊快了數個量級。

**狀態一致**：單一進程意味著不需要分散式狀態管理。所有模組可以共享同一個事件迴圈，確保操作的順序性和一致性。

**除錯方便**：一個進程意味著一份日誌、一個 PID、一個堆疊追蹤。當出現問題時，所有資訊都在同一個地方。

### Hub/Spoke 拓撲

雖然核心模組在同一個進程中運行，但頻道適配器（Channel Adapters）作為獨立進程運行，透過 WebSocket 連接到 Gateway。這形成了經典的 Hub/Spoke 拓撲：

```
                    ┌──────────────┐
                    │  Telegram    │
                    │  Adapter     │ ← 獨立進程
                    │  (PID 2001)  │
                    └──────┬───────┘
                           │ WebSocket
        ┌──────────────┐   │   ┌──────────────┐
        │  LINE         │   │   │  Discord     │
        │  Adapter      │───┼───│  Adapter     │
        │  (PID 2002)   │   │   │  (PID 2003)  │
        └──────┬───────┘   │   └──────┬───────┘
               │           │          │
               │     ┌─────┴─────┐    │
               └─────│  Gateway  │────┘
                     │  (Hub)    │
                     │ (PID 1234)│
                     └───────────┘
                           │
                     ┌─────┴─────┐
                     │   CLI     │
                     │  Adapter  │ ← 可能是同一進程
                     │           │   或獨立進程
                     └───────────┘
```

**Hub/Spoke 拓撲的優勢：**

1. **隔離性**：每個頻道適配器在獨立進程中運行。如果 Telegram 適配器崩潰，LINE 適配器和核心系統完全不受影響。
2. **獨立升級**：可以單獨更新某個頻道適配器，而不需要重啟整個系統。
3. **資源管控**：可以為每個適配器設定獨立的資源限制（CPU、記憶體）。
4. **安全隔離**：頻道適配器只能透過 Gateway 的 WebSocket 介面存取系統，無法直接存取記憶檔案或技能腳本。
5. **彈性擴展**：需要支援新的通訊頻道？只需開發一個新的適配器並連接到 Gateway。

### 生命週期管理

Gateway 守護程式的生命週期如下：

```
啟動階段：
1. 讀取配置檔 (~/.openclaw/config/gateway.yaml)
2. 初始化內部模組（Brain、Memory、Heartbeat、Skills）
3. 啟動 WebSocket 伺服器
4. 啟動 Heartbeat 計時器
5. 等待頻道適配器連線
6. 記錄啟動完成日誌

運行階段：
- 事件迴圈持續運行
- 處理入站訊息
- 執行排程任務
- 監控系統健康

關閉階段：
1. 收到關閉信號（SIGTERM / SIGINT）
2. 停止接受新連線
3. 等待進行中的請求完成（優雅關閉）
4. 通知所有已連線的適配器
5. 執行最後的記憶寫入
6. 關閉 WebSocket 伺服器
7. 記錄關閉完成日誌
```

### 擴展進程模型

雖然預設的單一進程模型適合大多數場景，但 OpenClaw 也支援分散式部署模式，適用於需要更高性能或可用性的場景：

**分離式 Brain**：Brain 模組可以作為獨立進程運行，透過 WebSocket 連接到 Gateway。這在需要使用 GPU 加速本地模型推理時特別有用。

**分離式 Memory**：Memory 模組可以連接到外部向量資料庫（如 Qdrant、Milvus），將語意搜尋能力擴展到大規模記憶庫。

**多 Gateway**：在需要支援大量使用者的部署場景中，可以運行多個 Gateway 實例，使用負載平衡器分發流量。

```
進階部署範例：

┌──────────┐     ┌──────────┐     ┌──────────┐
│ Gateway  │     │ Gateway  │     │ Gateway  │
│ Node 1   │     │ Node 2   │     │ Node 3   │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     └────────┬───────┘────────┬───────┘
              │                │
         ┌────┴────┐    ┌─────┴─────┐
         │  Brain  │    │  Memory   │
         │ Server  │    │  Service  │
         │ (GPU)   │    │ (Vector)  │
         └─────────┘    └───────────┘
```

不過，對於個人使用場景（OpenClaw 的主要目標場景），單一進程模型幾乎總是最佳選擇。它足以處理個人使用的工作負載，同時保持了架構的簡單性和可維護性。

## 通訊協議

### WebSocket 協議

OpenClaw 選擇 WebSocket 作為核心通訊協議，而非 HTTP REST 或 gRPC。這個選擇有幾個重要的理由：

**全雙工通訊**：WebSocket 支援伺服器主動推送訊息，這對於串流回應和 Heartbeat 通知至關重要。使用 HTTP 的話，每次推送都需要使用長輪詢（Long Polling）或 Server-Sent Events（SSE），增加了不必要的複雜性。

**低延遲**：WebSocket 連線建立後，訊息的傳輸開銷極小（沒有 HTTP 頭部的重複傳輸）。對於需要即時互動的 AI 助理場景，這一點尤為重要。

**持久連線**：WebSocket 是持久連線，適合需要長期維持的頻道適配器連線。頻道適配器連上 Gateway 後，連線會一直保持，直到適配器關閉或網路中斷。

**原生支援多種語言**：大多數程式語言都有成熟的 WebSocket 函式庫，方便開發者用任何語言實作頻道適配器。

```
WebSocket 連線流程：

Client                    Gateway
  │                          │
  │──── HTTP Upgrade ───────▶│
  │                          │
  │◀─── 101 Switching ──────│
  │     Protocols            │
  │                          │
  │──── Auth Message ───────▶│
  │     {"type":"auth",      │
  │      "token":"xxx"}      │
  │                          │
  │◀─── Auth Confirm ───────│
  │     {"type":"auth.ok"}   │
  │                          │
  │◀────── 雙向通訊 ────────▶│
  │                          │
```

### JSON Schema 驗證

所有經由 WebSocket 傳輸的訊息都必須符合預定義的 JSON Schema。這提供了幾個重要的保障：

**型別安全（Type Safety）**

每個訊息類型都有嚴格定義的結構，包括必要欄位和可選欄位、每個欄位的資料型別、允許的取值範圍、巢狀物件的結構。

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "OpenClaw Message",
  "type": "object",
  "required": ["type", "version", "payload"],
  "properties": {
    "type": {
      "type": "string",
      "enum": [
        "message.incoming",
        "message.outgoing",
        "brain.request",
        "brain.response",
        "skill.invoke",
        "skill.result",
        "heartbeat.trigger",
        "system.status"
      ]
    },
    "version": {
      "type": "string",
      "pattern": "^\\d+\\.\\d+$"
    },
    "payload": {
      "type": "object"
    },
    "metadata": {
      "type": "object",
      "properties": {
        "timestamp": {"type": "string", "format": "date-time"},
        "request_id": {"type": "string"},
        "sender_id": {"type": "string"}
      }
    }
  }
}
```

**安全防護（Security）**

JSON Schema 驗證還起到了安全防護的作用：
- 防止格式畸形的訊息進入系統
- 限制欄位的大小和數量，防止記憶體溢出攻擊
- 過濾不在白名單中的訊息類型
- 確保必要的安全欄位（如 sender_id）不會被遺漏

**文件即規格（Schema as Documentation）**

JSON Schema 本身就是訊息格式的完整文件。開發者可以直接閱讀 Schema 檔案來理解訊息結構，而不需要額外的 API 文件。

### 型別化訊息

OpenClaw 定義了一套完整的型別化訊息系統。每種訊息類型都有明確的語義和用途：

| 訊息類型 | 方向 | 用途 |
|---------|------|------|
| `message.incoming` | 適配器 → Gateway | 使用者輸入的訊息 |
| `message.outgoing` | Gateway → 適配器 | 系統生成的回應 |
| `brain.request` | Gateway → Brain | 推理請求 |
| `brain.response` | Brain → Gateway | 推理回應 |
| `brain.stream` | Brain → Gateway | 串流回應片段 |
| `skill.invoke` | Brain → Skills | 技能調用請求 |
| `skill.result` | Skills → Brain | 技能執行結果 |
| `memory.query` | Brain → Memory | 記憶查詢 |
| `memory.write` | Brain → Memory | 記憶寫入 |
| `heartbeat.trigger` | Heartbeat → Brain | 心跳觸發通知 |
| `heartbeat.result` | Brain → Heartbeat | 心跳處理結果 |
| `system.status` | 任何 → 任何 | 系統狀態報告 |
| `system.error` | 任何 → 任何 | 錯誤報告 |
| `auth.request` | 適配器 → Gateway | 認證請求 |
| `auth.response` | Gateway → 適配器 | 認證結果 |

每種訊息類型都有對應的 JSON Schema 定義，確保型別安全。

### 訊息信封格式

所有 OpenClaw 訊息都遵循統一的「信封」格式：

```json
{
  "type": "message.incoming",
  "version": "1.0",
  "id": "msg_2025011610300001",
  "timestamp": "2025-01-16T10:30:00.000Z",
  "sender": {
    "type": "adapter",
    "id": "telegram-adapter-001",
    "name": "Telegram Adapter"
  },
  "payload": {
    // 訊息類型特定的資料
  },
  "metadata": {
    "trace_id": "trace_abc123",
    "correlation_id": "corr_def456",
    "ttl": 30000
  }
}
```

- `type`：訊息類型，決定了 payload 的結構
- `version`：協議版本，支援向後相容
- `id`：唯一訊息識別碼
- `timestamp`：訊息建立時間（ISO 8601 格式）
- `sender`：發送者資訊
- `payload`：訊息本體（結構依類型而定）
- `metadata`：元資料（追蹤 ID、關聯 ID、存活時間等）

## 與傳統微服務架構的比較

為了更好地理解 OpenClaw 的架構定位，我們將其與傳統的微服務架構進行比較。

### 架構對比表

| 維度 | 傳統微服務架構 | OpenClaw 架構 |
|------|-------------|-------------|
| **進程模型** | 多個獨立服務進程 | 單一 Gateway 進程 + 獨立頻道適配器 |
| **通訊方式** | HTTP REST / gRPC / 訊息佇列 | WebSocket + JSON Schema |
| **服務發現** | Consul / etcd / Kubernetes DNS | 不需要（固定 Hub 地址） |
| **負載平衡** | Nginx / HAProxy / K8s Service | 不需要（單一進程處理） |
| **狀態管理** | 分散式快取（Redis） | 本地檔案系統 |
| **資料儲存** | PostgreSQL / MongoDB / S3 | Markdown 檔案 + 可選向量資料庫 |
| **部署方式** | Container Orchestration | 單一二進位檔 + systemd |
| **可觀測性** | Prometheus + Grafana + Jaeger | 結構化日誌檔案 |
| **擴展方式** | 水平擴展（多實例） | 垂直擴展 + 選擇性分離 |
| **複雜度** | 高（需要大量基礎設施） | 低（最小依賴） |
| **目標場景** | 高流量生產服務 | 個人/小團隊 AI 助理 |
| **啟動時間** | 數分鐘（容器啟動、服務註冊） | 數秒 |
| **除錯難度** | 高（分散式追蹤） | 低（單一進程、本地日誌） |

### 為什麼 OpenClaw 不需要微服務

微服務架構的核心優勢在於：獨立擴展、獨立部署、技術多樣性、故障隔離。然而，這些優勢主要適用於大型團隊開發的高流量服務。對於 OpenClaw 的目標場景——個人 AI 助理——這些優勢幾乎不存在：

**獨立擴展**：個人助理的流量極低（每天可能只有幾十到幾百個請求），單一進程綽綽有餘。

**獨立部署**：個人使用場景下，使用者通常是一次性更新整個系統，不需要逐服務部署。

**技術多樣性**：OpenClaw 的核心模組都用同一種語言實作，不需要支援多種技術棧。技能腳本雖然可以用任何語言撰寫，但這是透過 Shell 調用實現的，不需要微服務架構的支援。

**故障隔離**：唯一需要隔離的是頻道適配器（因為它們依賴外部服務），而這已經透過 Hub/Spoke 拓撲實現了。

### 何時考慮分散式部署

雖然單一進程模型是預設選擇，但在以下場景中可能需要考慮分散式部署：

1. **多使用者共用**：如果 OpenClaw 被用作團隊共享的 AI 助理（例如公司的 IT 支援助理），可能需要多個 Gateway 實例來處理併行請求。

2. **GPU 加速**：如果使用本地 LLM（如透過 Ollama），且 GPU 在另一台機器上，則需要將 Brain 模組分離到 GPU 伺服器上。

3. **大規模記憶庫**：如果記憶檔案累積到數 GB 以上，可能需要將 Memory 模組分離到專用的向量資料庫伺服器上。

4. **高可用性需求**：如果系統需要 24/7 不間斷運行，可能需要多 Gateway 部署來實現故障轉移。

即使在這些場景中，OpenClaw 的「漸進式擴展」設計也確保了從單一進程到分散式部署的遷移是平滑的。每個模組都設計了網路介面，可以在不改動程式碼的情況下，從進程內呼叫切換為網路呼叫。

## 設計權衡分析

每個架構決策都有其代價。本節坦誠地分析 OpenClaw 在設計過程中所做的權衡，以幫助讀者理解系統的優勢和局限性。

### 權衡 1：本地優先 vs 雲端便利

**選擇**：本地優先

**獲得的好處**：
- 完全的資料隱私和控制
- 離線能力
- 無訂閱費用（除 LLM API 費用外）
- 無供應商鎖定
- 極低的延遲（本地通訊）

**付出的代價**：
- 使用者需要自行管理硬體和軟體更新
- 初始設定比雲端服務複雜
- 不能跨設備無縫存取（需要額外配置同步）
- 計算資源受限於使用者硬體
- 不適合非技術使用者

**緩解措施**：
- 提供詳細的安裝與設定指南
- 支援 Git 同步進行跨設備共享
- 提供一鍵安裝腳本降低設定門檻
- 未來可能提供可選的雲端備份功能

### 權衡 2：檔案系統 vs 資料庫

**選擇**：檔案系統作為主要儲存

**獲得的好處**：
- 完全透明——可以用任何工具查看
- 與 Git 完美整合
- 不需要額外的資料庫服務
- 格式穩定——純文字永不過時
- 易於備份（cp -r 即可）
- 使用者可以手動編輯

**付出的代價**：
- 查詢效率低於關聯式資料庫
- 不支援複雜的結構化查詢
- 併行寫入需要額外處理
- 大量小檔案可能影響檔案系統性能
- 沒有內建的事務支援

**緩解措施**：
- 使用向量資料庫加速語意搜尋
- 使用檔案鎖定機制處理併行寫入
- 定期壓縮和歸檔舊檔案
- 為常用查詢建立索引快取

### 權衡 3：單一進程 vs 微服務

**選擇**：預設單一進程

**獲得的好處**：
- 極簡的部署和運維
- 無進程間通訊的開銷
- 啟動迅速
- 除錯簡單
- 資源消耗低

**付出的代價**：
- 單點故障（進程崩潰會影響所有功能）
- 垂直擴展有限
- 所有模組共享同一進程的資源
- 不能為不同模組使用不同的硬體

**緩解措施**：
- 使用 systemd 實現進程自動重啟
- 支援選擇性分離（Brain、Memory 可獨立部署）
- 內建健康檢查和自我修復
- 對於個人使用場景，單點故障的影響可接受

### 權衡 4：WebSocket vs HTTP REST

**選擇**：WebSocket

**獲得的好處**：
- 全雙工通訊（串流回應、主動推送）
- 低延遲
- 持久連線管理
- 適合即時互動場景

**付出的代價**：
- 比 HTTP REST 更難除錯（不能用 curl 測試）
- 不能使用標準的 HTTP 快取和代理
- 需要處理連線重建和心跳保活
- 文件和工具支援不如 REST 豐富

**緩解措施**：
- 提供 CLI 工具來測試 WebSocket 連線
- 提供可選的 HTTP REST 橋接端點
- 使用標準的 WebSocket 心跳機制（ping/pong）
- 提供完整的 JSON Schema 文件

### 權衡 5：Markdown 技能定義 vs 程式碼 SDK

**選擇**：Markdown 定義 + Shell 腳本

**獲得的好處**：
- 極低的技能開發門檻
- 非程式設計師也能理解和修改
- 跨語言（Shell 可以調用任何程式語言）
- 易於分享和版本控制
- 自文件化（SKILL.md 本身就是文件）

**付出的代價**：
- 表達能力不如程式碼 SDK
- Shell 腳本的除錯工具較弱
- 性能不如原生函數調用
- 缺少型別檢查
- 複雜技能的實現較為繁瑣

**緩解措施**：
- 支援混合模式（簡單技能用 Markdown，複雜技能用程式碼）
- 提供技能開發除錯工具
- 支援 MCP（Model Context Protocol）工具的整合
- 技能腳本可以調用任何複雜的外部程式

### 權衡 6：隱私優先 vs 功能完整

**選擇**：隱私優先

**獲得的好處**：
- 使用者資料永遠在本地
- 不需要信任第三方服務
- 符合資料保護法規（GDPR 等）
- 可以處理敏感資訊

**付出的代價**：
- 不能使用需要雲端資料的功能（如跨裝置同步、遠端存取）
- 無法使用大規模雲端運算資源
- 使用者需要自行管理備份
- 不能利用集體智慧（無法從其他使用者的資料中學習）

**緩解措施**：
- 提供可選的加密同步方案
- 支援端對端加密的遠端備份
- 透過 ClawHub 市集分享技能（而非使用者資料）
- 使用者可以選擇性地啟用特定的雲端功能

### 設計原則優先順序

當不同的設計原則產生衝突時，OpenClaw 遵循以下優先順序：

```
1. 安全性與隱私（不可妥協）
   │
2. 透明與可審計（高優先）
   │
3. 簡單性（優於功能豐富）
   │
4. 本地優先（優於雲端便利）
   │
5. 模組化（優於性能最佳化）
   │
6. 使用者體驗（持續改進）
```

這個優先順序表意味著：當隱私和便利性衝突時，選擇隱私；當簡單性和功能豐富衝突時，選擇簡單性；當透明度和性能衝突時，選擇透明度。

## 總結

OpenClaw 的系統架構是一個經過深思熟慮的設計，它在多個維度上做出了明確的選擇和權衡。以下是本文件的核心要點回顧：

### 設計哲學三支柱

1. **本地優先**：所有資料和運算在本地完成，確保隱私和控制
2. **模組化**：五大核心模組各司其職，介面清晰，可獨立替換
3. **透明可審計**：所有資料以人類可讀格式儲存，所有操作可追蹤

### 五大核心模組

| 模組 | 角色 | 關鍵特性 |
|------|------|---------|
| Gateway | 中央編排器 | WebSocket、Hub/Spoke、JSON Schema |
| Brain | 推理引擎 | LLM 抽象、ReAct、串流回應 |
| Memory | 記憶系統 | 多層次、檔案優先、語意搜尋 |
| Heartbeat | 自主排程 | 定時觸發、自動維護、降噪 |
| Skills | 技能系統 | Markdown 定義、安全門控、市集 |

### 架構選擇

- **通訊**：WebSocket + JSON Schema（全雙工、型別安全）
- **進程**：單一 Gateway 守護程式（簡單、高效）
- **拓撲**：Hub/Spoke（隔離、靈活）
- **儲存**：檔案系統 + 可選向量資料庫（透明、可審計）

### 適用場景

OpenClaw 最適合以下場景：
- 重視隱私的個人 AI 助理
- 需要高度自訂的技術使用者
- 需要離線能力的環境
- 小團隊的共享 AI 助手

不太適合的場景：
- 大規模多租戶 SaaS 服務
- 需要極低延遲的即時交易系統
- 非技術使用者的消費級產品（除非有完善的 GUI）
