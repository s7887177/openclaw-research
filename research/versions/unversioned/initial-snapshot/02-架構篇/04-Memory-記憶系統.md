## 引言

記憶是智慧的基石。沒有記憶的 AI Agent 就像一個每天醒來都忘記一切的人——它可能很聰明，但無法建立有意義的長期關係，無法從過去的經驗中學習，無法理解使用者的偏好和習慣。

OpenClaw 的 Memory 模組是整個系統中最獨特也最重要的模組之一。它實作了一個多層次的記憶架構，從轉瞬即逝的會話記憶到歷久彌新的核心記憶，形成了一個完整的記憶光譜。更重要的是，Memory 模組採用了「檔案即記憶」（File-first）的設計理念——所有記憶都以 Markdown 檔案的形式儲存在本地檔案系統中，使用者可以隨時查看、編輯和管理自己的記憶庫。

本文件將深入探討 Memory 模組的設計哲學、架構實作和最佳實踐。無論你是想要理解 AI 記憶系統的原理，還是想要優化自己的 OpenClaw 記憶配置，本文件都將提供全面的參考。

記憶系統的設計直接影響了 AI Agent 的「智慧感」。一個好的記憶系統能讓使用者感覺 Agent 真的「認識」自己、「理解」自己的需求。而 OpenClaw 的記憶系統正是為了實現這種深度的個人化互動而設計的。

## 為什麼記憶對 AI Agent 至關重要

### 無記憶 Agent 的困境

大多數 LLM 應用（如 ChatGPT 的基礎模式）都是「無狀態」的——每次對話都是從零開始。這意味著：

**重複自我介紹**：每次對話開始時，使用者都需要重新告訴 AI 自己是誰、做什麼工作、有什麼偏好。

```
第 1 天：
User: 我是 Eason，軟體工程師。幫我寫個排序演算法
AI: 好的，以下是排序演算法...

第 2 天：
User: 幫我寫個排序演算法
AI: 好的，請問你偏好哪種程式語言？  ← 忘記了使用者是工程師
```

**無法建立偏好**：AI 無法記住使用者喜歡什麼回答風格、慣用什麼術語、偏好什麼格式。每次都需要從頭溝通。

**無法追蹤長期任務**：對於跨越多天的專案，無記憶的 AI 無法追蹤進度、記住已完成的步驟、或保持一致的決策。

**缺乏上下文理解**：使用者說「繼續昨天的那個」，無記憶的 AI 完全不知道「那個」是什麼。

### 有記憶 Agent 的能力

有了記憶系統，OpenClaw 能夠：

**個人化互動**
```
User: 早安
AI: 早安 Eason！今天的天氣看起來不錯。
    昨天你提到想研究 OpenClaw 的 Memory 模組，
    要繼續昨天的討論嗎？
```

**累積知識**
```
User: 幫我寫個 API 端點
AI: 根據你之前的專案風格，我使用了：
    - Go 語言（你偏好的後端語言）
    - Chi 路由器（你之前用過的框架）
    - 結構化日誌（你在 1/10 提到的偏好）
    這是程式碼...
```

**追蹤長期任務**
```
User: 專案進度如何？
AI: 根據我的記錄，OpenClaw 研究專案目前的進度：
    ✅ 基礎篇（完成於 1/14）
    ✅ 架構篇前三章（完成於 1/15-1/16）
    🔄 架構篇第四章（進行中）
    ⬜ 安裝篇、記憶篇等待開始
```

**情感連結**
```
User: 最近工作壓力好大
AI: 我記得你上週也提到過工作壓力的問題。
    上次你說散步和聽音樂讓你感覺好多了。
    要不要現在休息一下？我可以幫你播放你喜歡的音樂清單。
```

### 記憶的分類

從認知科學的角度，記憶可以分為以下幾類，OpenClaw 的記憶系統借鑑了這些概念：

| 認知科學概念 | OpenClaw 對應 | 說明 |
|------------|-------------|------|
| 感覺記憶（Sensory Memory） | 原始訊息 | 未處理的原始輸入，短暫存在 |
| 短期記憶（Short-term Memory） | 會話記憶 | 當前對話的上下文 |
| 工作記憶（Working Memory） | 上下文窗口 | LLM 當前能處理的資訊量 |
| 長期記憶——情節記憶 | 每日筆記 | 特定事件和經歷的記錄 |
| 長期記憶——語義記憶 | MEMORY.md | 一般性知識和事實 |
| 程序記憶 | 技能定義 | 如何執行特定操作的知識 |

## 記憶層次結構

OpenClaw 的記憶系統由四個層次組成，從短期到長期、從具體到抽象：

```
┌──────────────────────────────────────────────┐
│                記憶層次金字塔                   │
│                                               │
│                  ╱╲                            │
│                 ╱  ╲                           │
│                ╱ 核 ╲        MEMORY.md         │
│               ╱ 心  ╲       長期核心記憶        │
│              ╱ 記 憶 ╲      很少變動            │
│             ╱────────╲                         │
│            ╱ 每日筆記  ╲     YYYY-MM-DD.md     │
│           ╱  (Daily    ╲    每天一個檔案        │
│          ╱   Notes)     ╲   中等更新頻率        │
│         ╱────────────────╲                     │
│        ╱  語意搜尋結果     ╲  向量索引檢索       │
│       ╱  (Semantic Search) ╲ 動態查詢           │
│      ╱──────────────────────╲                  │
│     ╱     會話記憶            ╲ 會話檔案        │
│    ╱     (Session Context)    ╲ 高頻更新        │
│   ╱────────────────────────────╲               │
└──────────────────────────────────────────────┘
```

### 第一層：會話記憶（Session Context）

會話記憶是最底層也是最活躍的記憶。它記錄了當前對話的完整歷史，包括使用者的每條訊息、Agent 的每個回應、工具調用的細節和結果。

**特性：**
- 壽命：會話持續期間（數分鐘到數小時）
- 更新頻率：每次訊息交互
- 儲存位置：記憶體中（活躍時）+ 檔案系統（持久化）
- 格式：結構化的對話記錄

**檔案結構：**

```markdown
<!-- ~/.openclaw/memory/sessions/sess_20250116_001.md -->

# 會話記錄：2025-01-16 10:00

**頻道**：telegram
**使用者**：Eason
**開始時間**：2025-01-16 10:00:05
**狀態**：active

---

## 對話記錄

### [10:00:05] User
今天天氣如何？

### [10:00:08] Agent
今天台北天氣晴朗，氣溫 25-30°C，適合外出活動。

> 🔧 工具調用：weather.get_forecast(location="台北", date="2025-01-16")
> 📋 結果：{"temp": "25-30°C", "condition": "晴天", "humidity": "65%"}

### [10:00:15] User
那明天呢？

### [10:00:18] Agent
明天台北預計多雲，氣溫 18-24°C，降雨機率 30%。建議攜帶雨傘。

> 🔧 工具調用：weather.get_forecast(location="台北", date="2025-01-17")
> 📋 結果：{"temp": "18-24°C", "condition": "多雲", "rain_chance": "30%"}
```

**會話記憶的管理：**
- 活躍會話的記錄保存在記憶體中，確保快速存取
- 每隔一定間隔（或每次交互後），記錄同步寫入檔案系統
- 會話結束後，完整記錄保留在檔案系統中
- 過期的會話記錄會被壓縮和歸檔

### 第二層：每日筆記（Daily Notes）

每日筆記是 OpenClaw 記憶系統的一個獨特設計。每天自動建立一個以日期命名的 Markdown 檔案，記錄當天的重要事件、對話摘要和使用者活動。

**特性：**
- 壽命：永久（除非使用者手動刪除）
- 更新頻率：每天，多次追加
- 儲存位置：`~/.openclaw/memory/daily/` 目錄
- 格式：Markdown

**檔案結構：**

```markdown
<!-- ~/.openclaw/memory/daily/2025-01-16.md -->

# 2025-01-16（週四）

## 今日摘要
- Eason 繼續進行 OpenClaw 研究專案
- 討論了 Memory 模組的架構設計
- 查詢了天氣資訊（今明兩天）
- 修改了一個會議時間

## 對話記錄

### 10:00 天氣查詢
- 查詢了今明兩天台北天氣
- 今天晴朗 25-30°C，明天多雲 18-24°C 有雨

### 11:30 技術討論
- 討論了 OpenClaw Memory 模組的檔案結構
- Eason 傾向使用 Markdown 格式而非 JSON
- 決定採用四層記憶架構

### 14:00 行程調整
- 將「產品會議」從上午 10 點改到下午 3 點
- 原因：上午的客戶拜訪延長了

## 觀察到的偏好/模式
- Eason 今天的工作重心是架構設計
- 偏好在上午處理技術問題
- 今天情緒積極，工作效率高

## 待辦/提醒
- [ ] 明天天氣可能下雨，提醒攜帶雨傘
- [ ] 產品會議改到下午 3 點
```

**每日筆記的自動生成：**

每日筆記不是由使用者手動撰寫的，而是由 Memory 模組自動維護的。每次對話結束後，Memory 模組會：

1. 分析對話內容，提取關鍵事件
2. 識別使用者的偏好和模式
3. 記錄工具調用和結果
4. 將以上資訊追加到當天的每日筆記中

**每日筆記也可手動編輯：**

由於每日筆記就是普通的 Markdown 檔案，使用者可以隨時用任何文字編輯器打開並修改。例如，使用者可以添加 Agent 不知道的資訊：

```markdown
## 手動添加
- 今天收到了升遷通知，下個月開始擔任技術主管
- 需要開始準備團隊管理相關的知識
```

這些手動添加的資訊會在下次對話時被 Memory 模組讀取，成為 Agent 知識的一部分。

### 第三層：長期記憶（MEMORY.md）

長期記憶是 OpenClaw 記憶系統的「核心知識庫」。它儲存了關於使用者和環境的穩定性資訊——那些不會每天改變的重要事實。

**特性：**
- 壽命：永久
- 更新頻率：較少（通常透過記憶提升機制更新）
- 儲存位置：`~/.openclaw/memory/MEMORY.md`
- 格式：結構化 Markdown

**檔案結構：**

```markdown
<!-- ~/.openclaw/memory/MEMORY.md -->

# OpenClaw 核心記憶

## 使用者基本資訊
- **姓名**：Eason
- **職業**：軟體工程師
- **公司**：[公司名]
- **工作年資**：8 年
- **所在地**：台北

## 技術偏好
- **偏好語言**：Go（後端）、TypeScript（前端）
- **編輯器**：VS Code + Vim 鍵綁定
- **作業系統**：macOS + Linux（伺服器）
- **版本控制**：Git（偏好 rebase workflow）
- **架構偏好**：模組化、KISS 原則、Clean Architecture

## 溝通偏好
- 喜歡簡潔直接的回答
- 技術問題時偏好看程式碼範例而非純文字說明
- 使用繁體中文
- 可以接受英文技術術語

## 工作模式
- 通常工作時間：09:00-18:00
- 最專注的時段：上午 10:00-12:00
- 午餐時間：12:00-13:30
- 週五下午通常較輕鬆

## 目前進行中的專案
- **OpenClaw 研究專案**：研究 OpenClaw AI Agent 架構
  - 進度：架構篇進行中
  - 重點：五大核心模組的設計

## 重要的過往決策
- 2025-01-10：決定使用 Markdown 檔案作為記憶儲存格式
- 2025-01-12：確定採用 Hub/Spoke 拓撲作為 Gateway 架構
- 2025-01-14：選擇 Claude 作為預設 LLM 提供者

## 人際關係
- Alice：同事，前端工程師，經常合作
- Bob：主管，技術總監
- Charlie：大學同學，偶爾技術討論
```

**長期記憶的維護規則：**

1. **只記錄穩定資訊**：不記錄每天變化的瑣事，只記錄長期有效的事實
2. **保持更新**：當穩定資訊發生變化（如換工作、搬家），及時更新
3. **控制大小**：長期記憶應該保持簡潔，建議不超過 2000 tokens
4. **結構化組織**：使用清晰的分類和層次結構

### 第四層：語意搜尋（Semantic Search）

語意搜尋是記憶系統的「智慧檢索層」。它不是一個獨立的儲存層，而是一個橫跨所有其他記憶層的檢索機制。

**運作方式：**

1. 所有記憶內容（每日筆記、長期記憶、會話記錄）被切分為語意片段
2. 每個片段被轉換為向量嵌入（Vector Embedding）
3. 向量被儲存在本地向量資料庫中
4. 當需要檢索相關記憶時，將查詢轉換為向量，進行最近鄰搜尋

```
記憶片段                    向量嵌入
"Eason 偏好 Go 語言"    → [0.23, -0.15, 0.87, ...]
"上次用 Chi 路由器"      → [0.21, -0.12, 0.85, ...]
"明天台北天氣多雲"       → [-0.45, 0.67, 0.12, ...]

查詢："推薦什麼程式語言？"
查詢向量：                  [0.25, -0.18, 0.82, ...]

最近鄰搜尋結果：
1. "Eason 偏好 Go 語言"    （相似度: 0.95）✅
2. "上次用 Chi 路由器"      （相似度: 0.89）✅
3. "明天台北天氣多雲"       （相似度: 0.12）❌ 不相關
```

**語意搜尋的配置：**

```yaml
# ~/.openclaw/config/memory.yaml

semantic_search:
  enabled: true
  
  # 嵌入模型
  embedding:
    provider: "ollama"          # 使用本地模型生成嵌入
    model: "nomic-embed-text"
    # 或使用雲端服務
    # provider: "openai"
    # model: "text-embedding-3-small"
  
  # 向量資料庫
  vector_store:
    type: "sqlite-vss"          # 使用 SQLite VSS 擴展
    path: "~/.openclaw/memory/vectors/embeddings.db"
  
  # 搜尋配置
  search:
    top_k: 10                   # 返回前 10 個最相關的結果
    min_similarity: 0.5         # 最低相似度閾值
    
  # 索引配置
  indexing:
    chunk_size: 500             # 每個片段的最大字數
    chunk_overlap: 50           # 片段之間的重疊字數
    update_on_write: true       # 寫入記憶時自動更新索引
```

## 檔案即記憶

### File-first 設計理念

「檔案即記憶」（File-first Approach）是 OpenClaw 記憶系統的核心設計理念。它的核心主張是：**記憶應該以人類可讀的檔案格式儲存，而非隱藏在資料庫中**。

這個設計理念的靈感來源：

- **Obsidian/Logseq** 等知識管理工具的「local-first」哲學
- **Unix 哲學**中的「一切皆檔案」
- **Git** 的純文字版本控制理念
- **Johnny.Decimal** 等檔案組織系統

### 檔案結構

```
~/.openclaw/memory/
│
├── MEMORY.md                    # 長期核心記憶
│
├── daily/                       # 每日筆記
│   ├── 2025-01-14.md
│   ├── 2025-01-15.md
│   ├── 2025-01-16.md
│   └── ...
│
├── sessions/                    # 會話記錄
│   ├── active/                  # 活躍會話
│   │   └── sess_20250116_001.md
│   ├── recent/                  # 最近的會話（7 天內）
│   │   ├── sess_20250115_001.md
│   │   └── sess_20250115_002.md
│   └── archive/                 # 歸檔的會話
│       ├── 2025-01/
│       │   ├── sess_20250101_001.md.gz
│       │   └── ...
│       └── ...
│
├── vectors/                     # 向量索引
│   └── embeddings.db
│
└── exports/                     # 匯出的記憶
    └── ...
```

### File-first 的好處

**好處 1：完全透明（Full Transparency）**

使用者可以隨時用任何文字工具查看所有記憶：

```bash
# 查看長期記憶
$ cat ~/.openclaw/memory/MEMORY.md

# 搜尋記憶中提到 "Go" 的內容
$ grep -r "Go" ~/.openclaw/memory/

# 查看今天的筆記
$ cat ~/.openclaw/memory/daily/$(date +%Y-%m-%d).md

# 查看最近 7 天的筆記標題
$ head -1 ~/.openclaw/memory/daily/2025-01-{10..16}.md
```

**好處 2：手動可編輯（Human Editable）**

使用者可以直接編輯記憶檔案：

```bash
# 用編輯器打開長期記憶
$ vim ~/.openclaw/memory/MEMORY.md

# 用 VS Code 打開整個記憶目錄
$ code ~/.openclaw/memory/
```

修改後的內容會在下次 Brain 讀取記憶時自動生效。

**好處 3：版本控制（Version Controllable）**

整個記憶目錄可以用 Git 進行版本控制：

```bash
$ cd ~/.openclaw/memory/
$ git init
$ git add .
$ git commit -m "記憶快照 2025-01-16"

# 後續自動定期提交
$ git add -A && git commit -m "每日自動快照 $(date)"
```

**好處 4：跨設備同步（Cross-device Sync）**

透過 Git 或任何檔案同步工具，記憶可以在多台設備之間同步：

```bash
# 推送記憶到遠端
$ git push origin main

# 在另一台設備拉取記憶
$ git pull origin main
```

**好處 5：備份簡單（Easy Backup）**

```bash
# 完整備份
$ tar czf openclaw-memory-backup-$(date +%Y%m%d).tar.gz ~/.openclaw/memory/

# 或使用 rsync
$ rsync -av ~/.openclaw/memory/ /backup/openclaw-memory/
```

**好處 6：隱私可控（Privacy Controllable）**

使用者可以隨時刪除任何不想保留的記憶：

```bash
# 刪除特定日期的筆記
$ rm ~/.openclaw/memory/daily/2025-01-10.md

# 清除所有會話記錄
$ rm -rf ~/.openclaw/memory/sessions/archive/*

# 從長期記憶中移除特定資訊
$ vim ~/.openclaw/memory/MEMORY.md  # 手動編輯移除
```

### Markdown 格式的選擇

選擇 Markdown 而非 JSON、YAML 或其他格式，是因為：

1. **人類可讀性最佳**：Markdown 是最接近純文字的格式化語言
2. **支援豐富的結構**：標題、列表、表格、程式碼區塊等
3. **廣泛的工具支援**：幾乎所有編輯器和預覽工具都支援
4. **Git 友好**：Markdown 的差異比較（diff）清晰易讀
5. **LLM 友好**：LLM 天生擅長閱讀和生成 Markdown

## 記憶壓縮機制

### 為什麼需要壓縮

隨著時間的推移，會話記錄和每日筆記會不斷累積。如果不進行壓縮，記憶庫的大小會持續增長，導致：

1. **Token 浪費**：過多的原始記錄被注入提示詞，佔用寶貴的上下文窗口空間
2. **搜尋效率下降**：記憶庫越大，搜尋越慢
3. **磁碟空間消耗**：雖然文字檔案很小，但長期累積仍可觀
4. **資訊噪音**：舊的瑣碎對話記錄會干擾對重要記憶的檢索

### 壓縮的類型

OpenClaw 實作了三種壓縮機制：

**類型 1：會話壓縮（Session Compaction）**

當一個會話結束或超過設定的大小閾值時，Memory 模組使用 LLM 將冗長的對話記錄壓縮為簡潔的摘要：

```
壓縮前（完整會話記錄 - 3000 tokens）：

User: 今天天氣如何？
AI: 今天台北天氣晴朗，氣溫 25-30°C...
User: 那明天呢？
AI: 明天預計多雲，氣溫 18-24°C...
User: 幫我查行程
AI: 您今天有三個會議...
User: 第三個改時間
AI: 好的，已將「產品會議」改到下午 3 點...
[... 更多對話 ...]

壓縮後（摘要 - 200 tokens）：

## 會話摘要：2025-01-16 10:00-11:30
- 查詢了今明兩天台北天氣（今天晴 25-30°C，明天多雲 18-24°C）
- 查看了今日行程，共 3 個會議
- 將「產品會議」從上午 10 點改到下午 3 點
- 討論了週末計畫（尚未做最終決定）
```

**類型 2：每日筆記壓縮（Daily Note Compaction）**

隨著時間推移，較舊的每日筆記會被進一步壓縮。例如，一週前的筆記可能被壓縮為幾行摘要：

```
壓縮前（2025-01-10 的完整每日筆記 - 1500 tokens）：
[包含 5 個詳細的對話記錄、偏好觀察、待辦事項等]

壓縮後（200 tokens）：
## 2025-01-10 摘要
- 決定使用 Markdown 檔案作為記憶儲存格式
- 討論了 OpenClaw 的設計哲學
- 安裝並測試了 Ollama 本地模型
- 重要決策：選擇 File-first 方案而非 SQLite
```

**類型 3：歸檔壓縮（Archive Compression）**

超過 30 天的舊記錄會被歸檔壓縮：
- 會話記錄被 gzip 壓縮
- 每日筆記只保留摘要版本
- 保留完整的向量索引以支援搜尋

### 壓縮觸發條件

```yaml
# ~/.openclaw/config/memory.yaml

compaction:
  session:
    # 會話超過此大小時觸發壓縮
    max_tokens: 5000
    # 會話閒置超過此時間時觸發壓縮
    idle_timeout: "24h"
    
  daily_notes:
    # 超過此天數的每日筆記會被壓縮
    compress_after_days: 7
    # 壓縮後保留的最大 token 數
    compressed_max_tokens: 500
    
  archive:
    # 超過此天數的記錄會被歸檔
    archive_after_days: 30
    # 是否啟用 gzip 壓縮
    gzip_enabled: true
```

### 壓縮的品質保證

壓縮是一個有損操作——原始資訊會被簡化。為了確保重要資訊不被遺失，OpenClaw 採用了以下策略：

1. **保留原始檔案**：壓縮後的摘要與原始檔案並存（歸檔目錄中），使用者可以隨時查看原始內容。

2. **重要性標記**：被手動標記為「重要」的內容不會被壓縮。

3. **壓縮前預覽**：在 Heartbeat 執行自動壓縮前，使用者可以配置為先生成壓縮預覽，確認後再執行。

4. **壓縮品質指標**：使用 LLM 評估壓縮摘要的品質，確保關鍵資訊被保留。

## 記憶提升流程

### 什麼是記憶提升

記憶提升（Memory Promotion）是指將短期記憶中的重要資訊「提升」到長期記憶中的過程。這類似於人類大腦在睡眠時將短期記憶鞏固為長期記憶的機制。

```
記憶提升流程：

會話記憶 ──提煉──▶ 每日筆記 ──提升──▶ 長期記憶
(具體對話)        (當日摘要)         (穩定事實)

範例：
會話中多次提到 "Go 語言"
    │
    ▼
每日筆記記錄 "Eason 今天使用 Go 語言完成了三個任務"
    │
    ▼（連續多天觀察到相同模式）
    │
長期記憶更新 "偏好語言：Go"
```

### 提升觸發條件

Memory 模組根據以下條件判斷是否需要進行記憶提升：

**頻率觸發**：如果某個資訊在多天的每日筆記中反覆出現，說明它是穩定的，應該被提升到長期記憶。

```
2025-01-10: "Eason 使用 Go 開發 API"
2025-01-12: "Eason 用 Go 重構了認證模組"
2025-01-14: "Eason 用 Go 處理了資料匯入"
2025-01-16: "Eason 用 Go 實作了快取層"
→ 觸發提升：將 "偏好語言：Go" 加入長期記憶
```

**重要性觸發**：某些資訊即使只出現一次，也足夠重要需要被記住。例如：
- 使用者明確表示的偏好（「我喜歡簡潔的回答」）
- 重要的個人資訊（「我下個月要換工作」）
- 使用者明確要求記住的事項（「記住這個密碼重設流程」）

**變化觸發**：當長期記憶中的資訊與每日筆記中的觀察出現矛盾時，觸發更新。

```
長期記憶：公司 = "公司A"
每日筆記出現多次：提到新公司 "公司B"
→ 觸發更新：確認後將公司資訊更新為 "公司B"
```

### 提升流程

```
步驟 1：收集候選資訊
├── 掃描最近 7 天的每日筆記
├── 識別反覆出現的模式
├── 識別標記為重要的資訊
└── 識別與長期記憶矛盾的資訊

步驟 2：生成提升建議
├── 使用 LLM 分析候選資訊
├── 生成結構化的提升建議
└── 評估每條建議的信心度

步驟 3：執行提升
├── 高信心度的建議自動執行
├── 低信心度的建議等待使用者確認
└── 更新 MEMORY.md 檔案

步驟 4：驗證
├── 檢查更新後的 MEMORY.md 格式正確
├── 確認沒有重複或矛盾的資訊
└── 記錄提升操作的審計日誌
```

### 提升建議範例

```json
{
  "promotion_suggestions": [
    {
      "action": "add",
      "section": "技術偏好",
      "content": "偏好使用 Docker Compose 進行本地開發環境配置",
      "evidence": [
        "2025-01-12: 使用 Docker Compose 設定開發環境",
```

## Token 預算管理

### 記憶的 Token 成本

每次 Brain 向 LLM 發送提示詞時，記憶上下文都會佔用一部分上下文窗口。Token 預算管理確保記憶注入不會「擠壓」其他重要的提示詞組件。

```
上下文窗口分配範例（以 128K token 模型為例）：

總上下文窗口：128,000 tokens
├── 回應保留：4,096 tokens
├── 系統提示：1,000 tokens
├── 人格設定：500 tokens
├── 可用技能列表：2,000 tokens
├── 使用者輸入：~500 tokens
└── 記憶預算：剩餘 ~119,904 tokens
    ├── 長期記憶（MEMORY.md）：最多 2,000 tokens
    ├── 今日筆記：最多 3,000 tokens
    ├── 語意搜尋結果：最多 5,000 tokens
    └── 對話歷史：最多 ~109,904 tokens
```

### 預算分配策略

```yaml
# ~/.openclaw/config/memory.yaml

token_budget:
  # 記憶組件的最大 token 數
  long_term_memory: 2000
  daily_notes: 3000
  semantic_search: 5000
  conversation_history: null  # null 表示使用剩餘空間
  
  # 動態調整
  dynamic_allocation:
    enabled: true
    # 當對話歷史較短時，將多餘的空間分配給語意搜尋
    rebalance_threshold: 0.5
```

### 記憶選擇的優先級

當 token 預算有限時，Memory 模組需要選擇最重要的記憶片段。選擇的依據包括：

1. **時間相關性**：最近的記憶優先於遠古的記憶
2. **語意相關性**：與當前查詢相關的記憶優先
3. **重要性標記**：被標記為重要的記憶優先
4. **頻率**：經常被引用的記憶優先
5. **層次**：長期記憶 > 今日筆記 > 歷史搜尋結果

## 向量搜尋 vs 關鍵字搜尋

### 兩種搜尋的比較

OpenClaw 的 Memory 模組同時支援向量搜尋和關鍵字搜尋，它們各有優勢：

| 維度 | 向量搜尋 | 關鍵字搜尋 |
|------|---------|----------|
| **原理** | 語意相似度 | 文字匹配 |
| **查詢方式** | 自然語言 | 精確關鍵字 |
| **優勢** | 理解語意、找到相關但用詞不同的記憶 | 精確、快速、無需嵌入模型 |
| **劣勢** | 需要嵌入模型、消耗較多資源 | 無法理解同義詞、需要精確的關鍵字 |
| **適用場景** | 模糊查詢、概念搜尋 | 精確查詢、已知關鍵字 |
| **資源需求** | 嵌入模型 + 向量資料庫 | 僅文字索引 |
| **離線可用** | 需要本地嵌入模型 | 完全離線 |

### 使用範例

**向量搜尋的優勢場景：**

```
使用者問："我之前用什麼框架做過 Web 開發？"

向量搜尋能找到：
- "用 Chi 路由器實作了 REST API"（語意相關，但沒有 "Web" 或 "框架" 這兩個詞）
- "Express.js 的中間件設計很好用"（語意相關）
- "React 前端專案用了 Next.js"（語意相關）

關鍵字搜尋可能只找到：
- 直接包含 "框架" 或 "Web" 的記錄
- 可能遺漏沒有使用這些關鍵字但語意相關的記憶
```

**關鍵字搜尋的優勢場景：**

```
使用者問："找到所有提到 Alice 的記錄"

關鍵字搜尋精確匹配：
- 包含 "Alice" 的所有記憶（精確、不遺漏）

向量搜尋可能返回：
- 包含 "Alice" 的記憶（正確）
- 包含 "同事" 的記憶（語意相關但不一定提到 Alice）
- 可能遺漏某些只是順帶提及 Alice 的記錄
```

### 混合搜尋策略

OpenClaw 預設使用混合搜尋策略——同時執行向量搜尋和關鍵字搜尋，然後合併結果：

```
查詢："我之前用什麼 Go 框架？"

向量搜尋結果（按語意相似度）：
1. "用 Chi 路由器實作了 REST API" (0.92)
2. "Go 的 Gin 框架效能很好" (0.88)
3. "Echo 框架的中間件設計" (0.85)

關鍵字搜尋結果（按文字匹配）：
1. "Go 框架比較：Chi vs Gin" (完全匹配)
2. "學習了 Go 的 Chi 框架" (部分匹配)

合併後（去重並按綜合分數排序）：
1. "Go 框架比較：Chi vs Gin" (綜合分數最高)
2. "用 Chi 路由器實作了 REST API"
3. "Go 的 Gin 框架效能很好"
4. "Echo 框架的中間件設計"
5. "學習了 Go 的 Chi 框架"
```

### 搜尋配置

```yaml
search:
  strategy: "hybrid"    # hybrid / vector_only / keyword_only
  
  hybrid:
    vector_weight: 0.6  # 向量搜尋的權重
    keyword_weight: 0.4 # 關鍵字搜尋的權重
    
  keyword:
    engine: "ripgrep"   # 使用 ripgrep 進行快速文字搜尋
    fuzzy: true         # 啟用模糊匹配
    
  vector:
    embedding_model: "nomic-embed-text"
    similarity_metric: "cosine"
    top_k: 20
```

## 記憶管理的最佳實踐

### 初始設定

**實踐 1：手動撰寫初始 MEMORY.md**

在開始使用 OpenClaw 之前，花 10-15 分鐘撰寫一份初始的 MEMORY.md。這就像向一個新助理做自我介紹：

```markdown
# 初始記憶

## 關於我
- 名字：[你的名字]
- 職業：[你的職業]
- 所在地：[你的城市]

## 我希望你知道的事
- [任何重要的偏好或習慣]
- [溝通風格偏好]
- [技術棧偏好]

## 你的語氣
- [你希望 Agent 用什麼語氣和你說話]
```

**實踐 2：設定適當的壓縮策略**

根據你的使用頻率調整壓縮參數：
- 高頻使用者（每天 20+ 對話）：較積極的壓縮（7 天壓縮、30 天歸檔）
- 中頻使用者（每天 5-10 對話）：中等壓縮（14 天壓縮、60 天歸檔）
- 低頻使用者（每天 1-3 對話）：較寬鬆的壓縮（30 天壓縮、90 天歸檔）

### 日常維護

**實踐 3：定期審查 MEMORY.md**

每月花幾分鐘審查長期記憶，確保資訊的準確性和時效性。刪除過時的資訊，添加新的重要事實。

**實踐 4：善用手動標記**

當對話中出現重要資訊時，告訴 Agent 記住它：

```
User: 記住：我的新電話號碼是 0912-345-678
Agent: 好的，我已經將你的新電話號碼記錄到記憶中。
```

**實踐 5：利用每日筆記的可編輯性**

如果 Agent 的每日筆記遺漏了重要資訊，直接編輯檔案補充。

### 進階技巧

**實踐 6：建立記憶的版本控制**

```bash
# 在記憶目錄初始化 Git
cd ~/.openclaw/memory/
git init
echo "vectors/" >> .gitignore  # 向量索引不需要版本控制

# 設定每日自動提交
# 可以在 HEARTBEAT.md 中配置
```

**實踐 7：記憶的匯出與分享**

如果你有多個 OpenClaw 實例（如工作用和個人用），可以選擇性地分享記憶：

```bash
# 只匯出技術偏好相關的記憶
grep -A 5 "技術偏好" ~/.openclaw/memory/MEMORY.md > tech-prefs-export.md
```

**實踐 8：記憶的隱私管理**

定期檢查記憶中是否包含敏感資訊，必要時進行清理：

```bash
# 搜尋可能包含密碼或金鑰的記憶
grep -ri "password\|secret\|key\|token" ~/.openclaw/memory/
```

## 與傳統資料庫方案的比較

### 為什麼不用 SQLite/PostgreSQL

許多 AI Agent 框架使用傳統資料庫來儲存記憶。OpenClaw 選擇了不同的路線。以下是詳細的比較：

| 維度 | 檔案系統方案 | 資料庫方案 |
|------|-----------|----------|
| **透明度** | ⭐⭐⭐⭐⭐ 直接可讀 | ⭐⭐ 需要工具查詢 |
| **可編輯性** | ⭐⭐⭐⭐⭐ 任何編輯器 | ⭐⭐ 需要 SQL 工具 |
| **版本控制** | ⭐⭐⭐⭐⭐ Git 完美支援 | ⭐ 需要額外方案 |
| **查詢效率** | ⭐⭐ grep 足夠 | ⭐⭐⭐⭐⭐ 索引查詢 |
| **結構化查詢** | ⭐ 不支援 SQL | ⭐⭐⭐⭐⭐ 完整 SQL |
| **併行存取** | ⭐⭐ 需要鎖定 | ⭐⭐⭐⭐⭐ 內建支援 |
| **備份簡單度** | ⭐⭐⭐⭐⭐ cp 即可 | ⭐⭐⭐ 需要 dump |
| **依賴** | ⭐⭐⭐⭐⭐ 無額外依賴 | ⭐⭐⭐ 需要資料庫 |
| **遷移性** | ⭐⭐⭐⭐⭐ 複製目錄即可 | ⭐⭐ 需要匯出/匯入 |
| **LLM 友好度** | ⭐⭐⭐⭐⭐ Markdown | ⭐⭐ 需要格式轉換 |

### 適用場景分析

**檔案系統方案適合：**
- 個人使用（OpenClaw 的主要場景）
- 重視透明度和可控性的使用者
- 需要手動編輯記憶的場景
- 記憶量不超過數千個檔案的場景

**資料庫方案適合：**
- 多使用者共用的場景
- 需要複雜結構化查詢的場景
- 記憶量極大（數百萬條記錄）的場景
- 需要嚴格事務控制的場景

### 混合方案

OpenClaw 實際上採用的是「混合方案」——主要記憶使用檔案系統，但向量搜尋使用 SQLite-VSS（一個 SQLite 擴展）。這結合了兩者的優勢：

```
主要記憶（透明、可編輯）      搜尋索引（高效檢索）
    ┌──────────────┐          ┌──────────────┐
    │  Markdown    │ ──索引──▶│  SQLite-VSS  │
    │  檔案系統     │          │  向量資料庫   │
    └──────────────┘          └──────────────┘
         ▲                         │
         │                         │
    使用者直接                  Brain 透過
    閱讀/編輯                  語意搜尋查詢
```

這種設計確保了：
- 使用者看到的和編輯的是 Markdown 檔案
- Brain 模組透過向量搜尋高效檢索相關記憶
- 兩者之間的索引由 Memory 模組自動維護
- 即使向量資料庫損壞，原始記憶檔案完好無損

---

## 總結

Memory 模組是 OpenClaw 實現「個人化 AI 助理」的關鍵。它透過多層次的記憶架構，在 LLM 有限的上下文窗口中，為 Agent 提供了「記憶」的能力。

### 核心設計要點

1. **四層記憶架構**：會話記憶 → 每日筆記 → 長期記憶 → 語意搜尋
2. **檔案即記憶**：所有記憶以 Markdown 檔案儲存，完全透明且可編輯
3. **記憶壓縮**：自動將冗長記錄壓縮為精簡摘要，節省空間和 token
4. **記憶提升**：將短期記憶中的穩定資訊提升到長期記憶中
5. **Token 預算**：精細管理記憶注入的 token 開銷
6. **混合搜尋**：結合向量搜尋和關鍵字搜尋的優勢
7. **版本控制友好**：支援 Git 進行記憶的版本管理和跨設備同步

### 記憶系統的設計哲學

> 「好的記憶系統不是記住所有事情，而是在對的時間記起對的事情。」

OpenClaw 的記憶系統正是為了實現這個目標而設計的。它不追求記住每一個細節（那是資料庫的工作），而是追求在使用者需要的時候，提供最相關、最有用的上下文資訊。
