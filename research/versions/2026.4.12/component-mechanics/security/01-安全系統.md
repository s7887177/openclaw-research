# OpenClaw 安全系統：稽核、掃描與信任模型完整剖析

## 本章摘要

OpenClaw 的安全系統（Security System）是整個框架中最關鍵的防禦層。它以「個人助理」（Personal Assistant）信任模型為核心前提——一位受信任的操作者（Operator）管理一到多個代理（Agent），而非多租戶（Multi-tenant）架構。

本章從原始碼出發，完整拆解 `src/security/` 目錄下的 27 個非測試原始檔案，涵蓋以下主題：

1. **主要稽核系統**（`audit.ts`）——統籌所有安全檢查的核心編排器（Orchestrator），1384 行程式碼
2. **同步稽核**（`audit-extra.sync.ts`）——純配置分析，無 I/O 操作
3. **非同步稽核**（`audit-extra.async.ts`）——涉及檔案系統與配置讀取的 I/O 檢查
4. **技能安全掃描器**（`skill-scanner.ts`）——偵測插件程式碼中的惡意模式
5. **上下文可見性**（`context-visibility.ts`）——控制補充上下文的可見性規則
6. **檔案系統權限檢查**（`audit-fs.ts`）——POSIX/Windows ACL 權限稽核
7. **危險配置旗標偵測**（`dangerous-config-flags.ts`）——識別不安全的設定項目
8. **工具拒絕清單**（`dangerous-tools.ts`）——Gateway HTTP 高風險工具封鎖
9. **安全修復系統**（`fix.ts`）——自動化權限與配置修復
10. **信任模型與事件回應**——來自 `SECURITY.md` 與 `INCIDENT_RESPONSE.md` 的設計哲學

## 2. 稽核型別定義 (audit.types.ts)

所有稽核結果都圍繞一組核心型別構建。`SecurityAuditReport` 是最終輸出，包含時間戳記（Timestamp）、摘要計數和所有發現（Findings）清單。

```typescript
// source-repo/src/security/audit.types.ts:1-31
export type SecurityAuditSeverity = "info" | "warn" | "critical";

export type SecurityAuditFinding = {
  checkId: string;
  severity: SecurityAuditSeverity;
  title: string;
  detail: string;
  remediation?: string;
};

export type SecurityAuditSummary = {
  critical: number;
  warn: number;
  info: number;
};

export type SecurityAuditReport = {
  ts: number;
  summary: SecurityAuditSummary;
  findings: SecurityAuditFinding[];
  deep?: {
    gateway?: {
      attempted: boolean;
      url: string | null;
      ok: boolean;
      error: string | null;
      close?: { code: number; reason: string } | null;
    };
  };
};
```

### 嚴重等級語義

| 等級 (Severity) | 意義 | 典型場景 |
|-----------------|------|----------|
| `critical` | 立即威脅，須優先處理 | 狀態目錄全域可寫、Gateway 無認證暴露 |
| `warn` | 潛在風險，建議修正 | 群組可寫權限、日誌遮罩關閉 |
| `info` | 資訊性提示，供參考 | Tailscale Serve 模式啟用 |

每個 `SecurityAuditFinding` 都包含唯一的 `checkId`（例如 `fs.state_dir.perms_world_writable`），方便程式化處理和追蹤。`remediation` 欄位提供具體的修復建議指令。

## 3. 主要稽核系統 (audit.ts)

`audit.ts` 是整個安全子系統的中樞，共 1384 行。它定義了稽核選項型別（`SecurityAuditOptions`）、執行上下文（`AuditExecutionContext`），並透過 `runSecurityAudit()` 函式編排所有檢查。

### 3.1 SecurityAuditOptions 選項型別


關鍵參數說明：

| 參數 | 型別 | 用途 |
|------|------|------|
| `config` | `OpenClawConfig` | 當前有效配置（已解析秘密值） |
| `sourceConfig` | `OpenClawConfig` | 原始配置（未解析秘密值，用於檢測秘密來源） |

| `includeFilesystem` | `boolean` | 啟用檔案系統權限檢查 |
| `includeChannelSecurity` | `boolean` | 啟用頻道安全檢查 |
| `deepTimeoutMs` | `number` | 深度 Gateway 探測逾時 |
| `configSnapshot` | `ConfigFileSnapshot` | 預載配置快照，避免重複讀取 |

OpenClaw 大量使用依賴注入（Dependency Injection）模式，使得 `execIcacls`、`execDockerRawFn`、`probeGatewayFn` 等參數可在測試時替換為模擬實作（Mock）。

### 3.2 runSecurityAudit 主入口

`runSecurityAudit()` 是稽核系統的唯一公開入口，它按順序收集所有安全發現，最後彙總為 `SecurityAuditReport`。

```typescript
// source-repo/src/security/audit.ts:1273-1384
export async function runSecurityAudit(opts: SecurityAuditOptions): Promise<SecurityAuditReport> {
  const findings: SecurityAuditFinding[] = [];
  const context = await createAuditExecutionContext(opts);
  const { cfg, env, platform, stateDir, configPath } = context;
  const auditNonDeep = await loadAuditNonDeepModule();

  findings.push(...auditNonDeep.collectAttackSurfaceSummaryFindings(cfg));
  findings.push(...auditNonDeep.collectSyncedFolderFindings({ stateDir, configPath }));
  findings.push(...collectGatewayConfigFindings(cfg, context.sourceConfig, env));
  findings.push(...(await collectPluginSecurityAuditFindings(context)));
  findings.push(...collectLoggingFindings(cfg));
  findings.push(...collectElevatedFindings(cfg));
  findings.push(...collectExecRuntimeFindings(cfg));
  findings.push(...auditNonDeep.collectHooksHardeningFindings(cfg, env));
  findings.push(...auditNonDeep.collectGatewayHttpNoAuthFindings(cfg, env));
  // ... 更多同步檢查 ...
  findings.push(...auditNonDeep.collectExposureMatrixFindings(cfg));
  findings.push(...auditNonDeep.collectLikelyMultiUserSetupFindings(cfg));

  // 條件性檔案系統檢查
  if (context.includeFilesystem) {
    findings.push(...(await collectFilesystemFindings({ stateDir, configPath, env, platform })));
    // ... 深度檔案系統、workspace skill、sandbox browser hash、插件信任 ...
    findings.push(...(await collectDeepCodeSafetyFindings({ cfg, stateDir, deep: context.deep })));
  }

  // 條件性頻道安全檢查
  if (shouldAuditChannelSecurity) {
    const { collectChannelSecurityFindings } = await loadAuditChannelModule();
    findings.push(...(await collectChannelSecurityFindings({ cfg, sourceConfig, plugins })));
  }

  // 深度即時 Gateway 探測
  const deepProbeResult = context.deep ? await maybeProbeGateway({ cfg, env }) : undefined;
  findings.push(...collectDeepProbeFindings({ deep, authWarning: deepProbeResult?.authWarning }));

  const summary = countBySeverity(findings);
  return { ts: Date.now(), summary, findings, deep };
}
```

|---|-----------|----------|
| 1 | `collectAttackSurfaceSummaryFindings` | 攻擊面總覽 |
| 2 | `collectSyncedFolderFindings` | 雲端同步資料夾偵測（iCloud / Dropbox / OneDrive） |
| 3 | `collectGatewayConfigFindings` | Gateway 繫結位址、認證、Control UI、允許來源、信任代理 |
| 4 | `collectPluginSecurityAuditFindings` | 插件安全稽核 |
| 5 | `collectLoggingFindings` | 日誌遮罩設定 |
| 6 | `collectElevatedFindings` | 提權執行允許清單 |
| 7 | `collectExecRuntimeFindings` | 沙箱模式與執行主機設定 |
| 8 | `collectHooksHardeningFindings` | Hooks 強化 |
| 9 | `collectGatewayHttpNoAuthFindings` | Gateway HTTP 無認證端點 |
| 10 | `collectSandboxDockerNoopFindings` | Sandbox Docker 空操作 |
| 11 | `collectSandboxDangerousConfigFindings` | Sandbox 危險配置 |
| 12 | `collectNodeDenyCommandPatternFindings` | Node 指令拒絕模式 |
| 13 | `collectSecretsInConfigFindings` | 配置中的秘密值 |
| 14 | `collectModelHygieneFindings` | 模型衛生 |
| 15 | `collectSmallModelRiskFindings` | 小模型風險 |

此函式檢查兩類路徑的檔案權限：

**狀態目錄（State Directory）檢查：**

| 條件 | 嚴重等級 | checkId |
|------|---------|---------|
| 全域可寫（World-writable） | `critical` | `fs.state_dir.perms_world_writable` |
| 群組可寫（Group-writable） | `warn` | `fs.state_dir.perms_group_writable` |
| 他人可讀（Readable by others） | `warn` | `fs.state_dir.perms_readable` |
| 符號連結（Symlink） | `warn` | `fs.state_dir.symlink` |

**配置檔（Config File）檢查：**

| 條件 | 嚴重等級 | checkId |
|------|---------|---------|
| 他人可寫（Writable by others） | `critical` | `fs.config.perms_writable` |
| 全域可讀（World-readable） | `critical` | `fs.config.perms_world_readable` |
| 群組可讀（Group-readable） | `warn` | `fs.config.perms_group_readable` |
| 符號連結（Symlink） | `warn` | `fs.config.symlink` |

全域可寫狀態目錄被視為 `critical`，因為其他使用者可以竄改 OpenClaw 的狀態資料。配置檔的全域可讀同樣是 `critical`，因為配置可能包含 Token 和私密設定。

這是稽核系統中最複雜的單一收集器，涵蓋 Gateway 的多個面向：

| 檢查 | checkId | 條件 | 等級 |
|------|---------|------|------|
| 非迴環繫結無認證 | `gateway.bind_no_auth` | bind≠loopback 且無 token/password | `critical` |
| 迴環無認證 | `gateway.loopback_no_auth` | bind=loopback 且無任何 auth | `critical` |
| Control UI 缺少允許來源 | `gateway.control_ui.allowed_origins_required` | 非迴環 + 空 allowedOrigins | `critical` |
| 允許來源含萬用字元 | `gateway.control_ui.allowed_origins_wildcard` | allowedOrigins 含 `*` | `critical`/`warn` |
| Host 標頭回退 | `gateway.control_ui.host_header_origin_fallback` | `dangerouslyAllowHostHeaderOriginFallback=true` | `critical`/`warn` |
| 缺少信任代理 | `gateway.trusted_proxies_missing` | loopback + 空 trustedProxies | `warn` |
| X-Real-IP 回退 | `gateway.real_ip_fallback_enabled` | `allowRealIpFallback=true` | `critical`/`warn` |
| Tailscale Funnel | `gateway.tailscale_funnel` | `tailscale.mode="funnel"` | `critical` |
| Tailscale Serve | `gateway.tailscale_serve` | `tailscale.mode="serve"` | `info` |
| mDNS 完整模式 | `discovery.mdns_full_mode` | `mdns.mode="full"` | `critical`/`warn` |
| HTTP 工具重啟 | `gateway.tools_invoke_http.dangerous_allow` | gateway.tools.allow 含拒絕清單工具 | `critical`/`warn` |
| 不安全認證 | `gateway.control_ui.insecure_auth` | `allowInsecureAuth=true` | `warn` |
| 裝置認證停用 | `gateway.control_ui.device_auth_disabled` | `dangerouslyDisableDeviceAuth=true` | `critical` |

值得注意的是，許多檢查採用「雙重嚴重等級」策略：當 Gateway 暴露於非迴環網路時升級為 `critical`，僅在迴環時降為 `warn`。

### 3.5 執行環境檢查 (collectExecRuntimeFindings)

```typescript
// source-repo/src/security/audit.ts:804-950
export function collectExecRuntimeFindings(cfg: OpenClawConfig): SecurityAuditFinding[] {
  // ...
}
```

此收集器關注命令執行（Exec）的安全設定：

- **`exec.host=sandbox` 但 `sandbox.mode=off`**：執行會「封閉式失敗」（fail closed），因為沒有可用的沙箱執行環境
- **`exec.security=full`**：完整執行信任已啟用，若同時偵測到開放頻道存取，升級為 `critical`
- **開放頻道 + 執行啟用**：任何人可透過開放的 DM/群組觸及啟用了 exec 的代理
- **`autoAllowSkills` 啟用**：隱式技能二進位允許清單擴大了主機 exec 信任範圍
- **直譯器允許清單缺少 `strictInlineEval`**：允許清單中有 python、node 等直譯器但未啟用嚴格行內求值

## 4. 同步稽核 (audit-extra.sync.ts)

```typescript
// source-repo/src/security/audit-extra.sync.ts:3-7

 * Synchronous security audit collector functions.
```

同步稽核模組的核心特性是**完全不進行 I/O**——僅分析配置物件（Config Object），因此可以在任何環境下安全執行。

### Gateway 遠端暴露檢測

```typescript
// source-repo/src/security/audit-extra.sync.ts:120-127
function isGatewayRemotelyExposed(cfg: OpenClawConfig): boolean {
  const bind = typeof cfg.gateway?.bind === "string" ? cfg.gateway.bind : "loopback";
  if (bind !== "loopback") {
    return true;
  }
  const tailscaleMode = cfg.gateway?.tailscale?.mode ?? "off";
  return tailscaleMode === "serve" || tailscaleMode === "funnel";
}
```

此函式判斷 Gateway 是否暴露於遠端網路。只要 `bind` 不是 `loopback`，或者 Tailscale 模式為 `serve` 或 `funnel`，就視為遠端暴露。

### 核心決策邏輯

```typescript

export function evaluateSupplementalContextVisibility(params: {
  mode: ContextVisibilityMode;
  kind: ContextVisibilityKind;
  senderAllowed: boolean;
}): ContextVisibilityDecision {
  if (params.mode === "all") {
    return { include: true, reason: "mode_all" };
  }
  if (params.senderAllowed) {
    return { include: true, reason: "sender_allowed" };
  }
  if (params.mode === "allowlist_quote" && params.kind === "quote") {
    return { include: true, reason: "quote_override" };
  }
  return { include: false, reason: "blocked" };
}
```

決策流程（Decision Flow）如下：

```
mode === "all" ? ─────── 是 → include (mode_all)
                  │ 否
senderAllowed? ──── 是 → include (sender_allowed)
                  │ 否
mode === "allowlist_quote"
  && kind === "quote"? ── 是 → include (quote_override)
                  │ 否
                  └──────── blocked
```

```typescript
export function scanSource(source: string, filePath: string): SkillScanFinding[] {
  const findings: SkillScanFinding[] = [];
  const lines = source.split("\n");
  // 行級規則：每規則最多一個發現
  for (const rule of LINE_RULES) {
    if (rule.requiresContext && !rule.requiresContext.test(source)) continue;
    for (let i = 0; i < lines.length; i++) {
      const match = rule.pattern.exec(lines[i]);
      if (!match) continue;
      // suspicious-network 特殊處理：檢查埠號
      if (rule.ruleId === "suspicious-network") {
        const port = parseInt(match[1], 10);
        if (STANDARD_PORTS.has(port)) continue;
      }
      findings.push({ ruleId, severity, file, line: i + 1, message, evidence });
      break; // 每檔每規則一個
    }
  }
  // 來源級規則：每 (ruleId+message) 組合最多一個發現
  for (const rule of SOURCE_RULES) {
    if (!rule.pattern.test(source)) continue;
    if (rule.requiresContext && !rule.requiresContext.test(source)) continue;
    findings.push({ ruleId, severity, file, line: matchLine, message, evidence });
  }
  return findings;
}
```

```typescript
  "apply_patch",    // Patch 可改寫任意檔案
  "sessions_spawn", // 遠端生成代理即為 RCE
  "sessions_send",  // 跨 Session 訊息注入
  "cron",           // 持久性自動化控制平面
  "gateway",        // 防止透過 HTTP 重新配置 Gateway
  "nodes",          // Node 指令中繼可達 system.run
  "whatsapp_login", // 互動式設定，HTTP 上會掛起
] as const;
```

這是 Gateway HTTP `POST /tools/invoke` 端點的預設拒絕清單。這些工具之所以被封鎖，是因為它們在非互動式 HTTP 表面上構成高風險：

- **立即 RCE**：`exec`、`spawn`、`shell` 允許直接執行任意命令
- **檔案系統變更**：`fs_write`、`fs_delete`、`fs_move`、`apply_patch` 可修改主機檔案
- **Session 編排**：`sessions_spawn`、`sessions_send` 可遠端控制代理
- **控制平面**：`cron`、`gateway`、`nodes` 可改變系統行為

操作者可以透過 `gateway.tools.allow` 明確重啟這些工具，但稽核系統會在此時發出 `gateway.tools_invoke_http.dangerous_allow` 警告。

### safeChmod 安全權限修改

```typescript
// source-repo/src/security/fix.ts:49-114
async function safeChmod(params: {
  path: string;
  mode: number;
  require: "dir" | "file";
}): Promise<SecurityFixChmodAction> {
  const st = await fs.lstat(params.path);
  // 安全防護：跳過符號連結
  if (st.isSymbolicLink()) {
    return { kind: "chmod", path, mode, ok: false, skipped: "symlink" };
  }
  // 驗證目標類型
  if (params.require === "dir" && !st.isDirectory()) {
    return { kind: "chmod", path, mode, ok: false, skipped: "not-a-directory" };
  }
  // 檢查是否已是目標權限
  if ((st.mode & 0o777) === params.mode) {
    return { kind: "chmod", path, mode, ok: false, skipped: "already" };
  }
  await fs.chmod(params.path, params.mode);
  return { kind: "chmod", path, mode, ok: true };
}
```

`safeChmod` 的安全防護措施：

1. **跳過符號連結**：避免透過符號連結修改非預期檔案的權限
2. **驗證目標類型**：確保目錄操作只作用於目錄，檔案操作只作用於檔案
3. **冪等性**：若權限已正確，標記為 `skipped: "already"` 並不做修改
4. **路徑不存在**：返回 `skipped: "missing"` 而非拋出例外

## 13. 輔助安全模組

### 13.1 外部內容安全 (external-content.ts)

此模組處理來自外部來源（電子郵件、Webhook、API）的不受信任內容，確保它們在傳遞給 LLM 之前被安全包裝。

```typescript
// source-repo/src/security/external-content.ts:18-33
const SUSPICIOUS_PATTERNS = [
  /ignore\s+(all\s+)?(previous|prior|above)\s+(instructions?|prompts?)/i,
  /disregard\s+(all\s+)?(previous|prior|above)/i,
  /you\s+are\s+now\s+(a|an)\s+/i,
  /system\s*:?\s*(prompt|override|command)/i,
  /rm\s+-rf/i,
  // ... 更多模式
];
```

`wrapExternalContent()` 函式使用隨機 ID 的邊界標記（Boundary Markers）包裝外部內容，並附加安全警告：

```typescript
// source-repo/src/security/external-content.ts:54-67
const EXTERNAL_CONTENT_START_NAME = "EXTERNAL_UNTRUSTED_CONTENT";
function createExternalContentMarkerId(): string {
  return randomBytes(8).toString("hex");
}
```

每個包裝都有唯一的隨機 ID，防止惡意內容注入假的邊界標記（Spoofing Attack）。此外，`replaceMarkers()` 函式會清除內容中可能存在的欺騙性標記，包括 Unicode 同形字（Homoglyphs）和全形字元。

支援的外部內容來源類型：`email`、`webhook`、`api`、`browser`、`channel_metadata`、`web_search`、`web_fetch`、`unknown`。

### 13.2 安全正則表達式 (safe-regex.ts)

`safe-regex.ts` 預防 ReDoS（Regular Expression Denial of Service）攻擊。它透過分析正則表達式的巢狀重複（Nested Repetition）來判斷是否安全。

```typescript
// source-repo/src/security/safe-regex.ts:33
export type SafeRegexRejectReason = "empty" | "unsafe-nested-repetition" | "invalid-regex";
```

核心函式 `compileSafeRegexDetailed()` 先檢查巢狀重複，再嘗試編譯。結果被快取（最大 256 條目）以避免重複分析。`testRegexWithBoundedInput()` 限制輸入長度至 2048 字元，進一步降低 DoS 風險。

### 13.3 常數時間秘密比較 (secret-equal.ts)

```typescript
// source-repo/src/security/secret-equal.ts:1-12
import { createHash, timingSafeEqual } from "node:crypto";

export function safeEqualSecret(
  provided: string | undefined | null,
  expected: string | undefined | null,
): boolean {
  if (typeof provided !== "string" || typeof expected !== "string") return false;
  const hash = (s: string) => createHash("sha256").update(s).digest();
  return timingSafeEqual(hash(provided), hash(expected));
}
```

透過先進行 SHA-256 雜湊再使用 `timingSafeEqual` 比較，防止時序攻擊（Timing Attack）。雜湊步驟確保兩個輸入的長度相同（固定 32 位元組），避免長度差異洩漏資訊。

### 13.4 DM 政策共用 (dm-policy-shared.ts)

`dm-policy-shared.ts` 提供 DM（Direct Message）和群組存取的共用決策邏輯，被頻道安全稽核和實際的訊息處理流程共同使用。

## 14. 信任模型 (SECURITY.md)

OpenClaw 的信任模型在 `SECURITY.md` 中有詳盡的文件化。以下是核心概念摘要：

### 個人助理模型

> OpenClaw's security model is "personal assistant" (one trusted operator, potentially many agents), not "shared multi-tenant bus."

> — `source-repo/SECURITY.md:162`

OpenClaw **不是**多租戶系統。一個 Gateway 實例服務一位受信任的操作者（Operator），該操作者可以管理多個代理（Agent）。

### 操作者信任邊界

| 原則 | 說明 | 來源 |
|------|------|------|
| 非多租戶 | 一個 Gateway 不構成多租戶、對抗性使用者邊界 | `SECURITY.md:97-98` |
| 認證=信任 | 已認證的 Gateway 呼叫者被視為受信任的操作者 | `SECURITY.md:99` |
| 預設無沙箱 | `sandbox.mode` 預設為 `off`（主機優先） | `SECURITY.md:115` |
| 自動 exec 主機 | `tools.exec.host` 預設為 `auto` | `SECURITY.md:116` |
| 插件=信任基礎 | 插件是受信任的計算基礎（Trusted Computing Base） | `SECURITY.md:120-127` |
| 插件行為預期 | 插件讀取 env/檔案、執行主機指令是預期行為 | `SECURITY.md:126` |
| 單使用者建議 | 建議每台機器/VPS 一位使用者、一個 Gateway | `SECURITY.md:112-113` |

### 上下文可見性與允許清單

OpenClaw 區分兩個層面：

1. **觸發授權**（Trigger Authorization）：誰可以觸發代理（`dmPolicy`、`groupPolicy`、允許清單、提及閘門）
2. **上下文可見性**（Context Visibility）：什麼補充上下文提供給模型

允許清單主要管控觸發和所有者式指令存取，不保證在所有頻道/表面上普遍遮罩補充上下文。

### 範圍外事項

以下情況不在安全漏洞回報的範圍內：
- 無邊界突破的提示注入攻擊
- 多個不受信任使用者共用同一 Gateway 的情境
- 需要寫入受信任本地狀態的報告
- 已認證使用者觸發本地動作被視為特權提升

## 15. 事件回應計畫 (INCIDENT_RESPONSE.md)

OpenClaw 定義了完整的五階段事件回應流程：

### 五階段流程

| 階段 | 名稱 | 內容 |
|------|------|------|
| 1 | 偵測與分類（Detection and Triage） | 監控 GHSA、GitHub Issues、自動化信號（Dependabot、CodeQL） |
| 2 | 評估（Assessment） | 按嚴重等級分類 |
| 3 | 回應（Response） | 確認、重現、修補、發布 |
| 4 | 溝通（Communication） | GHSA、Release Notes、回報者追蹤 |
| 5 | 恢復與追蹤（Recovery and Follow-up） | 驗證修復、事後檢討、後續強化 |

### 嚴重等級定義

```
// source-repo/INCIDENT_RESPONSE.md:19-24
- Critical: 套件/發布/儲存庫入侵、主動利用、或未認證的信任邊界突破
- High:     已驗證的信任邊界突破（需有限前提條件）
- Medium:   顯著安全弱點，但可利用性受限或有重大前提
- Low:      縱深防禦、有限範圍的 DoS、或無已證明信任邊界突破的強化缺口
```

### 溝通策略

- **Critical/High**：協調揭露（Coordinated Disclosure），適當時發放 CVE
- **Low**：可能僅記錄在 Release Notes 或 Advisory 中，不發 CVE
- 始終有報告者追蹤和狀態回報
