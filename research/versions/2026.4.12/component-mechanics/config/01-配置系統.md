# OpenClaw 配置系統深度解析

## 本章摘要

OpenClaw 的配置系統（Configuration System）是整個框架的基石——它決定了 AI 代理（Agent）如何連接模型供應商（Provider）、透過哪些通道（Channel）與使用者互動、記憶系統如何運作、安全策略如何執行。本章將從原始碼出發，完整剖析 `src/config/` 目錄下超過 140 個非測試原始碼檔案的設計，涵蓋配置載入管線（Loading Pipeline）、JSON5 解析、環境變數替換（Environment Variable Substitution）、`$include` 模組化引入、Zod Schema 驗證、合併補丁策略（Merge-Patch）、執行期實體化（Runtime Materialization）、快照管理（Snapshot Management），以及各關鍵配置區段的型別定義。無論你是想理解 OpenClaw 配置為何如此設計，還是想動手修改或擴展它，本章都是你的起點。

## 3. `io.ts` — 配置檔 I/O 核心

`io.ts` 是整個配置系統中最大的檔案（約 1813 行），包含了配置載入、讀取、寫入、快照觀察等所有 I/O 操作。

### 3.1 `loadConfig()` — 公開載入函式

最外層的 `loadConfig()` 函式定義在第 1731 行：

```typescript
// source-repo/src/config/io.ts:1731-1736
export function loadConfig(): OpenClawConfig {
  // First successful load becomes the process snapshot. Long-lived runtimes
  // should swap this snapshot via explicit reload/watcher paths instead of
  // reparsing openclaw.json on hot code paths.
  return loadPinnedRuntimeConfig(() => createConfigIO().loadConfig());
}
```

關鍵設計：**首次成功載入會被釘選（Pin）為行程快照**。長駐執行期（如 Gateway 伺服器）應該透過明確的重新載入路徑或檔案監控（Watcher）來更新快照，而不是在每個熱路徑（Hot Path）重新解析 `openclaw.json`。

### 3.4 `writeConfigFile()` — 配置寫入

寫入流程（第 1763 行起）涉及多個精巧步驟：

```typescript
// source-repo/src/config/io.ts:1763-1800（簡化摘要）
export async function writeConfigFile(
  cfg: OpenClawConfig,
  options: ConfigWriteOptions = {},
): Promise<void> {
  const io = createConfigIO();
  let nextCfg = cfg;
  // 1. 從執行期快照計算合併補丁
  const runtimePatch = createMergePatch(runtimeConfigSnapshot!, cfg);
  nextCfg = applyMergePatch(runtimeConfigSourceSnapshot!, runtimePatch);
  // 2. 寫入檔案（含環境變數參照還原）
  const writeResult = await io.writeConfigFile(nextCfg, { ... });
  // 3. 完成執行期快照更新
  await finalizeRuntimeSnapshotWrite({ ... });
  // 4. 通知配置寫入監聽器
  notifyRuntimeConfigWriteListeners({ ... });
}
```

這裡的核心洞見是：寫入時不是直接覆寫檔案，而是計算「使用者實際想改什麼」（Merge Patch），然後將這個補丁套用到原始來源配置上，避免將執行期衍生的預設值寫回檔案。同時，含有 `${VAR}` 參照的欄位會被還原為原始參照形式，而非展開後的值。

> **引用來源**：`source-repo/src/config/io.ts:1763-1800`

## 6. 配置格式：JSON5

### 6.1 為什麼選擇 JSON5

OpenClaw 的配置檔使用 **JSON5** 格式。JSON5 是 JSON 的超集，支援：
- 註解（`//` 和 `/* */`）
- 結尾逗號（Trailing Commas）
- 單引號字串
- 未加引號的物件鍵
- 多行字串

這讓手動編輯配置檔的體驗大幅改善。解析使用 `json5` 套件：

```typescript
// source-repo/src/config/io.ts:5
import JSON5 from "json5";
```

### 6.2 配置檔路徑解析

路徑解析邏輯定義在 `paths.ts`：

```typescript
// source-repo/src/config/paths.ts:21-24
const LEGACY_STATE_DIRNAMES = [".clawdbot"] as const;
const NEW_STATE_DIRNAME = ".openclaw";
const CONFIG_FILENAME = "openclaw.json";
const LEGACY_CONFIG_FILENAMES = ["clawdbot.json"] as const;
```

路徑解析遵循以下優先順序：

| 優先順序 | 路徑來源 | 範例 |
|---------|---------|------|
| 1（最高） | `OPENCLAW_CONFIG_PATH` 環境變數 | 任意路徑 |
| 2 | `OPENCLAW_STATE_DIR` 環境變數 + 設定檔名 | `$STATE_DIR/openclaw.json` |
| 3（預設） | 使用者家目錄 + `.openclaw` | `~/.openclaw/openclaw.json` |
| 4（舊版相容） | 使用者家目錄 + `.clawdbot` | `~/.clawdbot/clawdbot.json` |

### 6.3 狀態目錄解析

```typescript
// source-repo/src/config/paths.ts:60-89（簡化）
export function resolveStateDir(
  env: NodeJS.ProcessEnv = process.env,
  homedir: () => string = envHomedir(env),
): string {
  const override = env.OPENCLAW_STATE_DIR?.trim();
  if (override) {
    return resolveUserPath(override, env, effectiveHomedir);
  }
  const newDir = newStateDir(effectiveHomedir);     // ~/.openclaw
  if (fs.existsSync(newDir)) return newDir;
  const existingLegacy = legacyDirs.find(fs.existsSync);
  if (existingLegacy) return existingLegacy;        // ~/.clawdbot
  return newDir;                                     // 回退到 ~/.openclaw
}
```

解析邏輯會自動偵測舊版目錄（`.clawdbot`）並向後相容，新安裝則使用 `.openclaw`。

## 7. 環境變數替換（Environment Variable Substitution）

### 7.1 語法規則

環境變數替換引擎定義在 `env-substitution.ts`（共 203 行），語法為 `${VAR_NAME}`：

```typescript
// source-repo/src/config/env-substitution.ts:27
const ENV_VAR_NAME_PATTERN = /^[A-Z_][A-Z0-9_]*$/;
```

**規則摘要**：

| 語法 | 行為 |
|------|------|
| `${VAR_NAME}` | 替換為環境變數的值 |
| `$${VAR_NAME}` | 跳脫（Escape）：輸出字面值 `${VAR_NAME}` |
| 僅大寫字母 | 只匹配 `[A-Z_][A-Z0-9_]*` 的變數名 |

### 7.2 使用範例

```json5
// 從 env-substitution.ts 檔頭文件註解
{
  models: {
    providers: {
      "vercel-gateway": {
        apiKey: "${VERCEL_GATEWAY_API_KEY}"
      }
    }
  }
}
```

### 7.3 遺失變數處理

當參照的環境變數不存在或為空時：

```typescript
// source-repo/src/config/env-substitution.ts:113-124
if (token?.kind === "substitution") {
  const envValue = env[token.name];
  if (envValue === undefined || envValue === "") {
    if (opts?.onMissing) {
      opts.onMissing({ varName: token.name, configPath });
      // Preserve the original placeholder so the value is visibly unresolved.
      chunks.push(`\${${token.name}}`);
      i = token.end;
      continue;
    }
    throw new MissingEnvVarError(token.name, configPath);
  }
  // ...
}
```

有兩種模式：
1. **嚴格模式**（預設）：拋出 `MissingEnvVarError`
2. **寬鬆模式**：提供 `onMissing` 回呼（Callback），保留原始佔位符（Placeholder）並記錄警告

### 7.4 遞迴替換

`resolveConfigEnvVars()` 函式遞迴走訪整個配置物件，對所有字串值執行替換：

```typescript
// source-repo/src/config/env-substitution.ts:197-203
export function resolveConfigEnvVars(
  obj: unknown,
  env: NodeJS.ProcessEnv = process.env,
  opts?: SubstituteOptions,
): unknown {
  return substituteAny(obj, env, "", opts);
}
```

內部的 `substituteAny` 函式處理三種型別：
- **字串**：執行替換
- **陣列**：遞迴每個元素
- **物件**：遞迴每個值
- **基本型別**（數字、布林、null）：原樣通過

## 8. 密鑰參照（Secret References）

### 8.1 SecretRef 型別

OpenClaw 提供了一套密鑰參照系統，讓配置中的敏感值不必以明文儲存：

```typescript
// source-repo/src/config/types.secrets.ts:3-16
export type SecretRefSource = "env" | "file" | "exec";

export type SecretRef = {
  source: SecretRefSource;
  provider: string;
  id: string;
};
```

三種密鑰來源：

| 來源 | 說明 | 範例 |
|------|------|------|

## 11. 配置合併策略

### 11.1 `merge-patch.ts` — RFC 7396 風格合併補丁

合併補丁演算法（97 行）實作了 RFC 7396（JSON Merge Patch）的精神，並加入了 **以 ID 合併陣列**（Merge Arrays by ID）的擴展：

```typescript
// source-repo/src/config/merge-patch.ts:62-97
export function applyMergePatch(
  base: unknown,
  patch: unknown,
  options: MergePatchOptions = {},
): unknown {
  if (!isPlainObject(patch)) return patch;

  const result: PlainObject = isPlainObject(base) ? { ...base } : {};

  for (const [key, value] of Object.entries(patch)) {
    if (isBlockedObjectKey(key)) continue;           // 原型污染防護
    if (value === null) { delete result[key]; continue; } // null = 刪除
    if (options.mergeObjectArraysById && ...) { ... } // ID 陣列合併
    if (isPlainObject(value)) {                       // 物件遞迴合併
      result[key] = applyMergePatch(...);
      continue;
    }
    result[key] = value;                              // 基本值直接覆蓋
  }
  return result;
}
```

**合併規則摘要**：

| 情境 | 行為 |
|------|------|
| Patch 值為 `null` | 從結果中刪除該鍵 |
| Patch 值為物件 | 遞迴深度合併 |
| Patch 值為陣列（啟用 ID 合併） | 依 `id` 欄位比對合併 |
| Patch 值為基本型別 | 直接覆蓋 |
| 鍵為 `__proto__` / `prototype` / `constructor` | 跳過（安全防護） |

```typescript
  // 基底陣列必須全部含有 string 型別的 id
```

**深度合併規則**（`deepMerge` 函式）：

```typescript
// source-repo/src/config/includes.ts:69-80
/** Deep merge: arrays concatenate, objects merge recursively, primitives: source wins */
export function deepMerge(target: unknown, source: unknown): unknown {
  if (Array.isArray(target) && Array.isArray(source)) {
    return [...target, ...source];           // 陣列：串接
  }
  if (isPlainObject(target) && isPlainObject(source)) {
    const result = { ...target };
    for (const key of Object.keys(source)) {
      if (isBlockedObjectKey(key)) continue;
      result[key] = key in result ? deepMerge(result[key], source[key]) : source[key];
    }
    return result;                           // 物件：遞迴合併
  }
  return source;                             // 基本值：來源覆蓋
}
```

注意：`$include` 的 `deepMerge` 和 `merge-patch.ts` 的 `applyMergePatch` 行為不同——前者陣列是串接，後者陣列是依 ID 合併或覆蓋。

```typescript
export function materializeRuntimeConfig(
  config: OpenClawConfig,
  mode: ConfigMaterializationMode,
): RuntimeConfig {
  const profile = MATERIALIZATION_PROFILES[mode];
  let next = applyMessageDefaults(config);        // 1. 訊息預設值
  if (profile.includeLoggingDefaults) {
    next = applyLoggingDefaults(next);             // 2. 日誌預設值
  }
  next = applySessionDefaults(next);               // 3. 會話預設值
  next = applyAgentDefaults(next);                 // 4. 代理預設值
  if (profile.includeContextPruningDefaults) {
    next = applyContextPruningDefaults(next);      // 5. 上下文剪枝預設值
  }
  if (profile.includeCompactionDefaults) {
    next = applyCompactionDefaults(next);          // 6. 壓縮預設值
  }
  next = applyModelDefaults(next);                 // 7. 模型預設值
  next = applyTalkConfigNormalization(next);       // 8. 語音配置正規化
  if (profile.normalizePaths) {
    normalizeConfigPaths(next);                    // 9. 路徑正規化
  }
  normalizeExecSafeBinProfilesInConfig(next);      // 10. 安全執行檔設定正規化
  return asRuntimeConfig(next);
}
```

## 15. 配置系統設計哲學總結

綜觀 OpenClaw 的配置系統，可以歸納出以下設計原則：

1. **嚴格驗證，友善提示**：使用 Zod Schema 進行嚴格的型別檢查，但產生的錯誤訊息包含完整路徑和建議
2. **安全第一**：原型污染防護、密鑰參照系統、配置脫敏顯示
3. **漸進式複雜度**：簡單場景只需幾行 JSON5，進階場景可使用 `$include`、環境變數替換、密鑰參照
4. **向後相容**：舊版目錄名與配置檔名的自動偵測與遷移
5. **防禦性 I/O**：健康指紋偵測配置被破壞、備份輪轉確保可恢復、寫入時保留環境變數參照
6. **關注點分離**：142 個非測試檔案各自負責一個明確的責任，透過 Barrel 檔案統一匯出
