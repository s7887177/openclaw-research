# 診斷與可觀測性

## 本章摘要

本章介紹 OpenClaw 的進階診斷與可觀測性工具，包括 OpenTelemetry Extension 整合、Session Logs Skill、Model Usage Skill 與 Doctor 命令的診斷能力。這些工具讓操作者能夠追蹤 AI 使用量、分析會話日誌，並在系統出現異常時快速定位問題。

---

## 1. OpenTelemetry 診斷 Extension

### 1.1 概述

OpenClaw 提供 `diagnostics-otel` Extension，將系統的 Trace、Metrics 和 Logs 匯出到支援 OpenTelemetry（OTEL）的後端（如 Jaeger、Grafana、Datadog）。

### 1.2 Plugin 清單

```json
{
  "id": "diagnostics-otel",
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {}
  }
}
```
> 引用：`source-repo/extensions/diagnostics-otel/openclaw.plugin.json:1-8`

### 1.3 架構

Extension 由以下檔案組成：

| 檔案 | 功能 |
|------|------|
| `index.ts` | Plugin 入口，服務註冊 |
| `api.ts` | 從 `openclaw/plugin-sdk/diagnostics-otel` re-export |
| `src/service.ts` | OTEL 服務核心實作 |
| `src/service.test.ts` | 單元測試 |

### 1.4 服務實作

```typescript

```

核心函式：

| 函式 | 行號 | 功能 |
|------|------|------|
| `normalizeEndpoint()` | 17-20 | 正規化 OTEL 端點（移除尾端斜線）|

在 Docker 建構時加入 Extension：

```bash

```

## 4. Doctor 診斷能力

### 4.1 Doctor 工作流

Doctor 是 OpenClaw 最強大的診斷工具。完整的工作流：

```
openclaw doctor
    │
    ├── 檢查更新 → maybeOfferUpdateBeforeDoctor()
    ├── UI 協定新鮮度 → maybeRepairUiProtocolFreshness()
    ├── 原始碼安裝問題 → noteSourceInstallIssues()
    ├── 啟動最佳化提示 → noteStartupOptimizationHints()
    ├── 配置載入與遷移 → loadAndMaybeMigrateDoctorConfig()
    │
    └── Health Contributions:
        ├── [認證] OAuth Profile、Auth Profile、Codex Override
        ├── [Gateway] Health、Memory、Daemon、Service Config
        ├── [Plugin] Manifests、Bundled Deps
        ├── [Sandbox] Images、Scope Warnings
        ├── [記憶] Recall、Search
        ├── [Channel] Plugin Startup Maintenance
        ├── [Cron] Legacy Store Migration
        ├── [安全] Security Warnings
        ├── [狀態] State Integrity、Workspace Backup
        ├── [平台] macOS LaunchAgent、systemd Linger
        └── [其他] Bootstrap Size、Chrome Browser、Claude CLI
```
> 引用：`source-repo/src/flows/doctor-health.ts:33-62`, `source-repo/src/flows/doctor-health-contributions.ts:1-55`

### 4.2 診斷分類

Doctor 的檢查可分為三類：

1. **note*（觀察）**：以 `note` 為前綴的函式只回報狀態，不做修改
2. **maybeRepair*（修復）**：以 `maybeRepair` 為前綴的函式會嘗試自動修復
3. **check*（檢查）**：以 `check` 為前綴的函式執行深度檢查

當配置檔無效時，大多數 CLI 命令會被阻擋，但 Doctor 能繞過這個防護：

## 5. 常見問題排除

### 5.1 Gateway 無法啟動

| 檢查項目 | 命令 |
|---------|------|
| 埠號衝突 | `lsof -i :18789` |
| 配置錯誤 | `openclaw doctor` |
| Daemon 狀態 | `openclaw daemon status` |
| 日誌檢查 | `openclaw logs` |

### 5.2 Channel 連線失敗

| 症狀 | 可能原因 | 解決方式 |
|------|---------|---------|
| 斷線後不重連 | Token 過期 | 重新設定 Channel 認證 |
| 連線逾時 | 網路問題 | 檢查防火牆設定 |
| 重複重啟 | Channel 程式錯誤 | 檢查 Channel 健康日誌 |

Channel 健康監控器會在以下情況自動重啟 Channel：
- `stale-socket`：Socket 超過 30 分鐘無事件
- `disconnected`：Channel 意外斷線
- `stuck`：Channel 卡在某個狀態

> 引用：`source-repo/src/gateway/channel-health-policy.ts:41-46, 55-56`

### 5.3 記憶系統異常

```bash
openclaw doctor   # 會自動檢查並修復記憶系統
```

Doctor 的記憶相關檢查：
- `maybeRepairMemoryRecallHealth` — 修復記憶召回
- `noteMemorySearchHealth` — 檢查搜尋索引
- `probeGatewayMemoryStatus` — 探測 Gateway 記憶狀態

> 引用：`source-repo/src/flows/doctor-health-contributions.ts:30-33`

### 5.4 Cron 任務不執行

```bash
openclaw cron list    # 檢查排程狀態
openclaw doctor       # 修復 Cron 儲存
```

Doctor 的 `maybeRepairLegacyCronStore` 會修復舊版 Cron 資料格式。

> 引用：`source-repo/src/flows/doctor-health-contributions.ts:22`
