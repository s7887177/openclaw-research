# 第一部分：願景與架構設計

## 本章摘要

本章從第一性原理出發，定義「Discord 語音社交機器人」的完整使用場景與核心需求，然後基於 OpenClaw v2026.4.12 的實際程式碼，設計五大技術管線：聆聽管線（Listening Pipeline）、語境追蹤引擎（Context Tracking Engine）、插話決策引擎（Interjection Decision Engine）、語音輸出管線（Speaking Pipeline）、記憶與人格系統（Memory & Personality System）。每個管線都有完整的資料流定義與 OpenClaw 程式碼引用。

---

## 1. 完整使用場景

### 1.1 場景描述

想像一個典型的 Discord 語音頻道場景：

> 週五晚上十點，Alice、Bob、Carol 三個人在 #chill-zone 語音頻道裡閒聊。ClawBot 也在頻道裡——它是兩週前被邀請加入的 AI。
>
> Alice 正在講她今天在公司遇到的事：「我跟你們說，今天那個新來的 PM 又改需求了⋯⋯」
>
> Bob 附和：「每次都這樣啊，上次不是也是嗎？」
>
> 過了一會兒，話題轉到晚餐。Carol 說：「我今天吃了那家新開的拉麵，超好吃。」
>
> ClawBot 插話：「Carol，你上次不是說想試那家嗎？怎麼樣，湯頭是豚骨的嗎？」
>
> Carol 驚喜：「對！你記得啊！是豚骨的，而且叉燒超厚。」
>
> 後來話題轉到技術討論，Bob 問了一個 Python 問題但沒人回答。沉默了十秒後，ClawBot 主動回答了這個問題。
>
> 整個晚上，ClawBot 大約插了五六次話——不多不少，就像一個安靜但參與度剛好的朋友。

### 1.2 與傳統語音 Bot 的根本差異

傳統語音 Bot 是「命令—回應」模式（Command-Response Pattern）：

```
傳統模式：
用戶說 "Hey Bot, 今天天氣怎樣？" → Bot 回答天氣

社交模式（我們的目標）：
[持續聆聽所有人對話]
[理解話題脈絡]
[判斷：現在適合說話嗎？]
  → 不適合 → 繼續安靜聆聽
  → 適合 → 生成回應 → 說話
```

這個根本差異帶來三個核心挑戰：

| 挑戰 | 傳統 Bot | 社交 Bot |
|------|----------|----------|
| **觸發機制** | 明確的喚醒詞/命令 | 自主判斷何時插話 |
| **上下文** | 單一指令 | 持續的多人對話串流 |
| **記憶** | 無狀態或短期 | 跨 session 的長期人際記憶 |

### 1.3 核心需求拆解

#### 需求 1：聆聽（Listening）

持續接收語音頻道中所有人的聲音，不遺漏任何片段。

- **技術需求**：連接 Discord Voice Gateway，接收所有參與者的 Opus 音訊串流
- **關鍵指標**：零丟包（在網路正常的情況下）、支援同時 10+ 人說話

#### 需求 2：理解（Understanding）

將語音轉為文字，並知道「誰」在說「什麼」。

- **技術需求**：即時 STT（Speech-to-Text）、語者辨識（Speaker Identification）
- **關鍵指標**：STT 延遲 < 2 秒、語者辨識準確率 > 95%

#### 需求 3：記憶（Memory）

記住每個人的名字、個性、過去的對話。

- **技術需求**：per-person 向量記憶、語義搜尋、記憶壓縮
- **關鍵指標**：能記住過去 100+ 次互動的重點

#### 需求 4：判斷（Judgment）

知道什麼時候該說話、什麼時候該沉默。

- **技術需求**：規則觸發器 + LLM 判斷 + 信心閾值
- **關鍵指標**：插話命中率 > 80%（插話後被正面回應的比例）

#### 需求 5：發言（Speaking）

用自然的語音回應。

- **技術需求**：TTS 合成、語音個性一致、低延遲播放
- **關鍵指標**：端到端延遲（從決定說話到聲音輸出）< 3 秒

#### 需求 6：社交（Social）

有禮貌、有個性、能建立長期關係。

- **技術需求**：人格系統、關係追蹤、情感模型
- **關鍵指標**：使用者主動邀請 Bot 加入頻道的意願

## 2. 語音輸入管線（Listening Pipeline）

### 2.1 Discord Voice Channel 音訊串流接入

OpenClaw 的 Discord 語音管理器（`DiscordVoiceManager`）已實現完整的語音頻道連接流程。以下是核心連接邏輯：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:446-455
const voiceSdk = loadDiscordVoiceSdk();
const connection = voiceSdk.joinVoiceChannel({
  channelId,
  guildId,
  adapterCreator,
  selfDeaf: false,   // 不靜音自己——需要聽到所有人
  selfMute: false,   // 不靜麥——需要能說話
  daveEncryption,
  decryptionFailureTolerance,
});
```

關鍵觀察：`selfDeaf: false` 確保 Bot 能接收所有人的音訊。這是「聆聽」能力的基礎。

### 2.2 多人語音分軌處理

Discord.js Voice 的一個關鍵能力是**按用戶分軌接收**（per-user audio streams）。OpenClaw 已利用此能力：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:679-684
const stream = entry.connection.receiver.subscribe(userId, {
  end: {
    behavior: voiceSdk.EndBehaviorType.Manual,
  },
});
const generation = beginVoiceCapture(entry.capture, userId, stream);
```

這段程式碼展示了一個關鍵的設計決策：

1. **`receiver.subscribe(userId)`**：Discord.js 提供 per-user 的 Opus 串流，而非混音後的音訊
2. **`EndBehaviorType.Manual`**：手動控制串流結束，允許自訂靜默偵測邏輯
3. **`beginVoiceCapture`**：使用 generation 計數器管理並發捕獲，避免競態條件

**Voice Capture State 的精巧設計**：

```typescript
// source-repo/extensions/discord/src/voice/capture-state.ts:1-27
export type VoiceCaptureState = {
  activeSpeakers: Set<string>;           // 當前正在說話的用戶
  activeCaptureStreams: Map<string, VoiceCaptureEntry>;  // userId → 串流
  captureFinalizeTimers: Map<string, VoiceCaptureFinalizeTimer>;
  captureGenerations: Map<string, number>;  // 防止過時回調
};
```

這個狀態機解決了語音捕獲中的幾個關鍵問題：
- **重疊說話**：多人同時說話時，每人有獨立串流
- **短暫停頓**：`CAPTURE_FINALIZE_GRACE_MS = 1_200`（1.2 秒寬限期），避免把一句話切成兩段
- **Generation 計數**：防止延遲回調覆蓋新的捕獲

### 2.3 Opus 解碼

Discord 傳輸的音訊格式是 Opus 編碼。OpenClaw 支援兩種 Opus 解碼器：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:178-203
const factories: OpusDecoderFactory[] = [
  {
    name: "@discordjs/opus",      // 首選：原生 C++ 綁定，效能最佳
    load: () => {
      const DiscordOpus = require("@discordjs/opus");
      return new DiscordOpus.OpusEncoder(SAMPLE_RATE, CHANNELS);
    },
  },
  {
    name: "opusscript",           // 備選：純 JS 實現，相容性好
    load: () => {
      const OpusScript = require("opusscript");
      return new OpusScript(SAMPLE_RATE, CHANNELS, OpusScript.Application.AUDIO);
    },
  },
];
```

音訊參數：

| 參數 | 值 | 說明 |
|------|-----|------|
| Sample Rate | 48,000 Hz | Discord 標準 |
| Channels | 2 (Stereo) | 雙聲道 |
| Bit Depth | 16-bit | PCM 格式 |
| Min Segment | 0.35s | 最短可處理片段 |

引用來源：`source-repo/extensions/discord/src/voice/manager.ts:54-58`

### 2.4 STT 轉寫

解碼後的 PCM 音訊被封裝為 WAV 檔案，然後送入轉寫引擎：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:293-305
async function transcribeAudio(params: {
  cfg: OpenClawConfig;
  agentId: string;
  filePath: string;
}): Promise<string | undefined> {
  const result = await getDiscordRuntime().mediaUnderstanding.transcribeAudioFile({
    filePath: params.filePath,
    cfg: params.cfg,
    agentDir: resolveAgentDir(params.cfg, params.agentId),
    mime: "audio/wav",
  });
  return normalizeOptionalString(result.text);
}
```

目前的轉寫是**批次模式**（batch mode）：等一段話說完，轉為 WAV，再送入轉寫。這對「回應式」互動已足夠，但對社交 Bot 的「持續聆聽」場景有改進空間。

OpenClaw 同時有**串流轉寫**的基礎設施——Real-time Transcription Provider：

```typescript
// source-repo/src/realtime-transcription/provider-types.ts:17-33
export type RealtimeTranscriptionSessionCallbacks = {
  onPartial?: (partial: string) => void;     // 即時部分轉寫
  onTranscript?: (transcript: string) => void; // 完成的轉寫
  onSpeechStart?: () => void;                  // 偵測到語音開始
  onError?: (error: Error) => void;
};

export type RealtimeTranscriptionSession = {
  connect(): Promise<void>;
  sendAudio(audio: Buffer): void;   // 持續餵入音訊
  close(): void;
  isConnected(): boolean;
};
```

**社交 Bot 的改進方向**：使用 Realtime Transcription Provider 取代批次轉寫，實現「邊聽邊轉」。這需要將 Discord per-user Opus 串流直接接入 `RealtimeTranscriptionSession.sendAudio()`。

### 2.5 語者辨識（Speaker Identification）

OpenClaw 目前的語者辨識是基於 **Discord userId 映射**，而非聲紋（Voice Print）：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:1048-1089
private async resolveSpeakerIdentity(
  guildId: string,
  userId: string,
): Promise<{
  id: string;
  label: string;        // 暱稱或用戶名
  name?: string;        // Discord 用戶名
  tag?: string;         // 例如 "Alice#1234"
  memberRoleIds: string[];
}> {
  try {
    const member = await this.params.client.fetchMember(guildId, userId);
    return {
      id: userId,
      label: member.nickname ?? member.user?.globalName ?? username ?? userId,
      name: username,
      tag: member.user ? formatDiscordUserTag(member.user) : undefined,
      memberRoleIds: /* ... */,
    };
  } catch { /* fallback to fetchUser */ }
}
```

**重要洞察**：因為 Discord.js 的 `connection.receiver.subscribe(userId)` 天然提供 per-user 串流，我們**不需要傳統的語者分離（Speaker Diarization）**。Discord 的語音架構已經替我們解決了「誰在說話」的問題。

這是一個巨大的技術優勢——語者分離是語音 AI 中最困難的問題之一，而 Discord 的架構直接幫我們繞過了它。

## 3. 語境追蹤引擎（Context Tracking Engine）

### 3.1 設計目標

語境追蹤引擎是社交 Bot 的「大腦皮層」——它維護一個對頻道當前狀態的理解。傳統的聊天 Bot 只需要一個訊息歷史列表，但語音社交 Bot 需要更豐富的狀態：

```typescript
// 需要新開發的 Room State 資料結構
type RoomState = {
  // 誰在場
  participants: Map<string, ParticipantInfo>;
  
  // 當前對話
  activeDialogue: DialogueSegment[];    // 滾動 context window
  currentTopic: TopicInfo | null;
  topicHistory: TopicInfo[];
  
  // Bot 自身狀態
  lastSpokeAt: number | null;
  speakCount: number;                   // 本次 session 的發言次數
  silenceSince: number | null;          // 最後一次有人說話的時間
};

type ParticipantInfo = {
  userId: string;
  displayName: string;
  joinedAt: number;
  lastSpokeAt: number | null;
  turnCount: number;                    // 說了多少輪
  recentTopics: string[];               // 最近聊的話題
};

type DialogueSegment = {
  speakerId: string;
  speakerName: string;
  text: string;
  timestamp: number;
  isBot: boolean;
};

type TopicInfo = {
  summary: string;
  keywords: string[];
  startedAt: number;
  initiatedBy: string;
};
```

### 3.2 對話歷史的滾動 Context Window

語音頻道的對話可能持續數小時。我們不可能把所有對話都放進 LLM 的 context window。需要一個智慧的滾動策略：

```
Window 策略：
├── 即時 Buffer（最近 2 分鐘）：完整逐字記錄
├── 短期摘要（2-10 分鐘前）：每分鐘一個摘要句
├── 中期摘要（10-60 分鐘前）：每 5 分鐘一個摘要
└── 長期：寫入記憶系統（LanceDB）
```

OpenClaw 的 Context Engine 已有類似的分層機制：

```typescript
// source-repo/src/context-engine/types.ts:6-13
export type AssembleResult = {
  messages: AgentMessage[];           // 組裝好的消息
  estimatedTokens: number;            // token 預算管理
  systemPromptAddition?: string;      // 動態系統提示補充
};
```

Context Engine 的 `assemble()` 方法負責在 token 預算內組裝最相關的上下文。我們需要擴展它來處理語音頻道的滾動對話。

### 3.3 話題偵測與轉換追蹤

話題偵測是插話決策的重要依據。當話題轉到 Bot 擅長的領域時，這是一個適合插話的信號。

實作方式：

1. **輕量級方法**：用關鍵字提取（TF-IDF 或簡單的名詞提取）追蹤話題
2. **LLM 方法**：每 N 輪對話後，用 LLM 生成一個話題摘要
3. **混合方法**：關鍵字做初篩，話題變化時用 LLM 確認

建議採用混合方法——關鍵字提取幾乎不耗成本，LLM 只在話題可能變化時調用。

### 3.4 與 OpenClaw Context Engine 的整合

OpenClaw 的 Context Engine 是可插拔的（pluggable registry）：

```typescript

// registerContextEngineForOwner() — 受信任路徑（可替換引擎）
```

## 4. 插話決策引擎（Interjection Decision Engine）

### 4.1 核心挑戰

**「何時插話？」是整個系統最關鍵的設計問題。**

插話太多 → 令人煩躁，像個話太多的人
插話太少 → 不像在參與，像個旁聽者
插話時機不對 → 打斷別人，不禮貌

這需要一個多層次的決策系統：

```
                    觸發事件
                       │
                       ▼
              ┌────────────────┐
              │  Rule Triggers  │  ← 快速、零成本
              │  (規則層)       │
              └───────┬────────┘
                      │ 候選事件
                      ▼
              ┌────────────────┐
              │  LLM Judge      │  ← 精確、有成本
              │  (判斷層)       │
              └───────┬────────┘
                      │ 信心分數
                      ▼
              ┌────────────────┐
              │  Throttle &     │  ← 頻率控制
              │  Politeness     │
              │  (禮貌層)       │
              └───────┬────────┘
                      │ 最終決定
                      ▼
               Speak / Stay Silent
```

### 4.2 規則層：零成本觸發器

規則觸發器（Rule Triggers）不呼叫 LLM，基於簡單的條件判斷：

```typescript
// 需要新開發的 Rule Triggers
type InterjectionTrigger = {
  id: string;
  priority: number;          // 1 = 最高
  check: (state: RoomState, event: DialogueEvent) => TriggerResult;
};

type TriggerResult = {
  triggered: boolean;
  confidence: number;        // 0.0 - 1.0
  reason: string;
  skipLlmJudge?: boolean;    // 高信心時跳過 LLM
};
```

**觸發器清單**：

| 觸發器 | 條件 | 信心度 | 是否跳過 LLM |
|--------|------|--------|--------------|
| **被直接點名** | 有人說 Bot 名字 / @mention | 0.95 | ✅ 是 |
| **被問問題** | 有人對 Bot 提問 | 0.90 | ✅ 是 |
| **問題無人回答** | 有人問問題，沉默 > 8 秒 | 0.70 | ❌ 否 |
| **長時間沈默** | 所有人沉默 > 15 秒 | 0.50 | ❌ 否 |
| **話題相關** | 討論到 Bot 知道的話題 | 0.40 | ❌ 否 |
| **打招呼** | 新人加入頻道 | 0.60 | ❌ 否 |
| **有趣/搞笑時刻** | 偵測到笑聲/有趣言論 | 0.30 | ❌ 否 |

### 4.3 判斷層：LLM 精確評估

當規則層觸發但信心不足時（`confidence < 0.85 && !skipLlmJudge`），送入 LLM 判斷：

```typescript
// LLM Judge 的 prompt 結構
const judgePrompt = `
You are a social awareness module for an AI participating in a voice chat.

Current room state:
- Participants: ${participants.map(p => p.displayName).join(', ')}
- Current topic: ${currentTopic}
- Last speaker: ${lastSpeaker} (${timeSinceLastSpeech}s ago)
- Bot's last speech: ${timeSinceBotLastSpoke}s ago
- Bot's speech count this session: ${speakCount}

Recent dialogue (last 2 minutes):
${recentDialogue}

A potential interjection was triggered:
- Trigger: "${trigger.reason}"
- Confidence: ${trigger.confidence}

Should the AI speak now? Consider:
1. Would speaking add value to the conversation?
2. Would it be polite/natural at this moment?
3. Is the timing right (no one is mid-sentence)?

Respond with JSON:
{
  "shouldSpeak": boolean,
  "confidence": number (0.0-1.0),
  "suggestedResponse": "brief hint of what to say",
  "reasoning": "why this decision"
}
`;
```

### 4.4 禮貌層：頻率控制與社交禮儀

即使 LLM 判斷「應該說話」，禮貌層仍會進行最終檢查：

```typescript
// 禮貌策略規則
type PolitenessRules = {
  // 等人說完：語音活動偵測（VAD）靜默期
  minSilenceBeforeSpeaking: number;       // 至少等 1.5 秒靜默
  
  // 頻率控制
  minIntervalBetweenSpeaks: number;       // 兩次發言至少間隔 30 秒
  maxSpeaksPerMinute: number;             // 每分鐘最多 2 次
  maxConsecutiveSpeaks: number;           // 不連續說超過 2 次
  
  // 對話佔比
  maxTurnRatio: number;                   // Bot 發言不超過總輪次的 20%
  
  // 不打斷規則
  neverInterruptOngoing: boolean;         // 有人正在說話時絕不插嘴
  respectTurnTaking: boolean;             // 尊重發言順序
};
```

### 4.5 VAD（Voice Activity Detection）整合

OpenClaw 的 Voice Capture State 已實現基礎的 VAD 邏輯：

### 5.2 語音回應的完整流程

OpenClaw 的 Discord 語音管理器已實現完整的 TTS → 播放管線：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:805-851
// 1. 解析 TTS 配置
const { cfg: ttsCfg, resolved: ttsConfig } = resolveVoiceTtsConfig({
  cfg: this.params.cfg,
  override: this.params.discordConfig.voice?.tts,
});

// 2. 解析語音指令（支援行內覆寫）
const directive = parseTtsDirectives(replyText, ttsConfig.modelOverrides, {
  cfg: ttsCfg,
  providerConfigs: ttsConfig.providerConfigs,
});

// 3. 清理文字（去掉 emoji、標籤等不適合語音的內容）
const speakText = sanitizeVoiceReplyTextForSpeech(rawSpeakText, speaker.label);

// 4. TTS 合成
const ttsResult = await getDiscordRuntime().tts.textToSpeech({
  text: speakText,
  cfg: ttsCfg,
  channel: "discord",
  overrides: directive.overrides,
});

// 5. 排隊播放（避免多個回應重疊）
this.enqueuePlayback(entry, async () => {
  const resource = voiceSdk.createAudioResource(audioPath);
  entry.player.play(resource);
  // 等待播放開始
  await voiceSdk.entersState(entry.player, voiceSdk.AudioPlayerStatus.Playing, ...);
  // 等待播放結束
  await voiceSdk.entersState(entry.player, voiceSdk.AudioPlayerStatus.Idle, ...);
});
```

### 5.3 延遲管理

端到端延遲的組成：

```
STT 轉寫:     ~1-3s  (取決於片段長度和 provider)
LLM 生成回應:  ~1-3s  (取決於模型和回應長度)
TTS 合成:      ~1-2s  (取決於文字長度和 provider)
Discord 播放:  ~0.2s  (本地音訊資源載入)
─────────────────────
總計:          ~3-8s
```

**降低延遲的策略**：

1. **串流 TTS**：不等完整回應生成完畢就開始 TTS（需要 TTS provider 支援 streaming）
2. **預熱連接**：保持 TTS provider 連接活躍，避免冷啟動
3. **短回應優先**：社交場景的回應應該簡短（1-2 句話），而非長篇大論
4. **本地 TTS fallback**：延遲敏感時切換到 sherpa-onnx 本地推理

### 5.4 中斷處理

OpenClaw 已實現「被打斷時停止播放」的邏輯：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:675-677
// 當有人開始說話時，停止當前播放
if (entry.player.state.status === voiceSdk.AudioPlayerStatus.Playing) {
  entry.player.stop(true);   // force stop
}
```

這個邏輯確保 Bot 不會「自顧自地說」而忽略別人。當任何人開始說話，Bot 立即停止。這是社交禮貌的基本要素。

## 6. 記憶與人格系統（Memory & Personality System）

### 6.1 Per-Person 記憶架構

社交 Bot 的記憶不同於一般 chatbot——它需要「認識人」。每個互動過的人都應該有一份記憶檔案：

```typescript
// 需要新開發的 Per-Person Memory Schema
type PersonMemory = {
  // 基礎資訊
  userId: string;
  displayName: string;
  knownNames: string[];           // 可能有多個暱稱
  firstMetAt: number;
  lastSeenAt: number;
  totalInteractions: number;
  
  // 語者特徵（未來可加入聲紋）
  voiceCharacteristics?: {
    typicalVolume?: "quiet" | "normal" | "loud";
    speakingSpeed?: "slow" | "normal" | "fast";
    // 未來：voiceEmbedding?: number[];
  };
  
  // 個性特徵標籤
  personalityTags: string[];      // e.g., ["幽默", "技術宅", "貓奴"]
  interests: string[];            // e.g., ["Python", "動漫", "咖啡"]
  
  // 互動歷史摘要
  recentInteractions: InteractionSummary[];  // 最近 20 次互動摘要
  memorableQuotes: string[];      // 印象深刻的話
  
  // 關係指標
  relationship: {
    closeness: number;            // 0.0 - 1.0 親密度
    sentiment: number;            // -1.0 到 1.0 情感傾向
    trustLevel: number;           // 0.0 - 1.0 信任度
    interactionQuality: number;   // 歷史互動品質
  };
  
  // 偏好
  preferences: {
    humorStyle?: string;
    communicationStyle?: "casual" | "formal" | "mixed";
    topicsToAvoid?: string[];
    favoriteTopics?: string[];
  };
};

type InteractionSummary = {
  timestamp: number;
  context: string;          // "語音頻道閒聊" / "技術討論"
  highlights: string[];     // 關鍵互動點
  sentimentDelta: number;   // 這次互動對關係的影響
};
```

### 6.2 與 OpenClaw 記憶系統的整合

OpenClaw 的 LanceDB 記憶系統可以作為 per-person 記憶的儲存後端：

