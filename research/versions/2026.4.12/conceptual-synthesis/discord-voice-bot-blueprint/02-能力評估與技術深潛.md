## 2. 關鍵程式碼深度分析

### 2.1 processSegment()——現有管線的核心

`DiscordVoiceManager.processSegment()` 是整個語音互動的核心方法。理解它就理解了現有管線的全貌：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:719-852
// 完整的 processSegment 流程

private async processSegment(params: {
  entry: VoiceSessionEntry;
  wavPath: string;
  userId: string;
  durationSeconds: number;
}) {
  const { entry, wavPath, userId, durationSeconds } = params;

  // 步驟 1：解析說話者身份
  const speaker = await this.resolveSpeakerContext(entry.guildId, userId);
  const speakerIdentity = await this.resolveSpeakerIdentity(entry.guildId, userId);

  // 步驟 2：權限驗證
  const access = await authorizeDiscordVoiceIngress({
    cfg: this.params.cfg,
    discordConfig: this.params.discordConfig,
    guildId: entry.guildId,
    channelId: entry.channelId,
    memberRoleIds: speakerIdentity.memberRoleIds,
    sender: { id: speakerIdentity.id, name: speakerIdentity.name, tag: speakerIdentity.tag },
  });
  if (!access.ok) return;

  // 步驟 3：音訊轉文字
  const transcript = await transcribeAudio({
    cfg: this.params.cfg,
    agentId: entry.route.agentId,
    filePath: wavPath,
  });
  if (!transcript) return;

  // 步驟 4：格式化 prompt
  const prompt = formatVoiceIngressPrompt(transcript, speaker.label);
  // 產出格式：'Voice transcript from speaker "Alice":\n你好大家'

  // 步驟 5：送入 Agent 處理
  const result = await agentCommandFromIngress({
    message: prompt,
    sessionKey: entry.route.sessionKey,
    agentId: entry.route.agentId,
    messageChannel: "discord",
    senderIsOwner: speaker.senderIsOwner,
    allowModelOverride: false,
    deliver: false,                    // 不直接發送，取回結果
  }, this.params.runtime);

  // 步驟 6：提取回應文字
  const replyText = (result.payloads ?? [])
    .map((payload) => payload.text)
    .filter((text) => typeof text === "string" && text.trim())
    .join("\n").trim();
  if (!replyText) return;

  // 步驟 7：TTS 合成
  const ttsResult = await getDiscordRuntime().tts.textToSpeech({
    text: speakText,
    cfg: ttsCfg,
    channel: "discord",
    overrides: directive.overrides,
  });
  if (!ttsResult.success || !ttsResult.audioPath) return;

  // 步驟 8：排隊播放
  this.enqueuePlayback(entry, async () => {
    const resource = voiceSdk.createAudioResource(audioPath);
    entry.player.play(resource);
    await voiceSdk.entersState(entry.player, AudioPlayerStatus.Playing, ...);
    await voiceSdk.entersState(entry.player, AudioPlayerStatus.Idle, ...);
  });
}
```

**擴展點分析**：

要將這個「被動回應」管線轉變為「主動社交」管線，需要在以下位置插入新邏輯：

```
步驟 3 之後（轉文字完成）
    ↓
┌─ [新增] 更新 Room State（對話歷史、話題追蹤）
│
├─ [新增] 記憶捕獲（捕獲有趣的資訊到 per-person memory）
│
├─ [修改] 不是每次都回應——送入插話決策引擎
│   │
│   ├─ 決定不說話 → return（安靜聆聽）
│   │
│   └─ 決定說話 → 繼續到步驟 4
│
步驟 4（生成回應）
    ↓
├─ [修改] prompt 需要包含 Room State context
│
├─ [新增] 注入 per-person 記憶
│
步驟 7 之後（TTS 完成）
    ↓
└─ [新增] 更新 Bot 自身狀態（lastSpokeAt, speakCount）
```

### 2.2 Session Key 架構——per-channel 的限制

目前的 session key 格式：

```
agent:AGENT_ID:discord:channel:CHANNEL_ID
```

引用：`source-repo/extensions/discord/src/voice/manager.ts:477-483`

```typescript
const route = resolveAgentRoute({
  cfg: this.params.cfg,
  channel: "discord",
  accountId: this.params.accountId,
  guildId,
  peer: { kind: "channel", id: sessionChannelId },
});
```

**問題**：語音頻道中所有人共用同一個 session。這意味著：
- Agent 的對話歷史混合了所有人的發言
- 記憶查詢是 session-scoped 而非 person-scoped
- 無法為每個人維護獨立的上下文

**解決方案**：

方案 A（推薦）：在記憶層做 person 隔離

```typescript
// 不改變 session key，但在記憶操作時加入 personId 標識
// 優點：不影響 OpenClaw 核心，改動最小
// 缺點：session 級的對話歷史仍然混合
```

方案 B：引入 voice-room session 概念

```typescript
// 為語音頻道建立特殊的 session，內含 per-participant 子 session
// 優點：完整隔離
// 缺點：需要改動核心 session 管理
```

方案 C（混合）：Session 共用 + 記憶隔離 + Context Engine 自訂

```typescript
// 使用共用 session 保持「房間對話」的連貫性
// 用 Context Engine 的 assemble() 將 per-person 記憶注入 context
// 這樣 Agent 既能看到完整對話，又能回憶起每個人的歷史
```

**推薦方案 C**，因為社交場景中 Bot 需要理解完整的對話流（誰對誰說了什麼），同時又需要 per-person 的記憶能力。

### 2.3 handleSpeakingStart()——語音事件處理

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:648-717

private async handleSpeakingStart(entry: VoiceSessionEntry, userId: string) {
  // 忽略自己的聲音
  if (this.botUserId && userId === this.botUserId) return;
  
  // 如果這個人已經在捕獲中，取消 finalize timer（延續當前語句）
  if (isVoiceCaptureActive(entry.capture, userId)) {
    // 人還在說話，只是有短暫停頓——繼續捕獲
    return;
  }
  
  // 新的語音片段開始
  // ⚡ 重要：如果 Bot 正在播放，立即停止
  if (entry.player.state.status === voiceSdk.AudioPlayerStatus.Playing) {
    entry.player.stop(true);
  }
  
  // 訂閱該用戶的 Opus 串流
  const stream = entry.connection.receiver.subscribe(userId, {
    end: { behavior: voiceSdk.EndBehaviorType.Manual },
  });
  const generation = beginVoiceCapture(entry.capture, userId, stream);
  
  // 解碼 Opus → PCM
  const pcm = await decodeOpusStream(stream);
  
  // 寫入 WAV 檔案
  const { path: wavPath, durationSeconds } = await writeWavFile(pcm);
  
  // 過短片段丟棄
  if (durationSeconds < MIN_SEGMENT_SECONDS) return;
  
  // 排隊處理
  this.enqueueProcessing(entry, async () => {
    await this.processSegment({ entry, wavPath, userId, durationSeconds });
  });
}
```

**對社交 Bot 的啟示**：

1. **Bot 主動讓步**：`player.stop(true)` 確保有人說話時 Bot 立即閉嘴。這是好的社交行為。
2. **Processing Queue**：`enqueueProcessing` 確保多人同時說話時，處理是有序的。社交 Bot 需要保持這個特性，避免亂序回應。
3. **Segment Duration Check**：0.35 秒的最短片段可以過濾噪音和意外觸發。

## 3. 技術瓶頸深度分析

### 3.1 瓶頸一：即時 STT 延遲

**問題**：目前的 batch STT 流程（說完 → WAV → 上傳 → 轉寫）增加了 1-3 秒延遲。

**分析**：

```
目前流程（batch mode）：
  說話中... ──[靜默 1.2s]──▶ finalize ──▶ Opus decode ──▶ WAV write ──▶ API call ──▶ transcript
  │                                                                                    │
  └──────────────── total: 2-4s after speech ends ─────────────────────────────────────┘

改進流程（streaming mode）：
  說話中... ──▶ 即時送入 Deepgram streaming API ──▶ partial transcript ──▶ final transcript
  │                                                                        │
  └──────────────── total: 0.3-0.5s after speech ends ────────────────────┘
```

**解決方案**：

OpenClaw 已有 `RealtimeTranscriptionSession` 介面：

```typescript
// source-repo/src/realtime-transcription/provider-types.ts:28-33
export type RealtimeTranscriptionSession = {
  connect(): Promise<void>;
  sendAudio(audio: Buffer): void;   // ← 持續餵入 Opus/PCM 音訊
  close(): void;
  isConnected(): boolean;
};
```

需要做的是：將 Discord per-user Opus 串流接入 `sendAudio()`，用 `onPartial` 回調追蹤正在說的話（用於話題偵測），用 `onTranscript` 回調觸發後續處理。

**預估工作量**：約 2-3 週，主要工作是 Opus → PCM 的即時轉換管道和 RealtimeTranscription provider 的 Discord 適配。

### 3.2 瓶頸二：持續 LLM 判斷的成本

**問題**：如果每次有人說話都要問 LLM「我應該插嘴嗎？」，API 費用會很高。

**分析**：

假設語音頻道中 4 個人活躍聊天，平均每人每分鐘說 3 句：
- 12 句/分鐘 → 如果每句都觸發 LLM judge → 12 次 API call/分鐘
- 1 小時 = 720 次 API call
- 每次 ~500 input tokens + ~100 output tokens ≈ 600 tokens
- GPT-4o-mini: ~$0.15/1M input + $0.60/1M output
- 720 × 600 tokens = 432K tokens/hour ≈ $0.07-0.30/hour

**解決方案**：多層過濾大幅降低 LLM 調用：

```
所有語音事件: ~720/hour
    │
    ▼ Rule Triggers 過濾（零成本）
    │
觸發的事件: ~50/hour（約 7%）
    │
    ▼ 高信心直接跳過 LLM
    │
需要 LLM 判斷的: ~20/hour（約 3%）
    │
    ▼ LLM Judge
    │
最終說話次數: ~8/hour（約 1%）
```

實際成本：20 次/hour × 600 tokens = 12K tokens/hour ≈ $0.002-0.01/hour

**結論**：通過 Rule Triggers 的漏斗過濾，LLM 判斷的成本可以忽略不計。

### 3.3 瓶頸三：Discord API 限制

**Discord Voice Gateway 限制**：

| 限制 | 值 | 影響 |
|------|-----|------|
| 每 Guild 只能加入 1 個語音頻道 | 硬限制 | Bot 無法同時在多個頻道聆聽 |
| Bot 需要語音權限 | 需配置 | 需要 `CONNECT` + `SPEAK` + 需要確保有接收權限 |
| DAVE 加密 | 可選 | OpenClaw 已處理，但可能影響穩定性 |
| Opus 封包格式 | 固定 | 48kHz, stereo, Opus |
| Rate Limits | 全局 | 過多 API 呼叫可能被限制 |

**重要發現**：Discord.js 的 voice receiver 需要特別注意**接收端解密**問題。OpenClaw 已實現 DAVE receive passthrough 和 decrypt failure recovery：

```typescript
// source-repo/extensions/discord/src/voice/receive-recovery.ts:3-6
const DECRYPT_FAILURE_WINDOW_MS = 30_000;
const DECRYPT_FAILURE_RECONNECT_THRESHOLD = 3;
```

3 次解密失敗就會觸發自動重連。這在穩定性上很重要——長時間語音通話中偶發的解密錯誤不會導致整個 Bot 卡死。

### 3.4 瓶頸四：多人同時說話的處理

**問題**：當多人同時說話時，每個人的語音片段會近乎同時到達。

**OpenClaw 的解決方案**：

```typescript
// source-repo/extensions/discord/src/voice/manager.ts:619-622
private enqueueProcessing(entry: VoiceSessionEntry, task: () => Promise<void>) {
  entry.processingQueue = entry.processingQueue
    .then(task)
    .catch((err) => logger.warn(`...`));
}
```

Processing Queue 是一個 Promise 鏈——確保處理是序列的。但這意味著如果 Alice 和 Bob 同時說話，Bot 會先處理完 Alice 再處理 Bob，可能導致回應延遲。

**社交 Bot 的改進**：

```typescript
// 改進方案：並行 STT，但序列決策
// STT 可以並行（不同用戶的音訊互不影響）
// 決策必須序列（需要完整的對話上下文）

async processParallel(segments: VoiceSegment[]) {
  // 並行轉寫
  const transcripts = await Promise.all(
    segments.map(seg => transcribeAudio(seg))
  );
  
  // 按時間排序，序列更新 Room State
  const sorted = transcripts.sort((a, b) => a.timestamp - b.timestamp);
  for (const t of sorted) {
    roomState.addDialogue(t);
  }
  
  // 統一做一次插話決策（而非每句話都問一次）
  const decision = await interjectionEngine.evaluate(roomState);
}
```

### 3.5 瓶頸五：端到端延遲預算

社交場景中，回應速度極其重要。如果有人問了一個問題，5 秒後 Bot 才回答，這個回應就失去了意義（人類可能已經回答了）。

**延遲預算分解**：

```
目標：從「決定說話」到「聲音輸出」< 3 秒

步驟                          現有延遲    優化後
────────────────────────────────────────────────
VAD + 靜默偵測                  1.2s       0.5s    ← 縮短 grace period
STT 轉寫                       1.5-3s     0.3-0.5s ← 串流 STT
Context + Memory Query          0.3s       0.2s    ← cache
LLM 回應生成                    1.5-3s     1.0-1.5s ← 短回應 + 快模型
TTS 合成                        1.0-2.0s   0.5-1.0s ← 串流 TTS / 本地
Discord 播放載入                 0.2s       0.2s
────────────────────────────────────────────────
總計                            5.7-10.4s   2.7-3.9s
```

**關鍵優化策略**：

1. **短回應**：社交場景不需要長篇大論。限制回應在 1-2 句話
2. **快模型**：用 GPT-4o-mini 或 Claude Haiku 做回應生成
3. **串流 STT + TTS**：邊聽邊轉、邊生成邊合成
4. **預測性準備**：當 Rule Trigger 觸發時，就開始準備 context 和記憶查詢，不等 LLM judge 結果
5. **本地 TTS fallback**：sherpa-onnx 在本地運行，延遲 < 0.5s

```typescript
client.on('voiceStateUpdate', (oldState, newState) => {

  }
```

### 4.3 Speaking Events 的詳細行為


### 5.2 文字清理管線

```typescript
// source-repo/extensions/discord/src/voice/sanitize.ts:19-32
export function sanitizeVoiceReplyTextForSpeech(text: string, speakerLabel?: string): string {
  let cleaned = stripInlineDirectiveTagsForDisplay(text).text.trim();
  
  // 去掉 "Alice: " 前綴（Agent 有時會加上說話對象的名字）
  const label = speakerLabel?.trim();
  if (label) {
    const prefix = new RegExp(`^${escapeRegExp(label)}\\s*:\\s*`, "i");
    cleaned = cleaned.replace(prefix, "").trim();
  }
  
  // 去掉 emoji（TTS 不該讀出 emoji）
  return stripEmojiForSpeech(cleaned);
}
```

這個清理函數對社交 Bot 很重要——Agent 生成的文字可能包含 Markdown、emoji、提及格式等，這些都不適合直接朗讀。

### 6.2 混合搜尋策略

OpenClaw 的 `MemoryIndexManager` 實現了混合搜尋：

```
搜尋流程：
1. Vector Search（語義相似度）→ 候選集 A
2. FTS Search（全文關鍵字）→ 候選集 B
3. Score Merging（合併排序）→ 最終結果
4. Temporal Decay（越新的分數越高）
```

這個混合策略對社交 Bot 特別有用：
- 語義搜尋：「那個喜歡拉麵的人」→ 找到 Carol 的記憶
- 關鍵字搜尋：「Python」→ 找到 Bob 討論過 Python 的記錄
- 時間衰減：最近的互動記憶權重更高

### 6.3 Memory Capture 規則

Active Memory 有明確的捕獲觸發規則：

```
觸發捕獲的關鍵字/模式：
- "remember" / "記住"
- 偏好表達（"I like", "I prefer"）
- 決策表達（"I decided", "let's go with"）
- 聯繫資訊
- 實體資訊（人名、地點）

阻止捕獲的情況：
- Prompt injection 嘗試
- 系統內容
- Agent 自己生成的摘要
```

對社交 Bot，需要擴展捕獲規則：
- 興趣表達：「我最近在看...」「我週末去了...」
- 個性流露：「我覺得...」「我通常會...」
- 情感表達：「我好開心」「最近有點累」
