# 01 — 目標定義與技術路徑分析

> **本節摘要**：定義 Copilot CLI 整合的具體目標與需求，接著深入分析五條技術路徑——MCP Server、ACP Agent、copilot-proxy 擴展、Gateway 橋接器、Shell CLI 橋接——每條路徑都從 OpenClaw 原始碼層級出發，附帶關鍵程式碼片段、架構圖與可行性評分。

---

## 第一部分：目標定義與需求分析

### 1.1 什麼是 GitHub Copilot CLI？

GitHub Copilot CLI（正式名稱 Copilot CLI / `@anthropic-ai/copilot-cli`）是 GitHub 提供的終端 AI 助手。它的核心架構包含：

- **主對話迴圈（Main conversation loop）**：使用者在終端中與 AI 互動，AI 可以讀寫檔案、執行命令、搜尋程式碼
- **工具系統（Tool system）**：內建工具（bash、edit、view、grep、glob 等）+ MCP 外部工具
- **MCP Client**：Copilot CLI 原生支援 Model Context Protocol，可透過設定檔載入外部 MCP Server
- **子代理（Sub-agents）**：支援 explore、task、general-purpose 等不同類型的子代理
- **背景代理（Background agents）**：可在背景運行長時間任務

**關鍵特性**：Copilot CLI 就是一個增強型終端，它的 AI 可以呼叫 shell 命令、讀寫檔案、運行測試。它的強項是**程式碼操作**，但它**不具備**多通道通訊、記憶系統、人格引擎等 OpenClaw 的能力。

### 1.2 我們想要的整合效果

整合可以從三個方向思考：

#### 方向一：在 Copilot CLI 中使用 OpenClaw 能力（Copilot → OpenClaw）

| 使用情境 | 範例 | 需要的 OpenClaw 能力 |
|----------|------|---------------------|
| 查看多通道訊息 | 「列出 Telegram 最近的對話」 | `sessions.list` + `chat.history` |
| 透過 Copilot 回覆訊息 | 「回覆上一則 iMessage」 | `send` |
| 查詢 OpenClaw 記憶 | 「recall 上週討論的架構決定」 | `memory_recall` (plugin tool) |
| 存儲研究筆記到記憶 | 「把這段程式碼分析存入記憶」 | `memory_store` (plugin tool) |
| 使用 OpenClaw Skills | 「用 mcporter 查詢 Linear issues」 | Skills + tool invocation |
| 監控 Agent 狀態 | 「OpenClaw design agent 目前在做什麼？」 | `sessions.list` + `agents.list` |
| 審批執行請求 | 「允許 OpenClaw 執行 npm install」 | `exec.approval.resolve` |

#### 方向二：讓 OpenClaw 利用 Copilot 能力（OpenClaw → Copilot）

這個方向的核心是讓 OpenClaw 的 Agent 能夠呼叫 Copilot 的程式碼理解能力，例如：

- OpenClaw Agent 在處理 GitHub PR review 時，能呼叫 Copilot 進行程式碼分析
- OpenClaw 的「開發者助手」Agent 能透過 Copilot 執行 refactoring

**但這個方向目前技術路徑不明確**——Copilot CLI 是一個終端應用，沒有暴露為可被外部呼叫的 API。因此，本藍圖**聚焦方向一**。

#### 方向三：雙向整合

理想的最終狀態是雙向整合，但需要分階段實現。方向一是基礎，也是投入產出比最高的起點。

### 1.3 成功標準

在開始技術路徑分析之前，我們定義清晰的成功標準，這樣每條路徑都可以被客觀地評估：

| 層級 | 標準 | 驗證方式 |
|------|------|---------|
| **最低可用（L1）** | 能從 Copilot CLI 查詢 OpenClaw 的對話列表和訊息內容 | 在 Copilot CLI 中說「列出 OpenClaw 的對話」並得到結構化結果 |
| **基本功能（L2）** | 能從 Copilot CLI 發送訊息到 OpenClaw 的通道 | 在 Copilot CLI 中說「回覆 Telegram 最新訊息」並成功發送 |
| **即時性（L3）** | Copilot CLI 能即時感知 OpenClaw 的新事件（新訊息、審批請求等） | 在另一個通道發送訊息，Copilot CLI 在 30 秒內感知到 |
| **記憶整合（L4）** | 能從 Copilot CLI 讀寫 OpenClaw 的記憶系統 | `memory_recall` 和 `memory_store` 可用 |
| **完整管理（L5）** | 能從 Copilot CLI 管理 Agents、Skills、Config | 建立 Agent、安裝 Skill、修改設定 |

**Phase 1（Shell）達成 L1-L2**，**Phase 2（MCP）達成 L1-L4**，**Phase 3（Gateway）達成 L1-L5**。

### 路徑 A：OpenClaw 作為 MCP Server → Copilot CLI 作為 MCP Client

#### A.1 技術事實

OpenClaw 已經實作了**兩個 MCP Server**：

#### A.4 Channel Bridge 的事件處理機制深度分析

`OpenClawChannelBridge` 不只是一個靜態的 tool 代理——它維護了一個**完整的事件佇列系統**，讓 MCP Client 可以透過 long-poll 接收即時事件。

事件佇列機制（`source-repo/src/mcp/channel-bridge.ts:244-271`）：

```typescript
async waitForEvent(filter: WaitFilter, timeoutMs = 30_000): Promise<QueueEvent | null> {
  const existing = this.queue.find((event) => matchEventFilter(event, filter));
  if (existing) {
    return existing;
  }
  return await new Promise<QueueEvent | null>((resolve) => {
    const waiter: PendingWaiter = {
      filter,
      resolve: (value) => {
        this.pendingWaiters.delete(waiter);
        resolve(value);
      },
      timeout: null,
    };
    if (timeoutMs > 0) {
      waiter.timeout = setTimeout(() => { waiter.resolve(null); }, timeoutMs);
    }
    this.pendingWaiters.add(waiter);
  });
}
```

佇列支援的事件類型（`source-repo/src/mcp/channel-shared.ts:76-105`）：

| 事件類型 | 說明 | 觸發時機 |
|---------|------|---------|
| `message` | 新訊息到達 | 任何通道收到新訊息 |
| `claude_permission_request` | Claude 要求使用者授權 | Claude Agent 需要執行敏感操作 |
| `exec_approval_requested` | 執行審批請求 | Agent 要求執行 shell 命令 |
| `exec_approval_resolved` | 執行審批完成 | 使用者批准或拒絕 |
| `plugin_approval_requested` | Plugin 審批請求 | Plugin 要求使用者授權 |
| `plugin_approval_resolved` | Plugin 審批完成 | 使用者批准或拒絕 |

佇列限制為 1,000 個事件（`QUEUE_LIMIT`），超過則 FIFO 淘汰最舊的事件。

**這對 Copilot CLI 的意義**：透過 `events_wait` tool，Copilot CLI 可以**即時感知** OpenClaw 的所有活動——收到新訊息時 AI 可以主動通知使用者，有審批請求時可以直接處理。這不是一般的 polling，而是真正的 long-poll 機制，效率接近 push notification。

#### A.5 Claude Channel 通知機制

`channel-bridge.ts` 還實作了一個 Claude 專用的通知管道（`source-repo/src/mcp/channel-bridge.ts:440-502`）。當有新的使用者訊息時，bridge 會透過 MCP Server 的 notification 機制推送 `notifications/claude/channel` 事件給 MCP Client。

這個機制原本是為了讓 Claude Desktop 等支援 notifications 的 MCP Client 能接收即時訊息。Copilot CLI 是否支援 MCP notifications 仍需驗證，但即使不支援，`events_wait` tool 提供了等效的 polling 替代方案。

同時，bridge 還處理了 Claude 的權限審批流程——當 Claude 要求使用者批准某個操作時，bridge 會追蹤 permission request，並將使用者在通道中的回覆（如「yes abcde」格式）自動轉譯為 `allow`/`deny` 回應。

#### A.6 評估

| 維度 | 評分 | 說明 |
|------|------|------|
| **可行性** | ★★★★★ | 雙方都已支援 MCP，直接設定即可使用 |
| **複雜度** | 低 | 只需一個 JSON 設定檔 |
| **功能完整度** | 高 | 8 個 channel tools + 全部 plugin tools |
| **即時性** | 高 | `events_wait` 支援 long-poll，佇列支援 6 種事件類型 |
| **維護成本** | 低 | 依賴 OpenClaw 官方維護的 MCP Server |
| **限制** | - | 僅暴露 channel 和 plugin 層能力，不涵蓋 Gateway 全部 60+ methods |

### 路徑 B：OpenClaw 作為 ACP Agent

#### B.1 技術事實

ACP（Agent Client Protocol）是一個為 IDE-Agent 互動設計的協議。OpenClaw 的 ACP bridge 實作在 `source-repo/src/acp/server.ts`。

核心啟動流程（`source-repo/src/acp/server.ts:15-123`）：

```typescript
export async function serveAcpGateway(opts: AcpServerOptions = {}): Promise<void> {
  const cfg = loadConfig();
  const bootstrap = await resolveGatewayClientBootstrap({ config: cfg, /* ... */ });

  const gateway = new GatewayClient({
    url: bootstrap.url,
    token: bootstrap.auth.token,
    // ...
    onEvent: (evt) => { void agent?.handleGatewayEvent(evt); },
    onHelloOk: () => { resolveGatewayReady(); agent?.handleGatewayReconnect(); },
  });

  gateway.start();
  await gatewayReady;

  const input = Writable.toWeb(process.stdout);
  const output = Readable.toWeb(process.stdin);
  const stream = ndJsonStream(input, output);
  const _connection = new AgentSideConnection((conn) => {
    agent = new AcpGatewayAgent(conn, gateway, opts);
    agent.start();
    return agent;
  }, stream);
}
```

**ACP 與 MCP 的根本差異**：

- **ACP** 是一個**完整的 Agent 對話協議**——它管理 session、支援 prompt/response 流式傳輸、處理 tool_call events。ACP 的設計目標是讓 IDE 可以與一個完整的 AI Agent 互動。
- **MCP** 是一個**工具暴露協議**——它讓 client 可以列出和呼叫 server 提供的 tools。MCP 不管理對話狀態。

#### B.2 ACP 在 Zed Editor 的先例

`source-repo/docs.acp.md:107-143` 記載了 Zed editor 的整合方式：

```json
{
  "agent_servers": {
    "OpenClaw ACP": {
      "type": "custom",
      "command": "openclaw",
      "args": ["acp"],
      "env": {}
    }
  }
}
```

這允許 Zed 使用者在 Agent panel 中直接與 OpenClaw 對話。

#### B.3 Copilot CLI 是否支援 ACP？

**不支援**。GitHub Copilot CLI 目前只支援 MCP 作為外部工具協議。它沒有 ACP client 的實作，也沒有 `agent_servers` 類似的設定。

Copilot CLI 與外部系統的互動方式是：
1. MCP Servers（tools/resources）
2. Shell 命令（bash tool）
3. 沒有第三種

#### B.4 理論可行的迂迴路線

雖然 Copilot CLI 不直接支援 ACP，但理論上可以：

1. 將 `openclaw acp` 包裝為一個 MCP Server，把 ACP 的 prompt/response 映射到 MCP tool calls
2. 或者寫一個 proxy 把 ACP 翻譯成 MCP

但這樣做比直接用路徑 A（`openclaw mcp serve`）複雜得多，且無明顯優勢。

#### B.5 ACP 的獨特價值

ACP 有一些 MCP 不具備的能力（參見 `source-repo/docs.acp.md:28-41` 的相容性矩陣）：

| 能力 | ACP | MCP |
|------|-----|-----|
| Session 管理 | ✅ `newSession`, `loadSession` | ❌ |
| 對話流式傳輸 | ✅ streaming events | ❌ |
| Tool call 追蹤 | ✅ `tool_call` / `tool_call_update` | ❌ |
| Session 模式切換 | ✅ `session/set_mode` | ❌ |
| Cancel/Abort | ✅ `cancel` → `chat.abort` | ❌ |

但這些能力在 Copilot CLI 場景中不是剛需——Copilot CLI 自己管理 session，它只需要工具。

#### B.6 評估

| 維度 | 評分 | 說明 |
|------|------|------|
| **可行性** | ★★★☆☆ | Copilot CLI 不支援 ACP，需要額外轉譯層 |
| **複雜度** | 中 | 需寫 ACP-to-MCP 轉譯層 |
| **功能完整度** | 中高 | ACP 提供完整 Agent 對話能力 |
| **維護成本** | 中 | 轉譯層是額外的維護負擔 |
| **限制** | - | 架構上不自然，ACP 的對話模型與 MCP 的工具模型不匹配 |

### 路徑 C：利用 extensions/copilot-proxy/

#### C.1 技術事實——它做什麼？

`copilot-proxy` 是一個 **OpenClaw provider plugin**，它的功能是：**讓 OpenClaw 使用 GitHub Copilot 的語言模型**。

這是一個**方向相反**的整合——不是「Copilot 使用 OpenClaw」，而是「OpenClaw 使用 Copilot 的模型」。

核心程式碼（`source-repo/extensions/copilot-proxy/index.ts:68-153`）：

```typescript
export default definePluginEntry({
  id: "copilot-proxy",
  name: "Copilot Proxy",
  description: "Local Copilot Proxy (VS Code LM) provider plugin",
  register(api) {
    api.registerProvider({
      id: "copilot-proxy",
      label: "Copilot Proxy",

          id: "local",
          label: "Local proxy",
          kind: "custom",

            const baseUrlInput = await ctx.prompter.text({
              message: "Copilot Proxy base URL",
```

2. OpenClaw 的 copilot-proxy plugin 將這個 API 註冊為 model provider
3. OpenClaw 的 Agent 就可以使用 Copilot 的模型（GPT-5.2, Claude Sonnet 4.5 等）

#### C.2 能否擴展用於 CLI 整合？

**不能直接使用**。這個 plugin 的功能完全是 model provider 層面的——它不暴露任何 tools，也不提供 MCP/ACP 能力。

理論上的擴展路線：

1. **在 copilot-proxy plugin 中加入 tool registration**——但這偏離了它的設計意圖，且 Copilot CLI 不會消費 OpenClaw 的 plugin tools
2. **用類似的 plugin SDK 架構寫一個新 plugin**——但新 plugin 的目標是暴露 OpenClaw 能力給外部，這更適合用 MCP Server 實現

#### C.3 copilot-proxy 的間接價值

雖然 copilot-proxy 不能直接用於 Copilot CLI 整合，但它有一個重要的間接價值：

> 如果 OpenClaw 使用 copilot-proxy 作為 model provider，那麼 OpenClaw 的 Agent 就不需要獨立的 API key——它們可以透過使用者的 Copilot 訂閱來取得模型能力。

這在成本和便利性方面有意義，但不是本藍圖的核心目標。

#### C.4 評估

| 維度 | 評分 | 說明 |
|------|------|------|
| **可行性** | ★★☆☆☆ | 方向錯誤，需要大幅改造 |
| **複雜度** | 高 | 需要重新設計 plugin 架構 |
| **功能完整度** | 低 | 原始功能只是 model provider |
| **維護成本** | 高 | 偏離 plugin 設計意圖 |
| **限制** | - | 本質上是讓 OpenClaw 用 Copilot 的模型，不是讓 Copilot 用 OpenClaw |

#### D.1 技術事實——Gateway 的能力

OpenClaw Gateway 是整個系統的核心 API 層。所有外部通訊都透過它（WebSocket on port 18789/18790）。

#### D.2 自訂 MCP Server 的設計

目標：寫一個 MCP Server，將 Gateway 的關鍵 methods 包裝為 MCP tools，提供比 `openclaw mcp serve` 更完整的能力。

**架構**：

```
Copilot CLI ──MCP stdio──→ custom-mcp-server ──WebSocket──→ Gateway(:18789)
                              │
                              ├── sessions_list_all → sessions.list
                              ├── session_chat_send → chat.send
                              ├── session_chat_history → chat.history
                              ├── agents_list → agents.list
                              ├── skills_search → skills.search
                              ├── models_list → models.list
                              ├── config_get → config.get
                              ├── talk_speak → talk.speak
                              └── ... (更多 Gateway methods)
```

**關鍵參考——Gateway Client 的連接方式**（`source-repo/src/mcp/channel-bridge.ts:90-128`）：

```typescript
const bootstrap = await resolveGatewayClientBootstrap({
  config: this.cfg,
  gatewayUrl: this.params.gatewayUrl,
  explicitAuth: {
    token: this.params.gatewayToken,
    password: this.params.gatewayPassword,
  },
  env: process.env,
});

this.gateway = new GatewayClient({
  url: bootstrap.url,
  token: bootstrap.auth.token,
  password: bootstrap.auth.password,
  clientName: GATEWAY_CLIENT_NAMES.CLI,
  clientDisplayName: "OpenClaw MCP",
  clientVersion: VERSION,
  mode: GATEWAY_CLIENT_MODES.CLI,
  scopes: [READ_SCOPE, WRITE_SCOPE, APPROVALS_SCOPE],
  onEvent: (event) => { void this.handleGatewayEvent(event); },
  onHelloOk: () => { void this.handleHelloOk(); },

this.gateway.start();
```

#### D.4 Gateway WebSocket 協議深度分析

要實作路徑 D，必須理解 Gateway 的 WebSocket 協議。以下是從原始碼提取的關鍵細節。

**連接建立流程**（基於 `source-repo/src/gateway/client.ts` 分析）：

```
Client                                         Gateway (:18789)
  │                                                │
  │ ── WebSocket Connect ──────────────────────→   │
  │                                                │
  │ ── { type: "req", id: "1",                     │
  │      method: "connect",                        │
  │      params: {                                 │
  │        clientName: "cli",                      │
  │        clientDisplayName: "OpenClaw MCP",      │
  │        clientVersion: "2026.4.12",             │
  │        mode: "cli",                            │
  │        scopes: ["read", "write", "approvals"], │
  │        token: "...",                           │
  │        password: "..."                         │
  │      }                                         │
  │    } ──────────────────────────────────────→   │
  │                                                │
  │ ←── { type: "res", id: "1",                    │
  │       result: { ok: true, ... }                │
  │     } ─────────────────────────────────────    │
  │                                                │
  │ ── { type: "req", id: "2",                     │
  │      method: "sessions.subscribe",             │
  │      params: {}                                │
  │    } ──────────────────────────────────────→   │
  │                                                │
  │ ←── 事件推送 { type: "event",                   │
  │       event: "session.message",                │
  │       payload: { sessionKey, message, ... }    │
  │     } ─────────────────────────────────────    │
```

**Client 認證的三種方式**（基於 `source-repo/src/gateway/method-scopes.ts` 和 client bootstrap 邏輯）：

1. **Token auth**：`gateway.remote.token` 設定或 `OPENCLAW_GATEWAY_TOKEN` 環境變數
2. **Password auth**：`gateway.remote.password` 設定或 `OPENCLAW_GATEWAY_PASSWORD` 環境變數  
3. **Local loopback**：當 Gateway 在本機運行時，`127.0.0.1` 連接可能免認證（取決於設定）

**Scopes 與權限控制**：

自訂 MCP Server 在連接 Gateway 時需要聲明所需的 scopes。Channel Bridge 使用 `[READ_SCOPE, WRITE_SCOPE, APPROVALS_SCOPE]`（`source-repo/src/mcp/channel-bridge.ts:112`）。若要暴露管理功能（agents.create, skills.install 等），需要加入 `ADMIN_SCOPE`。

**錯誤處理模式**：

Gateway 的錯誤回應使用標準 JSON-RPC error codes（`source-repo/src/gateway/protocol/index.js`）：
- `INVALID_REQUEST`：無效請求（認證失敗、scope 不足等）
- `UNAVAILABLE`：方法暫時不可用（Gateway 啟動中）

自訂 MCP Server 應該將 Gateway 的錯誤碼翻譯成 MCP 的 error response，讓 Copilot CLI 的 AI 能理解失敗原因。

#### D.5 MCP HTTP Loopback 的啟示

Gateway 內部已經有一個 MCP HTTP loopback server（`source-repo/src/gateway/mcp-http.ts:34-129`），用於讓 mcporter bridge 呼叫 OpenClaw 的工具。這證明 Gateway 團隊已經在 MCP 包裝方面有成熟的模式。

這個 loopback server 的實作提供了幾個重要參考：
- **Token-based auth**：使用隨機生成的 token 保護 HTTP endpoint
- **綁定到 127.0.0.1**：僅允許本機連接，避免安全風險
- **Tools 動態註冊**：工具列表從 Gateway 的 tools catalog 動態載入，不是硬編碼

自訂 MCP Server（路徑 D）可以參考這個模式，從 loopback server 的 tool 註冊邏輯中學習如何動態暴露 Gateway 的工具。

#### D.6 評估

| 維度 | 評分 | 說明 |
|------|------|------|
| **可行性** | ★★★★☆ | 技術完全可行，但需要寫新程式碼 |
| **複雜度** | 中高 | 需要寫 TypeScript MCP Server + Gateway Client 邏輯 |
| **功能完整度** | **最高** | 可暴露 Gateway 全部能力 |
| **維護成本** | 中高 | 自維護的程式碼，需跟隨 Gateway API 變動更新 |
| **限制** | - | 需自行處理 WebSocket 連線管理、錯誤重試、auth |

### 路徑 E：透過 Shell/CLI 橋接

#### E.1 技術事實

Copilot CLI 內建 `bash` tool，可以直接執行 shell 命令。OpenClaw 提供完整的 CLI 工具：

**MCP 相關 CLI 命令**（`source-repo/src/cli/mcp-cli.ts:26-150`）：

```bash
openclaw mcp serve         # 啟動 MCP Server（stdio）
openclaw mcp list          # 列出已設定的 MCP servers
openclaw mcp show [name]   # 顯示特定 MCP server 設定
openclaw mcp set <name> <json>  # 設定 MCP server
openclaw mcp unset <name>  # 移除 MCP server
```

**ACP 命令**（`source-repo/src/acp/server.ts:216-234`）：

```bash
openclaw acp               # 啟動 ACP server（stdio）
openclaw acp --session agent:main:main  # 指定 session
openclaw acp --verbose     # 除錯模式
```

此外，OpenClaw CLI 應該還有其他命令（基於 `source-repo/src/gateway/server-methods.ts` 中的 handlers，CLI 通常會有對應的子命令），例如：

```bash
openclaw status            # 系統狀態
openclaw chat              # 互動式聊天
openclaw send              # 發送訊息到通道
openclaw skills            # 管理 Skills
openclaw config            # 設定管理
openclaw agents            # Agent 管理
```

#### E.2 Copilot CLI 如何呼叫

在 Copilot CLI 中，使用者可以直接要求 AI 執行 `openclaw` 命令。例如：

```
User: 列出 OpenClaw 的所有對話

Copilot CLI（自動執行）:
$ openclaw sessions list --json
[{ "key": "agent:main:main", "channel": "telegram", ... }, ...]

Copilot CLI: 找到以下對話：
1. agent:main:main (Telegram)
2. agent:design:main (Discord)
...
```

#### E.3 常用 OpenClaw CLI 命令映射表

以下是 Copilot CLI 使用者最可能需要的 OpenClaw CLI 命令，基於 Gateway methods 和 CLI 入口分析：

| 使用者意圖 | 推薦的 CLI 命令 | 對應 Gateway Method |
|-----------|---------------|-------------------|
| 查看系統狀態 | `openclaw status --json` | `health` |
| 列出通道對話 | `openclaw sessions list --json --limit 20` | `sessions.list` |
| 讀取對話記錄 | `openclaw chat history --session <key> --json --limit 10` | `chat.history` |
| 發送通道訊息 | `openclaw send --channel <ch> --to <target> --message "<msg>"` | `send` |
| 列出 Agent | `openclaw agents list --json` | `agents.list` |
| 查看 Skill 狀態 | `openclaw skills status --json` | `skills.status` |
| 搜尋 Skills | `openclaw skills search "<query>" --json` | `skills.search` |
| 查看可用模型 | `openclaw models list --json` | `models.list` |
| 查看設定 | `openclaw config get <key>` | `config.get` |
| 列出排程任務 | `openclaw cron list --json` | `cron.list` |
| 查看工具目錄 | `openclaw tools catalog --json` | `tools.catalog` |

**重要提示**：所有命令都應使用 `--json` 旗標。這讓 Copilot CLI 的 AI 能夠結構化地解析輸出，而不是靠文字匹配。沒有 `--json` 時，CLI 輸出的是人類可讀的格式，AI 解析可能出錯。

#### E.4 Shell 整合的進階技巧

**技巧一：建立 shell alias 加速呼叫**

```bash
# 在 .bashrc 或 .zshrc 中
alias ocl='openclaw'
alias ocls='openclaw sessions list --json'
alias oclh='openclaw chat history --json'
```

**技巧二：使用 jq 做預處理**

Copilot CLI 的 bash tool 可以管道 (pipe) 命令。結合 `jq` 可以在 shell 層面就過濾和格式化輸出：

```bash
# 只列出 Telegram 通道的對話
openclaw sessions list --json | jq '[.[] | select(.channel == "telegram")]'

# 取得最新一則使用者訊息
openclaw chat history --session agent:main:main --json --limit 5 | \
  jq '[.[] | select(.role == "user")] | last'
```

**技巧三：組合命令實現複雜操作**

```bash
# 找到最近更新的對話並讀取其最新訊息
SESSION=$(openclaw sessions list --json --limit 1 | jq -r '.[0].key')
openclaw chat history --session "$SESSION" --json --limit 3
```

#### E.5 效能考量

每次 `openclaw` CLI 命令都需要：
1. 啟動 Node.js 進程
2. 載入 OpenClaw 設定
3. 建立 Gateway WebSocket 連接
4. 執行命令
5. 斷開連接、退出進程

對於頻繁的呼叫（如連續查詢多個對話），這個啟動成本會累積。但對於 MVP 來說完全可接受。

**優點**：
- 零設定成本
- 不需要任何中間層
- 支援 OpenClaw CLI 的全部命令
- 每次呼叫都是獨立的，無需管理連線

**缺點**：
- 每次呼叫都需要啟動新的進程（效能較低）
- 不支援 streaming 或即時事件
- 輸出需要被 Copilot CLI 的 AI 解析（可能有誤解）
- 不是結構化的 tool 介面，沒有 schema validation

#### E.6 評估

| 維度 | 評分 | 說明 |
|------|------|------|
| **可行性** | ★★★★★ | 無需任何額外工作 |
| **複雜度** | **最低** | 零設定、零程式碼 |
| **功能完整度** | 中 | CLI 命令覆蓋主要功能，但缺乏 streaming |
| **維護成本** | **最低** | 完全依賴 OpenClaw 官方 CLI |
| **限制** | - | 效能較低、無 streaming、非結構化輸出 |
