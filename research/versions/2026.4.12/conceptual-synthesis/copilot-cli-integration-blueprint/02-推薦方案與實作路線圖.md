# 02 — 推薦方案與實作路線圖

> **本節摘要**：基於五條技術路徑的分析，推薦「Shell CLI 快速驗證 → MCP Server 結構化整合 → Gateway 橋接完整能力」的三階段策略。每個 Phase 都附帶具體的實作步驟、預期效果展示、設定範例。最後評估技術風險與維護負擔。

## 第三部分：推薦方案

### 3.1 選擇邏輯

我們以四個維度評估五條路徑：

| 維度 | 權重 | 說明 |
|------|------|------|
| **可行性** | 35% | 現在就能做到嗎？需要多少前置條件？ |
| **功能完整度** | 25% | 能暴露多少 OpenClaw 能力？ |
| **複雜度（反向）** | 25% | 越簡單越好——減少出錯機率 |
| **維護成本（反向）** | 15% | 長期的維護負擔 |

加權評分：

| 路徑 | 可行性 (35%) | 功能 (25%) | 簡潔 (25%) | 維護 (15%) | **加權總分** |
|------|-------------|-----------|-----------|-----------|------------|
| A (MCP) | 5×0.35=1.75 | 4×0.25=1.00 | 4×0.25=1.00 | 4×0.15=0.60 | **4.35** |
| B (ACP) | 3×0.35=1.05 | 3.5×0.25=0.88 | 3×0.25=0.75 | 3×0.15=0.45 | 3.13 |
| C (proxy) | 2×0.35=0.70 | 1×0.25=0.25 | 1×0.25=0.25 | 1×0.15=0.15 | 1.35 |
| D (Gateway) | 4×0.35=1.40 | 5×0.25=1.25 | 2×0.25=0.50 | 2×0.15=0.30 | 3.45 |
| E (Shell) | 5×0.35=1.75 | 3×0.25=0.75 | 5×0.25=1.25 | 5×0.15=0.75 | **4.50** |

**結論**：路徑 E（Shell）和路徑 A（MCP）分別在不同面向領先。最佳策略是**漸進式推進**：

1. **立即可用**：路徑 E（Shell CLI 橋接）— MVP，零成本驗證
2. **結構化整合**：路徑 A（MCP Server）— 增強功能，正式整合
3. **完整能力**：路徑 D（Gateway 橋接）— 進階需求時再投入

```typescript
export async function main() {
  const server = new McpServer({
    name: "openclaw-full",
    version: "1.0.0",

  const gw = await createGatewayConnection({
    url: process.env.OPENCLAW_GATEWAY_URL ?? "ws://127.0.0.1:18789",

        sessionKey: args.session_key,

    },

      limit: z.number().optional(),

      return { content: [{ type: "text", text: JSON.stringify(result, null, 2) }] };
    },
  );

  server.tool(
    "skills_search",
    "Search available OpenClaw skills on ClawHub",
    { query: z.string() },
```

#### Step 4：Gateway WebSocket 協議實作要點

基於 `source-repo/src/gateway/client.ts` 的分析，Gateway WebSocket 協議的關鍵要素：

1. **連接握手（Hello）**：連接後需發送 `connect` method 進行認證
2. **Request/Response**：JSON-RPC 風格的 `{ type: "req", id, method, params }` / `{ type: "res", id, result, error }`
3. **事件推送**：server 推送 `{ type: "event", event, payload }`
4. **Scopes**：client 連接時需聲明所需的 scopes（READ, WRITE, APPROVALS 等）
5. **Heartbeat**：需要處理連接保活

建議的實作方式是：如果能直接 import OpenClaw 的 `GatewayClient` 類別最理想，避免重新實作協議。否則需要自行實作精簡版 WebSocket client。

## 第五部分：困難度評估與風險

### 5.1 技術風險

| 風險 | 嚴重度 | 可能性 | 影響 | 緩解方案 |
|------|--------|--------|------|----------|
| Gateway WebSocket 協議變更 | 中 | 低 | Phase 3 的自訂 MCP Server 需更新 | Pin OpenClaw 版本，用整合測試驗證 |
| MCP SDK 版本不相容 | 低 | 低 | MCP Server 啟動失敗 | 跟隨 OpenClaw 使用的 `@modelcontextprotocol/sdk` 版本 |
| `openclaw mcp serve` 行為變更 | 低 | 中 | Phase 2 的 tool 集合可能改變 | 依賴官方 CHANGELOG，行為變更通常是向後相容的 |
| Gateway auth token 過期/旋轉 | 中 | 中 | MCP Server 斷線 | 使用 `--token-file` 讓 token 可更新而不需重啟 |
| 多 MCP Server 同時連接 Gateway 的 session 衝突 | 低 | 低 | 事件路由混亂 | 每個 MCP Server 使用不同的 client display name |

### 5.2 相依於外部變化

#### Copilot CLI 的 MCP 支援

Copilot CLI 的 MCP 支援是這個整合方案的基石。目前 Copilot CLI 穩定支援 MCP，但需要注意：

- **MCP 設定格式**可能在未來版本中調整
- **MCP protocol 版本**升級可能需要同步更新 OpenClaw 的 MCP SDK
- **Copilot CLI 本身**是 GitHub 的產品，其更新節奏由 GitHub 控制

**緩解策略**：

1. Phase 1（Shell 橋接）完全不依賴 MCP，是最穩定的退路
2. Phase 2 依賴的是 OpenClaw 官方維護的 `openclaw mcp serve`，由上游負責相容性
3. Phase 3 才需要自己處理相容性

#### OpenClaw 的 MCP 立場

`source-repo/VISION.md:72-85` 明確表示：

> "OpenClaw supports MCP through mcporter... We prefer this bridge model over building first-class MCP runtime into core."

這意味著：
- OpenClaw 將持續支援 MCP，但透過 bridge 模式而非核心內建
- `openclaw mcp serve` 是官方支援的 MCP 暴露方式
- 不太可能出現 MCP 支援被移除的情況

### 5.3 維護負擔

| Phase | 維護項目 | 頻率 | 工作量 |
|-------|---------|------|--------|
| Phase 1 | 無 | - | 無 |
| Phase 2 | 確認 `openclaw mcp serve` 仍正常 | 每次 OpenClaw 升級 | < 5 分鐘 |
| Phase 2 | 更新 MCP 設定（如果格式變更） | 極少 | < 10 分鐘 |
| Phase 3 | Gateway API 變更同步 | 每個 OpenClaw 大版本 | 1-4 小時 |
| Phase 3 | 自訂 MCP Server 依賴更新 | 每月 | 30 分鐘 |
| Phase 3 | 整合測試維護 | 每次變更 | 30 分鐘 |

```
├── 只是偶爾查看訊息/狀態 → Phase 1 就夠了
├── 經常需要在 coding 時收發 OpenClaw 訊息 → Phase 2
├── 需要從 Copilot CLI 深度管理 Agents/Skills/Config → Phase 3
└── 需要讓 OpenClaw Agent 呼叫 Copilot（反向） → 另一個研究主題
```

### 5.5 不做什麼

以下是明確排除的方向：

1. **不寫 Copilot CLI 的 extension/plugin**——Copilot CLI 不是一個可擴展的平台，它透過 MCP 和 shell 提供外部整合
2. **不修改 OpenClaw core**——所有整合都透過已有的 MCP Server 和 CLI，不需要 fork 或 PR
3. **不建立中間 proxy server**——過度工程化，直接用 stdio MCP 或 shell 更簡潔
4. **不嘗試讓 Copilot CLI 直接連接 Gateway WebSocket**——Copilot CLI 的 MCP 實作只支援 stdio/SSE transport，不支援 raw WebSocket

### 5.6 常見問題與疑難排解

#### Q1：MCP Server 啟動後 Copilot CLI 看不到工具？

**可能原因與排查步驟**：

1. **MCP 設定路徑不對**：Copilot CLI 讀取 `~/.copilot/mcp-config.json`（全域）或專案根目錄的 `.mcp.json`（專案級）。確認設定檔位於正確位置。
2. **`openclaw` 不在 PATH 中**：MCP Server 透過 `command` 啟動，如果 `openclaw` 不在系統 PATH 中，Server 無法啟動。測試：`which openclaw`
3. **Gateway 未運行**：Channel MCP Server 需要連接 Gateway。確認 OpenClaw Gateway 正在運行：`curl -s http://127.0.0.1:18789/health`（注意：Gateway 使用 WebSocket，此 HTTP 端點可能不存在，改用 `openclaw status`）
4. **MCP protocol 握手失敗**：查看 Copilot CLI 的 MCP 日誌。在啟動 Copilot CLI 時添加 `--verbose` 或 `--debug` 旗標可能有幫助。

**快速診斷命令**：

```bash
# 測試 OpenClaw CLI 是否可用
openclaw --version

# 測試 MCP Server 是否可啟動（手動運行，看是否有錯誤輸出）
openclaw mcp serve 2>&1 | head -5

# 測試 Gateway 連接
openclaw status --json
```

#### Q2：`events_wait` 長時間沒有回應？

`events_wait` 是一個 long-poll 工具，預設等待 30 秒。如果在此期間沒有新事件，會回傳 `null`。這是正常行為。

**注意**：Copilot CLI 的 tool 呼叫可能有自己的 timeout。如果 Copilot CLI 的 timeout 短於 `events_wait` 的等待時間，呼叫會被中斷。建議在呼叫時設定較短的 `timeout_ms`：

```json
{
  "tool": "events_wait",
  "arguments": {
    "filter": { "event_type": "message" },
    "timeout_ms": 10000
  }
}
```

#### Q3：Phase 3 的自訂 MCP Server 需要多少程式碼？

基於 OpenClaw 現有的 `channel-server.ts`（~102 行）和 `channel-tools.ts`（~188 行）作為參考，一個覆蓋主要 Gateway 功能的自訂 MCP Server 大約需要：

| 模組 | 預估行數 | 說明 |
|------|---------|------|
| Gateway Client 連接層 | ~80 行 | WebSocket 連接、認證、重連邏輯 |
| Tool 定義（Read 類） | ~200 行 | sessions, agents, skills, config, models 查詢 |
| Tool 定義（Write 類） | ~150 行 | chat.send, sessions.create, agents 管理 |
| Tool 定義（Admin 類） | ~100 行 | config.apply, skills.install, agents.create |
| 事件訂閱與推送 | ~60 行 | session 事件的訂閱與 MCP notification |
| 錯誤處理與日誌 | ~50 行 | Gateway 錯誤碼到 MCP 錯誤的轉譯 |
| CLI 入口 | ~30 行 | 命令列參數解析、啟動邏輯 |
| **合計** | **~670 行** | 不含測試 |

加上測試（預計 ~300 行），總共約 ~1,000 行 TypeScript。這是一個中等規模的專案，一個人可以在 1-2 週內完成。

#### Q4：Plugin Tools MCP Server 暴露了哪些工具？

Plugin Tools MCP Server（`plugin-tools-serve.ts`）動態掃描所有已安裝的 OpenClaw plugins，將它們註冊的工具透過 MCP 暴露。具體工具取決於安裝的 plugins，但預設安裝通常包括：

| 工具名稱 | 來源 Plugin | 功能 |
|---------|-----------|------|
| `memory_recall` | memory plugin | 從長期記憶中搜尋相關資訊 |
| `memory_store` | memory plugin | 儲存新的記憶片段 |
| `web_search` | web plugin | 執行網頁搜尋 |
| `web_fetch` | web plugin | 獲取指定 URL 的內容 |

Plugin Tools Server 的工具集會隨安裝的 plugins 變動。`openclaw tools catalog --json` 可以查看完整的工具列表。

#### Q5：兩個 MCP Server 可以同時使用嗎？

可以。Copilot CLI 支援同時連接多個 MCP Server。在 `mcp-config.json` 中定義多個 server，它們的工具會合併到同一個工具列表中。每個 tool 會以 server 名稱作為前綴來區分。

```json
{
  "mcpServers": {
    "openclaw-channels": {
      "command": "openclaw",
      "args": ["mcp", "serve"]
    },
    "openclaw-plugins": {
      "command": "node",
      "args": ["path/to/plugin-tools-serve.js"]
    }
  }
}
```

Copilot CLI 會看到如 `openclaw-channels__conversations_list`、`openclaw-plugins__memory_recall` 這樣的工具名稱（具體命名格式取決於 Copilot CLI 的 MCP tool 命名規則）。

### 5.7 與 Discord 語音機器人專案的關聯

這份 Copilot CLI 整合藍圖看似是一個獨立的研究主題，但它與 Eason 的另一個核心目標——「讓 OpenClaw 加入 Discord 語音頻道」——有深層的技術關聯。

**共通的技術基礎**：

1. **Gateway 理解**：無論是 Copilot CLI 整合還是 Discord 機器人，都需要深入理解 Gateway 的 WebSocket 協議、sessions 管理、事件系統。Phase 3 的 Gateway 橋接器程式碼可以直接作為 Discord 機器人的 Gateway 客戶端基礎。

2. **MCP 生態系統**：Discord 機器人可能也需要透過 MCP 整合外部工具（如搜尋引擎、日曆等）。理解 OpenClaw 的 MCP 架構對兩個專案都有幫助。

3. **Memory 系統**：Discord 機器人需要記住每個人的個性特徵和互動歷史。Plugin Tools MCP Server 暴露的 `memory_recall`/`memory_store` 工具，可以讓 Copilot CLI 在開發 Discord 機器人時直接操作 OpenClaw 的記憶系統——例如，在 Copilot CLI 中查詢記憶、調試記憶內容、批量匯入記憶。

4. **Agent 管理**：Discord 機器人本質上就是一個 OpenClaw Agent。Phase 2/3 的整合讓 Eason 可以從 Copilot CLI 中管理 Agent 配置，而不需要開啟 OpenClaw 的 Web UI。

**實際的開發工作流**：

```
開發 Discord 機器人時的工作流：

Copilot CLI (Terminal 1)          OpenClaw (背景運行)        Discord
   │                                    │                      │
   │ ── agents.create ──────────────→   │                      │
   │ ← agent 建立成功 ─────────────     │                      │
   │                                    │ ── 連接 Discord ──→  │
   │ ── chat.history ───────────────→   │                      │
   │ ← Agent 與使用者的對話記錄 ────     │                      │
   │                                    │                      │
   │ ── memory_recall "John" ───────→   │                      │
   │ ← John 的記憶片段 ────────────     │                      │
   │                                    │                      │
   │ ── config.apply (調整人格) ────→   │                      │
   │ ← 設定更新成功 ───────────────     │                      │
```

這樣，Copilot CLI 就成為了 Discord 機器人的**開發控制台**——在寫程式碼的同時，即時觀察 Agent 的行為、調試記憶系統、調整設定。

## 結語

這份藍圖的核心洞見是：**OpenClaw 和 Copilot CLI 的整合不需要發明新東西**。

OpenClaw 已經有完整的 MCP Server（`openclaw mcp serve`），暴露 8 個 channel tools 和全部 plugin tools。Copilot CLI 已經原生支援 MCP。兩者之間只需要一行 JSON 設定就能連接。

真正的投入產出最大化策略是：

1. **今天就開始用 Phase 1**——零成本，立即可用
2. **本週設定 Phase 2**——一行 JSON，結構化工具
3. **按需推進 Phase 3**——只有在 Phase 2 不夠用時才投入

### 回顧：我們學到了什麼

從技術分析的角度，這份藍圖產出了幾個重要的認知：

1. **OpenClaw 的 MCP 實作是成熟的**——Channel Bridge 的事件佇列系統、long-poll 機制、權限審批流程，都是經過測試的生產品質程式碼。Channel Server 的測試檔（`channel-server.test.ts`）有 ~500 行的測試，覆蓋了完整的 tool 呼叫流程。

2. **方向理解很重要**——`copilot-proxy` 擴展讓 OpenClaw 使用 Copilot 的模型（OC → Copilot），而我們要的是讓 Copilot 使用 OpenClaw 的能力（Copilot → OC）。在初始分析時如果搞反方向，會浪費大量時間走錯路。

3. **Gateway 是 OpenClaw 的神經系統**——60+ WebSocket methods 幾乎覆蓋了所有可能的操作。任何外部整合最終都是在和 Gateway 對話。理解 Gateway 的 scope 系統（READ/WRITE/ADMIN/APPROVALS）是理解 OpenClaw 權限模型的關鍵。

4. **漸進式策略優於一步到位**——Phase 1 可以在 5 分鐘內開始使用，Phase 2 只需要 10 分鐘設定，Phase 3 需要 1-2 週開發。根據實際需求選擇投入程度，避免過度工程化。

5. **VISION.md 的 MCP 立場是穩定的**——OpenClaw 明確將 MCP 定位為 bridge 模式，不太可能在短期內改變。這讓我們的整合方案有堅實的上游支援基礎。
